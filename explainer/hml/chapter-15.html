PART II: NEURAL NETWORKS & DEEP LEARNING

Transformers for Natural Language Processing & Chatbots

From "Attention Is All You Need" to ChatGPT

The Transformer architecture revolutionized NLP by replacing recurrence with attention mechanisms. In this chapter, we'll dissect multi-head attention, positional encoding, and the encoder-decoder framework. Then we'll explore how BERT, GPT, and modern chatbots emerge from this single architecture, and how you can fine-tune and use them with Hugging Face.

The Transformer Architecture

The Big Idea

Self-Attention = Every token asks "Who matters to me?" and learns to weight every other token accordingly. No recurrence, no convolutions—just learned, dynamic relationships between all token pairs.

The original 2017 Transformer is an encoder-decoder model. The encoder processes the source sequence (English tokens) and outputs contextualized embeddings. The decoder processes the target so far (Spanish tokens) and learns to predict the next token, attending to both the encoder's output and itself.

Core Components

Encoder Stack

  • N identical layers (typically 6)
  • Self-Attention layer
  • Feed-Forward Network
  • Layer Normalization & Residual connections

Decoder Stack

  • N identical layers (typically 6)
  • Masked Self-Attention (causal)
  • Cross-Attention to encoder
  • Feed-Forward Network

Key Innovation: No Recurrence

Unlike RNNs, the Transformer processes all tokens in parallel. This enables training on massive parallel GPU hardware and makes training much faster. The trade-off: we need positional encoding to capture word order.

Self-Attention & Multi-Head Attention

Mental Model

Self-Attention = A learned weighted average. For each token (query), compute a relevance score to every token (key), then use these weights to combine all token values. Do this in parallel for multiple "attention heads," each focusing on different linguistic patterns.

Scaled Dot-Product Attention

The core building block. For a single head:

\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right) V

Where:

  • Q (Query): What am I looking for?
  • K (Key): What information do I have?
  • V (Value): What features should I extract?
  • $d_k$: Dimension of keys (scaling factor prevents vanishing gradients)

Multi-Head Attention (MHA)

Instead of one attention head, use $h$ heads in parallel, each with different learned projections:

\text{MultiHead}(Q, K, V) = \text{Concat}(\text{head}_1, \ldots, \text{head}_h) W^O

Each head can specialize: one might focus on short-range dependencies (noun-verb agreement), another on long-range (resolving pronouns), another on syntactic structure.

Masked Self-Attention (Causal)

During decoding, we must prevent tokens from attending to future tokens (which haven't been generated yet). A mask sets attention weights to -∞ for future positions before softmax:

\text{Attention}_{\text{masked}}(Q, K, V) = \text{softmax}\left(\frac{QK^T + M}{\sqrt{d_k}}\right) V

Where $M_{ij} = -\infty$ if $j > i$ (future), and $0$ otherwise.

Positional Encoding

The Problem It Solves

Without position info, "The cat ate the dog" and "The dog ate the cat" look identical to the model! Since we process all tokens in parallel (no recurrence), we need to inject position information into embeddings.

Sinusoidal Positional Encoding

The original Transformer uses sinusoids at different frequencies:

PE(pos, 2i) = \sin\left(\frac{pos}{10000^{2i/d_{\text{model}}}}\right)
PE(pos, 2i+1) = \cos\left(\frac{pos}{10000^{2i/d_{\text{model}}}}\right)

Why sinusoids? Different frequencies encode different time scales. Low frequencies (large periods) capture broad position ranges; high frequencies capture fine local structure. The model can learn to extrapolate to longer sequences than seen in training.

Implementation Note

In practice, each token embedding is added to its positional encoding: $\mathbf{x}_t + PE_t$. This can be precomputed since it doesn't depend on the input values.

Encoder-Only Models: BERT & Friends

The Idea

BERT = Bidirectional Encoder Representations from Transformers. Take the encoder stack, pretrain on masked language modeling (guess the hidden word), and you get rich contextual embeddings for text classification, search, and more.

BERT Pretraining Strategy

Masked Language Modeling (MLM): Randomly mask 15% of tokens, train the model to predict them using context from both left and right.

Next Sentence Prediction (NSP): Given two sentences, predict whether the second follows the first (helps with understanding discourse).

BERT vs. GPT: Design Comparison

Aspect BERT (Encoder-only) GPT (Decoder-only)
Architecture Transformer Encoder Stack Transformer Decoder Stack
Attention Type Bidirectional (self-attention) Causal (masked self-attention)
Pretraining Task Masked LM (predict hidden words) Causal LM (predict next word)
Primary Use Classification, embeddings, semantic search Text generation, few-shot learning
Input Efficiency Processes full input at once Must generate token-by-token (slower)

BERT Fine-tuning for Text Classification

Once pretrained, BERT's encoder can be adapted to many tasks:

Common Fine-tuning Patterns

  • Classification: Add a linear layer on top of [CLS] token embedding
  • Token Tagging: Add a linear layer on top of each token
  • Semantic Search: Use token or sentence embeddings directly as vectors
  • Clustering: Extract embeddings and apply k-means or DBSCAN

BERT with Hugging Face

from transformers import AutoTokenizer, AutoModel
import torch

# Load pretrained BERT
tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased")
model = AutoModel.from_pretrained("bert-base-uncased", output_hidden_states=True)

# Tokenize and encode text
text = "The Transformer architecture is elegant."
inputs = tokenizer(text, return_tensors="pt")

# Get embeddings
with torch.no_grad():
    outputs = model(**inputs)
    last_hidden = outputs.last_hidden_state  # [batch, seq_len, 768]
    cls_embedding = last_hidden[:, 0, :]     # [batch, 768]

# cls_embedding can now be used for downstream tasks
print(f"Sentence embedding shape: {cls_embedding.shape}")

RoBERTa, ALBERT, DistilBERT

Many improvements on BERT exist:

RoBERTa

Improved training procedure. Larger batch size, longer training, better hyperparameters.

ALBERT

Parameter reduction via factorization. Smaller, faster, still competitive performance.

DistilBERT

Knowledge distilled from BERT. 40% smaller, 60% faster, retains 97% performance.

DeBERTa

Enhanced masking strategy and disentangled attention. Outperforms BERT on many benchmarks.

Decoder-Only Models: GPT & Text Generation

The Idea

GPT = Generative Pre-trained Transformer. Use only the decoder stack with causal attention. Pretrain on next-token prediction (language modeling). Result: a model that generates coherent text and can adapt to many tasks via prompting or fine-tuning.

Autoregressive Generation

GPT generates text token-by-token, always conditioning on what came before:

  1. Start with a prompt (or empty context)
  2. Feed all tokens so far to the model
  3. Get probability distribution over next token
  4. Sample or select the highest-probability token
  5. Append to context and repeat until EOS token

GPT Architecture Variants

GPT-1 (2018)

12 layers, 117M params. Showed LLMs can transfer-learn to diverse tasks.

GPT-2 (2019)

48 layers, 1.5B params. Zero-shot abilities surprised the world.

GPT-3 (2020)

96 layers, 175B params. Few-shot learning, in-context learning abilities.

GPT-3.5 / GPT-4

Enhanced with RLHF, instruction-tuning, reasoning. Multimodal variants.

Causal Language Modeling Loss

Train on predicting the next token given all previous tokens:

\mathcal{L} = -\frac{1}{N} \sum_{i=1}^{N} \log P(x_i | x_1, \ldots, x_{i-1}; \theta)

GPT with Hugging Face

from transformers import GPT2Tokenizer, GPT2LMHeadModel
import torch

tokenizer = GPT2Tokenizer.from_pretrained("gpt2")
model = GPT2LMHeadModel.from_pretrained("gpt2")

prompt = "The future of AI is"
input_ids = tokenizer.encode(prompt, return_tensors="pt")

# Generate next 20 tokens
output = model.generate(
    input_ids,
    max_length=input_ids.shape[1] + 20,
    num_return_sequences=1,
    temperature=0.7,
    top_p=0.9,
    do_sample=True
)

generated_text = tokenizer.decode(output[0], skip_special_tokens=True)
print(generated_text)

Generation Strategies

Strategy How It Works Pros Cons
Greedy Always pick highest prob token Fast, deterministic Repetitive, locally optimal
Sampling Sample from full distribution Diverse, creative Can be incoherent
Top-k / Top-p Sample from top-k or cumulative top-p Balanced diversity & quality Hyperparameter tuning
Beam Search Keep k best hypotheses, expand High quality Slower, can still repeat

Hugging Face Transformers Library

Why Hugging Face?

The dominant open-source library for transformers. 50,000+ pretrained models, unified API, easy fine-tuning, integrated with major frameworks (PyTorch, TensorFlow). If you're using transformers in production, you're likely using HF.

Core Concepts

  • Model: The neural network (weights and architecture)
  • Tokenizer: Converts text to token IDs and back
  • Config: Hyperparameters (num_layers, hidden_size, etc.)
  • Pipeline: High-level API for common tasks (classification, generation, etc.)

Auto Classes Pattern

from transformers import AutoTokenizer, AutoModel, AutoModelForSequenceClassification

# Automatically detect model type and load
model_name = "bert-base-uncased"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModel.from_pretrained(model_name)

# Or for a specific task
model_clf = AutoModelForSequenceClassification.from_pretrained(
    model_name,
    num_labels=2
)

Fine-tuning a Classifier

from datasets import load_dataset
from transformers import Trainer, TrainingArguments, AutoTokenizer, \
                          AutoModelForSequenceClassification
import torch

# Load data
dataset = load_dataset("imdb")

# Tokenize
tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased")

def preprocess(batch):
    return tokenizer(batch["text"], truncation=True, padding=True)

dataset = dataset.map(preprocess, batched=True)

# Load model
model = AutoModelForSequenceClassification.from_pretrained(
    "bert-base-uncased",
    num_labels=2
)

# Train
training_args = TrainingArguments(
    output_dir="./results",
    num_train_epochs=3,
    per_device_train_batch_size=8,
    per_device_eval_batch_size=8,
    warmup_steps=500,
    weight_decay=0.01,
    logging_dir="./logs",
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=dataset["train"],
    eval_dataset=dataset["test"],
)

trainer.train()

Fine-tuning Pretrained Models

Pretraining is expensive (millions of GPU hours). But fine-tuning a pretrained model is fast and requires much less data. This is the secret weapon for practical NLP.

Transfer Learning Benefits

  • Parameter Efficiency: Only train task-specific layers (or use LoRA)
  • Data Efficiency: Pretrained knowledge helps with small datasets
  • Speed: Convergence in hours, not weeks
  • Quality: Often beats training from scratch

LoRA: Low-Rank Adaptation

Don't fine-tune all weights. Instead, add low-rank matrices alongside frozen weights:

W' = W + \Delta W = W + AB

Where $A \in \mathbb{R}^{d \times r}$ and $B \in \mathbb{R}^{r \times d}$ with $r \ll d$.

Benefits: 10-100x fewer trainable parameters, fits on smaller GPUs, faster training, minimal memory overhead.

LoRA with Hugging Face (peft library)

from peft import LoraConfig, get_peft_model
from transformers import AutoModelForSequenceClassification

model = AutoModelForSequenceClassification.from_pretrained(
    "bert-base-uncased",
    num_labels=2
)

# Configure LoRA
lora_config = LoraConfig(
    r=8,
    lora_alpha=32,
    target_modules=["query", "value"],
    lora_dropout=0.1,
    bias="none",
)

# Wrap model with LoRA
model = get_peft_model(model, lora_config)

# Now train as usual—only LoRA parameters are trainable
print(model.print_trainable_parameters())
# trainable params: 123,456 || all params: 108,000,000 || trainable%: 0.11%

Instruction Fine-tuning

Modern LLMs are often fine-tuned on instruction-response pairs, not raw text. This helps the model follow user instructions.

instruction_dataset = [
    {
        "instruction": "Summarize this text in one sentence.",
        "input": "The Transformer architecture...",
        "output": "Transformers are neural networks that use attention mechanisms instead of recurrence."
    },
    ...
]

RLHF: Reinforcement Learning from Human Feedback

Modern chatbots (ChatGPT, Claude) use RLHF to align with human preferences. A reward model learns to score outputs, then the LLM is fine-tuned to maximize reward. This is why they're more helpful and safer.

Building Chatbots with Transformers

The Architecture

A chatbot is a language model trained on dialogue. It learns to:

  1. Understand user intent from context
  2. Generate contextually appropriate responses
  3. Maintain coherence over multi-turn conversations
  4. Refuse harmful requests (with RLHF)

Conversation Models: Design Choices

Context Window

How many previous messages to include? Longer = more context, but higher compute. Modern LLMs: 2k-100k+ tokens.

System Prompt

A preamble that sets the chatbot's behavior: "You are a helpful, harmless assistant." Crucial for steering output.

Simple Multi-turn Chat Loop

from transformers import AutoTokenizer, AutoModelForCausalLM

tokenizer = AutoTokenizer.from_pretrained("gpt2")
model = AutoModelForCausalLM.from_pretrained("gpt2")

system_prompt = "You are a helpful assistant."
conversation_history = []

while True:
    user_input = input("You: ")

    # Build conversation context
    messages = [
        {"role": "system", "content": system_prompt},
        *conversation_history,
        {"role": "user", "content": user_input}
    ]

    # Convert to text format (GPT-2 doesn't have chat tokens, so simple format)
    text = "\n".join(f"{m['role']}: {m['content']}" for m in messages)

    # Tokenize and generate
    input_ids = tokenizer.encode(text, return_tensors="pt")

    output = model.generate(
        input_ids,
        max_length=input_ids.shape[1] + 100,
        temperature=0.7,
        top_p=0.9,
        do_sample=True,
        eos_token_id=tokenizer.eos_token_id
    )

    assistant_response = tokenizer.decode(output[0][input_ids.shape[1]:])

    print(f"Assistant: {assistant_response}\n")

    # Update history
    conversation_history.append({"role": "user", "content": user_input})
    conversation_history.append({"role": "assistant", "content": assistant_response})

Modern Chat Templates (Chat ML)

Most modern models use structured chat templates:

<|im_start|>system
You are a helpful assistant.
<|im_end|>
<|im_start|>user
What is the capital of France?
<|im_end|>
<|im_start|>assistant
The capital of France is Paris.
<|im_end|>

This format is explicit, parseable, and easy for the model to learn.

Prompt Engineering Basics

Key Principles

  • Be specific: "List 5 benefits of X for Y" beats "Tell me about X"
  • Use examples: Few-shot prompting often beats zero-shot
  • Chain of thought: "Let's think step-by-step" helps reasoning
  • Role play: "You are an expert in..." can improve quality
  • Temperature: Lower (0.1-0.3) for factual, higher (0.7-0.9) for creative

Safety & Alignment

Chatbots need guardrails:

  • Content Filtering: Block NSFW, hateful, etc. in outputs
  • RLHF: Train to refuse harmful requests
  • System Prompts: "I can't help with that" instructions
  • Rate Limiting: Prevent abuse
  • Logging: Monitor for misuse

Conversation Models on Hugging Face

from transformers import pipeline

# Use a specialized chat model
chatbot = pipeline(
    "text-generation",
    model="microsoft/DialoGPT-medium",
    trust_remote_code=True
)

history_ids = None
user_input = "Hello!"

# DialoGPT is designed for multi-turn dialogue
while True:
    user_input = input("You: ")
    new_input_ids = chatbot.tokenizer.encode(
        user_input + chatbot.tokenizer.eos_token,
        return_tensors="pt"
    )

    bot_input_ids = (
        torch.cat([history_ids, new_input_ids], dim=-1)
        if history_ids is not None
        else new_input_ids
    )

    history_ids = chatbot.model.generate(
        bot_input_ids,
        max_length=1000,
        do_sample=True,
        top_p=0.95,
        top_k=50,
        temperature=0.75,
        eos_token_id=chatbot.tokenizer.eos_token_id
    )

    response = chatbot.tokenizer.decode(
        history_ids[0][new_input_ids.shape[-1]:],
        skip_special_tokens=True
    )
    print(f"Bot: {response}\n")

War Room: Socratic Deep Dives

Scenario 1: The Vanishing Context Problem

  • You're building a chatbot for a 10,000-token conversation. Why might you lose information from the first message by the end?
  • How does the Transformer's attention mechanism help, compared to RNNs?
  • What's the trade-off with very long contexts?
  • How do sliding-window attention or sparse attention help? (Hint: Chapter 16!)

Scenario 2: Attention Head Specialization

  • You visualize a 12-head attention model. Head 3 shows a "diagonal" pattern (each token mostly attends to itself), while Head 7 shows broad distribution. What might each be learning?
  • Why isn't it a problem if some heads seem "redundant"?
  • Could you prune redundant heads? What's the trade-off?
  • How would you empirically test whether a head is important?

Scenario 3: Positional Encoding Design

  • Why do we use sinusoids instead of just learned position embeddings (like in RNNs)?
  • The paper claims sinusoids allow "extrapolation to longer sequences." Why?
  • What happens if you train on sequences up to 512 tokens, then use the model on 2048 tokens?
  • Modern solutions: ALiBi (attention with linear biases), rotary embeddings—what are they trying to fix?

Scenario 4: BERT vs. GPT in the Wild

  • You want to classify tweets as toxic or not. Would you use BERT or GPT? Why?
  • Now you want to generate product descriptions from keywords. BERT or GPT?
  • What if you wanted to do both? (Hint: Consider encoder-decoder models like T5.)
  • Why can't you use GPT's causal attention for classification?

Scenario 5: Fine-tuning Gone Wrong

  • You fine-tune BERT on your domain-specific data (medical texts) for 5 epochs with learning rate 0.001. Validation loss increases after epoch 2. What's happening?
  • How do you know if you're overfitting vs. catastrophic forgetting?
  • What's a safe learning rate for fine-tuning? How's it different from pretraining?
  • Would LoRA have helped? Why or why not?

Scenario 6: Prompt Engineering Strategies

  • You ask GPT: "Summarize this article." vs. "You are a journalist. Summarize this article in a punchy headline and 2 sentences." Which works better? Why?
  • What's few-shot prompting, and when does it beat fine-tuning?
  • Why does "Let's think step-by-step" improve reasoning?
  • What's chain-of-thought prompting, and how does it relate to the model's internal "reasoning"?

Scenario 7: Chatbot Safety & Alignment

  • A user tricks your chatbot into generating harmful content by rewording their request. What layers of defense exist?
  • What's RLHF doing differently than supervised fine-tuning on instruction-response pairs?
  • Is it possible to "jailbreak" a well-aligned model? How?
  • What's the relationship between model size and alignment difficulty? (Hint: Scaling laws and emergent behaviors.)

Scenario 8: Production Deployment Trade-offs

  • You're deploying a chatbot to phones. bert-base-uncased (110M params) vs. DistilBERT (66M params) vs. TinyBERT (14M params). What's the latency-quality trade-off?
  • How do quantization, distillation, and pruning help? Which is easiest to implement?
  • When would you use an API (OpenAI's GPT-4) vs. running your own model?
  • What's the cost-benefit analysis for open-source vs. proprietary?
