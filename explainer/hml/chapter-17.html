KV Caching: Reusing Computed Values

Mental Model: KV Caching is like not re-reading the entire book each time you turn a page. Instead, you only read the new page and reference your notes from before.

In generative transformers, we produce tokens one at a time during decoding. Without optimization, each new token requires recomputing attention over all previous tokens—wasting enormous computation.

The Problem: Redundant Computation

When computing attention for token position $i$, we compute queries $Q$, keys $K$, and values $V$ for all positions $0...i$. But when computing token $i+1$, we recompute the same $K$ and $V$ for positions $0...i$ again!

Attention Operation (Standard)
$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$

The Solution: Cache Keys & Values

Instead of recomputing $K$ and $V$ each time, we cache them and reuse. For each new token, we only compute its query and combine with cached keys/values:

KV Cache (Incremental Attention)
$$\text{new\_output} = \text{softmax}\left(\frac{Q_{new}[K_{cached}; K_{new}]^T}{\sqrt{d_k}}\right)[V_{cached}; V_{new}]$$

Memory Trade-off

Standard: Compute attention from scratch each step → $O(n^2 d)$ FLOPs at step $n$

KV Cache: Cache $K, V$ matrices → $O(n d)$ FLOPs per step, but $O(n d)$ memory overhead

Metric Without Cache With Cache
Computation (per token) $O(n \cdot d)$ $O(d)$
Memory (cache) $0$ $O(n \cdot d_{model})$
Speedup 1x 2-5x typical

PyTorch Implementation

class KVCache: def __init__(self, max_seq_len, batch_size, num_heads, head_dim): self.K = torch.zeros(batch_size, num_heads, max_seq_len, head_dim) self.V = torch.zeros(batch_size, num_heads, max_seq_len, head_dim) self.pos = 0 def update(self, K_new, V_new): batch_size = K_new.shape[0] seq_len = K_new.shape[2] self.K[:batch_size, :, self.pos:self.pos+seq_len] = K_new self.V[:batch_size, :, self.pos:self.pos+seq_len] = V_new self.pos += seq_len return self.K[:batch_size, :, :self.pos], \ self.V[:batch_size, :, :self.pos] def reset(self): self.pos = 0

Speculative Decoding: Draft Model Fast Path

Mental Model: Speculative Decoding is like having an assistant draft your email quickly, then you verify and correct it. Usually they're right, so you save work.

The Challenge: Autoregressive Bottleneck

Transformers generate text token-by-token sequentially. Even with KV caching, large models are slow. We cannot parallelize across the sequence because each token depends on previous tokens.

The Idea: Speculate & Verify

Use a smaller "draft" model to generate several candidate tokens quickly, then use the large "verifier" model to accept or reject them in parallel:

  1. Draft: Small model generates $\gamma$ tokens (typically 4-16)
  2. Verify: Large model checks all $\gamma$ tokens in parallel
  3. Accept/Reject: Compare probability distributions; accept if draft ≈ large model
  4. Regenerate: If any token rejected, re-generate from that position

Why It Works

  • Typical acceptance rate: 50-80% of speculated tokens
  • Speedup: 2-3x for many language models
  • Quality preserved: Large model makes final decisions
Acceptance Criterion
$$\text{accept}_i = \begin{cases} 1 & \text{if } p_{\text{large}}(t_i) \geq p_{\text{draft}}(t_i) \\ \text{with prob } p_{\text{large}}(t_i) / p_{\text{draft}}(t_i) & \text{otherwise} \end{cases}$$

When to Use Speculative Decoding

  • ✓ Large language models (7B+ parameters)
  • ✓ Latency-critical applications
  • ✓ When you have access to smaller model
  • ✗ Small models (limited speedup)

Parallelizing Text Generation

Mental Model: Instead of writing one sentence after another, imagine multiple writers working on different branches, then combine the best results.

Techniques for Parallel Generation

1. Batch Decoding

Generate multiple sequences in parallel using mini-batches. If batch size is $B$, you generate $B$ sequences simultaneously.

Limitation: All sequences must complete at same time (padding overhead).

2. Beam Search

Generate multiple continuations at each step, forming a tree structure. Prune low-probability branches.

  • Beam width: $k=3$ to $k=5$ typical
  • Standard in machine translation

3. Non-Autoregressive Generation

Predict all tokens in parallel (iterate, mask randomly, predict, repeat):

  • Fast: 10-50x speedup
  • Trade-off: lower quality
Method Parallelization Quality Speed
Greedy Decoding None Low 1x (baseline)
Batch Decoding Batch Level Low B x (with padding)
Beam Search Intra-sequence High 1-2x typical
Speculative Token Level High 2-3x
Non-Autoregressive Full Medium 10-50x

Sparse Attention Patterns

Mental Model: Standard attention reads every word to understand a book. Sparse attention reads every $k$-th word plus nearby words—and still gets the gist.

Problem: Quadratic Complexity

Standard attention is $O(n^2)$ in sequence length $n$. For long documents (4K-100K tokens), this is prohibitive.

Complexity of Standard Attention
$$\text{Time: } O(n^2 d), \quad \text{Memory: } O(n^2 + nd)$$

Sparse Patterns

  • Local Attention: Each token attends to window of size $w$ → $O(n \cdot w \cdot d)$
  • Strided: Attend every $s$-th token → captures global context
  • Dilated: Exponential spacing (1, 2, 4, 8...) → $O(n \log n)$
  • BigBird/Longformer: Hybrid (local + strided + random) → $O(n \log n)$
Pattern Complexity Window Use Case
Local $O(n \cdot w)$ 256-512 Document classification
Strided $O(n \cdot n/s)$ stride=64 Long-range + local
BigBird $O(n \log n)$ hybrid Long documents (4K+)

Approximate Attention Mechanisms

Mental Model: Instead of computing exact attention scores, approximate them using statistical methods. Like using a hash table summary instead of checking every element.

Linformer: Low-Rank Projection

Observation: Attention matrices are often low-rank. Project to lower dimension before computing attention:

Linformer Projection
$$\text{Attention}_{\text{approx}} = \text{softmax}\left(\frac{Q(K^TP)^T}{\sqrt{d_k}}\right)V, \quad k \ll n$$
  • Complexity: $O(n \cdot k \cdot d)$ instead of $O(n^2 d)$
  • Performance: 2-5% accuracy drop, 3-5x faster

Performer: Kernel Approximation

Use random features to approximate softmax kernel:

FAVOR+ Approximation
$$\exp\left(\frac{QK^T}{\sqrt{d_k}}\right) \approx \phi(Q)\phi(K)^T$$
  • Complexity: $O(n \cdot d)$ (linear!)
  • Accuracy: 1-3% drop typical

Nyströmformer: Landmark Sampling

Sample $m$ landmark tokens, compute full attention on samples, interpolate back:

Nyström Approximation
$$A \approx A_{nm}(A_{mm})^{-1}A_{mn}$$
Method Mechanism Complexity Accuracy Drop
Standard Full $O(n^2)$ $O(n^2 d)$ 0%
Linformer Low-rank $O(n \cdot k \cdot d)$ 2-5%
Performer Kernel approx $O(n \cdot d)$ 1-3%
Nyströmformer Landmark sampling $O(n \cdot m)$ 1-2%

Sharing Projections: GQA & Multi-Query Attention

Mental Model: Standard multi-head attention is like 8 independent listeners. Grouped-query is like them sharing 2 stenographers. Much cheaper, similar insights.

Standard Multi-Head Attention (MHA)

Each of $h$ heads has independent $Q, K, V$ projections. Cost: $h$ separate $K, V$ projections (expensive in KV cache).

Multi-Query Attention (MQA)

Share single $K, V$ projection across all heads:

Multi-Query Attention
$$\text{head}_i = \text{Attention}(Q_iW_i^Q, KW^K, VW^V)$$
  • KV cache reduction: From $h \cdot d$ to $d$ per layer
  • Inference speedup: 2-4x for long sequences
  • Accuracy: ~1-2% drop typical

Grouped-Query Attention (GQA)

Middle ground: share $K, V$ across groups of heads with $g$ groups:

  • Each group of $h/g$ query heads shares one $K, V$ projection
  • Typical: $g=4$ to $g=8$ with $h=32$ or $h=64$
  • Used in: Llama 2, Mistral
Metric MHA (8 heads) GQA (4 groups) MQA (1 group)
KV Projections 8 2 1
Cache Memory $8d$ $2d$ $d$
Speedup (seq=2K) 1x 1.8x 3.2x

FlashAttention: Memory-Efficient Computation

Mental Model: Standard attention stores the full $n \times n$ attention matrix. FlashAttention processes in tiles: load chunks, compute, write back—never materializing the full matrix.

The Problem: Memory Bottleneck

  • Attention scores: $O(n^2)$ memory (not compute-bound)
  • For $n=4096, d=64, h=8$: ~4GB just for attention matrices!
  • Results in slow memory I/O

FlashAttention Solution: IO-Aware Algorithm

Process attention in blocks to maximize cache reuse:

Algorithm Overview

  1. Tile Q, K, V: Load small blocks into fast SRAM
  2. Compute attention: $S = Q_i K_j^T / \sqrt{d_k}$
  3. Online softmax: Update running statistics without storing full matrix
  4. Output block: Write to main memory
  5. Repeat for all tile pairs
FlashAttention Memory Complexity
$$\text{Standard: } O(n^2) \quad \text{FlashAttention: } O(n)$$ $$\text{Speedup: } 2-4x \text{ on modern GPUs}$$

Key Insights

  • Online Softmax: Compute incrementally as you load blocks
  • Tile Size: Matches GPU SRAM (96KB on A100)
  • Trade-off: Extra computation (online softmax) for fewer memory accesses
  • Exactness: Mathematically exact (no approximation)

FlashAttention-2: Further optimizations (work partitioning, reduced sync overhead) → ~2x faster than original.

Mixture of Experts: Scaling to Trillions

Mental Model: Instead of one massive expert, have dozens of specialized experts. A router decides which to use per token. You pay for only a few experts per token, not all.

The Challenge: Scaling Laws

Transformers follow: larger = better. But 10x model size = 10x compute. For trillion-parameter models, this is prohibitive.

Mixture of Experts Solution

Replace dense FFN with sparse mixture:

Standard FFN vs MoE
$$\text{Dense: } y = W_2 \text{ReLU}(W_1 x)$$ $$\text{MoE: } y = \sum_{i=1}^{N} \text{Gate}(x)_i \cdot \text{Expert}_i(x)$$

Architecture

  • Gate/Router: Learns per-token routing weights
  • Experts: Independent FFN modules (e.g., 128 experts)
  • Sparsity: Select top-$k$ experts per token (typically $k=2$)
  • Load Balancing: Loss term to prevent all tokens routing to same expert

GShard 6144 2 600B 97B
Switch Transformers 2048 1 1.6T 16B
Mixtral 8x7B 8 2 56B 12.9B

Key Insight: Enables trillion parameters while keeping inference cheap (2-3 experts per token).

Parameter-Efficient Fine-Tuning (PEFT)

Mental Model: LoRA is like giving an expert a small notebook to jot updates, rather than rewriting their entire knowledge base.

The Problem: Expensive Fine-Tuning

Fine-tuning a 7B model requires ~100GB memory (params + gradients + optimizer states). Solution: PEFT adapts only a small subset.

Low-Rank Adaptation (LoRA)

Key observation: weight updates have low rank. Instead of updating $W \in \mathbb{R}^{d_{out} \times d_{in}}$:

LoRA Update Rule
$$W_{\text{new}} = W + AB^T$$ $$\text{where } A \in \mathbb{R}^{d_{out} \times r}, \quad B \in \mathbb{R}^{d_{in} \times r}, \quad r \ll \min(d_{out}, d_{in})$$

Memory Savings

Configuration Tunable Params Memory Performance
7B, full fine-tune 7B ~80GB 100%
7B, LoRA (r=8) 4.2M ~12GB 97-99%
7B, QLoRA (r=8) 4.2M ~6GB 94-98%

Other PEFT Techniques

Prefix Tuning

Add learnable "prefix" tokens before input. Parameters: $p \times d_{model}$ (typically $p=100-500$).

Adapters

Insert small bottleneck modules in each layer. Between FFN and attention: $d \rightarrow r \rightarrow d$ projection.

QLoRA: Quantized LoRA

Freeze weights in 4-bit, train LoRA in full precision. Memory: ~6GB for 7B model on single GPU.

Efficient Training Techniques

Activation Checkpointing (Gradient Checkpointing)

Mental Model: Don't save all intermediate activations for backprop. Only save at checkpoints, recompute the rest. Trade compute for memory.
  • Memory saved: 50-70% typical
  • Compute overhead: ~30% slower training
  • When to use: Memory constrained

Sequence Packing

Instead of padding short sequences, concatenate multiple sequences:

  • Efficiency gain: ~50-80% on typical data
  • Handle sequence boundaries with special tokens

Gradient Accumulation

Simulate large batch size without large GPU memory:

  • Process small batches sequentially
  • Accumulate gradients without updating weights
  • Update weights after $k$ accumulation steps
Technique Memory Saved Speed Impact Implementation
Checkpointing 50-70% -30% speed Simple decorator
Sequence Packing 50-80% +10% speed Preprocessing
Grad Accumulation ~1/k -20% speed Training loop
Mixed Precision 40-50% +10-20% speed torch.autocast

Distributed Training & Parallelism

Data Parallelism (DP)

Replicate model on each GPU, split data. Each GPU: 1 model copy, different batch. Synchronize gradients after backward pass.

Tensor Parallelism (TP)

Split weight matrices across GPUs. Linear layer $W \in \mathbb{R}^{d \times d}$ split row-wise or column-wise.

Pipeline Parallelism (PP)

Split model layers across GPUs. GPU 0: layers 0-3, GPU 1: layers 4-7, etc. Data flows through pipeline.

Fully Sharded Data Parallel (FSDP)

Combines data and tensor parallelism:

  • Each GPU: 1/N of parameters, full batch copy
  • During forward: AllGather parameters, compute
  • After backward: ReduceScatter gradients
  • Scales: Trains 70B models on cluster of A100s
Method Model Memory Communication Typical Use
Data Parallel (DP) N x model AllReduce grads 8 GPUs, small model
Tensor Parallel (TP) Model / N AllGather/Scatter Model > GPU memory
Pipeline Parallel (PP) Model / N Layer outputs Very deep models
FSDP Model / N AllGather/Scatter 70B+ models
