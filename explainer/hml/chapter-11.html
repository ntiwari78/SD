PART II: NEURAL NETWORKS & DEEP LEARNING

Training Deep Neural Networks

The Art of Making Deep Networks Reliable

Deep networks are powerful but fragile. This chapter is your toolkit: proper initialization, batch normalization, gradient clipping, learning rate scheduling, regularization, and transfer learning—all techniques to make deep training stable, fast, and successful.

Vanishing & Exploding Gradients

The Core Problem

As gradients flow backward through many layers during backpropagation, they can shrink exponentially (vanish) or grow exponentially (explode). In deep networks with sigmoid activations, gradients become nearly zero in lower layers, preventing learning.

Interactive Gradient Flow Visualizer

Watch how gradient magnitude changes through network depth with different activation functions:

Gradient magnitude per layer (log scale)

Key Insight: Sigmoid activation outputs lie in [0, 1], and its derivative is at most 0.25. Through just 10 layers, the gradient becomes approximately 0.25^10 ≈ 10^-6, causing lower layers to learn extremely slowly.

Why ReLU Helps

ReLU's derivative is 1 for positive inputs, so gradients flow more reliably. However, ReLU can suffer from "dying ReLU" problem where neurons become permanently inactive.

Weight Initialization Strategies

Proper initialization prevents extreme gradient issues at the start of training. The goal is to maintain similar gradient magnitudes across all layers.

Xavier (Glorot) Initialization

For sigmoid/tanh activations, set the variance of weights to:

$$\text{Var}(w) = \frac{1}{n_{\text{in}} + n_{\text{out}}}$$

This keeps activations within a reasonable range, avoiding saturation.

He Initialization

For ReLU and variants, use:

$$\text{Var}(w) = \frac{2}{n_{\text{in}}}$$

The factor of 2 accounts for ReLU's behavior of zeroing out negative values.

LeCun Initialization

For SELU activations (with specific conditions), use:

$$\text{Var}(w) = \frac{1}{n_{\text{in}}}$$

Initialization Comparison

Activation Initializer Variance Formula PyTorch Code
Sigmoid/Tanh Xavier (Glorot) 1/(n_in + n_out) torch.nn.init.xavier_uniform_
ReLU He 2/n_in torch.nn.init.kaiming_uniform_
SELU LeCun 1/n_in torch.nn.init.lecun_uniform_

PyTorch Default Behavior

PyTorch's nn.Linear layers use a default initialization with variance = 1/n_in. For He initialization, explicitly call torch.nn.init.kaiming_uniform_ after layer creation.

Batch Normalization

Batch normalization (BN) standardizes layer inputs during training, addressing vanishing/exploding gradients at a deeper level than initialization alone.

The BN Algorithm

$$1. \quad \mu_B = \frac{1}{m_B} \sum_{i=1}^{m_B} \mathbf{x}^{(i)}$$
$$2. \quad \sigma_B^2 = \frac{1}{m_B} \sum_{i=1}^{m_B} \left(\mathbf{x}^{(i)} - \mu_B\right)^2$$
$$3. \quad \hat{\mathbf{x}}^{(i)} = \frac{\mathbf{x}^{(i)} - \mu_B}{\sqrt{\sigma_B^2 + \epsilon}}$$
$$4. \quad \mathbf{z}^{(i)} = \gamma \odot \hat{\mathbf{x}}^{(i)} + \beta$$

where μ_B is the batch mean, σ_B is the batch std, γ (scale) and β (shift) are learnable parameters, and ε ≈ 10^-5 prevents division by zero.

Interactive BN Comparison

Training loss over epochs: with vs. without batch norm

Benefits of Batch Normalization

  • Faster convergence: Networks often converge 14× faster with BN
  • Higher learning rates: BN allows larger learning rates without divergence
  • Regularization effect: Acts as a regularizer, reducing need for dropout
  • Reduced initialization sensitivity: Less dependent on careful weight initialization

Train vs. Eval Mode (Critical!)

During training: BN uses batch statistics (μ_B, σ_B).

During evaluation: BN uses running averages of batch statistics, estimated during training.

Common Mistake

Forgetting to call model.eval() during evaluation causes incorrect predictions. BN layers behave differently in train vs. eval mode!

PyTorch Implementation

import torch.nn as nn

model = nn.Sequential(
    nn.Flatten(),
    nn.Linear(784, 300),
    nn.BatchNorm1d(300),
    nn.ReLU(),
    nn.Linear(300, 100),
    nn.BatchNorm1d(100),
    nn.ReLU(),
    nn.Linear(100, 10)
)

Batch Norm Variations

  • BatchNorm1d: For 2D inputs [batch_size, features] (fully connected layers)
  • BatchNorm2d: For 4D inputs [batch, channels, height, width] (conv layers)
  • LayerNorm: Normalizes across feature dimensions instead of batch—better for RNNs and transformers

Gradient Clipping

Essential for RNNs and very deep networks. Clip gradients during backpropagation to prevent exploding gradients.

Two Clipping Strategies

$$\text{Clip by value: } g_i \leftarrow \text{clip}(g_i, -\text{threshold}, +\text{threshold})$$
$$\text{Clip by norm: } \mathbf{g} \leftarrow \frac{\text{threshold}}{\|\mathbf{g}\|}\mathbf{g} \text{ if } \|\mathbf{g}\| > \text{threshold}$$

Clip by norm (more common) preserves gradient direction while reducing magnitude.

PyTorch Usage

for epoch in range(n_epochs):
    for X_batch, y_batch in train_loader:
        y_pred = model(X_batch)
        loss = loss_fn(y_pred, y_batch)
        loss.backward()

        # Clip gradients
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

        optimizer.step()
        optimizer.zero_grad()

Transfer Learning: Reusing Pretrained Layers

Instead of training from scratch, reuse lower layers from a model trained on a similar task. Lower layers learn generic feature detectors; only higher layers need adjustment.

When to Use Transfer Learning

  • Small dataset + similar domain: Freeze most layers, fine-tune only top layers
  • Large dataset + different domain: Freeze fewer layers, retrain more
  • Small dataset + different domain: Keep most layers frozen, only train new output layer

Transfer Learning Strategy

  1. Load pretrained model
  2. Replace output layer for your task
  3. Freeze lower layers (set requires_grad=False)
  4. Train only the new output layer
  5. Unfreeze some upper layers and fine-tune with lower learning rate

PyTorch Example

import copy

# Load pretrained model
model_A = nn.Sequential([...])  # pretrained
reused_layers = copy.deepcopy(model_A[:-1])  # all but output

# New model for task B
model_B = nn.Sequential(
    *reused_layers,
    nn.Linear(100, 1)  # new output layer for binary task
)

# Freeze reused layers initially
for layer in model_B[:-1]:
    for param in layer.parameters():
        param.requires_grad = False

# Train with high learning rate on new layer only
optimizer = torch.optim.Adam(model_B[-1].parameters(), lr=0.001)

# After a few epochs, unfreeze upper layers
for param in model_B[-2].parameters():
    param.requires_grad = True

# Fine-tune with lower learning rate
optimizer = torch.optim.Adam(model_B.parameters(), lr=0.0001)

Why It Works

Transferring 92.5% accuracy with only 20 labeled examples is far better than training from scratch (71.6%). Pretrained models have already learned edge detection, texture patterns, and shapes—generic features useful for many vision tasks.

Faster Optimizers: Beyond Vanilla SGD

Different optimizers adapt the learning rate per parameter and accumulate momentum, enabling faster convergence.

Momentum Optimization

Accumulate a momentum vector that accelerates convergence on consistent gradient directions:

$$1. \quad \mathbf{m} \leftarrow \beta\mathbf{m} - \eta\nabla_\theta J(\theta)$$
$$2. \quad \theta \leftarrow \theta + \mathbf{m}$$

Typical β = 0.9. If gradient is constant, terminal velocity = η / (1 - β) = 10η, giving 10× speedup!

Nesterov Accelerated Gradient (NAG)

Measure gradient at a position slightly ahead in the momentum direction:

$$\mathbf{m} \leftarrow \beta\mathbf{m} - \eta\nabla_\theta J(\theta + \beta\mathbf{m})$$

Converges slightly faster than momentum by correcting direction earlier.

AdaGrad

Scales learning rate by accumulated squared gradients (adaptive per parameter):

$$1. \quad \mathbf{s} \leftarrow \mathbf{s} + \nabla_\theta J(\theta) \odot \nabla_\theta J(\theta)$$
$$2. \quad \theta \leftarrow \theta - \eta\nabla_\theta J(\theta) \oslash \sqrt{\mathbf{s} + \epsilon}$$

Problem: Learning rate decays monotonically—may converge too early.

RMSProp

Fixes AdaGrad by decaying old gradients exponentially:

$$1. \quad \mathbf{s} \leftarrow \alpha\mathbf{s} + (1-\alpha)\nabla_\theta J(\theta) \odot \nabla_\theta J(\theta)$$
$$2. \quad \theta \leftarrow \theta - \eta\nabla_\theta J(\theta) \oslash \sqrt{\mathbf{s} + \epsilon}$$

α typically 0.9. Much better than AdaGrad for deep learning.

Adam: Combining Momentum + RMSProp

Adam (Adaptive Moment Estimation) combines momentum and per-parameter learning rates:

$$1. \quad \mathbf{m} \leftarrow \beta_1\mathbf{m} + (1-\beta_1)\nabla_\theta J(\theta)$$
$$2. \quad \mathbf{s} \leftarrow \beta_2\mathbf{s} + (1-\beta_2)\nabla_\theta J(\theta) \odot \nabla_\theta J(\theta)$$
$$3. \quad \hat{\mathbf{m}} \leftarrow \frac{\mathbf{m}}{1-\beta_1^t}, \quad \hat{\mathbf{s}} \leftarrow \frac{\mathbf{s}}{1-\beta_2^t}$$
$$4. \quad \theta \leftarrow \theta + \eta\hat{\mathbf{m}} \oslash \sqrt{\hat{\mathbf{s}} + \epsilon}$$

Default: β₁ = 0.9, β₂ = 0.999, lr = 0.001. Requires very little tuning!

Adam Variants

  • AdamW: Decouples weight decay from gradient updates—works better with regularization
  • NAdam: Adam + Nesterov momentum—slightly faster convergence
  • AdaMax: Uses ℓ∞ norm instead of ℓ₂—more stable on some tasks

PyTorch Optimizer Examples

import torch.optim as optim

# Momentum SGD (classic, reliable)
optimizer = optim.SGD(model.parameters(), lr=0.05, momentum=0.9)

# Nesterov momentum
optimizer = optim.SGD(model.parameters(), lr=0.05, momentum=0.9, nesterov=True)

# RMSProp
optimizer = optim.RMSprop(model.parameters(), lr=0.001, alpha=0.9)

# Adam (most popular)
optimizer = optim.Adam(model.parameters(), lr=0.001, betas=(0.9, 0.999))

# AdamW (recommended for modern deep learning)
optimizer = optim.AdamW(model.parameters(), lr=0.001, weight_decay=1e-4)

Optimization Gotcha

Adaptive methods (Adam, RMSProp) can generalize poorly on some datasets. If Adam underperforms, try Nesterov momentum instead—it's often better for production models.

Learning Rate Scheduling

A constant learning rate is rarely optimal. Start high for fast progress, then reduce to fine-tune. PyTorch's torch.optim.lr_scheduler automates this.

Power Scheduling

Decay learning rate by a power of epoch number:

$$\eta(t) = \eta_0 \cdot (1 + t/s)^{-c}$$

where s and c are hyperparameters (often s=1, c=0.5).

Exponential Decay

$$\eta(t) = \eta_0 \cdot \gamma^t$$

γ typically 0.1 to 0.99. Faster drop than power scheduling.

Cosine Annealing

Smoothly decrease learning rate using cosine function:

$$\eta_t = \eta_{\min} + \frac{1}{2}(\eta_{\max} - \eta_{\min})\left(1 + \cos\left(\frac{t\pi}{T_{\max}}\right)\right)$$

Better than exponential—keeps learning rate high longer, then drops sharply at end.

1-Cycle Policy (State-of-the-Art)

Start at low LR, ramp up to max at mid-training, decay back down. Often achieves best results!

Warmup Phase

For sensitive architectures (RNNs, large batch sizes), increase learning rate gradually over first few epochs to stabilize training.

PyTorch Scheduler Examples

# Exponential decay
scheduler = optim.lr_scheduler.ExponentialLR(optimizer, gamma=0.9)

# Cosine annealing
scheduler = optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=200, eta_min=0.00001)

# 1-Cycle (Leslie Smith's super-convergence)
scheduler = optim.lr_scheduler.OneCycleLR(
    optimizer, max_lr=0.1, total_steps=1000, pct_start=0.3, anneal_strategy='cos'
)

# Step-based decay (reduce by factor every N epochs)
scheduler = optim.lr_scheduler.StepLR(optimizer, step_size=10, gamma=0.5)

# Reduce on plateau (when validation metric stops improving)
scheduler = optim.lr_scheduler.ReduceLROnPlateau(
    optimizer, mode='min', factor=0.1, patience=10
)

# Training loop
for epoch in range(200):
    for X_batch, y_batch in train_loader:
        # ...training step...
        optimizer.step()

    scheduler.step()  # Update learning rate after each epoch

Pro Tip: ReduceLROnPlateau

This scheduler is adaptive—it monitors validation performance and reduces LR only when stuck on a plateau. Perfect for preventing overfitting while squeezing out every bit of accuracy.

Avoiding Overfitting Through Regularization

With thousands or millions of parameters, deep networks easily overfit. Regularization techniques prevent this.

ℓ₁ and ℓ₂ Regularization (Weight Decay)

Add a penalty term proportional to weight magnitude:

$$\text{Loss}_{\text{total}} = \text{Loss}_{\text{data}} + \lambda \left(\frac{1}{2}\sum_i w_i^2\right) \quad (\text{ℓ}_2)$$
$$\text{Loss}_{\text{total}} = \text{Loss}_{\text{data}} + \lambda \sum_i |w_i| \quad (\text{ℓ}_1)$$

ℓ₂ drives weights toward zero. ℓ₁ drives some weights exactly to zero (sparse model).

# ℓ₂ regularization via SGD weight decay
optimizer = torch.optim.SGD(model.parameters(), lr=0.05, weight_decay=1e-4)

Dropout: Preventing Co-Adaptation

At each training step, randomly drop p% of neurons (set to 0), forcing the network to learn redundant representations:

  • Training: Randomly drop neurons, scale remaining by 1/(1-p)
  • Testing: Use all neurons (no dropout)

Typical p = 0.2-0.5 for hidden layers.

model = nn.Sequential(
    nn.Flatten(),
    nn.Linear(784, 300), nn.ReLU(),
    nn.Dropout(p=0.2),
    nn.Linear(300, 100), nn.ReLU(),
    nn.Dropout(p=0.2),
    nn.Linear(100, 10)
)

# Training mode (dropout active)
model.train()

# Evaluation mode (no dropout)
model.eval()

Monte Carlo Dropout (Uncertainty Estimation)

Keep dropout active even during inference, take multiple stochastic predictions, and average. Gives uncertainty estimates!

model.eval()
X_new = ...  # some new images
predictions = []

for _ in range(100):
    # Keep dropout on
    for module in model.modules():
        if isinstance(module, nn.Dropout):
            module.train()

    y_pred = model(X_new)
    predictions.append(y_pred)

y_pred_mean = torch.stack(predictions).mean(dim=0)
y_pred_std = torch.stack(predictions).std(dim=0)
# Use y_pred_std for confidence/uncertainty

Max-Norm Regularization

Constrain incoming weights of each neuron to have norm ≤ r:

$$\text{if } \|\mathbf{w}\|_2 > r \text{ then } \mathbf{w} \leftarrow \mathbf{w} \cdot \frac{r}{\|\mathbf{w}\|_2}$$

Prevents any single neuron from dominating.

Data Augmentation

Artificially expand training set: crop, rotate, flip, scale, add noise. Network sees variations, learns more robust features.

Early Stopping

Monitor validation loss. Stop training when it stops improving for N epochs. Simple and effective!

best_loss = float('inf')
patience = 10
patience_count = 0

for epoch in range(1000):
    train_loss = train_one_epoch()
    val_loss = evaluate(model, val_loader)

    if val_loss < best_loss:
        best_loss = val_loss
        patience_count = 0
        torch.save(model.state_dict(), 'best_model.pt')
    else:
        patience_count += 1
        if patience_count >= patience:
            print("Early stopping!")
            break

War Room: Scenario Analysis

Scenario: The Deep Network Puzzle

Situation: Your 20-layer deep network achieves 98% training accuracy but only 71% test accuracy. Training took 2 hours. Management asks: "Why isn't the fancy deep network working?"

Key Symptoms:

  • Training loss is very low (nearly perfect)
  • Validation loss is high (large gap)
  • Training is slow (2 hours for modest dataset)

Diagnostic Questions:

  • How many parameters? Is the network WAY larger than the training set?
  • What regularization is active? (Dropout? Weight decay? Early stopping?)
  • Are you using batch norm? If not, why?
  • What's your learning rate? Is it adaptive (Adam) or fixed?
  • Can you visualize training vs. validation curves? At what epoch does val loss diverge?

Most Likely Root Causes & Solutions:

1. Severe Overfitting (98% train vs. 71% test = 27% gap)

  • Add or increase dropout (start with p=0.3-0.5 on hidden layers)
  • Add ℓ₂ regularization (weight_decay=1e-4 to 1e-3)
  • Enable early stopping on validation accuracy
  • Try data augmentation if applicable (rotations, crops, etc.)

2. Slow Training (2 hours = possible learning rate too low?)

  • Switch to AdamW optimizer (usually 5-10× faster than vanilla SGD)
  • Try learning rate scheduling (OneCycleLR often speeds training)
  • Add batch normalization (accelerates convergence 5-14×)

3. Potential Gradient Issues (deep network, slow convergence)

  • Add batch norm before or after ReLU activations
  • Use He initialization for ReLU layers
  • Try better activation: Swish or Mish instead of ReLU

Quick Win Priority Order:

  1. Add batch norm + AdamW + OneCycleLR scheduler
  2. Add dropout (p=0.2-0.3) to hidden layers
  3. Use ReduceLROnPlateau to reduce LR when stuck
  4. Consider transfer learning (does a pretrained model exist?)

Expected Outcome: With proper regularization + optimization, you should see training time cut in half, test accuracy jump to 85-90%, and the overfitting gap shrink dramatically.

Debugging Checklist

Before Blaming Your Model:

  • Is data normalized (mean 0, std 1)?
  • Are train/test splits clean (no data leakage)?
  • Is model in correct mode? (model.train() vs. model.eval())
  • Are gradients flowing? (Check param.grad is not None)
  • Is optimizer updating weights? (Check loss actually decreases)

Complete Training Pipeline Example

Putting it all together—a production-ready training loop with all Chapter 11 techniques:

import torch
import torch.nn as nn
import torch.optim as optim
from torch.optim.lr_scheduler import OneCycleLR

# 1. Define model with proper initialization & batch norm
model = nn.Sequential(
    nn.Flatten(),
    nn.Linear(784, 300), nn.BatchNorm1d(300), nn.ReLU(), nn.Dropout(0.2),
    nn.Linear(300, 100), nn.BatchNorm1d(100), nn.ReLU(), nn.Dropout(0.2),
    nn.Linear(100, 50), nn.BatchNorm1d(50), nn.ReLU(), nn.Dropout(0.1),
    nn.Linear(50, 10)
).to(device)

# 2. Loss, optimizer, scheduler
loss_fn = nn.CrossEntropyLoss()
optimizer = optim.AdamW(model.parameters(), lr=0.001, weight_decay=1e-4)
scheduler = OneCycleLR(optimizer, max_lr=0.01, total_steps=100*len(train_loader))

# 3. Training loop
best_val_acc = 0
for epoch in range(100):
    # Training
    model.train()
    train_loss = 0
    for X_batch, y_batch in train_loader:
        X_batch, y_batch = X_batch.to(device), y_batch.to(device)

        y_pred = model(X_batch)
        loss = loss_fn(y_pred, y_batch)

        optimizer.zero_grad()
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        optimizer.step()
        scheduler.step()

        train_loss += loss.item()

    # Validation
    model.eval()
    val_loss = 0
    val_acc = 0
    with torch.no_grad():
        for X_val, y_val in val_loader:
            X_val, y_val = X_val.to(device), y_val.to(device)

            y_pred = model(X_val)
            loss = loss_fn(y_pred, y_val)
            val_loss += loss.item()

            acc = (y_pred.argmax(1) == y_val).float().mean()
            val_acc += acc.item()

    val_acc /= len(val_loader)

    if val_acc > best_val_acc:
        best_val_acc = val_acc
        torch.save(model.state_dict(), 'best_model.pt')

    if epoch % 10 == 0:
        print(f"Epoch {epoch}: train_loss={train_loss/len(train_loader):.4f}, "
              f"val_loss={val_loss/len(val_loader):.4f}, val_acc={val_acc:.4f}")

# 4. Load best model and evaluate on test set
model.load_state_dict(torch.load('best_model.pt'))
model.eval()
