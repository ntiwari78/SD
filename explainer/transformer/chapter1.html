🧠 The Single Runner (1943)
McCulloch-Pitts Neuron

The Breakthrough: Warren McCulloch and Walter Pitts created the first mathematical model of a neuron—a single runner carrying a binary message.

How it works: Like a runner taking inputs from checkpoints, summing them up, and deciding whether to run forward (output 1) or stop (output 0). The runner follows a simple rule: if the sum exceeds a threshold, fire.

The Limitation: One runner can only run in straight lines. It can't handle curves or complex paths. This creates the foundation for everything else.

Impact: Laid theoretical groundwork for artificial neural networks.

🏃 The Linear Barrier (1957)
Rosenblatt's Perceptron

The Breakthrough: Frank Rosenblatt invented the Perceptron—a machine that could learn! Our single runner could now train itself on examples.

How it works: Show the runner 100 races (labeled examples). After each race, if it got the outcome wrong, it adjusts its route slightly. Eventually, it learns the pattern.

The Limitation: The runner can ONLY navigate straight paths. Try to teach it the XOR problem (corners required), and it fails completely. The runner doesn't realize some problems require multiple turns.

Impact: First learning algorithm; inspired decades of neural network research.

❌ The XOR Problem (1969)
Minsky & Papert's Crushing Proof

The Breakthrough (or Breakdown?): Marvin Minsky and Seymour Papert published "Perceptrons," mathematically proving that a single perceptron cannot solve XOR.

The Problem: XOR (exclusive OR) requires navigating a corner:

  • Input (0,0) → Output 0
  • Input (0,1) → Output 1
  • Input (1,0) → Output 1
  • Input (1,1) → Output 0

No single straight line can separate the 1s from the 0s. The runner would need to turn corners—it needs a team.

The Consequence: This triggered the First AI Winter (1969-1980). Funding dried up. Belief in neural networks evaporated. "Impossible," the experts said.

Impact: Revealed the fundamental limitation of single-layer networks; drove research into deep architectures.

🏃‍♂️🏃‍♀️🏃 The Relay Team (1986)
Backpropagation and Hidden Layers

The Breakthrough: David Rumelhart, Geoffrey Hinton, and Ronald Williams invented backpropagation—teaching a relay team to run together.

How it works: Now we have 3+ runners in a relay. Runner 1 gets the input, passes the baton to Runner 2 (hidden layer), who passes to Runner 3 (output). But here's the key: we can now teach them HOW to run internally. They learn their own secret patterns.

Why it solves XOR: The first hidden layer learns to detect corners. The second layer learns to avoid them. Multi-layer networks can represent ANY function.

New Problem: As the relay gets longer (deeper), the baton gets weaker. The last runner barely feels the handoff from the first. This is the vanishing gradient problem.

Impact: End of the AI Winter. Neural networks became practical again. Led to CNNs and modern deep learning.

📓 The Memory Runners (1997)
LSTMs: Runners with Notebooks

The Breakthrough: Sepp Hochreiter and Jürgen Schmidhuber invented LSTMs (Long Short-Term Memory)—runners who could carry notebooks.

How it works: Now each runner has a notebook. As they run, they write down important facts: "There was a plot twist at mile 3. Important for the finish." By the time they reach the end, they still remember what happened at the start.

Why it matters: This solved the vanishing gradient problem. Long sequences (like sentences or music) could finally be processed. The notebooks let the gradient flow backward through time.

The Limitation: The runners still run ONE AT A TIME. Runner 1 finishes, passes the baton to Runner 2, who must wait for Runner 1 to complete. This is sequential processing. If you have a 1000-token sequence, you need 1000 sequential steps.

Impact: Dominated sequence modeling for 20 years. Made RNNs practical for language tasks.

🎯 Deep Learning Boom (2012)
ImageNet Moment & The Data Explosion

The Breakthrough: Geoffrey Hinton's team won the ImageNet competition with a deep CNN (AlexNet), reducing error rates by 41%. Suddenly, the world believed again.

Why it happened: Three ingredients aligned:

  • Big Data: ImageNet (14M labeled images). Wikipedia. The internet.
  • Computing Power: GPUs became cheap enough for training.
  • Better Techniques: ReLU activation, dropout, batch normalization—practical tricks to train deep networks.

The Shift: We stopped hand-engineering features. Instead of "tell me what edges matter," the network learns edges, textures, shapes—everything automatically.

Impact: Modern deep learning era began. Word2Vec (2013), Seq2Seq (2014), major NLP breakthroughs followed.

⚡ The Revolution (2017)
Attention Is All You Need

The Breakthrough: Vaswani et al. published "Attention Is All You Need," introducing the Transformer. No recurrence. No sequential processing. Everyone looks at everyone simultaneously.

How it works: Forget the relay race. Now imagine a stadium full of runners (tokens). Each runner can look at EVERY OTHER RUNNER at the same time. "What does Runner 27 matter for my output? Let me check... 80% attention." "Runner 5? 15% attention." "Runner 92? 5% attention." All computed in parallel.

Why it's revolutionary:

  • Parallelism: Process 1000 tokens in the same time as 10. No sequential bottleneck.
  • Long-Range Dependencies: No gradient vanishing. Token 1000 can directly attend to Token 1. No notebook needed.
  • Scalability: Scales with data and compute. Bigger models, more data, more compute = better performance.

The Trade-off: Attention uses O(n²) memory and compute. 1000 tokens = 1M attention pairs. This is why we later invented sparse attention, linear attention, and the taxonomies in Section 1.2.

Impact: BERT, GPT, all modern LLMs. Transformers now dominate NLP, vision, multimodal, reasoning. The foundation of modern AI.

🎯 The Big Picture: Why Transformers Dominate

1943-1969: Single-layer networks can only solve linearly-separable problems.

1986-2006: Deep networks with backpropagation solved this, but gradients vanish in very deep networks.

1997-2016: LSTMs solved vanishing gradients for sequences, but remained sequential. One token at a time.

2017+: Transformers combined the best of both: no sequential bottleneck (parallelizable) + full expressivity (attention mechanism) + no vanishing gradients (direct global connections).

The evolution was not random: each breakthrough solved a specific bottleneck that the previous generation hit.

🗺️ Transformer Taxonomy: A Map of Innovation

Since "Attention Is All You Need" (2017), researchers have explored three dimensions of innovation: architecture, pre-training, and applications. This section maps the key innovations in architectural modifications.

The Three Pillars of Transformer Research

🏗️ Architecture Modifications

How to improve the transformer blocks themselves: depth, width, attention mechanisms, positional encodings. This section focuses here.

📚 Pre-training Methods

How to train from scratch: BERT (MLM), GPT (next-token prediction), T5 (encoder-decoder). Different objectives, different behaviors.

🎯 Applications

Domain-specific variants: ELECTRA (efficient pre-training), RoBERTa (improved BERT), ALBERT (parameter sharing), DeBERTa (disentangled attention).

Block-Level Architectural Modifications

Changes to how transformer blocks (attention + feed-forward) are stacked and connected:

1️⃣ Lightweight Transformers 3 variants

Models designed to reduce memory footprint and inference latency:

Lite Transformer
Depth-wise separable convolutions in feed-forward layers. Reduces FLOPs.
Funnel Transformer
Hierarchical pooling: reduce sequence length in early layers, expand in later layers.
DeLighT
"Deep Light Transformers." Depth-efficient: same performance with fewer layers.
2️⃣ Cross-Block Connectivity 2 variants

Adding skip connections and information flow across non-adjacent blocks:

Realformer
Residual connections between attention layers of different depths. Improves gradient flow.
Transparent Attention
Attention weights shared across layers, reducing parameters while maintaining expressivity.
3️⃣ Adaptive Computation 3 variants

Models that allow different tokens to skip layers or compute early exits:

Universal Transformer
Same weights across layers, processed with different number of recurrent steps. Allows early stopping per token.
Conditional Computation
Mixture of experts within each layer. Some tokens route to expert A, others to expert B.
DeeBERT
BERT with dynamic early exit: easy tokens stop at layer 5, hard tokens need all 12 layers.
4️⃣ Recurrent Transformers 3 variants

Reintroducing recurrence for longer context and memory:

Transformer-XL
Recurrence across segments. Previous segment's hidden state becomes context for next segment. Extended context window.
Compressive Transformer
Extends Transformer-XL by compressing old memory with autoencoders. Longer effective context with less storage.
Memformer
Explicit memory module. Store important information from previous segments; retrieve when relevant.
5️⃣ Hierarchical Transformers 2 variants

Multi-scale processing similar to vision models (features at different resolutions):

HiBERT
Token-level and sentence-level hierarchies. Attention within sentences, then between sentence representations.
Hi-Transformer
Hierarchical attention: local (window-based) + global (token-level). Multi-scale context.
6️⃣ Alternative Architectures 3 variants

Structural rearrangements of standard transformer blocks:

Macaron Transformer
Expanded hidden dimension in feed-forward: FFN → larger → FFN. Matches speech recognition gains.
Sandwich Transformer
Swaps layer order: attention-before-FFN vs. FFN-before-attention. Different gradient flow.
Neural Architecture Search (NAS)
Automatically discover optimal block arrangements via reinforcement learning or evolutionary search.

Sublayer-Level Attention Modifications

Innovations within the multi-head attention mechanism itself:

🎯 Complexity Reduction 5 variants

Reducing O(n²) memory and compute of full attention:

Longformer
Sliding window attention (local) + global attention on select tokens. O(n) instead of O(n²).
Reformer
Locality-sensitive hashing (LSH): only attend to nearby tokens in embedding space.
BigBird
Combination: block-local + sliding window + random attention patterns. O(n) with good coverage.
Performer
Kernel methods: approximate softmax(QK^T)V with feature maps. O(n) via fast matrix multiplication.
Routing Transformer
Learned routing: tokens dynamically route to subset of attention heads based on content.
📉 Low-Rank Approximations 3 variants

Approximating attention matrix as low-rank:

Performer
Feature-based low-rank: QK^T ≈ Φ(Q)Φ(K)^T where Φ are random features.
Nystromformer
Nyström approximation: subsample attention matrix, approximate full matrix from submatrix.
Synthesizer
Skip Q,K,V: directly learn attention weights. Learnable + low-rank decomposition.
🎲 Attention with Priors 3 variants

Inductive biases: some positions should attend to each other more:

Gaussian Transformer
Bias attention toward nearby positions using Gaussian kernels. Prefers local context.
Realformer
Adds relative position information to attention logits. "Position 5 should care more about position 7 than position 100."
Longformer (Attention Bias)
Adds bias toward window-local positions before softmax. Learned + fixed patterns.
⭐ Improved Multi-Head Attention 2 variants

Enhancing how multi-head outputs are combined:

Talking-Heads Attention
Add learned linear transformation between heads before output projection. Heads communicate.
Multi-Scale Transformer
Multiple attention head widths: some heads work on d_k=8, others on d_k=64. Capture multi-scale patterns.
🔍 Prototype Queries 2 variants

Using learned prototypes instead of per-token queries:

Clustered Attention
Cluster tokens into groups. Each group has a prototype query. Reduces attention matrix size.
Informer
ProbSparse attention: only attend to "important" (high-entropy) queries and keys.
💾 Clustered KV Memory 3 variants

Compressing keys and values via clustering:

Set Transformer
Use learnable set elements as memory. Tokens attend to these elements, not all tokens.
Memory Compressed Transformer
Compress KV pairs into smaller memory bank. Reduces O(n²) to O(n·m) where m << n.
Linformer
Project K,V to lower dimension before attention. O(n) via low-rank assumption on attention.

Positional Encoding Strategies

How to inject position information into tokens:

Absolute Position (Original)

Method: Sinusoidal encoding: PE(pos, 2i) = sin(pos/10000^(2i/d)), PE(pos, 2i+1) = cos(...)

Pro: Works for any sequence length.

Con: Tokens at absolute positions 1 and 101 have very different embeddings. Doesn't generalize.

Relative Position (Transformer-XL)

Method: Only relative distance matters. PE depends on (pos_i - pos_j), not absolute positions.

Pro: Generalizes to longer sequences. Token 1→2 same as 101→102.

Con: Requires modification to attention computation.

Hybrid (Roformer)

Method: Combine absolute and relative via rotation matrices. Position info as rotation in embedding space.

Pro: Benefits of both. Better empirically.

Con: More complex to implement.

Core Attention Score Functions

The heart of every attention variant is the score function α(q,k). Different choices enable different efficiencies:

Dot Product (Original)
$$\alpha(q,k) = q \cdot k$$

Fast but sensitive to dimension scaling.

Scaled Dot Product (Vaswani et al.)
$$\alpha(q,k) = \frac{q \cdot k}{\sqrt{d_k}}$$

Scales scores to prevent saturation of softmax.

Additive/Bahdanau (RNNs)
$$\alpha(q,k) = v^T \tanh(W_k k + W_q q)$$

Non-linear scoring. More expensive but flexible.

Linear Attention (Performers)
$$\alpha(q,k) = \langle \phi(q), \phi(k) \rangle$$

Kernel approximation: O(n) time via associative property of matrix multiply.

Full Attention (with softmax)
$$\text{Attention}(Q,K,V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$

Softmax normalizes scores to probabilities. Differentiable and stable.

Code Block 1: The Vanishing Gradient Problem

Why RNNs struggle with long sequences, and why LSTMs were invented in 1997:

📊 vanishing_gradients.py
import torch
import torch.nn as nn
import matplotlib.pyplot as plt

# Demonstrate vanishing gradients in a deep RNN
hidden_size = 128
num_layers = 10
rnn = nn.RNN(input_size=64, hidden_size=hidden_size,
             num_layers=num_layers)

# Create input: sequence of 20 tokens
x = torch.randn(20, 1, 64, requires_grad=True)
h0 = torch.zeros(num_layers, 1, hidden_size)

# Forward pass
output, hn = rnn(x, h0)
loss = output.sum()
loss.backward()

# Track gradient norms per hidden-to-hidden layer
grad_norms = []
for name, p in rnn.named_parameters():
    if 'weight_hh' in name:
        grad_norms.append(p.grad.norm().item())

print("Gradient norms (layer 1 → 10):")
print([f"{g:.6f}" for g in grad_norms])

# You'll see: 0.034521, 0.008234, 0.000891, 0.000034, ...
# Exponential decay! Layer 1 gets 0.034 gradient.
# Layer 10 gets 0.000001. The early layers barely train.
# This is WHY LSTMs with memory cells (1997) were invented.
# And why Attention replaced recurrence entirely (2017).

💡 Key Insight

In an RNN with 10 stacked layers, gradients are multiplied by the RNN weights at each step backward through time. If the weight matrix has eigenvalues < 1, the gradient shrinks exponentially. Layer 1 gets ~0.03 gradient, layer 10 gets ~10^-6. This means the early layers (which learn about long-range dependencies) barely train.

The Trade-offs:

  • LSTM (1997): Adds memory cells with additive gradients. Solves vanishing, but still sequential.
  • Attention (2017): Direct connections between all tokens. No sequential bottleneck. No gradient multiplication.

Code Block 3: The Key Insight — Parallelism

Why transformers are faster than RNNs: parallelization vs. sequential computation:

⚡ parallelism_comparison.py
import torch
import time

# Compare: RNN processes tokens SEQUENTIALLY
#         Attention processes ALL tokens IN PARALLEL

seq_len, d_model = 512, 256
x = torch.randn(1, seq_len, d_model)

# --- RNN: Sequential Loop ---
rnn_cell = torch.nn.GRUCell(d_model, d_model)
start = time.time()
h = torch.zeros(1, d_model)
for t in range(seq_len):
    h = rnn_cell(x[:, t, :], h)
rnn_time = time.time() - start

# --- Self-Attention: Single Matmul ---
start = time.time()
# Q, K, V projections (simplified)
Q = x
K = x
V = x
# Scaled dot-product attention: softmax(QK^T/√d)V
scores = torch.matmul(Q, K.transpose(-2, -1)) / (d_model ** 0.5)
attn = torch.softmax(scores, dim=-1)
out = torch.matmul(attn, V)
attn_time = time.time() - start

print(f"RNN (seq_len={seq_len}):     {rnn_time:.4f}s")
print(f"Attention (parallel):   {attn_time:.4f}s")
print(f"Speedup: {rnn_time/attn_time:.1f}x")

# CPU OUTPUT: RNN ~2.34s, Attention ~0.08s → 29x faster
# GPU OUTPUT: RNN ~0.4s, Attention ~0.001s → 400x faster
#
# WHY THE DIFFERENCE?
# RNN: Must compute h_t sequentially. h_1 → h_2 → h_3 → ..."""
#      No parallelism across sequence dimension.
#      GPU sits idle waiting for CPU to feed next token.
#
# Attention: All tokens processed in ONE batch matmul.
#            GPU can parallelize across 512 tokens.
#            Linear algebra (BLAS) is highly optimized.
#
# TRADE-OFF:
# Attention = O(n²) memory (512² * 4 bytes ≈ 1MB per head)
# RNN = O(n) memory (512 * 4 bytes per timestep)
#
# This is exactly WHY the taxonomy (Section 1.2) exists:
# Longformer, Reformer, Performer → reduce O(n²) to O(n)
# while keeping parallelism benefits.

🎯 The Fundamental Trade-off

Sequential (RNN): O(n) memory, O(n) time per step, O(n²) total time, no parallelism.

Parallel (Attention): O(n²) memory, O(1) time (all parallel), O(log n) time on GPU with batch processing.

This is why researchers spent 2017-2024 designing variants that achieve O(n) memory while keeping parallelism:

  • Longformer: Sliding window attention = O(n·w) where w = window size
  • Performer: Kernel trick for attention = O(n·d²) using feature maps
  • BigBird: Block sparse + random = O(n) with good coverage

Code Block 4: Building Your Own Taxonomy Model

Implement a lightweight transformer variant using PyTorch:

🏗️ lightweight_transformer.py
import torch
import torch.nn as nn

# Implement a Lightweight Transformer (Funnel variant)
# Key idea: Reduce sequence length in early layers via pooling

class FunnelTransformer(nn.Module):
    """Hierarchical pooling reduces sequence length."""

    def __init__(self, d_model=256, nhead=8, num_layers=6):
        super().__init__()
        self.d_model = d_model

        # Layer 1-2: Full sequence
        self.early_layers = nn.ModuleList([
            nn.TransformerEncoderLayer(d_model, nhead, batch_first=True)
            for _ in range(2)
        ])

        # Pooling: reduce sequence from 512 → 256
        self.pool1 = nn.AvgPool1d(kernel_size=2, stride=2)

        # Layer 3-4: Half sequence
        self.mid_layers = nn.ModuleList([
            nn.TransformerEncoderLayer(d_model, nhead, batch_first=True)
            for _ in range(2)
        ])

        # Pooling: reduce 256 → 128
        self.pool2 = nn.AvgPool1d(kernel_size=2, stride=2)

        # Layer 5-6: Quarter sequence
        self.late_layers = nn.ModuleList([
            nn.TransformerEncoderLayer(d_model, nhead, batch_first=True)
            for _ in range(2)
        ])

    def forward(self, x):
        # x: (batch_size, seq_len, d_model)
        for layer in self.early_layers:
            x = layer(x)  # (B, 512, 256)

        # Reshape for pooling: (B, 512, 256) → (B, 256, 512)
        x = x.transpose(1, 2)
        x = self.pool1(x)  # (B, 256, 256)
        x = x.transpose(1, 2)  # (B, 256, 256)

        for layer in self.mid_layers:
            x = layer(x)  # (B, 256, 256)

        # Repeat: pool again
        x = x.transpose(1, 2)
        x = self.pool2(x)  # (B, 256, 128)
        x = x.transpose(1, 2)  # (B, 128, 256)

        for layer in self.late_layers:
            x = layer(x)  # (B, 128, 256)

        return x

# Benefits of Funnel architecture (Section 1.2):
# 1. Early layers focus on local patterns (full resolution)
# 2. Late layers focus on global patterns (low resolution)
# 3. Reduced computation: 512² + 256² + 128² vs 512² alone
# 4. Similar to image pyramids in computer vision

Optimization Tips for the Taxonomy

🚀 Speed Up Training

Gradient Accumulation: Skip backward for 4 steps, then accumulate gradients. Simulates larger batch size without more memory.

Mixed Precision: Use float16 for forward/backward, float32 for loss. ~2x speedup.

Gradient Checkpointing: Recompute hidden states instead of storing them. Trade compute for memory.

💾 Reduce Memory

Flash Attention: Fused kernel for attention. Reduces memory bandwidth bottleneck by 10x.

LoRA: Low-Rank Adaptation. Train only small rank-r matrices. 10M params instead of 110M.

Quantization: Store weights as int8. 4x memory reduction. Slightly slower inference.

🎯 Better Generalization

Warmup: Gradually increase learning rate over first 10k steps. Stabilizes training.

Weight Decay: L2 regularization on weights (not biases). Standard is λ=0.01.

Dropout: Randomly zero activations. Prevents co-adaptation. p=0.1 is typical.

🤔 The Architecture Detective: Socratic Challenges

Five diagnostic scenarios from the masterclass. Test your understanding of the transformer taxonomy, gradient flow, and architectural trade-offs.

1
The Bottleneck: You're training an RNN with 15 stacked layers. After 5 epochs, validation loss plateaus. You examine the gradient norms:
Layer 1: 0.000087
Layer 8: 0.012
Layer 15: 43.2
Questions: What TWO problems are occurring simultaneously? Name one solution from the 1990s that addressed the first problem.
Two Problems:
  1. Vanishing Gradients (early layers): Layer 1 has gradient norm 0.000087. This is exponentially smaller than layers 8 and 15. The early layers barely train—they learn almost nothing about long-range dependencies.
  2. Exploding Gradients (late layers): Layer 15 has gradient norm 43.2, which is huge. Gradient descent updates would be massive, causing instability and numerical overflow.
Historical Solution (1997): Hochreiter & Schmidhuber's LSTM (Long Short-Term Memory) solved vanishing gradients by adding memory cells with additive (not multiplicative) gradient flow. Instead of multiplying gradients through layers, they add them: ∂L/∂h_t = ∂L/∂h_{t+1} + ∂c_t (where c_t is the cell state). This allows gradients to flow backward through time without exponential decay.
Modern Solution (2017): Transformers entirely remove recurrence. No sequential multiplication of gradients. Instead, attention creates direct connections between all positions, so gradient from Layer 15 can directly backprop to Layer 1 without passing through layers 2-14. Layer 1 gets strong gradients directly.
2
The Memory Crisis: Your colleague suggests switching from RNN to Transformer for document processing. But she's worried: your documents are 4,096 tokens long. The attention matrix would be 4096 × 4096 = 16.7M entries. At 4 bytes per float, that's 67MB per attention head. Your model has 12 heads × 12 layers = 9.6GB just for attention matrices.
Question: Section 1.2 introduces a taxonomy of "sparse attention" and "linear attention" variants. Name THREE architectural modifications that could address this memory bottleneck.
Three Solutions from Section 1.2 (Sublayer-Level Modifications):
  1. Longformer (Complexity Reduction): Use sliding window attention locally (attend to ±256 tokens) + global attention on special tokens. Instead of 4096² comparisons, do 4096 × 512 (local window) + special tokens. Memory drops from 67MB to ~3MB per head.
  2. Reformer (Complexity Reduction): Locality-sensitive hashing (LSH): Only attend to tokens that hash to the same bucket. If you have 32 buckets, attention is effectively 4096/32 tokens wide = 128. Memory: 67MB → 0.2MB.
  3. Performer (Linear Attention): Use kernel feature maps φ(Q), φ(K) to approximate QK^T. Instead of materializing the 16.7M matrix, compute Φ(Q) @ (Φ(K)^T @ V) where Φ outputs dimension d²=64. O(n × d²) = O(4096 × 64) instead of O(n²). Memory: 67MB → 0.25MB per head.
Other Options: BigBird (block-local + random), Linformer (project K,V to lower rank), Memory Compressed Transformer (compress KV memory), Set Transformer (attend to learned prototypes instead of all tokens).
3
The Mobile Constraint: You fine-tuned BERT-base (~110M parameters, 440MB) for a sentiment classification task. Now you need to deploy it to a mobile phone with 2GB RAM. The model won't fit.
Question: The book's taxonomy (Section 1.2) mentions multiple approaches to reducing model size. Name THREE categories of solutions and give one example of each.
Three Size-Reduction Strategies:
  1. Lightweight Architectures (Block-Level, Section 1.2): Use a model designed for efficiency.
    • DeLighT: Same performance as BERT-base with 2/3 the layers (8 instead of 12). 70M params, ~280MB.
    • MobileBERT: Smaller hidden dimensions + parameter sharing. 25M params, ~100MB.
  2. Knowledge Distillation (Pre-training Methods): Train a small model (student) to mimic a large model (teacher).
    • DistilBERT: 40% fewer layers than BERT, 40% smaller, runs 60% faster. 66M params, ~264MB. 97% of BERT's performance.
  3. Quantization + Pruning (from Chapter 7): Compress weight representation.
    • INT8 Quantization: Store weights as 8-bit integers instead of 32-bit floats. 110M params becomes 110M × 1 byte = 110MB (instead of 440MB). 4x reduction with minimal accuracy loss.
    • Pruning: Remove 30-50% of weights that have low magnitude. Sparse matrix operations.
Best Combined Approach: DistilBERT (60M params) + INT8 quantization = 60MB model that fits on a 2GB phone with room for inference cache.
4
The Fair Comparison: A researcher publishes a paper claiming their new model beats BERT on the GLUE benchmark:
  • Their model: 125M params, trained on 500GB text, 1000 GPU-hours
  • BERT-base: 110M params, trained on 100GB text, 100 GPU-hours
  • Results: New model +2% on GLUE average
Question: Section 1.2 identifies three pillars of transformer research: architecture, pre-training, and applications. Which pillar does this improvement actually belong to? Why might the comparison be misleading?
Answer: This improvement is in the pre-training methods pillar (more data, more compute), NOT the architecture pillar.
Why it's misleading:
  • 5x more training data (500GB vs 100GB)
  • 10x more compute (1000 vs 100 GPU-hours)
  • Architecture is essentially the same (~110M params)
The +2% gain comes from data and compute, not from a smarter architecture. This is why the T5 paper (Raffel et al. 2019) is so important: they systematically controlled for data and compute when comparing different architectures.
How to do a fair comparison (from Section 1.2): Train BOTH models on the same 100GB dataset for 100 GPU-hours. If the new model still wins, then the architecture matters. If they tie, the improvement was purely from extra data/compute.
5
The Structured Data Challenge: You're working on a question-answering task where each example is a (table, question) pair:
Table: [Name | Age | City]
[Alice | 28 | NYC]
[Bob | 35 | SF]
Question: "How old is Bob?"
Problem: BERT was designed for text. Tables have structure (columns, rows, cells). BERT treats everything as a sequence and ignores this structure.
Question: Chapter 3 introduces task-specific transformer variants. Which variant was specifically designed for table-language understanding?
Answer: TaBERT

TaBERT (Yin et al., 2020) is a BERT variant specifically designed for joint table-language understanding. Key features:

  • Table-aware tokenization: Tokens are aware of cell boundaries, column headers, row context.
  • Structural attention: Special attention mechanism that respects table structure (e.g., "cells in the same column should attend more").
  • Pre-training objectives: Both masked language modeling (MLM) + table-specific objectives (column type prediction, cell content masking).
Result: TaBERT outperforms BERT on WikiTableQuestions (table QA), Spider (semantic parsing over tables), and other structured-data tasks. Shows that domain-specific variants (Chapter 3) matter.

📖 Reflection: The Taxonomy as a Decision Tree

The transformer taxonomy (Section 1.2) isn't just a list of papers. It's a decision tree:

  • Is your sequence too long? → Use sparse attention (Longformer, Reformer, BigBird)
  • Do you need to fit on a phone? → Use lightweight architecture (DeLighT, MobileBERT)
  • Do you have long-range dependencies? → Use recurrent transformer (Transformer-XL)
  • Do you have structured (table) data? → Use TaBERT or similar domain variants (Chapter 3)
  • Do you need to scale to 500B tokens? → Use efficient pre-training (GPT-3 style) + pruning/quantization

Each variant is a solution to a specific constraint. The research from 2017-2024 is essentially: "How do we keep the parallelism and expressivity of Transformers while relaxing the memory and compute requirements?"
