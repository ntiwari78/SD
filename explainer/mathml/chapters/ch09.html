Functions

Domain, image, operations on functions, callable objects in Python, and the function base class pattern

Part II — Calculus ~35 min read
Mental Model

Functions Are Machines

A function is a machine with an input slot and an output slot. You feed it a value, it does something deterministic, and it hands back a result. The same input always produces the same output — no randomness, no side effects. In ML, everything is a function: a neural network is a function from inputs to predictions, a loss function maps predictions and labels to a scalar, and an optimizer is a function from current parameters and gradients to updated parameters. Getting the mathematical concept of "function" razor-sharp is what makes the rest of calculus precise.

Section 9.1

What Is a Function?

Formal Definition

Informally, a function is a rule that assigns to each input exactly one output. The formal definition makes this precise using set theory.

Definition 9.1 — Function
A Map Between Sets

A function (or map) \(f : X \to Y\) from a set \(X\) to a set \(Y\) is a rule that assigns to each element \(x \in X\) a unique element \(f(x) \in Y\). The key requirement is well-definedness: every input has exactly one output.

Formally, \(f\) is a subset of \(X \times Y\) (a set of ordered pairs) such that for every \(x \in X\), there is exactly one \(y \in Y\) with \((x, y) \in f\). We write \(y = f(x)\).

The notation \(f : X \to Y\) specifies three things: the name of the function (\(f\)), what it accepts (\(X\)), and where the output lives (\(Y\)). Sometimes we write \(x \mapsto f(x)\) to describe the assignment rule explicitly — for example, \(x \mapsto x^2\) is the squaring function.

Domain, Codomain, and Image

Definition 9.2 — Domain, Codomain, Image
The Three Parts of a Function

For a function \(f : X \to Y\):

  • The domain is \(X\) — the set of all valid inputs.
  • The codomain is \(Y\) — the set where outputs are declared to live.
  • The image (or range) is \(\text{im}(f) = \{f(x) : x \in X\} \subseteq Y\) — the set of outputs actually produced.

The image is always a subset of the codomain, but may be strictly smaller. For instance, \(f : \mathbb{R} \to \mathbb{R}\) defined by \(f(x) = x^2\) has codomain \(\mathbb{R}\) but image \([0, \infty)\). 

Python
import numpy as np

# A simple function: f(x) = x²
def f(x):
    return x ** 2

# Domain: all reals (we sample a finite subset)
domain = np.linspace(-3, 3, 1000)

# Image: the set of actual outputs
image = f(domain)

print(f"Domain range: [{domain.min():.1f}, {domain.max():.1f}]")
print(f"Codomain: all of ℝ (declared)")
print(f"Image range: [{image.min():.4f}, {image.max():.4f}]")
print(f"Image ⊂ Codomain? Yes — image is [0, 9], a subset of ℝ")
print()

# Multivariate function: f(x, y) = x² + y²
def g(x, y):
    return x**2 + y**2

# Domain is ℝ², image is [0, ∞)
x_vals = np.random.randn(10000)
y_vals = np.random.randn(10000)
outputs = g(x_vals, y_vals)
print(f"g: ℝ² → ℝ, g(x,y) = x² + y²")
print(f"Min output observed: {outputs.min():.6f}")
print(f"All outputs ≥ 0? {np.all(outputs >= 0)}")
Domain Matters

In ML, domain restrictions are surprisingly practical. The logarithm function \(\log : (0, \infty) \to \mathbb{R}\) is undefined for non-positive inputs. The softmax function requires its inputs to be finite. Cross-entropy loss requires probabilities in \((0, 1)\). Getting the domain wrong causes NaN values, numerical instabilities, and silent bugs. Always know the domain of your functions.

Section 9.2

Injectivity, Surjectivity, and Bijectivity

Three fundamental properties describe how a function relates its domain to its codomain.

Definition 9.3 — Injective, Surjective, Bijective
Function Classification

A function \(f : X \to Y\) is:

  • Injective (one-to-one) if \(f(x_1) = f(x_2)\) implies \(x_1 = x_2\). Different inputs always produce different outputs.
  • Surjective (onto) if for every \(y \in Y\), there exists some \(x \in X\) with \(f(x) = y\). Every element of the codomain is "hit" by at least one input. Equivalently, \(\text{im}(f) = Y\).
  • Bijective if it is both injective and surjective. Every output is produced by exactly one input. A bijection has a well-defined inverse.
Python
import numpy as np

# Injective but not surjective: f(x) = eˣ, f: ℝ → ℝ
# Image is (0, ∞), not all of ℝ — so not surjective
# But f(x₁) = f(x₂) ⟹ x₁ = x₂ — injective

x = np.array([-1, 0, 1, 2, 3])
print("f(x) = eˣ — injective, not surjective onto ℝ")
print(f"Inputs:  {x}")
print(f"Outputs: {np.round(np.exp(x), 4)}")
print(f"All outputs positive? {np.all(np.exp(x) > 0)}")
print()

# Surjective but not injective: f(x) = x², f: ℝ → [0,∞)
# Both x=2 and x=-2 map to 4 — not injective
# But every y ≥ 0 has a preimage — surjective onto [0,∞)
print("f(x) = x² on ℝ → [0,∞) — surjective, not injective")
print(f"f(2) = {2**2}, f(-2) = {(-2)**2} — same output, different inputs")
print()

# Bijective: f(x) = 2x + 1, f: ℝ → ℝ
# Inverse: f⁻¹(y) = (y - 1) / 2
print("f(x) = 2x + 1 — bijective")
f = lambda x: 2*x + 1
f_inv = lambda y: (y - 1) / 2
test = np.array([-2, 0, 3, 7])
print(f"f(test):       {f(test)}")
print(f"f⁻¹(f(test)):  {f_inv(f(test))}")
print(f"Round-trip?     {np.allclose(f_inv(f(test)), test)}")

In ML, bijectivity is crucial for normalizing flows — a class of generative models that require invertible transformations to compute exact likelihoods. Each layer of a normalizing flow must be a bijection so the change-of-variables formula applies.

Section 9.3

Operations on Functions

Composition

Definition 9.4 — Function Composition
Chaining Functions

Given \(f : X \to Y\) and \(g : Y \to Z\), the composition \(g \circ f : X \to Z\) is defined by

$$(g \circ f)(x) = g(f(x)).$$

Note the right-to-left order: \(f\) is applied first, then \(g\). The composition is well-defined because the codomain of \(f\) matches the domain of \(g\).

Proposition 9.1 — Composition Properties

Function composition satisfies:

  • Associativity: \(h \circ (g \circ f) = (h \circ g) \circ f\).
  • Not commutative: in general, \(g \circ f \neq f \circ g\).
  • Identity: \(f \circ \text{id}_X = f = \text{id}_Y \circ f\) where \(\text{id}_X(x) = x\).

A neural network is precisely a composition of functions. A 3-layer network computes \(\hat{y} = f_3 \circ f_2 \circ f_1(\mathbf{x})\), where each \(f_i\) is an affine transformation followed by a non-linear activation. The chain rule of calculus — the foundation of backpropagation — tells us how to differentiate compositions. 

Python
import numpy as np

# Composition: a mini neural network layer
def affine(W, b):
    """Returns an affine function x ↦ Wx + b."""
    return lambda x: W @ x + b

def relu(x):
    """ReLU activation."""
    return np.maximum(x, 0)

# Two-layer network: f₂ ∘ relu ∘ f₁
np.random.seed(42)
W1 = np.random.randn(4, 3) * 0.5
b1 = np.zeros(4)
W2 = np.random.randn(2, 4) * 0.5
b2 = np.zeros(2)

layer1 = affine(W1, b1)
layer2 = affine(W2, b2)

# Forward pass = composition
x = np.array([1.0, -0.5, 0.3])
h = relu(layer1(x))       # hidden layer
y = layer2(h)              # output

print(f"Input:  {x}")
print(f"Hidden: {np.round(h, 4)}")
print(f"Output: {np.round(y, 4)}")
print()

# Composition is associative
network = lambda x: layer2(relu(layer1(x)))
print(f"Via composition: {np.round(network(x), 4)}")
print(f"Match? {np.allclose(y, network(x))}")
print()

# Composition is NOT commutative
f = lambda x: x ** 2
g = lambda x: x + 1
print(f"f(x) = x², g(x) = x + 1")
print(f"(g ∘ f)(3) = g(f(3)) = g(9) = {g(f(3))}")
print(f"(f ∘ g)(3) = f(g(3)) = f(4) = {f(g(3))}")
print(f"g ∘ f ≠ f ∘ g: {g(f(3)) != f(g(3))}")

Inverse Functions

Definition 9.5 — Inverse Function
Undoing a Function

If \(f : X \to Y\) is a bijection, its inverse \(f^{-1} : Y \to X\) satisfies

$$f^{-1}(f(x)) = x \quad \text{for all } x \in X, \qquad f(f^{-1}(y)) = y \quad \text{for all } y \in Y.$$

Equivalently, \(f^{-1} \circ f = \text{id}_X\) and \(f \circ f^{-1} = \text{id}_Y\).

Common function–inverse pairs in ML: \(\exp\) and \(\log\); the logit \(\sigma^{-1}(p) = \log\frac{p}{1-p}\) and the sigmoid \(\sigma(x) = \frac{1}{1+e^{-x}}\); affine transformations \(\mathbf{x} \mapsto A\mathbf{x} + \mathbf{b}\) and their inverses \(\mathbf{y} \mapsto A^{-1}(\mathbf{y} - \mathbf{b})\) (when \(A\) is invertible). 

Python
import numpy as np

# Sigmoid and logit are inverses
def sigmoid(x):
    return 1 / (1 + np.exp(-x))

def logit(p):
    return np.log(p / (1 - p))

# Test round-trip
x = np.array([-2, -1, 0, 1, 2], dtype=float)
print("sigmoid ∘ logit = id on (0, 1):")
p = sigmoid(x)
print(f"  x → sigmoid(x): {np.round(p, 6)}")
print(f"  → logit back:    {np.round(logit(p), 6)}")
print(f"  Round-trip match: {np.allclose(x, logit(sigmoid(x)))}")
print()

# exp and log are inverses on their valid domains
print("exp ∘ log = id on (0, ∞):")
vals = np.array([0.1, 1, 2.7, 10, 100])
print(f"  exp(log(vals)): {np.round(np.exp(np.log(vals)), 10)}")
print(f"  Match: {np.allclose(vals, np.exp(np.log(vals)))}")
Section 9.4

Functions as Python Objects

Callable Objects

In Python, a function is any object that can be "called" with parentheses. This includes plain def functions, lambdas, and — crucially for ML — any object with a __call__ method. PyTorch's nn.Module uses this pattern: a neural network is an object you call like a function. 

Python
# Three ways to define a function in Python

# 1. Named function
def square(x):
    return x ** 2

# 2. Lambda (anonymous function)
cube = lambda x: x ** 3

# 3. Callable object — the ML way
class Polynomial:
    """A callable polynomial object."""
    def __init__(self, coefficients):
        self.coefficients = coefficients

    def __call__(self, x):
        """Evaluate the polynomial at x."""
        result = 0
        for i, c in enumerate(self.coefficients):
            result += c * x ** i
        return result

    def __repr__(self):
        terms = []
        for i, c in enumerate(self.coefficients):
            if c != 0:
                if i == 0:
                    terms.append(f"{c}")
                elif i == 1:
                    terms.append(f"{c}x")
                else:
                    terms.append(f"{c}x^{i}")
        return " + ".join(terms) if terms else "0"

# Usage: all three are "functions" in Python
p = Polynomial([1, -3, 2])  # 1 - 3x + 2x²
print(f"Polynomial: {p}")
print(f"p(0) = {p(0)}, p(1) = {p(1)}, p(2) = {p(2)}")
print(f"square(5) = {square(5)}")
print(f"cube(3) = {cube(3)}")
print()

# All are callable
print(f"callable(square): {callable(square)}")
print(f"callable(cube):   {callable(cube)}")
print(f"callable(p):      {callable(p)}")

The Function Base Class Pattern

In ML frameworks, functions (models, losses, activations) are typically organized as subclasses of a base class that enforces a consistent interface. This is the pattern behind PyTorch's nn.Module and JAX's flax.nn.Module. 

Python
import numpy as np
from abc import ABC, abstractmethod

class MathFunction(ABC):
    """Base class for mathematical functions with common utilities."""

    @abstractmethod
    def forward(self, x):
        """Evaluate the function at x."""
        pass

    def __call__(self, x):
        """Make the function callable."""
        x = np.asarray(x, dtype=float)
        return self.forward(x)

    def tabulate(self, domain, n_points=10):
        """Print a table of input-output pairs."""
        xs = np.linspace(domain[0], domain[1], n_points)
        print(f"{'x':>10s} | {'f(x)':>12s}")
        print("-" * 25)
        for x in xs:
            print(f"{x:10.4f} | {self(x):12.6f}")

    def compose(self, other):
        """Return the composition self ∘ other."""
        parent = self
        class Composed(MathFunction):
            def forward(self, x):
                return parent.forward(other.forward(x))
        return Composed()

# Concrete implementations
class Sigmoid(MathFunction):
    def forward(self, x):
        return 1 / (1 + np.exp(-x))

class ReLU(MathFunction):
    def forward(self, x):
        return np.maximum(x, 0)

class Affine(MathFunction):
    def __init__(self, weight, bias):
        self.weight = weight
        self.bias = bias

    def forward(self, x):
        return self.weight * x + self.bias

# Usage
sigmoid = Sigmoid()
relu = ReLU()
scale = Affine(2.0, -1.0)  # f(x) = 2x - 1

print("Sigmoid table:")
sigmoid.tabulate((-4, 4), 9)

print("\nComposition: sigmoid ∘ affine")
composed = sigmoid.compose(scale)
composed.tabulate((-2, 3), 6)
Performance Deep-Dive

Why Callable Objects Beat Plain Functions

In ML, functions need to carry state: a linear layer stores weights and biases; a batch-norm layer stores running statistics; an optimizer stores momentum buffers. Plain Python functions can't do this cleanly. Callable objects bundle the state (learnable parameters) with the computation (forward pass) in one object. This is why PyTorch uses nn.Module (a callable with parameter tracking) rather than plain functions. 

Python
import numpy as np

class LinearLayer:
    """A simple linear layer: f(x) = Wx + b."""
    def __init__(self, in_features, out_features, seed=42):
        rng = np.random.default_rng(seed)
        # Xavier initialization
        scale = np.sqrt(2.0 / (in_features + out_features))
        self.weight = rng.normal(0, scale, (out_features, in_features))
        self.bias = np.zeros(out_features)

    def __call__(self, x):
        return self.weight @ x + self.bias

    def parameters(self):
        return {'weight': self.weight, 'bias': self.bias}

# The layer is a function with learnable state
layer = LinearLayer(3, 2)
x = np.array([1.0, -0.5, 0.3])
print(f"Input:      {x}")
print(f"Output:     {np.round(layer(x), 4)}")
print(f"Parameters: {layer.weight.shape[0] * layer.weight.shape[1] + layer.bias.shape[0]}")
print(f"Weight:\n{np.round(layer.weight, 4)}")
print(f"Bias: {np.round(layer.bias, 4)}")
Section 9.5

Important Function Classes for ML

Certain types of functions appear so frequently in machine learning that they deserve special mention. These form the vocabulary you'll use throughout the rest of the curriculum.

Definition 9.6 — Linear and Affine Functions
The Building Blocks of Neural Networks

A function \(f : \mathbb{R}^n \to \mathbb{R}^m\) is linear if \(f(\alpha \mathbf{x} + \beta \mathbf{y}) = \alpha f(\mathbf{x}) + \beta f(\mathbf{y})\) for all scalars \(\alpha, \beta\) and all \(\mathbf{x}, \mathbf{y}\). Every linear function is represented by a matrix: \(f(\mathbf{x}) = A\mathbf{x}\).

An affine function has the form \(f(\mathbf{x}) = A\mathbf{x} + \mathbf{b}\) — linear plus a translation. Neural network layers are affine (not linear), though the terminology is often blurred.

Proposition 9.2 — Key Function Types in ML

Functions commonly encountered in machine learning include:

  • Convex functions: satisfy \(f(\alpha x + (1-\alpha)y) \leq \alpha f(x) + (1-\alpha)f(y)\). Convex loss functions have no local minima — every critical point is a global minimum.
  • Monotonic functions: preserve (or reverse) order. Activation functions like sigmoid and tanh are monotonically increasing.
  • Lipschitz functions: satisfy \(|f(x) - f(y)| \leq L|x - y|\) for some constant \(L\). These have bounded "steepness" — important for training stability in GANs and reinforcement learning.
  • Piecewise linear functions: composed of linear segments. ReLU networks compute piecewise linear functions, which is why they can approximate any continuous function given enough neurons.
Python
import numpy as np

# Common ML activation functions and their properties
x = np.linspace(-5, 5, 1000)

activations = {
    'ReLU':    (lambda x: np.maximum(x, 0),
                "Piecewise linear, not bijective, not bounded"),
    'Sigmoid': (lambda x: 1 / (1 + np.exp(-x)),
                "Monotonic, bijective (0,1)→ℝ⁻¹, bounded"),
    'Tanh':    (lambda x: np.tanh(x),
                "Monotonic, bijective (-1,1)→ℝ⁻¹, bounded"),
    'Softplus':(lambda x: np.log1p(np.exp(x)),
                "Monotonic, smooth approximation of ReLU"),
}

for name, (fn, props) in activations.items():
    y = fn(x)
    print(f"{name:10s} | range: [{y.min():.4f}, {y.max():.4f}] | {props}")

# Verify Lipschitz property of sigmoid
# Derivative of sigmoid is σ(x)(1-σ(x)) ≤ 1/4
# So sigmoid is Lipschitz with L = 1/4
sigmoid = lambda x: 1 / (1 + np.exp(-x))
x1, x2 = np.random.randn(10000), np.random.randn(10000)
ratios = np.abs(sigmoid(x1) - sigmoid(x2)) / (np.abs(x1 - x2) + 1e-10)
print(f"\nSigmoid Lipschitz constant (empirical max): {ratios.max():.4f}")
print(f"Theoretical Lipschitz constant: 0.25")
Section 9.6

Summary

This chapter established the mathematical and computational foundations of functions — the atomic unit of calculus and machine learning. We defined functions formally via set theory (domain, codomain, image), classified them by their structural properties (injective, surjective, bijective), showed how composition chains simple functions into complex ones (the essence of deep learning), and bridged the gap to Python with callable objects and the base-class pattern used by every major ML framework.

Why This Matters for AI

Everything in ML Is a Function

  • Neural networks — A composition of affine functions and non-linear activations. The universal approximation theorem says sufficiently wide networks can approximate any continuous function.
  • Loss functions — Maps from (prediction, label) pairs to a scalar measuring error. Properties like convexity and smoothness determine whether optimization will converge.
  • Normalizing flows — Bijective functions with tractable Jacobians, enabling exact likelihood computation for generative modelling.
  • Activation functions — Non-linear functions applied element-wise. Their properties (monotonicity, boundedness, Lipschitz constant) affect gradient flow and training dynamics.
  • Feature engineering — Domain transformations (\(\log\), polynomial, interaction terms) that map raw inputs into representations where linear models work well.

Socratic Knowledge Check

You are designing a deep learning framework from scratch. You need to define the core abstractions for layers, activations, and loss functions. Think through these questions about the mathematical properties your framework needs to enforce.
1
Your framework's Layer base class has a forward(x) method. A user creates a layer where forward(x) returns different values when called twice with the same input (due to dropout). Is this a "function" in the mathematical sense? What implications does this have?
No — a mathematical function must be deterministic: the same input always produces the same output. Dropout violates this during training (it randomly zeroes neurons), making the forward pass a random function. This is why frameworks distinguish between training mode (model.train() — stochastic) and evaluation mode (model.eval() — deterministic). At evaluation time, dropout is disabled and the layer becomes a proper function. The mathematical machinery of calculus (derivatives, chain rule) applies to the expected value of the stochastic forward pass, which is what batch training approximates.
2
You want to implement a normalizing flow layer. What mathematical property must the layer's forward method satisfy, and what additional method would you add to the base class?
The layer must be a bijection — injective and surjective — so that it has a well-defined inverse. You'd add an inverse(y) method to the base class that satisfies inverse(forward(x)) == x for all valid x. You'd also need a log_det_jacobian(x) method, because the change-of-variables formula for probability densities requires the absolute determinant of the Jacobian: \(p_Y(y) = p_X(f^{-1}(y)) \cdot |\det J_{f^{-1}}(y)|\). Normalizing flow architectures (RealNVP, GLOW, Neural Spline Flows) are specifically designed so that the Jacobian is triangular, making the determinant cheap to compute.
3
Your cross-entropy loss function takes predicted probabilities in \((0, 1)\) as input. A user accidentally passes 0.0 as a prediction. What goes wrong, and how does the concept of "domain" help you design a safer API?
Cross-entropy computes \(-\log(p)\), and \(\log(0) = -\infty\). The domain of \(\log\) is \((0, \infty)\), so 0 is outside the domain. The result is a NaN or inf that poisons all subsequent gradients. To design a safer API: (1) explicitly document the domain as \((0, 1)\), (2) add input validation that clips or rejects out-of-domain values, (3) use numerically stable formulations like log_softmax that avoid ever computing \(\log(0)\), or (4) add a small epsilon: \(\log(p + \varepsilon)\). PyTorch's F.cross_entropy takes raw logits precisely to avoid this domain issue — the softmax and log are fused for stability.
4
A 10-layer network is a composition \(f_{10} \circ f_9 \circ \cdots \circ f_1\). If each layer's output dimension is 256, how many distinct compositions exist with the same architecture? Does the order of composition matter?
Each layer \(f_i\) is parameterised by its own weight matrix and bias vector, so different parameter values produce different functions even if the architecture is the same. The number of distinct compositions is infinite (continuous parameter space). The order absolutely matters: \(f_2 \circ f_1 \neq f_1 \circ f_2\) in general. Even swapping two layers with identical dimensions changes the network's function. However, composition is associative: \((f_3 \circ f_2) \circ f_1 = f_3 \circ (f_2 \circ f_1)\), which is why we can compute the forward pass layer-by-layer without worrying about grouping.
5
ReLU is not bijective (it maps all negative values to 0). Why doesn't this prevent neural networks from being universal approximators? What would go wrong if you used a bijective activation everywhere instead?
The universal approximation theorem requires each neuron to introduce a non-linearity, but doesn't require each individual activation to be bijective. ReLU's non-injectivity (many-to-one mapping of negatives to 0) is actually a feature: it creates sparse activations (many neurons output exactly 0), which improves efficiency and acts as implicit regularization. The network as a whole doesn't need to be bijective to approximate arbitrary functions — it just needs enough neurons. If you used a bijective activation (like tanh or sigmoid) everywhere, the network would still be a universal approximator, but you'd face vanishing gradients in deep networks because these functions saturate (derivatives → 0 for large inputs), making training much harder.
