Chapter 8: Sequential Games I

🌳 Thinking Backwards: The Art of Looking Ahead by Working Back

The Road Trip Analogy: Chess masters don't think "what should I do first?" — they think "what's the endgame I want, and how do I get there?" Imagine you're planning a road trip: you start by deciding your destination, then figure out the last highway exit, then the route to that exit, all the way back to your driveway. This is backward induction. In sequential games, the same principle applies: work backwards from the end to find the optimal strategy.

Key Concepts

1. Sequential vs. Simultaneous Games: Unlike Chapters 2-7 where players move at the same time, now players take turns. You can see what your opponent did before you respond. This is the defining feature of sequential games.
2. Game Trees: Like a choose-your-own-adventure book — each choice branches into new possibilities, ending at outcomes with payoffs. The root is the starting position, branches are moves, leaves are payoffs.
3. Game Digraphs: When different paths lead to the same situation, we can merge them (like a GPS that only cares about your current location, not how you got there). For games where history doesn't matter (like NIM), the digraph is more efficient than the tree.
4. Backward Induction: "Life can only be understood backwards" (Kierkegaard). Start at the end, figure out what the last player would do, then what the second-to-last player would do knowing that, and so on. This is the algorithm for solving sequential games.
5. The Centipede Paradox: Backward induction says take the money immediately, but real humans cooperate for a while. This reveals the gap between theory and reality — rationality assumptions, altruism, and signaling matter.

In this chapter, we'll see that sequential games have a special structure: perfect information (all past moves are known) and no randomness allow us to completely solve the game with backward induction. The solution tells us what rational players should do at every decision point.

📐 Extensive Form Games and Backward Induction

Definitions and Core Concepts

Sequential Game: Players move one after another. A position is a situation where a player must make a move (choose from available options). End positions (or terminal nodes) have payoffs; all others have a player who must move next.

Game Tree: A directed tree with:

  • Root = start position
  • Internal nodes = decision points (labeled with the moving player)
  • Edges = moves (labeled with move names)
  • Leaves = end positions with payoff vectors

Game Digraph: Like a game tree but positions with identical futures are merged. Formally a DAG (directed acyclic graph) for finite games. This is more compact when history is irrelevant to future possibilities.

Finite Game: Has an acyclic game digraph. No cycles means no infinite loops — the game must eventually reach a terminal position.

The Backward Induction Algorithm

  1. Find a vertex V with no values yet, but all successors have values assigned.
  2. Identify the player X who moves at V. Scan all successor nodes and find the one W where X's payoff is highest.
  3. Copy W's payoff vector to V. Record that at V, player X should move toward W.
  4. Repeat until the start vertex has values and an optimal first move is identified.

Result: Every player has an optimal strategy at every position, and the game has a unique backward induction (BI) solution — the payoffs rational players should expect.

Zermelo's Theorem (1913)

Zermelo's Theorem: Every finite perfect-information sequential game without random moves can be analyzed using backward induction. The resulting payoffs are what rational players should expect if both players are rational and this is common knowledge.

This is a cornerstone of game theory. It says: for these special games, the solution is completely determined by backward induction. No uncertainty, no strategic subtlety — just logic.

Security Level Analysis

Instead of assuming rational opponents, assume hostile opponents:

  • At your own moves: choose the successor that maximizes your payoff
  • At opponent moves: assume they choose to minimize your payoff

This gives guaranteed payoffs (security levels) for each player — the worst you can guarantee yourself even if opponents play perfectly against you.

In 2-player zero-sum games: Backward induction = security level analysis. Because minimizing your payoff IS maximizing theirs. In non-zero-sum games, they differ because opponents have other priorities.

Example Games

We'll analyze these classic sequential games:

  • NIM(6): Remove 1, 2, or 3 stones. Last player to move wins. Game digraph version.
  • Senate Race II: Senator chooses leftist/rightist platform, then Challenger responds. Different voter preferences.
  • Legislators' Vote: 3-player sequential voting game (A→B→C vote raise/no raise). Coalitions matter.
  • Centipede(6): 6-round pass/take game with growing stakes. Backward induction vs. cooperation.
  • Ultimatum(5): Ann proposes a split of $5, Beth accepts or rejects. Monetary payoffs vs. fairness.

Python Implementation Sketch

class GameTree:
    def __init__(self):
        self.nodes = {}  # id -> {player, children: [{move, target}], payoffs}

    def add_node(self, id, player=None, payoffs=None):
        """Add a node. player=None means terminal node with payoffs."""
        self.nodes[id] = {
            'player': player,
            'children': [],
            'payoffs': payoffs,
            'bi_payoffs': None,
            'bi_move': None
        }

    def add_edge(self, from_id, to_id, move_name):
        """Add a directed edge (move) from from_id to to_id."""
        self.nodes[from_id]['children'].append({
            'move': move_name,
            'target': to_id
        })

    def backward_induction(self, node_id=None):
        """Compute BI payoffs and optimal moves recursively."""
        if node_id is None:
            node_id = 'root'

        node = self.nodes[node_id]

        # Terminal node: return payoffs
        if node['payoffs'] is not None:
            return node['payoffs'], None

        # Non-terminal: compute BI for all children
        player = node['player']
        best_payoffs = None
        best_move = None

        for child in node['children']:
            child_payoffs, _ = self.backward_induction(child['target'])

            # Does this child give player higher payoff?
            if best_payoffs is None or \
               child_payoffs[player] > best_payoffs[player]:
                best_payoffs = child_payoffs
                best_move = child['move']

        node['bi_payoffs'] = best_payoffs
        node['bi_move'] = best_move
        return best_payoffs, best_move

    def security_analysis(self, node_id, player_index):
        """Compute security level (worst-case guaranteed payoff)."""
        node = self.nodes[node_id]

        # Terminal: return payoff for player_index
        if node['payoffs'] is not None:
            return node['payoffs'][player_index]

        # Non-terminal
        if node['player'] == player_index:
            # Your move: maximize your payoff
            return max(
                self.security_analysis(c['target'], player_index)
                for c in node['children']
            )
        else:
            # Opponent's move: they minimize your payoff
            return min(
                self.security_analysis(c['target'], player_index)
                for c in node['children']
            )

    def get_bi_path(self, node_id=None):
        """Trace the backward induction optimal path from root to leaf."""
        if node_id is None:
            node_id = 'root'

        path = [node_id]
        node = self.nodes[node_id]

        while node['payoffs'] is None and node['bi_move'] is not None:
            # Find the child we should move to
            next_id = None
            for child in node['children']:
                if child['move'] == node['bi_move']:
                    next_id = child['target']
                    break

            if next_id is None:
                break

            path.append(next_id)
            node = self.nodes[next_id]

        return path

🤔 Sequential Reasoning Reflection

Expand each question below to explore deeper thinking about sequential games and backward induction:

Q1: Why does backward induction say take immediately in the Centipede game, but experiments show players pass?

The Theory vs. Reality Gap:

Backward induction assumes perfect rationality and common knowledge of rationality. Under these assumptions, Player 1 knows Player 2 will defect at the last round, so Player 1 should take immediately.

But real players deviate because:

  • Bounded Rationality: Players may not think through all 6 rounds perfectly.
  • Doubt about Rationality: Players may doubt if the opponent is rational, so passing might be worth the risk.
  • Altruism/Fairness: Some players prefer equal splits. Their true payoff isn't just money but includes fairness (utility ≠ money).
  • Signaling Cooperation: Passing in early rounds signals cooperative intent, possibly encouraging reciprocal passing later.
  • Reciprocity: If Opponent 2 cooperated, Player 1 wants to repay that kindness by passing more.

Key Insight: Backward induction is logically correct given its assumptions. But those assumptions (perfect rationality, common knowledge, selfish monetary preferences) don't always hold in the real world.

Q2: Why use a game digraph instead of a game tree for NIM, and when would you prefer one?

History vs. Current State:

In the game tree, there are multiple nodes labeled "3 stones, White to move" — one reached by removing 1 stone from 4, another by removing 2 from 5, etc. Each has its own separate subtree. This is redundant because the future possibilities are identical.

In the digraph, all those "3W" nodes merge into one. The digraph only tracks current state, not the path taken to reach it.

Preference Rules:

  • Use the tree when history matters: Chess (castling rights depend on whether the king moved), Poker (the deal order shapes betting), Contract negotiation (previous offers constrain future ones).
  • Use the digraph when history is irrelevant: NIM (only current stone count matters), Checkers (position is position, doesn't matter how you got there), Turn-based board games (future depends only on current board state).

Computational Impact: The digraph can be exponentially smaller, making backward induction much faster. For NIM(6), the tree has 189 nodes; the digraph has only 21.

Q3: Why does "backward induction = security level analysis" in zero-sum games but not in non-zero-sum games?

Zero-Sum Insight:

In a 2-player zero-sum game, your payoff + opponent's payoff = 0 (or constant). So:

  • Opponent maximizing their payoff = opponent minimizing YOUR payoff.
  • Security analysis says: assume opponent plays to minimize your payoff.
  • Rational play analysis says: opponent will maximize their own payoff, which happens to minimize yours.
  • These are the same!

Non-Zero-Sum Divergence:

In a non-zero-sum game (like Chess but with both players trying to maximize their own score):

  • Opponent's payoff and your payoff are not perfectly opposed. They care about their own score, not yours.
  • Backward induction: assumes opponent wants to maximize their own payoff (rational self-interest).
  • Security analysis: assumes opponent wants to minimize your payoff (hostile), regardless of their own.
  • These lead to different choices and different predicted outcomes.

Example: In the Ultimatum Game (non-zero-sum), BI says Ann should offer $0 if Beth only cares about money. But security analysis (assume Beth wants to punish Ann) predicts Ann offers roughly $2.50 out of fear of rejection.

Q4: Zermelo's theorem requires finite games. Chess has a 50-move rule. Why?

The Problem with Infinite Games:

Backward induction must start at terminal positions and work backwards. If there are no guaranteed terminal positions (the game might cycle forever), the procedure can never begin.

Chess without the 50-move rule:

  • The same position can repeat indefinitely (the position itself doesn't change).
  • Players could technically cycle positions forever, never reaching checkmate.
  • Backward induction breaks down: there are no definite leaf nodes to start from.

The 50-move rule solution: If 50 moves pass without a pawn advance or capture, the game is a draw. This guarantees eventual termination. Combined with the finiteness of chess (only finitely many positions), the game becomes finite and Zermelo's theorem applies.

Similarly, the threefold-repetition rule: If a position repeats 3 times, the player to move can claim a draw. Again, this forces termination.

Real-world implication: Any formal game system that wants to be "solved" (analyzed completely by backward induction) must guarantee that play eventually ends. Games with potential infinite loops are mathematically intractable.

Q5: In the Ultimatum Game, why do people reject low offers when backward induction says accept?

The BI Prediction:

With purely monetary payoffs, backward induction says:

  1. Beth (responder) will accept any offer > $0, because $0 is worse than something.
  2. Knowing this, Ann (proposer) offers Beth $1 and keeps $4.
  3. Ann's BI payoff: $4. Beth's BI payoff: $1.

The Experimental Reality:

When real people play:

  • Proposers offer roughly $2–$2.50 (a fair split).
  • Responders reject offers below $1–$1.50, even though rejecting gives them $0 instead of something.

Why the Difference?

Humans have fairness preferences and inequality aversion. The true payoff isn't just money; it includes:

  • Fairness satisfaction: Getting an equal split feels good (utility boost).
  • Unfairness penalty: Getting a very unequal split feels bad (utility loss).
  • Punishment satisfaction: Rejecting an unfair offer to punish the proposer feels good.

Revised utility functions might look like:

  • Proposer: money from split + fairness bonus/penalty
  • Responder: money from split - unfairness penalty (when offer is low) + punishment satisfaction (when rejecting)

With these more realistic payoff functions, backward induction does predict rejections of very low offers and more equal proposals!

Key Lesson: Backward induction is only as good as your payoff model. If you model payoffs incorrectly (ignoring fairness, emotion, punishment), your BI solution will be wrong.

Summary: The Power of Looking Backwards

Chapter 8 Key Takeaways:

  • Sequential games have a temporal structure: players move one after another, observing past moves.
  • Game trees and game digraphs represent this structure. Trees track full history; digraphs track only current state.
  • Backward induction is the algorithm to solve perfect-information finite sequential games: start at the end, work backwards, determine optimal play at each position.
  • Zermelo's Theorem guarantees that finite sequential games have a unique backward induction solution.
  • Security level analysis provides guaranteed payoffs when you assume hostile (or at least non-cooperative) opponents.
  • Theory vs. reality: Backward induction is logically sound but assumes perfect rationality and common knowledge. Real players deviate due to bounded rationality, doubt about others' rationality, fairness preferences, and reciprocity.
  • In 2-player zero-sum games: Rational BI analysis and security level analysis agree. In non-zero-sum games, they can differ significantly.

Next chapter, we'll extend this to games with imperfect information (hidden moves) and randomness, where backward induction becomes more subtle but remains central to game theory.
