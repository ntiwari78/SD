The Chess Grandmaster's Internal Dialogue

Imagine a chess grandmaster in deep thought, facing a critical decision. What's happening in her mind? She's running a conversation with herself:

"If I move my knight to e4, my opponent will likely respond with ...d5. Then I'll have the option to play Nxd5 or f4. If I take on d5, she'll recapture with the queen, giving her pressure on e4..."

This back-and-forth mental simulation—thinking several moves ahead, considering the opponent's best responses, and evaluating the resulting positions—is the essence of minimax. The AI is not just evaluating a single position; it's recursively exploring a tree of possible futures, alternating between maximizing its own score and minimizing the opponent's.

The Three Key Insights

1. Minimax: The Foundation

At every turn, MAX (the AI player) wants to choose the move that leads to the best outcome assuming perfect play. But MIN (the opponent) will always choose their move to minimize MAX's score. So MAX must assume the worst and choose the move where the "worst-case" outcome is best. That's minimax.

2. Alpha-Beta Pruning: Cutting Branches

The grandmaster doesn't explore every possible line to completion. At some point she thinks: "Wait, I've already found a move that guarantees me at least a +3 advantage. In this other line, even if everything goes my way, I can only get +2. I don't need to fully analyze this line." She prunes it. This is alpha-beta pruning—eliminating branches that can't possibly change the final decision.

Key Insight: Alpha-beta can reduce the effective branching factor from $b$ to approximately $\sqrt{b}$, making search exponentially faster. With a branching factor of 35 (typical in chess), this shrinks a search depth of 8 to roughly the computational cost of depth 4.

3. Evaluation Functions: Knowing When to Stop

The grandmaster can't think 100 moves ahead; she stops at a reasonable depth and evaluates the board position. She might estimate: "Material is roughly even, but I have better piece coordination and a safer king. I'd say this position is worth +0.7 for me." This evaluation function encodes chess knowledge as a weighted sum of features: material count, piece safety, pawn structure, and more.

Modern AI adds two more layers:

  • Monte Carlo Tree Search (MCTS): Instead of exhaustively evaluating all positions, sample random playouts from promising branches. This works beautifully for games like Go where random play is still informative.
  • Stochastic Games (Expectiminimax): When there's chance involved (dice rolls, card draws), average over the possible outcomes instead of taking min/max.
The Unifying Principle: All adversarial search is about exploring a game tree while making one key assumption: the opponent plays optimally. This assumption lets us work backward from hypothetical futures to decide what to do now.

Mathematical Ascent

Game Formulation

A finite, two-player, zero-sum, deterministic, fully observable game is formally defined as a tuple:

$$G = (S, s_0, \text{Actions}, \text{Result}, \text{Terminal}, \text{Utility})$$

Where:

  • $S$ = set of all possible game states
  • $s_0$ = the initial state (e.g., standard chess starting position)
  • $\text{Actions}(s)$ = set of legal moves available from state $s$
  • $\text{Result}(s, a)$ = the state resulting from taking action $a$ in state $s$
  • $\text{Terminal}(s)$ = true if $s$ is a terminal state (game over)
  • $\text{Utility}(s, p)$ = numeric value of terminal state $s$ for player $p$ (1 = win, 0 = draw, −1 = loss in win/loss games)

The Minimax Value

The minimax value of a state $s$ is the score that the MAX player (us) can achieve from that state assuming both players play optimally:

$$\text{Minimax}(s) = \begin{cases} \text{Utility}(s) & \text{if } \text{Terminal}(s) \\ \max_{a \in \text{Actions}(s)} \text{Minimax}(\text{Result}(s, a)) & \text{if } \text{Player}(s) = \text{MAX} \\ \min_{a \in \text{Actions}(s)} \text{Minimax}(\text{Result}(s, a)) & \text{if } \text{Player}(s) = \text{MIN} \end{cases}$$

This recursive definition says: if it's a terminal state, return its utility; if it's MAX's turn, pick the move with maximum value; if it's MIN's turn, pick the move with minimum value (worst for MAX).

Alpha-Beta Bounds

During minimax evaluation, we track two bounds:

  • $\alpha$ = the best value (highest) that MAX can guarantee so far
  • $\beta$ = the best value (lowest) that MIN can guarantee so far

At any node, if $\alpha \geq \beta$, we can prune remaining branches:

$$\text{If } \alpha \geq \beta \text{, then prune remaining actions}$$

Why? At a MAX node, $\alpha$ is the worst outcome MIN can force. At a MIN node, $\beta$ is the worst outcome MAX can force. Once they meet or cross, the parent node won't choose this branch anyway, so we stop exploring.

Evaluation Function

Rather than search to depth 200, we search to depth 4 and estimate the value of non-terminal states using an evaluation function. A common form is a weighted linear combination of features:

$$\text{Eval}(s) = w_1 f_1(s) + w_2 f_2(s) + \cdots + w_n f_n(s)$$

For chess, $f_1$ might be material balance (points for pieces), $f_2$ piece mobility, $f_3$ king safety, etc. The weights $w_i$ are often learned from games or derived from chess theory.

Upper Confidence Bounds (UCB1)

In Monte Carlo Tree Search, when deciding which child node to expand, we balance exploration vs. exploitation using the Upper Confidence Bound formula:

$$\text{UCT}(n) = \frac{U(n)}{N(n)} + C \sqrt{\frac{\ln N(\text{parent}(n))}{N(n)}}$$

Where:

  • $U(n)$ = total utility accumulated in playouts from node $n$
  • $N(n)$ = number of times node $n$ has been visited
  • $C$ = exploration constant (typically $\sqrt{2}$)
  • The first term favors high-value nodes (exploitation)
  • The second term favors under-explored nodes (exploration)

Expectiminimax for Stochastic Games

In games with chance (dice, card draws), we add chance nodes that average over possible outcomes:

$$\text{Expectiminimax}(s, p) = \begin{cases} \text{Utility}(s, p) & \text{if } \text{Terminal}(s) \\ \max_a \text{Expectiminimax}(\text{Result}(s, a), p) & \text{if Player}(s) = p \\ \min_a \text{Expectiminimax}(\text{Result}(s, a), p) & \text{if Player}(s) \neq p \\ \sum_r P(r) \cdot \text{Expectiminimax}(\text{Result}(s, r), p) & \text{if } s \text{ is a chance node} \end{cases}$$

The chance node case computes the expected value by summing over all outcomes $r$ weighted by their probability $P(r)$.

Clean-Code Implementation

Algorithm 1: Minimax Search

The foundational algorithm. Returns the best action for MAX from the given state by recursively evaluating all reachable positions.

def minimax_search(game, state):
    """Return the best action for the current player using minimax.

    Args:
        game: Game object with Actions, Result, Terminal, Utility, Player
        state: Current game state

    Returns:
        The best action to take from this state
    """
    # Find the action that maximizes the minimax value
    best_value = float('-inf')
    best_action = None

    for action in game.actions(state):
        # Recursively evaluate the value of this action
        value = min_value(game, game.result(state, action))

        if value > best_value:
            best_value = value
            best_action = action

    return best_action


def max_value(game, state):
    """Return the minimax value assuming MAX (us) is to move."""
    # Base case: terminal state
    if game.terminal_test(state):
        return game.utility(state, 'MAX')

    # Recursive case: MAX chooses action that maximizes value
    value = float('-inf')
    for action in game.actions(state):
        value = max(value, min_value(game, game.result(state, action)))

    return value


def min_value(game, state):
    """Return the minimax value assuming MIN (opponent) is to move."""
    # Base case: terminal state
    if game.terminal_test(state):
        return game.utility(state, 'MAX')

    # Recursive case: MIN chooses action that minimizes value
    value = float('inf')
    for action in game.actions(state):
        value = min(value, max_value(game, game.result(state, action)))

    return value

Algorithm 2: Alpha-Beta Pruning Search

Optimized minimax with pruning. Maintains alpha (best for MAX so far) and beta (best for MIN so far). Stops exploring when alpha ≥ beta.

def alpha_beta_search(game, state):
    """Return best action using minimax with alpha-beta pruning."""
    best_value = float('-inf')
    best_action = None

    for action in game.actions(state):
        # Initial alpha-beta call: alpha = -infinity, beta = infinity
        value = min_value_ab(game, game.result(state, action),
                            float('-inf'), float('inf'))
        if value > best_value:
            best_value = value
            best_action = action

    return best_action


def max_value_ab(game, state, alpha, beta):
    """Minimax with alpha-beta bounds for MAX player.

    Args:
        alpha: Best value MAX can guarantee
        beta: Best value MIN can guarantee

    Returns:
        Minimax value of state, with pruning
    """
    if game.terminal_test(state):
        return game.utility(state, 'MAX')

    value = float('-inf')
    for action in game.actions(state):
        # Recursively get MIN's best response
        value = max(value, min_value_ab(game, game.result(state, action),
                                        alpha, beta))

        # Pruning condition: if we found a move that guarantees at least
        # 'value', and MIN can guarantee at most 'beta', MIN won't let
        # us reach this node, so stop searching.
        if value >= beta:
            return value  # Prune!

        alpha = max(alpha, value)

    return value


def min_value_ab(game, state, alpha, beta):
    """Minimax with alpha-beta bounds for MIN player."""
    if game.terminal_test(state):
        return game.utility(state, 'MAX')

    value = float('inf')
    for action in game.actions(state):
        # Recursively get MAX's best move
        value = min(value, max_value_ab(game, game.result(state, action),
                                        alpha, beta))

        # Pruning condition: if we found a move that limits our loss
        # to at most 'value', and MAX can guarantee at least 'alpha',
        # MAX won't let us reach this node, so stop searching.
        if value <= alpha:
            return value  # Prune!

        beta = min(beta, value)

    return value

Algorithm 3: Monte Carlo Tree Search (MCTS)

Instead of exhaustively evaluating positions, MCTS samples random playouts and builds a tree of promising continuations. The four phases: selection (UCB1), expansion, simulation (random playout), and backpropagation.

import math
import random
from collections import defaultdict


class MCTSNode:
    """A node in the MCTS tree."""

    def __init__(self, state, parent=None, action=None):
        self.state = state
        self.parent = parent
        self.action = action  # The action that led to this node
        self.children = {}    # Dictionary: action -> MCTSNode
        self.visits = 0       # Number of times this node has been visited
        self.value = 0        # Total reward accumulated from playouts


def uct_value(node, exploration_constant=math.sqrt(2)):
    """Compute the UCT (Upper Confidence bound applied to Trees) value."""
    if node.visits == 0:
        return float('inf')  # Unvisited nodes have highest priority

    exploitation = node.value / node.visits
    exploration = exploration_constant * math.sqrt(
        math.log(node.parent.visits) / node.visits
    )
    return exploitation + exploration


def mcts(game, state, num_iterations=1000):
    """Monte Carlo Tree Search. Returns best action.

    Args:
        game: Game object with actions, result, terminal, utility
        state: Current state
        num_iterations: Number of MCTS iterations (playouts)

    Returns:
        Best action found by MCTS
    """
    root = MCTSNode(state)

    for _ in range(num_iterations):
        node = root
        current_state = state

        # PHASE 1: SELECTION
        # Traverse the tree using UCB1 until we reach a non-fully-expanded node
        while len(node.children) == len(game.actions(current_state)) and \
              not game.terminal_test(current_state):
            # All children have been visited; pick the best one by UCT
            action = max(
                game.actions(current_state),
                key=lambda a: uct_value(node.children[a])
            )
            current_state = game.result(current_state, action)
            node = node.children[action]

        # PHASE 2: EXPANSION
        # If not terminal, expand one new child
        if not game.terminal_test(current_state):
            unvisited_actions = [
                a for a in game.actions(current_state)
                if a not in node.children
            ]
            action = random.choice(unvisited_actions)
            current_state = game.result(current_state, action)
            node = MCTSNode(current_state, parent=node, action=action)
            node.parent.children[action] = node

        # PHASE 3: SIMULATION (Random Playout)
        # Play randomly until terminal state
        while not game.terminal_test(current_state):
            action = random.choice(game.actions(current_state))
            current_state = game.result(current_state, action)

        # PHASE 4: BACKPROPAGATION
        # Update all nodes on the path with the outcome
        outcome = game.utility(current_state, 'MAX')
        while node is not None:
            node.visits += 1
            node.value += outcome
            node = node.parent

    # Return the action with the most visits (most reliable)
    best_action = max(
        game.actions(state),
        key=lambda a: root.children[a].visits
    )
    return best_action

Algorithm 4: Expectiminimax for Stochastic Games

Extends minimax to handle chance nodes (dice rolls, random events).

def expectiminimax_search(game, state):
    """Minimax extended for games with chance nodes.

    Game must also provide:
      - is_chance_node(state): bool
      - chance_outcomes(state): list of (probability, next_state) tuples
    """
    best_value = float('-inf')
    best_action = None

    for action in game.actions(state):
        value = expectiminimax_value(game, game.result(state, action), 'MAX')
        if value > best_value:
            best_value = value
            best_action = action

    return best_action


def expectiminimax_value(game, state, player):
    """Recursively compute expectiminimax value.

    For chance nodes: expected value over outcomes
    For MAX nodes: max over actions
    For MIN nodes: min over actions
    """
    if game.terminal_test(state):
        return game.utility(state, player)

    # Chance node: average over possible outcomes
    if game.is_chance_node(state):
        expected_value = 0
        for probability, next_state in game.chance_outcomes(state):
            expected_value += probability * expectiminimax_value(
                game, next_state, player
            )
        return expected_value

    # MAX or MIN node
    if game.to_move(state) == player:
        # Maximizing node
        return max(
            expectiminimax_value(game, game.result(state, action), player)
            for action in game.actions(state)
        )
    else:
        # Minimizing node
        return min(
            expectiminimax_value(game, game.result(state, action), player)
            for action in game.actions(state)
        )

Socratic Sandbox

These Q&A pairs highlight common pitfalls and misconceptions. Click each question to reveal the diagnosis.

Q1: Your alpha-beta search explores exactly the same nodes as plain minimax. What went wrong? ▼

Diagnosis: Move ordering is poor. Alpha-beta's efficiency depends critically on examining the best moves first. With worst-case ordering, no pruning occurs and the complexity remains $O(b^d)$ instead of the optimal $O(b^{d/2})$.

Fix: Use heuristics to order moves so you evaluate likely-good moves first. In chess, try moves that capture opponent pieces or give check first. Better move ordering → more pruning → exponential speedup. In the best case, alpha-beta explores only about $2 \cdot b^{d/2}$ nodes instead of $b^d$.

Q2: Your evaluation function says White is winning, but White loses 5 moves later. What happened? ▼

Diagnosis: Horizon effect—your search stopped at a depth where a forced tactical sequence (like a capture or sacrifice) was underway. The evaluation function can't see past the horizon, so it misjudges the position.

Fix: Implement quiescence search. Don't stop evaluation at a fixed depth; continue searching "noisy" positions (where checks, captures, or threats are happening) until you reach a quiet state. This extends the search horizon only where it matters, catching the tactics your fixed-depth search missed.

Q3: Your MCTS agent plays well in Go but terribly in Chess. Why? ▼

Diagnosis: MCTS with random playouts struggles in games where random play is highly uninformative. Chess has many traps and tactical patterns that random playouts miss entirely; you'll lose games because your simulations don't reflect real play.

Fix: (1) Use a neural network policy to guide playouts, not random moves (like AlphaGo). (2) Combine MCTS with a learned evaluation function. (3) Add domain knowledge: heavy pieces shouldn't move into forks, castling is often good, etc. In Go, random playouts work because the game is less tactical; in Chess, you need guidance.

Q4: You're playing Backgammon and your expectiminimax takes forever. What's the issue? ▼

Diagnosis: Chance nodes create a massive branching factor. Backgammon has 21 possible dice outcomes per turn. The tree grows as $O(b^d \times n^d)$ where $n$ is the number of chance outcomes. With $b=20$ moves, $n=21$, and depth 8, that's over $10^{12}$ nodes—intractable.

Fix: You can't search deeply; use a depth cutoff with an evaluation function for non-terminal states. This trades accuracy for speed. Alternatively, sample outcomes (like MCTS) rather than expanding all of them.

Q5: In a card game, your AI keeps making decisions as if it can see opponent's cards. What's wrong? ▼

Diagnosis: Not handling partial observability. You're computing minimax on a single assumed state, but the opponent has hidden information. The AI needs to maintain a belief state—a set of all possible opponent hands consistent with observed play.

Fix: Compute the expected value over all consistent states: $\text{Value}(s) = \sum_{s' \in \text{consistent}(s)} P(s') \cdot \text{Minimax}(s')$. Each hidden state gets a probability based on Bayesian reasoning. This way your decisions are robust to uncertainty. (See Chapter 11 on partial observability for details.)
