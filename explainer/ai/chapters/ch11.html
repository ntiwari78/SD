Chapter 11: Planning and Acting in the Real World

From abstract hierarchies to messy reality: how rational agents decompose complex goals and handle the unexpected.

1. Feynman Mental Model: The Road Trip Analogy

Imagine planning a cross-country road trip from New York to Los Angeles. You have two ways to plan this:

  1. Classical Planning (Unrealistic): Before you leave, plan every single steering wheel movement, every gear shift, every traffic light stop. This is impossible and absurd.
  2. Hierarchical Planning (Real-world): Plan at a high level first: "Drive to Philadelphia, then to Pittsburgh, then to Chicago, then to Denver, then to LA." For each city, you zoom in and plan the highways to take. For each highway segment, you handle real-time detours around traffic.

This is Hierarchical Task Network (HTN) planning. The agent works at multiple levels of abstraction, refining abstract high-level actions into concrete primitives only when needed. Think of it as:

Key Insight: A rational agent doesn't commit to low-level details until it has a high-level plan it believes will work. This mirrors human planning: first figure out the strategy, then the tactics.

Why Classical Planning Fails in the Real World

Classical planning (from Chapter 10) assumes:

  • Fully observable: You know the exact state of the world.
  • Deterministic: Every action has a guaranteed effect.
  • Static: Nothing changes except through your actions.
  • Single agent: Only you are acting.

But the real world is messy:

  • Partially observable: The store might be closed, but you don't know until you arrive.
  • Stochastic: You plan to drive 200 miles, but a storm delays you by 3 hours.
  • Dynamic: Other drivers change lanes, accidents block roads, gas stations run out of fuel.
  • Multi-agent: You and your roommate both want the car. You need to coordinate or negotiate.

HTN planning addresses this by:

  • Conditional planning: Build branches into the plan. "If it rains, take the highway; else take the scenic route."
  • Belief-state planning: Reason about what you know and don't know, then gather information.
  • Multi-agent coordination: Model other agents' goals and negotiate or cooperate.
  • Replanning: When reality diverges from the plan, replan locally rather than from scratch.

The Road Trip: HTN in Action

Your decomposition might look like: 

Prepare trip
├── Plan high-level route
│   └── Choose cities to visit
│   └── Check season/weather
├── Prepare car
│   └── Gas station → fill tank
│   └── Mechanic → check tire pressure
├── Drive leg 1 (NY → Philadelphia)
│   ├── High-level: take I-95
│   └── Real-time: avoid traffic (observe road conditions)
└── Stay overnight & repeat for remaining legs

At each level, you refine further based on observed conditions. This is execution monitoring: you check if your actions are working as planned, and if not, you adapt.

Multi-Agent Coordination: The Roommate Problem

If you and your roommate both plan to cook dinner simultaneously:

  • You both need the stove (shared resource).
  • You both need to buy groceries (but can't both take the car).
  • You need to coordinate: decide who cooks first, or plan your menus to avoid stove conflicts.

In multi-agent planning, agents reason about:

  • Joint actions: Actions that require multiple agents (e.g., lifting a heavy object).
  • Conflicts: When agents' goals interfere (both want the car).
  • Nash equilibrium: A state where no agent can improve by unilaterally changing its plan.
  • Cooperation vs. competition: Sometimes you cooperate (share cooking chores), sometimes you compete (get the car first).

2. Mathematical Ascent: Formalizing HTN Planning

Hierarchical Task Networks

An HTN consists of:

  • A set of primitive actions (e.g., "Drive 50 miles", "Buy milk").
  • A set of high-level actions or methods that decompose into primitives or other high-level actions.
  • A set of decomposition rules that say how to break down each high-level action.

Formally, a task decomposition is:

$$T \rightarrow \{t_1, t_2, \ldots, t_n\}$$

This reads: "Task T can be decomposed into the sequence of subtasks t₁, t₂, ..., tₙ."

High-Level Actions (HLAs) and Refinements

A high-level action is an abstract action. For example:

  • HLA: Travel(Home, Airport)
  • Refinement 1: [Drive(Home, Parking), Walk(Parking, Terminal), CheckIn(Terminal)]
  • Refinement 2: [TakeUber(Home, Airport)]

The planner can choose which refinement to use. Each refinement is a valid decomposition of the high-level action.

Angelic Semantics: Optimistic and Pessimistic Reachability

When planning at an abstract level, the planner reasons about what states are reachable under different decompositions:

  • Optimistic reachability: "What's the best outcome I can possibly achieve?" (union over all decompositions)
  • Pessimistic reachability: "What's the worst outcome?" (intersection over all decompositions)

Formally:

$$\text{Reach}^+(s, h) = \bigcup_{d \in \text{Decomp}(h)} \text{Reach}^+(s, d)$$

The optimistic reachable set of state s under high-level action h is the union of reachable sets over all possible decompositions d of h.

$$\text{Reach}^-(s, h) = \bigcap_{d \in \text{Decomp}(h)} \text{Reach}^-(s, d)$$

The pessimistic reachable set is the intersection: only states reachable under ALL decompositions.

Insight: If the goal is in Reach⁺, the planner is optimistic and commits to the decomposition. If the goal is only in Reach⁻, the planner is guaranteed to reach it regardless of which decomposition is chosen.

Conditional Planning: If-Then-Else Branches

In a partially observable world, a plan must include branches:

$$\text{Plan} = \text{if}(\text{condition}, \text{Plan}_A, \text{Plan}_B)$$

This reads: "Observe the condition. If true, execute Plan_A; else execute Plan_B."

Example: "Plan dinner" 

if (store is open):
    Buy ingredients → Cook → Serve
else:
    Order takeout → Serve

Belief State Planning

When the world is partially observable, the agent maintains a belief state: a set of possible worlds consistent with observations.

$$b_0 = \{s_1, s_2, \ldots, s_k\}$$

Initially, the agent doesn't know the exact state, so it considers all k possibilities. After an observation:

$$b' = \{s \in \text{Succeed}(b, a) : o(s) = o'\}$$

The belief state is updated to only states consistent with the observation.

Multi-Agent Planning: Joint Actions and Nash Equilibrium

In multi-agent settings, a joint action is an action executed by multiple agents:

$$\vec{a} = (a_1, a_2, \ldots, a_n)$$

A Nash equilibrium is a state where no agent can improve its payoff by changing its action unilaterally:

$$\text{Payoff}_i(\vec{a}) \geq \text{Payoff}_i(a_i', \vec{a}_{-i}) \quad \forall a_i'$$

This means: "Agent i cannot do better by switching actions while all other agents keep their actions fixed."

3. Clean-Code Implementation: HTN Planner in Python

Core Classes: Primitive vs. High-Level Actions

class Action:
    """
    A primitive action the agent can execute directly.
    Represents the lowest level of abstraction.
    """
    def __init__(self, name, preconditions, effects):
        self.name = name
        # preconditions: set of fluents that must be true
        # fluents are boolean propositions about the world
        self.preconditions = preconditions
        # effects: dict mapping fluent -> True/False
        # True means add fluent, False means remove it
        self.effects = effects

    def __repr__(self):
        return f"Action({self.name})"


class HighLevelAction:
    """
    An abstract action that must be decomposed before execution.
    Cannot be directly executed; must first be refined into primitives.
    """
    def __init__(self, name, decompositions):
        self.name = name
        # Each decomposition is a sequence of actions (primitive or HLA)
        self.decompositions = decompositions

    def __repr__(self):
        return f"HLA({self.name}, {len(self.decompositions)} decompositions)"

The HTN Planner

class HTNPlanner:
    """
    Hierarchical Task Network planner.

    The agent reasons at multiple abstraction levels, decomposing
    high-level goals into primitive actions until executable.
    This mirrors how rational agents handle complex, real-world problems.
    """
    def __init__(self, actions, hla_map):
        # Store primitive actions by name
        self.actions = {a.name: a for a in actions}
        # Store high-level actions by name
        self.hla_map = hla_map

    def plan(self, state, tasks):
        """
        Recursively decompose tasks until all are primitive.
        Returns a list of primitive actions, or None if no valid plan found.

        Args:
            state: Set of fluents currently true
            tasks: List of tasks to accomplish

        Returns:
            List of primitive actions, or None
        """
        # Base case: no more tasks
        if not tasks:
            return []

        task = tasks[0]
        remaining = tasks[1:]

        # Case 1: task is a primitive action
        if task in self.actions:
            action = self.actions[task]
            # Check preconditions
            if action.preconditions.issubset(state):
                # Apply action to get new state
                new_state = self.apply(state, action)
                # Recursively plan the rest
                rest_plan = self.plan(new_state, remaining)
                if rest_plan is not None:
                    return [action] + rest_plan
            # Precondition failed; backtrack
            return None

        # Case 2: task is a high-level action
        if task in self.hla_map:
            hla = self.hla_map[task]
            # Try each decomposition in order
            for decomp in hla.decompositions:
                # Replace HLA with decomposition and replan
                result = self.plan(state, decomp + remaining)
                if result is not None:
                    return result

        # No valid decomposition found
        return None

    def apply(self, state, action):
        """Apply an action to a state, returning the new state."""
        new_state = set(state)
        for fluent, val in action.effects.items():
            if val:
                new_state.add(fluent)
            else:
                new_state.discard(fluent)
        return new_state

Example: Planning a Dinner Party

# Define primitive actions
buy_ingredients = Action(
    name="BuyIngredients",
    preconditions={"money", "store_open"},
    effects={"have_ingredients": True}
)

cook_meal = Action(
    name="CookMeal",
    preconditions={"have_ingredients", "stove_available"},
    effects={"meal_ready": True}
)

set_table = Action(
    name="SetTable",
    preconditions=set(),
    effects={"table_set": True}
)

# Define high-level actions with decompositions
prepare_dinner_hla = HighLevelAction(
    name="PrepareDinner",
    decompositions=[
        ["BuyIngredients", "CookMeal", "SetTable"],  # Preferred decomposition
        ["OrderTakeout", "SetTable"]                  # Fallback
    ]
)

order_takeout = Action(
    name="OrderTakeout",
    preconditions={"money", "phone"},
    effects={"meal_ready": True}
)

# Create planner
planner = HTNPlanner(
    actions=[buy_ingredients, cook_meal, set_table, order_takeout],
    hla_map={"PrepareDinner": prepare_dinner_hla}
)

# Initial state
initial_state = {"money", "store_open", "stove_available", "phone"}

# Plan
plan = planner.plan(initial_state, ["PrepareDinner"])
print("Plan:", [a.name for a in plan])

Conditional Planning with Observation

class ConditionalPlan:
    """
    A plan with branches based on observations.
    When the world is partially observable, the agent must prepare
    for multiple possible outcomes.

    This addresses the key insight: a rational agent doesn't just
    execute actions blindly—it observes, learns, and adapts.
    """
    def __init__(self, action=None, condition=None,
                 if_true=None, if_false=None):
        self.action = action              # Action to execute
        self.condition = condition        # Condition to observe
        self.if_true = if_true           # Plan if condition is true
        self.if_false = if_false         # Plan if condition is false

    def execute(self, state, observe_fn):
        """
        Execute the plan in a partially observable world.

        Args:
            state: Current state
            observe_fn: Function that observes a condition, returns bool

        Returns:
            Final state after plan execution
        """
        if self.action:
            # Execute the action first
            state = self.apply_action(state, self.action)

        if self.condition:
            # Observe the condition
            observation = observe_fn(self.condition)
            if observation:
                # Condition is true; follow if_true branch
                return self.if_true.execute(state, observe_fn)
            else:
                # Condition is false; follow if_false branch
                return self.if_false.execute(state, observe_fn)

        # No condition; plan is done
        return state

    def apply_action(self, state, action_name):
        """Placeholder for applying actions."""
        # In a real implementation, this would modify state
        return state

4. Socratic Sandbox: Failure Cases and Learning

Let's explore a realistic scenario where classical planning fails, and see how modern techniques address it.

Scenario: An HTN planner is tasked with organizing a dinner party. It decomposes Prepare Dinner into:
  1. Buy Ingredients (precondition: store is open)
  2. Cook Meal (precondition: have ingredients)
  3. Set Table (precondition: none)
The planner commits to this plan and executes step 1. But when the agent arrives at the store, it's closed. The planner gets stuck.
1. Why can't the classical HTN planner recover when the store is closed? +

The HTN planner made a critical assumption: that preconditions are true in the initial state or can be satisfied by action effects. It checked that store_open was true initially, but didn't account for the possibility that the store might become closed between planning time and execution time.

This is the closed-world assumption: the planner assumes that the world only changes through the agent's actions, and external events don't happen. In reality, the store might close due to an emergency, or the weather might change, or another agent might take the last ingredient.

The agent's assumption: "I planned based on store_open = true, so the store must still be open when I arrive."

Reality: The world changed, and the assumption no longer holds.

2. How would a conditional planner handle this? What observation would trigger the alternative branch? +

A conditional planner would build in an observation point after attempting to execute BuyIngredients 

if (buy_ingredients_succeeded):
    Cook meal → Set table
else:
    Order takeout → Set table

The key difference: the conditional planner observes the outcome of each action and branches accordingly. After trying to BuyIngredients, it observes: "Did I actually get the ingredients?" If yes, proceed with cooking. If no, fall back to OrderTakeout.

This is execution monitoring: the agent watches what happens and adapts in real time.

3. If two agents (you and a roommate) are both planning to cook dinner simultaneously, what new planning challenges arise? +

Multi-agent planning introduces several complications:

  1. Shared resources: You both need the stove. A classical planner assumes exclusive access. Now you must reason: "If I start cooking at 6pm and take 1 hour, and my roommate starts at 6:15pm and takes 1 hour, our plans conflict."
  2. Interdependent goals: Your goal is to cook Italian. Your roommate wants to cook Thai. Both need the same saucepan. You must negotiate: "I'll use the saucepan first from 6-6:30, then you use it 6:30-7."
  3. Communication and coordination: You need a communication protocol. Without it, you both show up at the stove at 6pm and conflict.
  4. Incomplete information: You don't know your roommate's exact plan. You must reason about their likely goals and constraints, and adjust accordingly.
  5. Nash equilibrium: The solution must be a state where neither of you benefits from unilaterally changing your plan. If you both commit to your schedules, neither should have incentive to "cheat" and go earlier.
4. The planner decomposed "Prepare Dinner" in a fixed order. Could a different decomposition avoid the failure? +

Absolutely. Consider these alternative decompositions:

Decomposition 1 (Original): 

BuyIngredients → CookMeal → SetTable
(Fails if store is closed)

Decomposition 2 (Check before committing): 

CheckStore → if(open): BuyIngredients else: OrderTakeout → CookMeal → SetTable
(Observes first, then commits to a plan)

Decomposition 3 (Parallel options): 

(BuyIngredients OR OrderTakeout) → CookMeal → SetTable
(Either option is valid; planner picks one)

This teaches us a critical lesson: the choice of decomposition can make or break a plan in the real world. A good HTN planner doesn't just decompose arbitrarily; it decomposes in a way that is robust to uncertainty.

5. How does belief-state planning differ from conditional planning? +

Both techniques handle partial observability, but they differ in approach:

Conditional Planning:

  • Builds branches into the plan: "if A, do Plan1; else do Plan2"
  • The agent decides during execution which branch to take based on observations
  • The plan is a tree structure; the agent navigates the tree at runtime
  • Example: "Try to call the store. If they answer, ask about hours. If they say 'open', go buy ingredients."

Belief-State Planning:

  • Maintains a set of possible worlds: "I don't know if the store is open, so I consider all worlds where it might be open or closed"
  • The planner reasons about states of knowledge, not individual states
  • The plan is a sequence of actions that ensure the goal is reached in ALL possible worlds (or gather information to narrow down the worlds)
  • Example: "I'll go to the store. If it's closed, the store closure state becomes impossible (I can now rule it out). If it's open, I buy ingredients. Either way, I've learned something."

When to use each:

  • Conditional planning: When you know the possible observations in advance and can plan explicit branches for each.
  • Belief-state planning: When you need to handle information gathering and exploration in a more general framework. It's more powerful but can be slower.
