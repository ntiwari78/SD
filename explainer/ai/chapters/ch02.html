Intelligent Agents: The Mind of Rational Behavior

How to design systems that perceive, reason, and act in uncertain environments—from vacuum cleaners to self-driving cars.

Section 1: Feynman Mental Model (Zero-Jargon Opening)

Imagine you're designing a robot butler for your home. This butler needs to do dishes, vacuum floors, greet guests, and handle unexpected situations. How do you program it? More fundamentally: what does it mean for this butler to act intelligently?

What is an Agent?

An agent is anything that:

  1. Perceives its environment through sensors (eyes, ears, thermometers, touch)
  2. Thinks about what it perceives
  3. Acts on the environment through actuators (hands, motors, speakers)

For our robot butler: sensors are cameras and touch sensors, actuators are motors and arms. The butler perceives "there's a dirty dish on the counter" and acts by picking it up and washing it.

The Agent Function

Everything an agent does can be described by its agent function: a mapping from what it has perceived to what it should do next. In other words:

Agent Function

Given every percept sequence ever seen (the complete history of sensations), what action should the agent take right now?

If the butler has seen: [empty room, person walks in with muddy shoes, dirty floor, person sits down], it should act: "vacuum the floor near the person."

What Makes a Good Agent? Rationality

This is crucial: a good agent is not one that knows everything or always makes the objectively best decision. Instead, a rational agent is one that:

Rationality Definition

Given what the agent knows right now (its percept sequence and built-in knowledge), it chooses the action most likely to succeed according to its performance criteria.

This is subtly different from "always making the best decision." The difference matters. Consider this example from the textbook:

The Cargo Door Example

A spaceship's cargo bay doors fail unpredictably. The commander must decide: close them or leave them open? The rational choice is "close them" because statistically, they stay closed 99% of the time. But on this particular run, they jam and cause a catastrophe. The commander made the rational decision—but got unlucky. Rationality is about expected outcomes, not guaranteed ones.

The PEAS Framework: A Job Description

Before building an agent, you must specify what you want it to do. The textbook introduces the PEAS framework—a template for describing any agent:

Performance Measure
How do we judge success? What is the score?
Environment
What is the setting? What can change?
Actuators
What can the agent do? What are its actions?
Sensors
What can the agent perceive? What are its inputs?

Case Study: The Autonomous Taxi Driver

Let's use a running example: designing a self-driving taxi. What is its PEAS?

Autonomous Taxi - PEAS Description
  • Performance Measure: Safe arrival, minimal time, fuel efficiency, passenger comfort, legal compliance
  • Environment: Roads, other vehicles, pedestrians, traffic lights, weather, GPS satellites, passenger requests
  • Actuators: Steering wheel, accelerator, brake, turn signals, windshield wipers, horn, display screens
  • Sensors: Cameras (front, sides, back), LiDAR (3D laser), radar, wheel encoders, GPS, accelerometers, thermometers, microphones, sonar

This PEAS description immediately tells us what the taxi agent must handle:

  • It must perceive multiple modalities (visual, spatial, temporal)
  • It operates in a dynamic, stochastic environment (other drivers, pedestrians, weather)
  • It must make real-time decisions (no time to compute optimal paths)
  • It has safety-critical consequences (mistakes harm people)

Section 2: Mathematical Ascent (Formalizing Rationality)

Now we climb from intuition to formalism. The Feynman model above is correct—but we need mathematics to reason precisely about agents.

The Agent Function (Formal Definition)

An agent function is a pure mathematical object:

$$f: \mathcal{P}^* \to \mathcal{A}$$

Where:

  • $\mathcal{P}^*$ = the set of all possible percept sequences (histories)
  • $\mathcal{A}$ = the set of all possible actions
  • $f$ = a function mapping each history to an action to perform

For the taxi: $\mathcal{P}^*$ includes all possible sequences of camera images, LiDAR scans, GPS readings, etc. $\mathcal{A}$ is {accelerate, brake, turn-left, turn-right, honk, ...}

Agent = Architecture + Program

The agent function is what we want. But we implement it on actual hardware:

$$\text{Agent} = \text{Agent Program} + \text{Physical Architecture}$$

The agent program is the code (the algorithm, the neural network, the decision tree). The physical architecture is the hardware (CPU, GPU, sensors, motors). Together, they realize the abstract agent function.

Rational Agent (Formal Definition)

Rationality is about maximizing expected utility:

Rational Agent (Textbook Definition)

For each possible percept sequence, a rational agent should select an action that is expected to maximize its performance measure, given:

  1. The evidence provided by the percept sequence
  2. Whatever built-in knowledge the agent has

Mathematically, this is the expected utility maximization principle:

$$a^* = \arg\max_{a \in A} \sum_{s'} P(s' | a, e) \cdot U(s')$$

Where:

  • $a^*$ = the rational action to take
  • $a$ = a candidate action
  • $e$ = the evidence (current percept and history)
  • $s'$ = a possible future state
  • $P(s' | a, e)$ = probability of reaching state $s'$ given action $a$ and evidence $e$
  • $U(s')$ = utility (goodness) of state $s'$

In plain English: try each possible action, estimate what state you'll end up in, compute how good that state is, weight by probability, and pick the action with the highest expected utility.

Performance Over Environment Sequences

The performance measure is typically a function over sequences of environment states:

$$V: S_0, S_1, S_2, \ldots \to \mathbb{R}$$

For the taxi:

  • $V = -\text{(travel time in seconds)} - \text{(fuel used in liters)} + 100 \times \text{(happy passengers)}$
  • $V = -\infty$ if the taxi crashes (hard safety constraint)

Environment Dimensions (Formal)

Every environment has properties. These shape which agent architectures will work:

$$\mathcal{E} = \langle \text{Observable}, \text{Agents}, \text{Determinism}, \text{Episodes}, \text{Static}, \text{Discrete} \rangle$$

Each dimension is binary (or multi-valued):

  • Observable: Fully observable (agent sees all) vs. Partially observable (agent sees some)
  • Agents: Single agent vs. Multi-agent
  • Determinism: Deterministic (action always has same outcome) vs. Stochastic (action outcomes vary)
  • Episodes: Episodic (agent's life divides into independent tasks) vs. Sequential (long-term dependencies)
  • Static: Static (environment doesn't change unless agent acts) vs. Dynamic (environment changes on its own)
  • Discrete: Discrete (finite actions/percepts) vs. Continuous (infinite values)
Examples
  • Chess: Fully observable, deterministic, episodic, static, discrete, single-agent
  • Poker: Partially observable (can't see opponent's hand), stochastic, episodic, static, discrete, multi-agent
  • Autonomous Taxi: Partially observable, stochastic, sequential, dynamic, continuous, multi-agent

The Lookup Table Problem

Naively, an agent could implement its function as a lookup table: "If you've seen percept sequence $p_1, p_2, \ldots, p_t$, then do action $a$."

How many entries would this table need? For a taxi with a camera seeing 30 frames per second for 8 hours:

$$\sum_{t=1}^{T} |\mathcal{P}|^t \quad \text{where } |\mathcal{P}| \approx 2^{24 \times 1920 \times 1080}$$

This is approximately $10^{600,000,000,000}$ entries. That's more atoms in the universe raised to itself. Lookup table agents are impossible.

Key Insight

Real agents must use generalization: learning patterns from limited data, exploiting structure in the environment, reasoning abstractly. This is where agent architectures come in.

Section 3: Clean-Code Implementation (Python)

Now we implement four real agent architectures in Python. Each maps directly to the theory above, with detailed comments.

1. Simple Reflex Agent

The simplest agent: if percept is X, do action Y. No memory, no planning. 

class SimpleReflexAgent:
    """
    Simple Reflex Agent - maps current percept directly to action.
    
    Agent Function: f: P -> A (current percept only, no history)
    Architecture: No internal state, just condition-action rules.
    
    Works in: Fully observable, deterministic environments.
    Example: A simple vacuum agent that cleans if dirt, moves otherwise.
    """
    
    def __init__(self):
        # Condition-action rules: state -> action
        self.rules = {
            ('A', 'Dirty'): 'Suck',
            ('A', 'Clean'): 'Right',
            ('B', 'Dirty'): 'Suck',
            ('B', 'Clean'): 'Left',
        }
    
    def perceive(self, location, status):
        """
        Sensor model: Map environment to percept.
        Input: location ('A' or 'B'), status ('Dirty' or 'Clean')
        """
        return (location, status)
    
    def decide(self, percept):
        """
        Agent function: percept -> action
        Direct lookup based on condition-action rules (Rational Agent Principle)
        """
        return self.rules.get(percept, 'NoOp')
    
    def act(self, action, environment):
        """
        Actuator model: Execute action on environment.
        """
        environment.apply_action(action)
    
    def run(self, environment, steps=10):
        """Execute the agent for N steps."""
        for _ in range(steps):
            percept = self.perceive(*environment.get_agent_percept())
            action = self.decide(percept)
            self.act(action, environment)
            print(f"Percept: {percept} -> Action: {action}")


class ModelBasedReflexAgent:
    """
    Model-Based Reflex Agent - maintains internal state model.
    
    Agent Function: f: P* -> A (percept history, via state)
    Architecture: Sensors -> State Update -> Condition-Action Rules -> Actuators
    
    Maintains:
    - State: Current knowledge about the world
    - Transition Model: How actions change the world
    - Sensor Model: How to interpret percepts
    
    Works in: Partially observable, deterministic environments.
    Example: Vacuum agent that tracks which room it's in (history matters).
    """
    
    def __init__(self):
        # Internal state model
        self.state = {'location': 'A', 'status': {'A': 'Dirty', 'B': 'Dirty'}}
        
        # Transition model: (state, action) -> new_state
        self.transition_model = self._build_transition_model()
        
        # Condition-action rules
        self.rules = {
            ('A', 'Dirty'): 'Suck',
            ('A', 'Clean'): 'Right',
            ('B', 'Dirty'): 'Suck',
            ('B', 'Clean'): 'Left',
        }
    
    def _build_transition_model(self):
        """Define how the world evolves: (state, action) -> state"""
        def transition(state, action):
            loc = state['location']
            status = state['status'].copy()
            
            if action == 'Suck':
                status[loc] = 'Clean'
            elif action == 'Left':
                loc = 'A'
            elif action == 'Right':
                loc = 'B'
            
            return {'location': loc, 'status': status}
        
        return transition
    
    def update_state(self, percept):
        """
        State Update Function: integrate new percept with current state.
        Implements the sensor model (percept -> state prediction).
        """
        location, status = percept
        self.state['location'] = location
        self.state['status'][location] = status
    
    def decide(self):
        """
        Agent function via state: use current state to decide.
        More robust than simple reflex (can handle partial observability).
        """
        location = self.state['location']
        status = self.state['status'][location]
        return self.rules.get((location, status), 'NoOp')
    
    def run(self, environment, steps=10):
        """Execute the agent."""
        for _ in range(steps):
            percept = environment.get_agent_percept()
            self.update_state(percept)
            action = self.decide()
            environment.apply_action(action)
            print(f"State: {self.state} -> Action: {action}")


class GoalBasedAgent:
    """
    Goal-Based Agent - has explicit goals and searches for plans.
    
    Agent Function: f: (P*, Goal) -> A
    Architecture: Sensors -> State Update -> Goal -> Search/Planner -> Actuators
    
    Key components:
    - Goal: A desired state or set of states
    - Transition Model: How actions affect the world
    - Search/Planner: Finds a sequence of actions to reach goal
    
    Works in: Partially observable, deterministic, sequential environments.
    Example: A taxi that has a goal "get passenger to airport".
    """
    
    def __init__(self, goal=None):
        self.state = {'location': 'A', 'status': {'A': 'Dirty', 'B': 'Dirty'}}
        self.goal = goal or lambda state: all(v == 'Clean' for v in state['status'].values())
        self.transition_model = self._build_transition_model()
        self.plan = []  # Sequence of actions to execute
    
    def _build_transition_model(self):
        """State transition model."""
        def transition(state, action):
            loc = state['location']
            status = state['status'].copy()
            
            if action == 'Suck':
                status[loc] = 'Clean'
            elif action == 'Left':
                loc = 'A'
            elif action == 'Right':
                loc = 'B'
            
            return {'location': loc, 'status': status}
        
        return transition
    
    def search_plan(self, state):
        """
        Simple breadth-first search to find a plan.
        Implements goal-based reasoning: find a sequence of actions
        that takes current state to a goal state.
        """
        from collections import deque
        
        queue = deque([(state, [])])
        visited = {str(state)}
        
        while queue:
            current_state, path = queue.popleft()
            
            # Check if goal reached
            if self.goal(current_state):
                return path
            
            # Try all actions
            for action in ['Suck', 'Left', 'Right']:
                next_state = self.transition_model(current_state, action)
                state_str = str(next_state)
                
                if state_str not in visited:
                    visited.add(state_str)
                    queue.append((next_state, path + [action]))
        
        return []  # No plan found
    
    def update_state(self, percept):
        """Update internal model from percept."""
        location, status = percept
        self.state['location'] = location
        self.state['status'][location] = status
    
    def run(self, environment, steps=20):
        """Execute the agent, replanning if needed."""
        while steps > 0:
            percept = environment.get_agent_percept()
            self.update_state(percept)
            
            # If no plan, compute one
            if not self.plan:
                self.plan = self.search_plan(self.state)
            
            if self.plan:
                action = self.plan.pop(0)
                environment.apply_action(action)
                print(f"Following plan. State: {self.state} -> Action: {action}")
            else:
                print(f"Goal reached! State: {self.state}")
                break
            
            steps -= 1


class UtilityBasedAgent:
    """
    Utility-Based Agent - maximizes expected utility.
    
    Agent Function: f: (P*, Performance Measure) -> A*
    Architecture: Sensors -> State Update -> Utility Function -> 
                  Expectation over futures -> Action with max expected utility
    
    Key principle: Choose action that maximizes expected utility.
    a* = argmax_a sum_s' P(s'|a,e) * U(s')
    
    Works in: Partially observable, stochastic, sequential environments.
    Example: A taxi that balances speed, safety, and passenger comfort.
    """
    
    def __init__(self, transition_model=None, sensor_model=None, utility_fn=None):
        self.state = {'location': 'A', 'status': {'A': 'Dirty', 'B': 'Dirty'}}
        self.transition_model = transition_model or self._default_transition_model()
        self.utility_fn = utility_fn or self._default_utility()
        self.sensor_model = sensor_model or self._default_sensor_model()
    
    def _default_transition_model(self):
        """Probabilistic transition model: (state, action) -> distribution over states"""
        def transition(state, action):
            loc = state['location']
            status = state['status'].copy()
            
            if action == 'Suck':
                status[loc] = 'Clean'
            elif action == 'Left':
                loc = 'A'
            elif action == 'Right':
                loc = 'B'
            
            return {'location': loc, 'status': status}
        
        return transition
    
    def _default_sensor_model(self):
        """Map true state to observation (may be noisy)."""
        return lambda state: state
    
    def _default_utility(self):
        """Utility function: state -> real number (higher is better)."""
        def utility(state):
            # Reward for clean squares, cost for being in dirty square
            score = sum(100 if s == 'Clean' else -10 for s in state['status'].values())
            return score
        return utility
    
    def expected_utility(self, action, state, depth=2):
        """
        Expected Utility: E[U | action, state] = sum_s' P(s'|a,state) * U(s')
        
        For depth > 0, recursively estimates future utility.
        This is a simplified version; real implementations use more sophisticated
        probabilistic reasoning.
        """
        next_state = self.transition_model(state, action)
        
        if depth == 0:
            return self.utility_fn(next_state)
        
        # Simplified: assume deterministic transitions for demo
        return self.utility_fn(next_state)
    
    def decide(self, state):
        """
        Rational decision: choose action with maximum expected utility.
        Implements: a* = argmax_a E[U | a, e]
        """
        actions = ['Suck', 'Left', 'Right', 'NoOp']
        utilities = {a: self.expected_utility(a, state) for a in actions}
        return max(utilities, key=utilities.get)
    
    def update_state(self, percept):
        """Update state from percept."""
        location, status = percept
        self.state['location'] = location
        self.state['status'][location] = status
    
    def run(self, environment, steps=10):
        """Execute the agent."""
        for _ in range(steps):
            percept = environment.get_agent_percept()
            self.update_state(percept)
            action = self.decide(self.state)
            environment.apply_action(action)
            utilities = {a: self.expected_utility(a, self.state) for a in ['Suck', 'Left', 'Right']}
            print(f"State: {self.state} -> Utilities: {utilities} -> Action: {action}")

class VacuumWorldEnvironment:
    """
    Vacuum World: Two squares (A and B) that can be dirty or clean.
    
    Environment Properties (from textbook Figure 2.6):
    - Observable: Partially (agent sees only current square status, not both)
    - Agents: Single agent
    - Determinism: Deterministic (actions have predictable outcomes)
    - Episodes: Sequential (long-term dependencies)
    - Static: Static (environment doesn't change unless agent acts)
    - Discrete: Discrete (finite states and actions)
    """
    
    def __init__(self, location_A_dirty=True, location_B_dirty=True):
        """Initialize environment."""
        self.status = {
            'A': 'Dirty' if location_A_dirty else 'Clean',
            'B': 'Dirty' if location_B_dirty else 'Clean',
        }
        self.agent_location = 'A'
        self.step_count = 0
        self.performance = 0
    
    def get_agent_percept(self):
        """
        Agent perceives: its current location and the status of that square.
        This is partial observability (doesn't see the other square).
        """
        return (self.agent_location, self.status[self.agent_location])
    
    def apply_action(self, action):
        """Execute agent's action."""
        self.step_count += 1
        
        if action == 'Suck':
            self.status[self.agent_location] = 'Clean'
            self.performance += 10
        elif action == 'Left':
            self.agent_location = 'A'
        elif action == 'Right':
            self.agent_location = 'B'
        elif action == 'NoOp':
            pass
        else:
            return  # Invalid action
        
        # Time penalty
        self.performance -= 1
    
    def all_clean(self):
        """Check if all squares are clean."""
        return all(s == 'Clean' for s in self.status.values())
    
    def reset(self):
        """Reset the environment."""
        self.status = {'A': 'Dirty', 'B': 'Dirty'}
        self.agent_location = 'A'
        self.step_count = 0
        self.performance = 0

3. Learning Agent Architecture

class LearningAgent:
    """
    Learning Agent - improves performance over time.
    
    Architecture components:
    1. Performance Element: Makes decisions (like utility-based agent)
    2. Learning Element: Improves the policy/value function based on feedback
    3. Critic: Evaluates performance vs. standard, provides reward signal
    4. Problem Generator: Suggests exploratory actions to improve future performance
    
    Agent Function: f(P*, history of rewards) -> A that learns from experience.
    
    Key insight: The learning element modifies the performance element's knowledge
    based on feedback from the critic.
    """
    
    def __init__(self):
        # Performance element: Q-values for state-action pairs
        self.q_values = {}
        self.learning_rate = 0.1
        self.discount_factor = 0.9
        self.epsilon = 0.1  # Exploration rate
        
        # Agent state
        self.state = {'location': 'A', 'status': {'A': 'Dirty', 'B': 'Dirty'}}
        self.last_action = None
        self.last_state = None
    
    def get_q_value(self, state_key, action):
        """Get Q-value for state-action pair (default to 0 if unseen)."""
        return self.q_values.get((state_key, action), 0.0)
    
    def select_action_performance_element(self, state):
        """
        Performance element: Use learned Q-values to select action.
        Implements epsilon-greedy: mostly exploit, occasionally explore.
        """
        state_key = str(state)
        actions = ['Suck', 'Left', 'Right', 'NoOp']
        
        # Exploration: random action
        if self._random() < self.epsilon:
            return self._random_choice(actions)
        
        # Exploitation: best known action
        q_vals = {a: self.get_q_value(state_key, a) for a in actions}
        return max(q_vals, key=q_vals.get)
    
    def _random(self):
        """Random number 0-1."""
        import random
        return random.random()
    
    def _random_choice(self, lst):
        """Random choice from list."""
        import random
        return random.choice(lst)
    
    def learning_element_update(self, state, action, reward, next_state):
        """
        Learning element: Update Q-values based on experience.
        Uses Q-learning: Q(s,a) := Q(s,a) + lr * (r + gamma * max Q(s',a') - Q(s,a))
        
        This implements the critic's feedback: reward signal tells learning element
        whether to increase or decrease the value of this action in this state.
        """
        state_key = str(state)
        next_state_key = str(next_state)
        
        # Current Q-value
        current_q = self.get_q_value(state_key, action)
        
        # Best future Q-value
        next_actions = ['Suck', 'Left', 'Right', 'NoOp']
        next_q_values = [self.get_q_value(next_state_key, a) for a in next_actions]
        max_next_q = max(next_q_values) if next_q_values else 0
        
        # Q-learning update
        new_q = current_q + self.learning_rate * (reward + self.discount_factor * max_next_q - current_q)
        self.q_values[(state_key, action)] = new_q
    
    def problem_generator_suggest_action(self, state):
        """
        Problem generator: Suggest exploratory actions.
        This is a simple version that just enforces epsilon-greedy.
        A sophisticated version would suggest states the agent hasn't explored yet.
        """

        # already handles this, but in principle, problem generator could be smarter.
        pass
    
    def run(self, environment, episodes=10):
        """Run the learning agent for multiple episodes."""
        for episode in range(episodes):
            environment.reset()
            done = False
            total_reward = 0
            
            while not done:
                percept = environment.get_agent_percept()
                self.update_state(percept)
                
                # Performance element selects action
                action = self.select_action_performance_element(self.state)
                
                # Execute action
                old_state = self.state.copy()
                environment.apply_action(action)
                self.update_state(environment.get_agent_percept())
                
                # Critic evaluates: reward is performance change
                reward = environment.performance
                total_reward += reward
                
                # Learning element updates Q-values
                self.learning_element_update(old_state, action, reward, self.state)
                
                # Check if done

            print(f"Episode {episode + 1}: Total reward = {total_reward:.1f}")

        location, status = percept

        self.state['status'][location] = status

Section 4: Socratic Sandbox (Failure Cases & Guiding Questions)

Understanding requires wrestling with hard cases. Below are five questions from the textbook and practice problems. Each has a collapsible answer. Try to answer before peeking.

The Problem: A simple reflex vacuum agent has only a dirt sensor (no location sensor). It perceives [Clean]. Should it go Left or Right? Why does this create an infinite loop?
Why This Fails: Without knowing its location, the agent cannot distinguish between square A and square B. Both squares showing [Clean] look identical. If the rule is "when you see [Clean], move right," then the agent will move right infinitely, never stopping.
Agent Type Fix: A model-based reflex agent maintains internal state: "I just moved right, so I must be in square B now." It remembers its location even though the sensor didn't tell it. The state update (transition model) prevents the infinite loop.
Rational Agent Principle: A rational agent must use all available information—not just current percepts, but also the history and internal models. This is why the agent function is $f: \mathcal{P}^* \to \mathcal{A}$ (percept sequence, not percept).
The Problem: A model-based taxi agent tracks other cars' positions using a transition model that assumes they follow traffic rules perfectly. During rush hour, cars cut in unexpectedly. Why did the agent's predictions fail? What property of the environment did the designer underestimate?
Why This Fails: The transition model was wrong. The designer assumed: Deterministic (if a car is in lane 2, it stays in lane 2 unless signaling). But the real environment is Stochastic: cars can make unpredictable lane changes. The model couldn't handle this variability.
Design Lesson: Misclassifying environment properties leads to failed agents. A deterministic model-based agent is useless in a stochastic world. The taxi needed either:
  1. A probabilistic transition model: $P(\text{next location} | \text{current location, action})$
  2. A goal-based or utility-based architecture that can plan for multiple futures
  3. A learning agent that updates its model as it sees unexpected behaviors
PEAS Insight: The environment dimension "Determinism" is critical. Get it wrong and your agent architecture is fundamentally inadequate.
The Problem: A goal-based agent is told "get the passenger to the airport." It drives 150mph on the highway during rush hour. The goal is achieved but the passenger is terrified. What's missing from a goal-based architecture that a utility-based agent would have?
Why This Fails: A goal-based agent only cares about: does the action achieve the goal? The taxi's goal—"passenger at airport"—is achieved, so the agent succeeded. But it ignored all the constraints: safety, comfort, legality, fuel efficiency.
The Missing Piece: A utility function. Goals are binary (achieved or not), but utility is continuous. A utility function can express preferences over multiple objectives:
$$U = 100 \times \text{(passenger safely at airport)} - 10 \times \text{(travel time in minutes)} - 50 \times \text{(passenger fear level)} - 5 \times \text{(fuel used)}$$
Agent Redesign: A utility-based agent chooses the action that maximizes expected utility, not just achieves a goal. The taxi would still deliver the passenger, but at a speed that balances arrival time against safety and comfort.
The Problem: A learning agent for chess plays thousands of games but keeps making the same endgame mistakes. Its performance element works fine, and its learning element correctly updates values based on outcomes. Which component of the learning agent architecture is failing?
The Learning Agent has Four Components:
  1. Performance Element (works): Chooses actions based on learned policy
  2. Learning Element (works): Updates policy from rewards
  3. Critic (unknown): Evaluates performance vs. standard, provides reward signal
  4. Problem Generator (likely culprit): Suggests exploratory actions to learn better policies
The Failure: The problem generator is not generating endgame positions for the agent to explore. It only plays full games from the start. Since endgames are rare in random play (games usually decide earlier), the agent never explores enough endgame variations to learn good strategies there.
The Fix: A better problem generator would intentionally create endgame scenarios: "Play these 1000 positions with 5 pieces left." This ensures the learning element gets enough data to learn in regions that matter.
The Problem: You're designing a PEAS description for a home thermostat. Is this environment fully or partially observable? Deterministic or stochastic? Episodic or sequential? What happens if you get these wrong?
Correct Classification:
  • Observable: Fully observable for the thermostat's purposes (it sees the temperature sensor; that's all it needs)
  • Determinism: Stochastic (opening a door, weather changes, people adjusting radiators—all unpredictable)
  • Episodes: Sequential (decisions today affect heating patterns tomorrow; if you learned "turn up heat at 6am," you need that knowledge tomorrow)
If You Get It Wrong:
Assume Fully Observable + Deterministic: You'd design a lookup table agent: "if temp is 68, turn heat up." But users open windows (stochastic event), breaking your model. The thermostat overshoots or undershoots.
Assume Episodic: You'd design a simple reflex agent that resets each day. It never learns that "people always turn it down at 10pm, so conserve energy then." A sequential learning agent improves by noticing patterns.
Correct Design: A model-based or learning agent that:
  • Maintains state (current temp, recent drift from setpoint)
  • Uses a probabilistic model of heating (accounts for variability)
  • Learns user preferences over time (sequential memory)

Conclusion: Building Intelligent Agents

This chapter gave us the theoretical scaffolding for AI: the rational agent model. We learned that:

  1. An agent is a mapping from percept sequences to actions. The agent function is the what; the program + architecture is the how.
  2. Rationality is about expected value, not perfect knowledge. A rational agent maximizes expected utility given what it knows.
  3. Environment properties matter enormously. A reflex agent works in fully observable, deterministic worlds. A utility-based agent can handle stochastic, multi-agent chaos.
  4. There is no free lunch. A lookup table implementing the true agent function is computationally infeasible. Real agents must exploit structure: models, goals, learned policies.
  5. PEAS is your design template. Before coding, specify Performance, Environment, Actuators, Sensors. Get these wrong and your agent fails.
