The GPS Analogy: Finding Your Way in a City

Imagine you've just arrived in a brand new city and need to get from your hotel to a restaurant across town. You have a map (the state space), your hotel is the start state, the restaurant is the goal state, and each road is an action. Different search strategies are fundamentally different approaches to navigation—each with trade-offs in speed, memory, and optimality.

💡 Key Insight: Every search algorithm is a different strategy for exploring a graph. The choice of strategy determines what you find, how quickly you find it, and how much of the city you need to remember.
🔍

Breadth-First Search

Explores every street at distance 1, then distance 2. Guarantees shortest path. Remembers EVERYTHING.

🧗

Depth-First Search

Picks a direction and walks as far as possible, then backtracks. Memory-efficient. May wander far from goal.

💰

Uniform-Cost Search

Like a taxi driver wanting the cheapest route. Explores outward by cumulative cost, not just hops. Optimal for costs.

🎯

Greedy Best-First

Always walks toward where the restaurant "looks closest" (straight-line distance). Fast but myopic—can be fooled by walls.

⭐

A* Search

The smart navigator. Combines cost so far + estimated remaining distance. "I've walked 3 blocks, restaurant looks 5 away, so 8 total." Optimal AND efficient.

The Trade-Off Landscape

BFS is like checking every intersection methodically. You WILL find the shortest route, but you'll explore an exponential number of streets. With a branching factor of 10 and a goal at depth 12, that's $10^{12}$ intersections—roughly 1 terabyte of memory just to track where you've been.

DFS is like committing to one street and following it to the end. You use much less memory (just one path at a time), but you might end up at the far edge of the city before realizing you went the wrong direction.

A* is the Goldilocks solution. It's guided by a heuristic—an estimate of remaining distance—so it favors promising directions without exploring exhaustively. If your heuristic is good (never overestimates true distance), A* finds the shortest path while exploring far fewer streets than BFS.

Mathematical Ascent

The formal structure of search problems is elegant and powerful. Let's build up from simple definitions to the deep theory of heuristics and optimality.

Problem Formulation: The Search Graph

At its heart, every search problem is a graph where:

Graph Representation

$G = (V, E)$ where:
$V$ = set of states (vertices)
$E$ = set of actions (edges)

A problem is formally a tuple:

Problem Definition

$(S, s_0, \text{Actions}(s), \text{Result}(s,a), \text{Goal}(s), \text{Cost}(s,a,s'))$

Where:
$S$ = state space
$s_0$ = initial state
$\text{Actions}(s)$ = applicable actions in state $s$
$\text{Result}(s,a)$ = state resulting from action $a$ in $s$
$\text{Goal}(s)$ = test whether $s$ is a goal
$\text{Cost}(s,a,s')$ = cost of transition $s \to s'$ via $a$

A solution is a sequence of actions $[a_1, a_2, \ldots, a_n]$ that transforms $s_0$ into a goal state.

Algorithm Complexity: Time and Space

Each algorithm has different scaling properties with:

  • $b$ = branching factor (avg. number of actions per state)
  • $d$ = depth of shallowest goal state
  • $m$ = maximum depth of search tree

Breadth-First Search

Time: $O(b^d)$
Space: $O(b^d)$

Must store entire frontier at depth $d$

Depth-First Search

Time: $O(b^m)$
Space: $O(bm)$

Only stores one path from root

Uniform-Cost Search

Time: $O(b^{1+\lfloor C^*/\epsilon \rfloor})$
Space: $O(b^{1+\lfloor C^*/\epsilon \rfloor})$

$C^*$ = optimal cost, $\epsilon$ = min step cost

A* Search

Time: $O(b^d)$ worst case
Space: $O(b^d)$

Much better in practice with good $h$

Heuristic Functions: The Art of Good Guesses

A heuristic $h(n)$ estimates the cost from state $n$ to the nearest goal. Good heuristics are the difference between practical search and computational explosion.

Admissibility: Never Overestimate

A heuristic is admissible if:
$$h(n) \leq h^*(n) \text{ for all } n$$

Where $h^*(n)$ = true cost from $n$ to goal

Intuition: If your estimate is always optimistic, A* finds optimal solutions.

Consistency: Local Admissibility

A heuristic is consistent (monotone) if:
$$h(n) \leq c(n, a, n') + h(n')$$

For every action $a$ from $n$ to successor $n'$

Intuition: No detours. Moving toward goal always decreases estimated total cost.
💡 Key Insight: Consistency implies admissibility. A consistent heuristic is even stronger—it guarantees A* never re-expands a state. This is why consistent heuristics are the gold standard.

A* Evaluation Function

$$f(n) = g(n) + h(n)$$

Where:
$g(n)$ = actual cost from start to $n$
$h(n)$ = estimated cost from $n$ to goal
$f(n)$ = estimated total path cost through $n$

A* always expands the node with lowest $f(n)$ value.

Heuristic Dominance: More Information is Better

When two admissible heuristics $h_1$ and $h_2$ satisfy $h_2(n) \geq h_1(n)$ for all $n$, we say $h_2$ dominates $h_1$.

Dominance Rule

If $h_2$ dominates $h_1$ and both are admissible:

A* with $h_2$ expands a subset of nodes expanded by A* with $h_1$

Always prefer the dominating heuristic.

Why? A higher heuristic value (while still admissible) gives A* more information about which paths to pursue. It naturally prunes away more inefficient directions.

Effective Branching Factor: Measuring Heuristic Quality

A good heuristic behaves as if the search tree has a much lower branching factor. We measure this with the effective branching factor $b^*$.

Effective Branching Factor Formula

$$N + 1 = 1 + b^* + (b^*)^2 + \ldots + (b^*)^d$$

Where $N$ = number of nodes expanded by A*

An effective $b^*$ close to 1.0 means the heuristic is excellent. $b^*$ = $b$ means the heuristic is useless.

For example, if a search with true branching factor 10 generates 113 node expansions, we can solve for $b^*$ and find it's about 1.5—much better than blind search!

Clean Code Implementation

Let's implement the core search framework in Python. These implementations follow the AIMA pseudocode directly and emphasize clarity over performance.

Problem.py — Problem Abstraction
class Problem:
    """Abstract class representing a formal search problem.

    A problem defines the space of possible states, actions that
    transform states, and tests to detect goal states. A concrete
    subclass must implement actions() and result().
    """

    def __init__(self, initial, goal=None):
        """Initialize the problem with initial state and goal(s)."""
        self.initial = initial
        self.goal = goal

    def actions(self, state):
        """Return the actions executable in this state.

        This is the core of the problem's structure—how the agent
        can move through the state space.
        """
        raise NotImplementedError

    def result(self, state, action):
        """Return the state that results from executing an action.

        Given a state and action, deterministically return the next state.
        This must be consistent with actions().
        """
        raise NotImplementedError

    def goal_test(self, state):
        """Return True if the state is a goal state.

        Can be a single goal or a list of goal states.
        """
        if isinstance(self.goal, list):
            return state in self.goal
        return state == self.goal

    def path_cost(self, c, state1, action, state2):
        """Return the cost of a path arriving at state2.

        Arguments:
            c: cost of path so far
            state1: the previous state
            action: the action taken
            state2: the resulting state

        Default: unit cost (all actions cost 1). Override for weighted graphs.
        """
        return c + 1


class Node:
    """A node in the search tree.

    Each node represents a partial solution path from the initial
    state to some state. Nodes are expanded to generate child nodes.
    """

    def __init__(self, state, parent=None, action=None, path_cost=0):
        """Initialize a node.

        state: the state represented by this node
        parent: the parent node in the search tree
        action: the action that led from parent to this node
        path_cost: g(n) — total cost from initial to this node
        """
        self.state = state
        self.parent = parent
        self.action = action
        self.path_cost = path_cost
        self.depth = parent.depth + 1 if parent else 0

    def expand(self, problem):
        """List the nodes reachable by one action from this node.

        This is the core operation of search: generating successors.
        Returns a list of child Node objects.
        """
        return [self.child_node(problem, action)
                for action in problem.actions(self.state)]

    def child_node(self, problem, action):
        """Return a child node created by applying action."""
        next_state = problem.result(self.state, action)
        next_cost = problem.path_cost(self.path_cost, self.state,
                                      action, next_state)
        return Node(next_state, self, action, next_cost)

    def solution(self):
        """Return the sequence of actions to reach this node from root."""
        return [node.action for node in self.path()[1:]]

    def path(self):
        """Return a list of nodes forming the path from root to this node."""
        node, path_back = self, []
        while node:
            path_back.append(node)
            node = node.parent
        return list(reversed(path_back))

    def __lt__(self, other):
        """Comparison for priority queue. Subclasses override."""
        return self.path_cost < other.path_cost
SearchAlgorithms.py — Core Algorithms
from collections import deque
import heapq


def breadth_first_search(problem):
    """BFS: Explores all nodes at depth d before depth d+1.

    Properties:
    - COMPLETE: Always finds a solution (if one exists)
    - OPTIMAL: Finds shortest path (for unit-cost problems)
    - TIME: O(b^d) where b=branching factor, d=depth
    - SPACE: O(b^d) - must store entire frontier

    BFS is like exploring every intersection at distance 1 km,
    then distance 2 km, etc. Guaranteed shortest path but huge memory.
    """
    node = Node(problem.initial)
    if problem.goal_test(node.state):
        return node

    frontier = deque([node])  # FIFO queue: shallowest first
    explored = set()  # Track visited states to avoid cycles

    while frontier:
        node = frontier.popleft()  # Always expand shallowest
        explored.add(node.state)

        for child in node.expand(problem):
            if child.state not in explored and child not in frontier:
                if problem.goal_test(child.state):
                    return child  # Goal found at optimal depth
                frontier.append(child)

    return None  # No solution exists


def depth_first_search(problem):
    """DFS: Follows one path deeply before backtracking.

    Properties:
    - COMPLETE: Only if max depth is finite
    - OPTIMAL: No, may find long roundabout paths
    - TIME: O(b^m) where m=max depth
    - SPACE: O(bm) - only stores one path (very efficient!)

    DFS is like picking a street and walking until you hit a dead end,
    then backtracking. Uses little memory but might explore far.
    """
    frontier = [Node(problem.initial)]  # LIFO stack: deepest first
    explored = set()

    while frontier:
        node = frontier.pop()  # Always expand deepest

        if problem.goal_test(node.state):
            return node

        explored.add(node.state)

        # Add children to frontier (they'll be deepest)
        for child in node.expand(problem):
            if child.state not in explored:
                frontier.append(child)

    return None


def uniform_cost_search(problem):
    """UCS: Expands nodes in order of path cost g(n).

    Properties:
    - COMPLETE: Yes
    - OPTIMAL: Yes, for any cost function
    - TIME: O(b^(1 + floor(C*/ε)))
    - SPACE: O(b^(1 + floor(C*/ε)))

    Like a taxi finding the cheapest route: expands outward by
    cumulative cost, not just hop count. Optimal for any costs.
    """
    node = Node(problem.initial)
    frontier = [(0, id(node), node)]  # Min-heap: lowest cost first
    frontier_states = {node.state: 0}
    explored = set()

    while frontier:
        _, _, node = heapq.heappop(frontier)

        if problem.goal_test(node.state):
            return node

        if node.state in explored:
            continue

        explored.add(node.state)

        for child in node.expand(problem):
            if child.state not in explored:
                # Only add if we haven't seen this state or found cheaper path
                if child.state not in frontier_states or \
                   child.path_cost < frontier_states[child.state]:
                    frontier_states[child.state] = child.path_cost
                    heapq.heappush(frontier, (child.path_cost,
                                               id(child), child))

    return None


def greedy_best_first_search(problem, h):
    """Greedy: Expands node with lowest h(n) estimate to goal.

    Properties:
    - COMPLETE: No, can get stuck
    - OPTIMAL: No, follows heuristic not cost
    - TIME: O(b^m) typically better than uninformed
    - SPACE: Similar to BFS

    Like always walking toward where the destination looks closest.
    Fast but myopic—fooled by obstacles.

    Args:
        h: heuristic function h(node) -> estimated cost to goal
    """
    node = Node(problem.initial)
    frontier = [(h(node), id(node), node)]
    explored = set()

    while frontier:
        _, _, node = heapq.heappop(frontier)

        if problem.goal_test(node.state):
            return node

        explored.add(node.state)

        for child in node.expand(problem):
            if child.state not in explored:
                heapq.heappush(frontier, (h(child), id(child), child))

    return None


def astar_search(problem, h):
    """A* Search: Expands node with lowest f(n) = g(n) + h(n).

    Properties:
    - COMPLETE: Yes
    - OPTIMAL: Yes (if h is admissible)
    - TIME: O(b^d) worst case, much better with good h
    - SPACE: O(b^d)

    The smart navigator. Combines:
    - g(n): actual cost so far
    - h(n): estimated remaining cost

    f(n) = g(n) + h(n) balances exploration with goal-directedness.

    Args:
        h: heuristic function h(node) -> estimated cost to goal
           MUST be admissible: h(n) <= true_cost(n, goal)
    """
    node = Node(problem.initial)

    # f(n) will be our priority
    frontier = [(h(node), id(node), node)]
    frontier_states = {node.state: (h(node), node.path_cost)}
    explored = set()

    while frontier:
        _, _, node = heapq.heappop(frontier)

        if problem.goal_test(node.state):
            return node

        if node.state in explored:
            continue

        explored.add(node.state)

        for child in node.expand(problem):
            if child.state not in explored:
                # f(n) = g(n) + h(n)
                f_child = child.path_cost + h(child)

                # Only add if new state or found cheaper path
                if child.state not in frontier_states or \
                   child.path_cost < frontier_states[child.state][1]:
                    frontier_states[child.state] = (f_child, child.path_cost)
                    heapq.heappush(frontier, (f_child, id(child), child))

    return None

Why These Implementations Matter

These are not optimized implementations—they're clear implementations. Each algorithm directly reflects its defining principle:

  • BFS: FIFO frontier. First in, first out. Shallowest nodes first.
  • DFS: LIFO stack. Last in, first out. Deepest nodes first.
  • UCS: Priority queue ordered by path cost $g(n)$. Cheapest first.
  • Greedy: Priority queue ordered by heuristic $h(n)$. Most promising first (myopically).
  • A*: Priority queue ordered by $f(n) = g(n) + h(n)$. Balance cost and promise.

The only difference between these algorithms is what ordering strategy you use for the frontier. This demonstrates the power of the Node/Problem abstraction: swap the data structure, get a new algorithm.

Socratic Sandbox: Questions to Deepen Understanding

Use these questions to test and refine your intuition about search algorithms. Click each question to reveal a detailed answer.

Why does BFS guarantee finding the shallowest goal but use enormous memory? ▼

The Memory Explosion: BFS stores the entire frontier—every node at the current depth level. At depth $d$ with branching factor $b$, the frontier holds $O(b^d)$ nodes.

Example: In a game or puzzle with $b = 10$ (10 possible actions per state) and goal at depth $d = 12$:

Frontier size $= 10^{12}$ nodes

At ~100 bytes per node = 100 gigabytes of memory!

For $d = 13$: 1 terabyte.

Why It Works: This exhaustive approach guarantees optimality. BFS explores ALL nodes at depth $d-1$ before exploring any at depth $d$. So the first goal found is guaranteed to be at the shallowest level. There's no way a deeper goal could have been found first.

The Trade-Off: Time complexity is $O(b^d)$ (must examine all nodes), and space is also $O(b^d)$ (must store them). For practical problems with moderate branching factors, BFS becomes infeasible. This is why heuristics (A*) are essential.

A robot using Greedy Best-First Search got stuck circling a lake. Why? ▼

The Problem: Greedy only looks at $h(n)$—the estimated distance to the goal. It has no awareness of cost so far ($g(n)$).

Suppose the robot wants to cross a lake:

  • Direct route blocked by a lake
  • Must detour around it
  • Each step along the shore initially moves AWAY from the goal
  • So $h(n)$ increases temporarily
  • Greedy prefers to backtrack (reduce $h$) rather than commit to the detour
  • Result: circles the same area indefinitely

Why A* Fixes It: A* considers $f(n) = g(n) + h(n)$. Once you've invested cost $g(n)$ in going around the lake, A* "commits"—backtracking would mean wasting that already-paid cost. A* prefers to continue forward even if $h$ temporarily increases.

Key Insight: Greedy is myopic; it only looks at the immediate promise of being close to the goal, ignoring the path already taken. A* is farsighted; it considers both where you've been and where you're going.

Two heuristics h₁ and h₂ are both admissible. h₂(n) ≥ h₁(n) for all n. Which should we prefer? ▼

Always prefer $h_2$. This is the principle of heuristic dominance.

Why: A higher heuristic value (while still admissible—never overestimating) gives A* more information:

  • $h_2(n) \geq h_1(n)$ means $h_2$ is "more informed" or "tighter"
  • This causes $f(n) = g(n) + h_2(n)$ to be larger
  • A* expands nodes in order of $f(n)$
  • Larger $f$ values mean higher priority in the frontier
  • Paradoxically, higher heuristic values prune more of the search space

Formal Result: Every node expanded by A* with $h_2$ is also expanded by A* with $h_1$. But the converse is false. So $h_2$ expands a subset—it's more efficient.

Intuition: More information = better decisions. If you know the goal is "fairly far" rather than just "not super close," you'll avoid pursuing false leads more effectively.

A robot solving the 8-puzzle used DFS and took 65,000 steps to solve a puzzle that BFS solves in 20 moves. What happened? ▼

DFS Found a Non-Optimal Path: DFS has no depth preference. It follows the first branch to its absolute end before backtracking. In the 8-puzzle (a state space where max depth is about 31), DFS might:

  1. Start with an arbitrary move
  2. Continue making moves down that path
  3. Reach depth 30 or 31 without finding the goal
  4. Backtrack and try another branch
  5. Eventually find a goal, but at depth 20 or more

The path discovered is not optimal. DFS explores $O(b^m)$ where $m$ is max depth. A 20-move solution exists, but DFS commits to a different branch first and wanders deep before finding anything.

Why BFS Finds 20: BFS explores all states at depth 1, then all at depth 2, etc. The moment it finds a goal at depth 20, it stops—and that's guaranteed to be optimal (shortest) because no shallower goal exists.

Key Lesson: DFS is not optimal. It's memory-efficient ($O(bm)$ space) but finds long, wasteful solutions. BFS is optimal (for unit cost) but memory-hungry. A* with a good heuristic gets the best of both worlds.

Why is straight-line distance admissible for route-finding but NOT the number of cities in between? ▼

Straight-Line Distance (SLD): Admissible

By the triangle inequality, the shortest road distance between two points is at least as long as the straight-line distance. Mathematically:

Straight-line distance $\leq$ actual road distance

$\text{SLD}(n, \text{goal}) \leq h^*(n, \text{goal})$

SLD never overestimates. ✓ Admissible.

Number of Cities: Not Admissible

This heuristic can easily overestimate:

  • Route A: Pass through 4 cities on short roads = 10 km total
  • Route B: Pass through 3 cities on long highways = 50 km total
  • Heuristic says "3 cities = 30 km estimated" (overestimate for route B!)

Since the heuristic can exceed the true cost, it violates admissibility.

Key Principle: For admissibility, the heuristic must be impossible to violate—it must be physically guaranteed by the problem structure (like the triangle inequality) rather than statistical (like average cities per route).

Understanding Heuristic Dominance

The Manhattan distance heuristic (h2) is said to dominate the misplaced tiles heuristic (h1) because:

h1: Misplaced Tiles

Count how many tiles are not in their goal position. For a tile 3 places away, h1 = 1.

Less informed, expands more nodes

h2: Manhattan Distance

Sum of distances each tile must move (horizontal + vertical). For a tile 3 places away, h2 = 3.

More informed, expands fewer nodes

Try solving with both heuristics on the same puzzle. You'll always see h2 expand fewer nodes, even though both find the optimal solution. This is heuristic dominance in action.
