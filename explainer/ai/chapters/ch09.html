Chapter 9 · Inference in First-Order Logic

Effective procedures for answering questions about what must be true given a knowledge base of logical sentences. From universal instantiation to resolution, we explore the landscape of algorithmic reasoning.

The Detective Agency's Playbook

The Core Problem

You have a knowledge base (KB) containing facts and rules expressed in first-order logic. Someone asks a question: "Is Colonel West a criminal?" How do you figure out the answer algorithmically?

This is the inference problem: given a set of logical sentences and a query, determine whether the query is entailed by the KB. Unlike propositional logic (Chapter 7), FOL sentences involve quantifiers and variables, making brute-force enumeration infeasible.

Strategy 1: Propositionalize (Brute Force)

Convert every general rule into specific instances for every object in your domain. For example:

  • Rule: "All greedy kings are evil" = $\forall x \; (King(x) \land Greedy(x) \Rightarrow Evil(x))$
  • After grounding: $King(John) \land Greedy(John) \Rightarrow Evil(John)$, $King(Richard) \land Greedy(Richard) \Rightarrow Evil(Richard)$, ...

Then apply propositional logic algorithms from Chapter 7 (e.g., SAT solvers).

The catch: With function symbols like $Father(x)$, the Herbrand universe becomes infinite: $Father(John), Father(Father(John)), Father(Father(Father(John))), \ldots$ You generate infinitely many ground atoms, making the approach impractical.

Strategy 2: Forward Chaining (Data-Driven)

Start from known facts and work forward, applying rules to derive new facts. Picture a detective who begins with the evidence:

  • Known: $American(West), Missile(M_1), Owns(Nono, M_1), Enemy(Nono, America)$
  • Apply rule: "A missile owned by a hostile nation and sold by an American is a threat..."
  • Derive: $Sells(West, M_1, Nono), Weapon(M_1), Hostile(Nono)$
  • Apply another rule: "Americans who sell weapons to hostile nations are criminals"
  • Conclude: $Criminal(West)$ ✓

This is a chain reaction where each new fact can trigger additional rules. Forward chaining is:

  • Complete for definite clauses (Horn clauses with a single positive conclusion)
  • Efficient when few rules fire (for Datalog, bounded by $p \cdot n^k$ ground facts)
  • Semidecidable with function symbols (may not terminate on false queries)

Strategy 3: Backward Chaining (Goal-Driven)

Start from the goal and work backward, asking "What rules could prove this?" Like a detective working backward from a suspect:

  • Goal: Prove $Criminal(West)$
  • Which rules conclude $Criminal(x)$? Rule: "American + Weapon + Sells + Hostile ⇒ Criminal"
  • Subgoals: Prove $American(West)$, $Weapon(y)$, $Sells(West,y,z)$, $Hostile(z)$
  • Recursively prove each subgoal...

Backward chaining implements AND/OR search: OR nodes try multiple rules, AND nodes must prove all premises. This is the foundation of Prolog.

  • Advantage: Only explores goal-relevant rules (more focused)
  • Disadvantage: Can loop on cyclic graphs; needs memoization or tabling

The Glue: Unification

The magic ingredient that makes lifted (non-grounded) inference possible. Instead of trying every possible substitution, unification finds the one right substitution that makes two expressions match.

Example: Given $Knows(John, x)$ and $Knows(John, Jane)$, unification instantly discovers $\{x \leftarrow Jane\}$ without searching.

Unification with occur check (to prevent infinite structures) is the key to efficient inference algorithms.

Resolution

The nuclear option: a single inference rule that works for any FOL sentence (not just definite clauses). The algorithm:

  1. Convert the KB and $\neg\alpha$ to Conjunctive Normal Form (CNF)
  2. Repeatedly resolve complementary literals using unification
  3. If you derive the empty clause, $\alpha$ is entailed (proof by contradiction)
  4. If you exhaust all possibilities, $\alpha$ is not entailed

Resolution is refutation-complete: it will find a proof if one exists, but may not terminate for false queries (semidecidable).

The Fundamental Limit

FOL inference is semidecidable (Turing, 1936; Church, 1936):

  • If a sentence is entailed, we will eventually prove it.
  • If a sentence is not entailed, we might search forever.
  • We cannot decide (in finite time) whether a false sentence is truly false or merely unprovable.

This is not a limitation of our algorithms—it's a fundamental property of FOL itself. Propositional logic is decidable (we can solve SAT in theory, though NP-complete in practice), but adding quantifiers takes us beyond decidability.

Mathematical Formalism

Universal Instantiation (UI)

Given a universally quantified sentence, substitute any ground term for the variable:

$$\frac{\forall v \; \alpha}{\text{SUBST}(\{v/g\}, \alpha)}$$

Example: From $\forall x \; (King(x) \land Greedy(x) \Rightarrow Evil(x))$, we can instantiate with $g = John$ to get:

$$King(John) \land Greedy(John) \Rightarrow Evil(John)$$

Existential Instantiation (EI)

Given an existentially quantified sentence, replace the variable with a new Skolem constant not appearing elsewhere:

$$\frac{\exists v \; \alpha}{\text{SUBST}(\{v/k\}, \alpha)}$$

where $k$ is a fresh constant. Example: From $\exists x \; (Crown(x) \land OnHead(x, John))$, we instantiate with a new constant $C_1$:

$$Crown(C_1) \land OnHead(C_1, John)$$

Key insight: If $\exists y$ appears inside $\forall x$, the choice of $y$ depends on $x$, so we use a Skolem function $F(x)$ instead of a constant.

Generalized Modus Ponens

For atomic sentences $p_i, p_i', q$ where $\text{SUBST}(\theta, p_i') = \text{SUBST}(\theta, p_i)$ for all $i$:

$$\frac{p_1', \; p_2', \; \ldots, \; p_n', \quad (p_1 \land p_2 \land \ldots \land p_n \Rightarrow q)}{\text{SUBST}(\theta, q)}$$

This combines UI and modus ponens: we find a substitution that makes the premises of a rule match known facts, then derive the conclusion under that substitution.

Unification

The unification algorithm finds a substitution $\theta$ that makes two expressions identical:

$$\text{UNIFY}(p, q) = \theta \quad \text{where} \quad \text{SUBST}(\theta, p) = \text{SUBST}(\theta, q)$$

Examples:

  • $\text{UNIFY}(Knows(John, x), Knows(John, Jane)) = \{x/Jane\}$ ✓
  • $\text{UNIFY}(Knows(John, x), Knows(y, Bill)) = \{x/Bill, y/John\}$ ✓
  • $\text{UNIFY}(Knows(John, x), Knows(y, Mother(y))) = \{y/John, x/Mother(John)\}$ ✓
  • $\text{UNIFY}(Knows(John, x), Knows(x, Elizabeth))$ = FAIL (same variable in two positions after standardizing apart)

The unifier is unique up to variable renaming—we call it the Most General Unifier (MGU).

Forward Chaining (FOL-FC-ASK)

Algorithm for proving queries from definite clauses (Horn clauses):

function FOL-FC-ASK(KB, α)
facts ← all ground atomic sentences in KB
rules ← all rules in KB
repeat
for each rule $p_1 \land \cdots \land p_n \Rightarrow q$ do
for each substitution $\theta$ matching premises against facts do
new_fact ← SUBST($\theta$, $q$)
if UNIFY(new_fact, α) = $\phi$ then return $\phi$
if new_fact ∉ facts then facts ← facts ∪ {new_fact}
until no new facts added
return FAIL

Complexity for Datalog (no function symbols): $\leq p \cdot n^k$ ground facts, where $p$ is the number of predicates, $n$ is the number of constants, and $k$ is the maximum arity.

Optimization: The Rete algorithm avoids redundant matching by building a dataflow network and updating only affected nodes when new facts arrive.

Backward Chaining (FOL-BC-ASK)

Goal-driven search using AND/OR expansion:

function FOL-BC-OR(KB, goal, $\theta$)
for each clause in KB where HEAD unifies with goal do
new_$\theta$ ← UNIFY(goal, HEAD)
if new_$\theta$ ≠ FAIL then
yield from FOL-BC-AND(KB, BODY, new_$\theta$)

Backward chaining returns a generator of substitutions (all ways to satisfy the goal).

CNF Conversion for FOL

Steps to convert any FOL sentence to Conjunctive Normal Form:

  1. Eliminate implications: $\alpha \Rightarrow \beta \equiv \neg\alpha \lor \beta$
  2. Move ¬ inward: $\neg(\alpha \land \beta) \equiv \neg\alpha \lor \neg\beta$, $\neg(\forall x \; \alpha) \equiv \exists x \; \neg\alpha$
  3. Standardize variables: Rename bound variables to avoid conflicts
  4. Skolemize: Replace $\exists$ with Skolem constants/functions
  5. Drop universal quantifiers: Remaining $\forall$ are implicit
  6. Distribute ∨ over ∧: $(\alpha \lor \beta) \land \gamma \equiv (\alpha \land \gamma) \lor (\beta \land \gamma)$

Skolemization example: $\forall x \; \exists y \; Loves(x,y)$ becomes $\forall x \; Loves(x, F(x))$ where $F$ is a Skolem function.

Resolution Rule (First-Order)

The fundamental inference rule for CNF sentences:

$$\frac{\ell_1 \lor \cdots \lor \ell_k \qquad m_1 \lor \cdots \lor m_n}{\text{SUBST}(\theta, \ell_1 \lor \cdots \lor \ell_{i-1} \lor \ell_{i+1} \lor \cdots \lor \ell_k \lor m_1 \lor \cdots \lor m_{j-1} \lor m_{j+1} \lor \cdots \lor m_n)}$$

where $\theta = \text{UNIFY}(\ell_i, \neg m_j)$.

Resolution removes complementary literals and applies the unifying substitution. It is refutation-complete: if KB $\models$ α, we will derive the empty clause from KB ∧ ¬α.

Equality Handling

FOL naturally includes equality. Three approaches to handle it:

  • Axiomatize: Add reflexive, symmetric, transitive, and substitution properties as rules
  • Demodulation: Replace literals using equality facts (e.g., if $x = y$ and we have $P(x)$, replace with $P(y)$)
  • Paramodulation: A resolution variant that handles equality directly by substituting equals

from dataclasses import dataclass
from typing import Dict, List, Optional, Set, Tuple, Union, Generator

# ── Core Data Structures ──

@dataclass(frozen=True)
class Var:
    """A logic variable, e.g., ?x"""
    name: str
    def __repr__(self): return f"?{self.name}"

@dataclass(frozen=True)
class Const:
    """A constant symbol, e.g., John, Nono"""
    name: str
    def __repr__(self): return self.name

@dataclass(frozen=True)
class Func:
    """A function term, e.g., Mother(John)"""
    name: str
    args: tuple
    def __repr__(self): return f"{self.name}({','.join(str(a) for a in self.args)})"

@dataclass(frozen=True)
class Pred:
    """A predicate (atomic sentence), e.g., King(John)"""
    name: str
    args: tuple
    negated: bool = False
    def __repr__(self):
        sign = "¬" if self.negated else ""
        return f"{sign}{self.name}({','.join(str(a) for a in self.args)})"
    def negate(self):
        return Pred(self.name, self.args, not self.negated)

@dataclass(frozen=True)
class DefiniteClause:
    """Horn clause: premises => conclusion
    premises: tuple of Pred (conjunction)
    conclusion: Pred (single positive literal)
    """
    premises: tuple
    conclusion: Pred
    def __repr__(self):
        if not self.premises:
            return str(self.conclusion)
        return f"{' ∧ '.join(str(p) for p in self.premises)} ⇒ {self.conclusion}"

Substitution = Dict[Var, object]


# ── Unification Algorithm ──

def unify(x, y, theta=None):
    """Find substitution θ such that SUBST(θ,x) = SUBST(θ,y).
    Returns substitution dict or None (failure).
    """
    if theta is None:
        theta = {}
    if x == y:
        return theta
    if isinstance(x, Var):
        return unify_var(x, y, theta)
    if isinstance(y, Var):
        return unify_var(y, x, theta)
    if isinstance(x, (Pred, Func)) and isinstance(y, (Pred, Func)):
        if type(x) != type(y):
            return None
        if x.name != y.name or len(x.args) != len(y.args):
            return None
        if isinstance(x, Pred) and x.negated != y.negated:
            return None
        for a, b in zip(x.args, y.args):
            theta = unify(a, b, theta)
            if theta is None:
                return None
        return theta
    return None

def unify_var(var, x, theta):
    """Handle variable unification with occur check."""
    if var in theta:
        return unify(theta[var], x, theta)
    if isinstance(x, Var) and x in theta:
        return unify(var, theta[x], theta)
    if occurs_in(var, x, theta):
        return None  # Occur check failed
    return {**theta, var: x}

def occurs_in(var, x, theta):
    """Does var appear inside x?"""
    if var == x:
        return True
    if isinstance(x, Var) and x in theta:
        return occurs_in(var, theta[x], theta)
    if isinstance(x, (Pred, Func)):
        return any(occurs_in(var, a, theta) for a in x.args)
    return False

def apply_subst(expr, theta):
    """Apply substitution θ to an expression."""
    if isinstance(expr, Const):
        return expr
    if isinstance(expr, Var):
        if expr in theta:
            return apply_subst(theta[expr], theta)
        return expr
    if isinstance(expr, Func):
        return Func(expr.name, tuple(apply_subst(a, theta) for a in expr.args))
    if isinstance(expr, Pred):
        return Pred(expr.name, tuple(apply_subst(a, theta) for a in expr.args), expr.negated)
    return expr

def standardize_apart(clause, counter=[0]):
    """Rename all variables in a clause to fresh names."""
    counter[0] += 1
    mapping = {}
    def rename(expr):
        if isinstance(expr, Var):
            if expr not in mapping:
                mapping[expr] = Var(f"{expr.name}_{counter[0]}")
            return mapping[expr]
        if isinstance(expr, Const):
            return expr
        if isinstance(expr, Func):
            return Func(expr.name, tuple(rename(a) for a in expr.args))
        if isinstance(expr, Pred):
            return Pred(expr.name, tuple(rename(a) for a in expr.args), expr.negated)
        return expr
    if isinstance(clause, DefiniteClause):
        new_prems = tuple(rename(p) for p in clause.premises)
        new_conc = rename(clause.conclusion)
        return DefiniteClause(new_prems, new_conc)
    return rename(clause)


# ── Forward Chaining ──

def fol_fc_ask(kb_clauses, query):
    """Forward chaining for definite clauses.
    Returns substitution if query is entailed, None otherwise.
    """
    facts = set()
    rules = []
    for c in kb_clauses:
        if not c.premises:
            facts.add(c.conclusion)
        else:
            rules.append(c)

    while True:
        new_facts = set()
        for rule in rules:
            std_rule = standardize_apart(rule)
            for theta in match_conjuncts(std_rule.premises, facts, {}):
                new_fact = apply_subst(std_rule.conclusion, theta)
                if new_fact not in facts and new_fact not in new_facts:
                    new_facts.add(new_fact)
                    phi = unify(new_fact, query)
                    if phi is not None:
                        return phi
        if not new_facts:
            return None
        facts.update(new_facts)

def match_conjuncts(conjuncts, facts, theta):
    """Match a conjunction of predicates against known facts."""
    if not conjuncts:
        yield theta
        return
    first = apply_subst(conjuncts[0], theta)
    rest = conjuncts[1:]
    for fact in facts:
        new_theta = unify(first, fact, dict(theta))
        if new_theta is not None:
            yield from match_conjuncts(rest, facts, new_theta)


# ── Backward Chaining ──

def fol_bc_ask(kb_clauses, query):
    """Backward chaining returns generator of substitutions."""
    return fol_bc_or(kb_clauses, query, {})

def fol_bc_or(kb, goal, theta):
    """OR node: try each rule whose conclusion matches the goal."""
    for clause in kb:
        std = standardize_apart(clause)
        rhs = std.conclusion
        lhs = std.premises
        new_theta = unify(rhs, apply_subst(goal, theta), dict(theta))
        if new_theta is not None:
            yield from fol_bc_and(kb, lhs, new_theta)

def fol_bc_and(kb, goals, theta):
    """AND node: prove all subgoals."""
    if theta is None:
        return
    if not goals:
        yield theta
        return
    first = goals[0]
    rest = goals[1:]
    for t1 in fol_bc_or(kb, apply_subst(first, theta), theta):
        yield from fol_bc_and(kb, rest, t1)

Socratic Sandbox: Five Critical Questions

Q1: Propositionalization generates infinite ground terms with function symbols. How do I handle this?
▼

With function symbols, the Herbrand universe is infinite: you generate $Father(John), Father(Father(John)), Father(Father(Father(John))), \ldots$ This makes propositionalization impractical.

Herbrand's theorem guarantees that if the sentence IS entailed, a proof using only a finite subset of these instantiations exists. The practical approach is iterative deepening: generate terms of depth 0, then depth 1, etc.

However, if the sentence is NOT entailed, iterative deepening never terminates. This is the fundamental semidecidability of FOL inference (Turing, 1936; Church, 1936). No algorithm can always correctly determine whether an arbitrary FOL sentence is entailed.

Q2: My forward chaining keeps re-matching rules against facts. It's inefficient. How do I optimize?
▼

Naive FOL-FC-ASK re-checks every rule against every fact each iteration—$O(n^2)$ behavior per iteration. The key optimization is the Rete algorithm: build a dataflow network where each node tests one condition.

Variable bindings flow through the network, and partial matches are cached. When a new fact arrives, only the affected parts of the network are updated. This is incremental forward chaining—each new fact derived on iteration $t$ must involve at least one fact from iteration $t-1$, so you never redo old work.

Additional heuristics: (1) Conjunct ordering (Most Restricted Variable heuristic) to minimize search space, (2) Early termination if the query is answered, (3) Caching intermediate results.

Q3: My Prolog backward chaining enters infinite loops on cyclic data. Why?
▼

Prolog uses depth-first backward chaining with no loop detection. If you write $path(X,Z) \gets path(X,Y), link(Y,Z)$ before $path(X,Z) \gets link(X,Z)$, Prolog expands $path(a,c)$ by trying the first clause, which generates subgoal $path(a,Y)$, which again tries the first clause... infinite recursion!

The fix: put the base case (ground facts) first. But even then, cycles in the data cause loops. Forward chaining doesn't have this problem because it derives facts bottom-up without recursion.

Modern Prologs use tabled logic programming (memoization): cache query results so that a repeated subgoal uses the cached answer instead of recursing again. This solves the loop problem while preserving soundness.

Q4: Why does Skolemization use a function F(x) for ∃y inside ∀x instead of a constant?
▼

When $\exists y$ is inside $\forall x$, the choice of $y$ can depend on $x$. Example: "everyone loves someone" = $\forall x \; \exists y \; Loves(x,y)$. Each person loves a potentially different someone.

If you replaced $y$ with a constant $c$, you'd say everyone loves the same entity $c$, which is strictly stronger (and wrong!). The correct Skolemization uses a function: $\forall x \; Loves(x, F(x))$ where $F$ is a Skolem function of $x$.

Standalone $\exists y$ (no enclosing $\forall$) has no dependencies, so a Skolem constant suffices. This is why variable scope matters in Skolemization.

Q5: Resolution proved my query by deriving the empty clause. How do I extract the variable bindings?
▼

Resolution proves $KB \models \alpha$ by showing $KB \land \neg\alpha$ is unsatisfiable. To extract variable bindings for existential queries like $\exists w \; Kills(w, Tuna)$, negate it as $\neg Kills(w, Tuna)$ and track the substitutions applied to $w$ throughout the proof.

When the empty clause is derived, compose all substitutions to find what $w$ was bound to. For instance, if the proof applied $w \leftarrow Curiosity$, you've found the answer.

However, resolution can produce "nonconstructive proofs" where no unique binding exists—the proof shows the query is true without pinpointing exactly which object satisfies it. In such cases, the query is entailed but not informative about the specific binding.

Forward Chaining: Data-Driven Explosion

Forward chaining starts with known facts and chains them forward through rules. Each new fact can trigger additional rules, creating a chain reaction of inferences.

The Crime Example: We start with 4 facts and 4 rules. Rules fire in sequence, deriving new facts each iteration until we derive $Criminal(West)$.

Backward Chaining: Goal-Directed Search

Backward chaining starts with the goal and works backward, asking "What rules could prove this?" It builds an AND/OR tree where OR nodes represent rule choices and AND nodes represent conjunctions of subgoals.

More focused than forward chaining (only expands relevant rules) but susceptible to infinite loops on cyclic data (requires tabling or occur check variants).

Unification: Pattern Matching for Variables

Unification finds the substitution that makes two expressions identical. Robinson's algorithm (with occur check) ensures the MGU is correct and most general.

The occur check prevents infinite structures: if you try to unify $x$ with $f(x)$, the check detects that $x$ appears in $f(x)$, so unification fails. Without the check, you could create $x = f(f(f(\cdots)))$ infinitely.
