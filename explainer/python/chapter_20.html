Classes & Objects

Understanding Object-Oriented Programming Foundations in Python

Why OOP? Programming Paradigms

Object-Oriented Programming (OOP) represents a fundamental shift in how we organize and think about code. Rather than viewing programs as sequences of imperative commands, OOP models software as a collection of interacting objects that encapsulate both data and behavior.

Mental Model: Paradigm Comparison

Imperative (Procedural): "Do this, then do that, then do that" — explicit sequences of steps. Good for simple scripts; becomes unwieldy for complex systems.

Functional: "Compose pure transformations" — functions as first-class values, immutability, avoiding state. Excellent for parallel processing and reasoning about correctness.

Object-Oriented: "Model the world as objects communicating via messages" — encapsulation, inheritance, polymorphism. Natural for modeling complex domains and managing large codebases.

Why Choose OOP?

  • Modularity: Code is organized into self-contained units (objects) with clear boundaries.
  • Reusability: Classes allow code patterns to be reused and extended (inheritance).
  • Maintainability: Changes to one object rarely break other parts of the system.
  • Scalability: Large systems become manageable through hierarchical organization.
  • Intuitive Modeling: Objects often correspond to real-world entities, making code more readable.

What are Classes and Objects?

Class as Blueprint

A class is a template or blueprint that defines a type of object. It specifies what attributes (data) and methods (behavior) objects of that type will have. Think of a class as an architectural blueprint for a house.

Object as Instance

An object is a concrete instantiation of a class — a specific realization of that blueprint. If the class is the blueprint, the object is the actual house built from that blueprint. Each object has its own state (attribute values) but shares behavior (methods) with other instances of the same class.

Real-World Analogy

Consider a Car class: it defines what attributes all cars have (make, model, color, speed) and what methods they can perform (accelerate, brake, honk). A specific instance like "my 2023 blue Toyota" is an object of the Car class with particular values for those attributes. 

# Class definition (blueprint)
class Car:
    def __init__(self, make, model, color):
        self.make = make
        self.model = model
        self.color = color
        self.speed = 0

    def accelerate(self):
        self.speed += 10
        return f"Car is now going {self.speed} km/h"

# Object instantiation (creating instances)
my_car = Car("Toyota", "Camry", "blue")
your_car = Car("Honda", "Civic", "red")

# Each object has its own state
print(my_car.color)      # Output: blue
print(your_car.color)    # Output: red

# But share the same methods
print(my_car.accelerate())   # Output: Car is now going 10 km/h
print(your_car.accelerate()) # Output: Car is now going 10 km/h

User-Defined Classes in Programming

The class Keyword

In Python, we use the class keyword to define a new class. By convention, class names use PascalCase (first letter capitalized, no underscores). 

class ClassName:
    """Docstring describing the class."""

    # Class variables (shared by all instances)
    class_variable = "shared value"

    # Instance method
    def method_name(self):
        pass

The __init__ Method (Constructor)

The __init__ method is the constructor — it runs automatically when a new object is created. It initializes the object's state by setting up its attributes.

The self Parameter

Every instance method must have self as its first parameter. self refers to the specific object calling the method. It allows the method to access and modify that object's attributes. When you call a method like obj.method(), Python automatically passes obj as the first argument. 

class Dog:
    def __init__(self, name, breed):
        self.name = name      # Instance attribute
        self.breed = breed    # Instance attribute
        self.tricks = []      # Instance attribute (initialized empty)

    def teach_trick(self, trick):
        self.tricks.append(trick)
        print(f"{self.name} learned {trick}!")

    def show_tricks(self):
        return f"{self.name} knows: {', '.join(self.tricks)}"

# Creating instances
dog1 = Dog("Buddy", "Golden Retriever")
dog2 = Dog("Max", "Pug")

# Each has independent state
dog1.teach_trick("sit")
print(dog1.show_tricks())  # Buddy knows: sit
print(dog2.show_tricks())  # Max knows:  (empty)

Deep Dive: What Happens During Object Creation?

▼

When you write obj = ClassName(args), Python performs two steps:

  1. __new__() creates a new, uninitialized instance of the class.
  2. __init__() is called on that instance to initialize it.

Normally you only override __init__. You rarely override __new__, which is useful only for immutable types or special metaclass work.

Access Convention: Visibility Levels

Python uses naming conventions (not enforced by the language) to indicate attribute visibility. This is by design — Python favors trust and convention over strict enforcement.

Public Attributes

Normal names (no leading underscores) are considered public. They're meant to be accessed directly.

Protected Attributes

Names with a single leading underscore (_attribute) signal "use this with care" — they're implementation details but not strictly private. It's a convention saying "this is an internal detail."

Private Attributes

Names with double leading underscores (__attribute) trigger name mangling. Python automatically renames them to _ClassName__attribute to prevent accidental name collisions in inheritance. This provides weak privacy—it prevents casual access but can be overridden. 

class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner              # Public
        self._internal_id = id(self)    # Protected (internal detail)
        self.__balance = balance        # Private (name-mangled)

    def deposit(self, amount):
        self.__balance += amount

    def get_balance(self):
        return self.__balance

account = BankAccount("Alice", 1000)

# Accessing public
print(account.owner)  # Works: Alice

# Accessing protected (works, but discouraged)
print(account._internal_id)  # Works, but shouldn't be relied upon

# Accessing private (not possible directly)
# print(account.__balance)  # AttributeError!

# But name mangling makes it technically accessible
print(account._BankAccount__balance)  # Works! (Don't do this)

# Proper access through methods
print(account.get_balance())  # Works: 1000

Deep Dive: Name Mangling Details

▼

Name mangling is Python's way of preventing accidental attribute name conflicts in inheritance. When you write self.__attr inside a class, Python rewrites it to self._ClassName__attr. 

class Parent:
    def __init__(self):
        self.__private = "parent"

class Child(Parent):
    def __init__(self):
        super().__init__()
        self.__private = "child"  # Doesn't overwrite parent's!

obj = Child()
print(obj._Parent__private)  # parent
print(obj._Child__private)   # child

# They're completely separate attributes due to name mangling

Class Variables and Methods

Instance Variables vs Class Variables

Instance variables (set in __init__) belong to individual objects. Class variables are shared by all instances — they're defined in the class body, outside methods. 

class Circle:
    pi = 3.14159  # Class variable (shared by all Circle instances)

    def __init__(self, radius):
        self.radius = radius  # Instance variable (unique to each Circle)

    def area(self):
        return Circle.pi * self.radius ** 2

circle1 = Circle(5)
circle2 = Circle(10)

# Both access the same class variable
print(circle1.pi)  # 3.14159
print(circle2.pi)  # 3.14159

# Instance variables are independent
print(circle1.radius)  # 5
print(circle2.radius)  # 10

# Modifying class variable affects all instances
Circle.pi = 3.14159265
print(circle1.pi)  # 3.14159265

@staticmethod

A static method doesn't have access to self (the instance) or cls (the class). It's just a regular function grouped with the class for organizational purposes.

@classmethod

A class method receives cls instead of self and can access/modify class variables. It's useful for alternative constructors or operations that affect the class as a whole. 

class Temperature:
    scale = "Celsius"  # Class variable

    def __init__(self, value):
        self.value = value

    @staticmethod
    def celsius_to_fahrenheit(celsius):
        """Convert without access to instance or class."""
        return (celsius * 9/5) + 32

    @classmethod
    def from_fahrenheit(cls, fahrenheit):
        """Alternative constructor using class variable."""
        celsius = (fahrenheit - 32) * 5/9
        return cls(celsius)

    @classmethod
    def set_scale(cls, scale):
        """Modify class variable."""
        cls.scale = scale

# Using static method
print(Temperature.celsius_to_fahrenheit(0))  # 32

# Using class method (alternative constructor)
temp = Temperature.from_fahrenheit(98.6)
print(temp.value)  # ~37

# Modifying class state
Temperature.set_scale("Kelvin")
print(Temperature.scale)  # Kelvin

Inspecting Objects: vars() and dir()

vars() Function

vars(obj) returns the __dict__ of an object — a dictionary of all instance attributes and their values. It's useful for debugging and introspection.

dir() Function

dir(obj) returns a list of all attributes and methods available on an object, including inherited ones. This shows the complete "interface" of an object. 

class Book:
    def __init__(self, title, author):
        self.title = title
        self.author = author

    def info(self):
        return f"{self.title} by {self.author}"

book = Book("1984", "George Orwell")

# vars() shows instance attributes
print(vars(book))
# Output: {'title': '1984', 'author': 'George Orwell'}

# dir() shows all attributes and methods
print(dir(book))
# Output: [..., '__class__', '__doc__', 'author', 'info', 'title', ...]

# Access __dict__ directly
print(book.__dict__)  # Same as vars(book)

Deep Dive: Built-in Attributes and Attribute Lookup Chain

▼

Every object has special attributes starting and ending with double underscores (dunders):

  • __dict__: Dictionary of instance attributes
  • __class__: The class object the instance belongs to
  • __doc__: The docstring
  • __module__: The module where the class is defined

When you access obj.attribute, Python searches in this order:

  1. Instance __dict__ (instance attributes)
  2. Class __dict__ (class attributes, methods)
  3. Parent class __dict__ (inherited attributes)
  4. Continue up the inheritance chain
  5. Raise AttributeError if not found
class Parent:
    class_var = "from parent"

class Child(Parent):
    def __init__(self):
        self.instance_var = "from instance"

obj = Child()

# Looking up obj.class_var:
# 1. Check obj.__dict__ -> not there
# 2. Check Child.__dict__ -> not there
# 3. Check Parent.__dict__ -> found! Returns "from parent"

print(obj.instance_var)  # Checks obj.__dict__ first
print(obj.class_var)     # Falls through to class hierarchy

Object Initialization and Default Values

__init__ is the constructor. It should initialize all attributes your objects need. You can provide default values for parameters, making some initialization optional. 

class Student:
    def __init__(self, name, student_id, grade=None, courses=None):
        self.name = name
        self.student_id = student_id
        self.grade = grade
        self.courses = courses if courses is not None else []

    def enroll(self, course):
        self.courses.append(course)

# Creating students with and without optional parameters
student1 = Student("Alice", 101)
student2 = Student("Bob", 102, grade="A", courses=["Math", "Physics"])

# Good practice: avoid mutable default arguments!
# Don't do this:
# def __init__(self, courses=[]):  # WRONG!
#     self.courses = courses
#
# All instances would share the same list! Use None and initialize instead.

Initialization Principles

  • Initialize ALL attributes in __init__, even if to None or empty values
  • Avoid mutable default arguments (use None and initialize in the method)
  • Keep __init__ simple — avoid heavy computation
  • Make it clear what attributes an object will have

Expert Insights: Advanced Object Concepts

Objects as Dictionaries: __dict__ and Direct Manipulation

▼

Python stores instance attributes in obj.__dict__, a regular dictionary. You can inspect, modify, or completely replace it: 

class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(3, 4)

# Inspect the dictionary
print(p.__dict__)  # {'x': 3, 'y': 4}

# Add attributes dynamically
p.__dict__['z'] = 0
print(p.z)  # 0 (Now the object has a z attribute)

# This flexibility is powerful but can lead to chaos!
# Use descriptors or __slots__ for more control.

__new__ vs __init__: Two-Phase Construction

▼

Object construction has two phases:

  1. __new__(cls) creates and returns a new uninitialized instance
  2. __init__(self) initializes that instance
class Singleton:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

    def __init__(self):
        self.value = None

# Only one instance ever created
s1 = Singleton()
s2 = Singleton()
print(s1 is s2)  # True! Same object

# Most of the time, override only __init__.
# Override __new__ only for immutable types or special patterns.

__del__ and Garbage Collection

▼

__del__ is a destructor called when an object is about to be garbage collected. Use it for cleanup (closing files, releasing resources), but be cautious:

  • You don't control when __del__ runs (depends on garbage collector)
  • Exceptions in __del__ are ignored
  • For reliable cleanup, use context managers (with statement) instead
class FileHandler:
    def __init__(self, filename):
        self.file = open(filename, 'r')

    def __del__(self):
        print(f"Closing file...")
        self.file.close()

# Bad way: relies on garbage collection timing
# fh = FileHandler('data.txt')
# # When will __del__ be called? Unpredictable!

# Good way: use context manager
with open('data.txt', 'r') as f:
    content = f.read()
# File is guaranteed to close when exiting the block

__slots__: Limiting Attributes and Saving Memory

▼

By default, Python stores instance attributes in a dictionary (__dict__), which uses memory. For classes with many instances, you can use __slots__ to pre-declare attributes and use a tuple instead: 

class Point:
    __slots__ = ('x', 'y')  # Only these attributes allowed

    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(3, 4)
print(p.x)  # Works

# p.z = 5  # AttributeError! Not in __slots__

# Memory savings: no __dict__!
print(hasattr(p, '__dict__'))  # False

# Advantages: less memory, faster attribute access
# Disadvantages: less flexibility, can't add dynamic attributes

Socratic Questions: Test Your Understanding

1. What's the difference between a class and an object?

▼
A class is a blueprint or template (abstract). An object is a concrete instantiation of that blueprint (concrete). A class is defined once; you can create many objects from it, each with its own state.

2. Why do all instance methods require self as the first parameter?

▼
self allows the method to know which object called it. When you write obj.method(arg), Python automatically passes obj as the first argument. Without self, methods wouldn't know which instance's attributes to access or modify.

3. What is name mangling and why does Python use double underscores?

▼
Name mangling rewrites __attr to _ClassName__attr to prevent accidental name collisions in inheritance. It's a weak privacy mechanism — it discourages casual access but doesn't truly prevent it. Python prefers convention over strict enforcement.

4. Why is it bad practice to use mutable defaults like def __init__(self, items=[])?

▼
Default arguments are evaluated once when the function is defined, not each time it's called. With a mutable default like a list, all instances share the same list! If one instance modifies it, all instances are affected. Use items=None and initialize with items if items is not None else [].

5. What does vars(obj) return?

▼
vars(obj) returns obj.__dict__, a dictionary of the object's instance attributes and their values. It's useful for introspection and debugging to see what state an object currently has.

6. Explain the order Python uses when looking up an attribute like obj.attr.

▼
Python follows the Attribute Resolution Order (ARO):
  1. Instance __dict__
  2. Class __dict__ (including methods)
  3. Parent class __dict__
  4. Continue up inheritance chain
  5. Raise AttributeError if not found

7. What's the difference between a class variable and an instance variable?

▼
Instance variables are unique to each object (defined in __init__). Class variables are shared by all instances (defined in the class body). Modifying a class variable affects all instances; modifying an instance variable affects only that object.

8. When would you use @staticmethod vs @classmethod?

▼
Use @staticmethod for utility functions grouped with the class but not needing instance or class access. Use @classmethod for alternative constructors or operations on class variables. Example: from_json() class method creates an instance from JSON; celsius_to_fahrenheit() static method is just a helper function.

9. What happens when you assign obj.new_attr = value dynamically?

▼
Python adds the attribute to obj.__dict__ immediately. In standard classes, you can add attributes dynamically at runtime. This flexibility is powerful but can cause confusion. To prevent it, use __slots__.

10. What's the difference between __new__ and __init__?

▼
__new__ creates a new uninitialized instance (called before __init__). __init__ initializes that instance. Override __init__ 99% of the time; override __new__ only for special patterns like immutable types or singletons.

11. Why is __del__ unreliable for resource cleanup?

▼
The garbage collector decides when to call __del__, and timing is unpredictable. You don't control if or when it runs. For reliable cleanup, use context managers (with statement) or explicit methods instead.

12. What do __slots__ do and when should you use them?

▼
__slots__ pre-declares which attributes instances can have, replacing the flexible __dict__ with a fixed tuple of attribute references. This saves memory (no dictionary overhead) and speeds up attribute access. Use them for classes with many instances where attributes are rarely added dynamically.
