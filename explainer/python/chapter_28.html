Synchronization & Thread Safety

Coordinate multiple threads safely and prevent data corruption from concurrent access

🧠 Mental Model: Thread-Safe Access

When multiple threads access shared data simultaneously, chaos ensues. Synchronization is the practice of coordinating thread access to ensure consistency.

Think of it like a bathroom: only one person can be inside at a time (mutual exclusion). Locks are the mechanism—one thread acquires the lock, uses the resource, then releases it for the next thread.

Race condition: Two threads read a value, both modify it, both write it back—the second write overwrites the first. Thread-safe code prevents this.

Synchronization and Race Conditions

What are Race Conditions?

A race condition occurs when multiple threads access shared data and the result depends on their timing:

import threading

counter = 0

def increment():
    global counter
    for _ in range(1_000_000):
        counter += 1  # NOT thread-safe!

# Create two threads
t1 = threading.Thread(target=increment)
t2 = threading.Thread(target=increment)

t1.start()
t2.start()
t1.join()
t2.join()

print(counter)  # Should be 2,000,000 but often much less!
# Why? The increment operation has 3 steps:
# 1. Read counter
# 2. Add 1
# 3. Write counter
# Threads interleave these steps, losing updates

The += operation is not atomic—it's three separate steps. If threads interleave, updates get lost.

Examples of Shared Resource Problems

Shared Counter

# WRONG: Shared counter without synchronization
class BankAccount:
    def __init__(self):
        self.balance = 0

    def deposit(self, amount):
        # Simulate some computation (to increase race condition likelihood)
        temp = self.balance
        temp += amount
        self.balance = temp  # Race condition here!

    def withdraw(self, amount):
        if self.balance >= amount:
            temp = self.balance
            temp -= amount
            self.balance = temp

# Two threads deposit concurrently
account = BankAccount()
account.balance = 100

t1 = threading.Thread(target=account.deposit, args=(50,))
t2 = threading.Thread(target=account.deposit, args=(50,))
t1.start()
t2.start()
t1.join()
t2.join()

print(account.balance)  # Should be 200, but might be 150!

Producer-Consumer

# WRONG: Producer and consumer without synchronization
items = []

def producer():
    for i in range(5):
        items.append(i)  # Not synchronized
        print(f"Produced {i}")

def consumer():
    while True:
        if items:
            item = items.pop(0)  # Race condition: list may be empty!
            print(f"Consumed {item}")

# This can crash if consumer tries to pop from empty list
Data Corruption vs Exceptions

Race conditions sometimes cause immediate crashes (pop from empty list) but often cause silent data corruption (wrong balance). Silent corruption is worse because you don't notice until the damage is done.

Synchronization Mechanisms

Lock (Mutual Exclusion)

A lock ensures only one thread can access a resource at a time:

import threading

class ThreadSafeCounter:
    def __init__(self):
        self.count = 0
        self.lock = threading.Lock()

    def increment(self):
        with self.lock:  # Acquire lock
            self.count += 1
            # Lock is released when exiting the with block

    def get_value(self):
        with self.lock:
            return self.count

# Usage
counter = ThreadSafeCounter()

def worker():
    for _ in range(100_000):
        counter.increment()

t1 = threading.Thread(target=worker)
t2 = threading.Thread(target=worker)

t1.start()
t2.start()
t1.join()
t2.join()

print(counter.get_value())  # Always 200,000 (correct!)

The with lock: pattern ensures the lock is acquired before the block and released after, even if an exception occurs.

Alternative (manual acquire/release):

lock = threading.Lock()

lock.acquire()
try:
    # Critical section
    pass
finally:
    lock.release()  # Ensure release even on exception

Always use the with statement—it's cleaner and safer.

RLock (Reentrant Lock)

A reentrant lock can be acquired multiple times by the SAME thread:

import threading

lock = threading.Lock()
rlock = threading.RLock()

def recursive_function_bad(n, lock):
    with lock:  # First acquire
        if n > 0:
            # Try to acquire again—DEADLOCK!
            recursive_function_bad(n - 1, lock)

def recursive_function_good(n, rlock):
    with rlock:  # Can acquire multiple times from same thread
        if n > 0:
            recursive_function_good(n - 1, rlock)

# recursive_function_bad(3, lock)  # Deadlocks!
recursive_function_good(3, rlock)  # Works fine

Use RLock for recursive functions or when a function might call another function holding the same lock.

Semaphore: Counting Access

A semaphore limits how many threads can access a resource simultaneously:

import threading

semaphore = threading.Semaphore(3)  # Allow 3 concurrent threads

def use_resource():
    with semaphore:
        print(f"{threading.current_thread().name} using resource")
        time.sleep(1)

# Create 5 threads, only 3 can use resource at a time
threads = [threading.Thread(target=use_resource) for _ in range(5)]
for t in threads:
    t.start()
for t in threads:
    t.join()

# BoundedSemaphore: prevents incrementing beyond initial value
bounded = threading.BoundedSemaphore(3)
# bounded.release()  # Would raise if count > initial value

Use semaphores for resource pooling (e.g., max 10 concurrent database connections).

Event: Signaling Between Threads

An event allows one thread to signal others:

import threading

event = threading.Event()

def worker():
    print("Worker waiting for event...")
    event.wait()  # Block until event is set
    print("Worker woken up!")

def signaler():
    time.sleep(2)
    print("Setting event")
    event.set()  # Wake up all waiting threads

t1 = threading.Thread(target=worker)
t2 = threading.Thread(target=signaler)

t1.start()
t2.start()
t1.join()
t2.join()

# Output:
# Worker waiting for event...
# (2 seconds later)
# Setting event
# Worker woken up!

Events are useful for start/stop signals and coordination.

Condition: Wait and Notify

A condition combines a lock with signaling, useful for producer-consumer patterns:

import threading
import time

condition = threading.Condition()
items = []

def producer():
    for i in range(5):
        with condition:
            items.append(i)
            print(f"Produced {i}")
            condition.notify_all()  # Wake up consumers
        time.sleep(0.5)

def consumer(id):
    while True:
        with condition:
            condition.wait_for(lambda: len(items) > 0)  # Wait until items available
            if items:
                item = items.pop(0)
                print(f"Consumer {id} consumed {item}")

prod = threading.Thread(target=producer)
cons1 = threading.Thread(target=consumer, args=(1,), daemon=True)
cons2 = threading.Thread(target=consumer, args=(2,), daemon=True)

prod.start()
cons1.start()
cons2.start()
prod.join()

Use wait() to block until signaled, and notify()` / `notify_all() to wake up waiting threads.

Deep Dive: queue.Queue (Thread-Safe Queue)

▼

For producer-consumer patterns, use queue.Queue instead of manually managing lists and locks:

import queue
import threading
import time

q = queue.Queue(maxsize=5)  # Thread-safe queue

def producer():
    for i in range(10):
        q.put(i)  # Block if queue full
        print(f"Produced {i}")
        time.sleep(0.5)

def consumer(id):
    while True:
        item = q.get()  # Block if queue empty
        if item is None:  # Sentinel value to stop
            break
        print(f"Consumer {id} got {item}")

prod = threading.Thread(target=producer)
cons1 = threading.Thread(target=consumer, args=(1,))
cons2 = threading.Thread(target=consumer, args=(2,))

prod.start()
cons1.start()
cons2.start()

prod.join()
# Signal consumers to stop
q.put(None)
q.put(None)
cons1.join()
cons2.join()

queue.Queue is thread-safe, handles blocking/unblocking, and is much simpler than manual condition variables.

Concurrent Patterns

Producer-Consumer Pattern

A classic pattern where producers generate data and consumers process it:

import queue
import threading
import time

def producer(q):
    for i in range(10):
        data = f"Item-{i}"
        q.put(data)
        print(f"Produced: {data}")
        time.sleep(0.5)
    q.put(None)  # Signal end

def consumer(q, id):
    while True:
        data = q.get()
        if data is None:
            break
        print(f"Consumer {id}: Processing {data}")
        time.sleep(1)  # Simulate work

q = queue.Queue()
prod = threading.Thread(target=producer, args=(q,))
cons1 = threading.Thread(target=consumer, args=(q, 1))
cons2 = threading.Thread(target=consumer, args=(q, 2))

prod.start()
cons1.start()
cons2.start()

prod.join()
cons1.join()
cons2.join()

Deadlock Prevention

Deadlock happens when threads wait for each other circularly. Prevention strategies:

# WRONG: Can deadlock
lock1 = threading.Lock()
lock2 = threading.Lock()

def thread1_code():
    with lock1:
        time.sleep(0.1)  # Give thread2 time to acquire lock2
        with lock2:  # Waits for lock2 (deadlock!)
            pass

def thread2_code():
    with lock2:
        time.sleep(0.1)
        with lock1:  # Waits for lock1 (deadlock!)
            pass

# CORRECT: Always acquire locks in the same order
lock_a = threading.Lock()
lock_b = threading.Lock()

def safe_thread1():
    with lock_a:
        with lock_b:
            pass

def safe_thread2():
    with lock_a:  # Same order as thread1
        with lock_b:
            pass

# Also: Use timeout to detect/break deadlocks
if lock.acquire(timeout=1):
    try:
        pass
    finally:
        lock.release()
else:
    print("Timeout: possible deadlock")

Deadlock prevention rules:

  • Always acquire locks in the same order across threads
  • Use timeouts to detect deadlock
  • Minimize lock scope (hold locks as briefly as possible)
  • Avoid nested locks when possible

Deep Dive: threading.local() for Thread-Local Storage

▼

Sometimes threads need their own isolated data without sharing:

import threading

# Global data shared by all threads (requires locking)
shared_counter = 0
lock = threading.Lock()

# Thread-local data (each thread gets its own copy)
thread_local = threading.local()

def worker(id):
    # Each thread has its own value
    thread_local.value = id * 100
    print(f"Thread {id}: {thread_local.value}")
    time.sleep(0.1)
    print(f"Thread {id} still has: {thread_local.value}")

threads = [threading.Thread(target=worker, args=(i,)) for i in range(3)]
for t in threads:
    t.start()
for t in threads:
    t.join()

# Output:
# Thread 0: 0
# Thread 1: 100
# Thread 2: 200
# Thread 0 still has: 0  (not affected by other threads)
# Thread 1 still has: 100
# Thread 2 still has: 200

Thread-local storage avoids the need for locks when each thread needs independent state.

Socratic Q&A: Test Your Understanding

What exactly happens in a race condition? Why does `counter += 1` fail in concurrent code? ▼

`counter += 1` is three operations: (1) read counter, (2) add 1, (3) write back. Threads interleave: Thread A reads 5, Thread B reads 5, Thread A writes 6, Thread B writes 6. Result: 6 instead of 7. One update is lost. A race condition where the outcome depends on timing.

Why is using `with lock:` better than manually calling `lock.acquire()` and `lock.release()`? ▼

The `with` statement guarantees release even if an exception occurs. Manual calls can skip release if an exception is raised, leading to deadlock. Also cleaner and less error-prone. Always use `with lock:` for context manager cleanup.

When would you use an RLock instead of a regular Lock? ▼

Use RLock when the same thread needs to acquire the lock multiple times (recursive functions). A regular Lock would deadlock on re-acquisition. RLock allows the same thread to acquire it multiple times; other threads still block.

How do Semaphore and Lock differ? When would you use each? ▼

Lock: only ONE thread can access (binary: 0 or 1). Semaphore: N threads can access (counting). Use Lock for simple mutual exclusion. Use Semaphore for resource pooling (e.g., max 10 concurrent database connections).

What causes a deadlock and how can you prevent it? ▼

Deadlock: circular waiting for locks. Thread A has Lock1, waits for Lock2. Thread B has Lock2, waits for Lock1. Prevention: (1) Always acquire locks in the same order, (2) Use timeouts to detect it, (3) Minimize lock scope, (4) Avoid nested locks. Order is most important.

How does an Event differ from a Condition? When would you use each? ▼

Event: Simple signaling (set/wait). One thread sets, others wake. Use for start/stop signals. Condition: Combines lock + signaling. Useful for producer-consumer (threads need to check state within a lock). Event is simpler; Condition is more powerful for complex synchronization.

Why is `queue.Queue` better than manually managing a list with locks? ▼

Queue is thread-safe, handles blocking (waits if empty/full), provides clean API (put/get). Manual lists require you to manage locks, conditions, and edge cases. Queue is battle-tested and less error-prone. Use Queue for producer-consumer patterns.

What is thread-local storage and why would you use it instead of a shared variable with a lock? ▼

Thread-local storage gives each thread its own isolated copy of a variable. No locking needed because threads never share. Use when each thread needs independent state (e.g., thread ID, connection, database handle). Avoids synchronization overhead.

If you acquire Lock A then Lock B, and another thread acquires them in reverse order, what happens? ▼

Potential deadlock. Thread 1: acquires A, waits for B. Thread 2: acquires B, waits for A. Circular dependency. Prevention: always acquire locks in the same order globally. In this case, both threads should acquire A first, then B.

How do you stop a worker thread cleanly without interrupting it mid-task? ▼

Use a sentinel value (None) or stop flag. Producer sends None to signal end. Consumer checks for it and breaks. Alternatively, use a stop_event: thread checks event periodically and exits if set. Don't forcefully kill threads—let them finish current task and check for stop signal.
