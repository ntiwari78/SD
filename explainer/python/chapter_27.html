Concurrency & Parallelism

Write responsive, high-performance applications by leveraging multiple CPU cores and asynchronous I/O

🧠 Mental Model: Concurrency vs Parallelism

Concurrency: Managing multiple tasks that can interleave. A single CPU core switches between tasks rapidly, making them appear simultaneous. Like a chef preparing multiple dishes by switching between them.

Parallelism: Executing multiple tasks simultaneously on different CPU cores. Like multiple chefs each working on a dish. Requires multiple processors.

Key insight: All parallelism is concurrent, but not all concurrency is parallel. Concurrency solves I/O delays; parallelism solves CPU-bound computation.

Concurrency and Parallelism Explained

Concurrency: Task Switching

Concurrency means multiple tasks are in progress simultaneously. The OS scheduler switches between them quickly, creating the illusion of parallelism on a single core.

# Simulating concurrency (single core, switching)
import time

start = time.time()

def task_a():
    for i in range(3):
        print("Task A")
        time.sleep(0.1)  # Simulates I/O wait

def task_b():
    for i in range(3):
        print("Task B")
        time.sleep(0.1)  # Simulates I/O wait

# Sequential (no concurrency): 0.6 seconds
task_a()
task_b()
print(f"Sequential: {time.time() - start:.2f}s")

# With concurrency, both run "together"
# While Task A waits for I/O, Task B can run

Concurrency is ideal for I/O-bound programs (network requests, file operations, database queries). While one task waits, another progresses.

Parallelism: Simultaneous Execution

Parallelism means tasks run simultaneously on different CPU cores. True parallelism requires multiple processors.

# Simulating parallelism (multiple cores, simultaneous)
import time

# CPU-bound task: crunching numbers
def cpu_intensive(n):
    result = 0
    for i in range(n):
        result += i ** 2
    return result

# Sequential: N seconds
start = time.time()
cpu_intensive(100_000_000)
cpu_intensive(100_000_000)
seq_time = time.time() - start
print(f"Sequential: {seq_time:.2f}s")

# With true parallelism (multiple processes): ~N/2 seconds
# Each process runs on a different core simultaneously

Parallelism is ideal for CPU-bound programs (heavy calculations, data processing). You need multiple cores to actually parallelize.

I/O-bound vs CPU-bound

I/O-bound Programs

What: Time is spent waiting for I/O (network, disk, database)

Examples: Web scraper, API client, file processor, database query app

Solution: Concurrency (threading, asyncio). Let other tasks run while waiting.

Speedup: Can be 10-100x with many concurrent tasks

CPU-bound Programs

What: Time is spent in CPU computation (no I/O waiting)

Examples: Machine learning, numerical computation, image processing

Solution: Parallelism (multiprocessing). Use multiple cores.

Speedup: Can be near-linear with number of cores (minus overhead)

Python's GIL: The Global Interpreter Lock

Python's GIL prevents true parallelism on a single process. Even with threading on multiple cores, only one thread executes Python bytecode at a time. Threading in Python works for I/O-bound tasks (the thread releases the GIL during I/O), but not for CPU-bound parallelism. Use multiprocessing for CPU parallelism (each process has its own GIL).

Threads: Lightweight Concurrency

What are Threads?

Threads are lightweight processes that share the same memory space. They're cheaper to create than full processes.

import threading
import time

def worker(name, duration):
    print(f"{name} started")
    time.sleep(duration)
    print(f"{name} done")

# Create and start threads
t1 = threading.Thread(target=worker, args=("Thread-1", 2))
t2 = threading.Thread(target=worker, args=("Thread-2", 1))

start = time.time()
t1.start()  # Start execution
t2.start()

# Wait for threads to finish
t1.join()  # Block until t1 completes
t2.join()  # Block until t2 completes

print(f"Total time: {time.time() - start:.2f}s")
# Sequential would be 3s. With threads: ~2s (parallelized I/O wait)

Threads run "concurrently" (interleaved on one core). For I/O-bound tasks, other threads can run while one waits.

Thread Properties

Threads have useful properties:

import threading

def worker():
    print(f"Running {threading.current_thread().name}")
    time.sleep(2)

# Create thread with a custom name
thread = threading.Thread(target=worker, name="MyWorker")

print(thread.name)  # 'MyWorker'
print(thread.is_alive())  # False (not started yet)

thread.start()
print(thread.is_alive())  # True (running)
print(thread.daemon)  # False (not a daemon)

thread.join()
print(thread.is_alive())  # False (finished)

# Daemon threads
daemon = threading.Thread(target=worker, daemon=True)
daemon.start()
# Program can exit while daemon is running (unlike normal threads)

Key properties:

  • name - Thread identifier
  • daemon - If True, program doesn't wait for this thread to finish
  • is_alive() - Check if thread is still running
  • join() - Wait for thread to complete

Launching Threads

Create and start threads with target function and arguments:

import threading

def work(task_id, delay):
    print(f"Task {task_id} starting")
    time.sleep(delay)
    print(f"Task {task_id} done")

# Method 1: Pass args
thread = threading.Thread(target=work, args=(1, 2))
thread.start()

# Method 2: Pass kwargs
thread = threading.Thread(target=work, task_id=2, delay=1)
thread.start()

# Method 3: Subclass Thread
class Worker(threading.Thread):
    def __init__(self, task_id):
        super().__init__()
        self.task_id = task_id

    def run(self):
        print(f"Task {self.task_id} starting")
        time.sleep(1)
        print(f"Task {self.task_id} done")

thread = Worker(3)
thread.start()

Passing Parameters to Threads

Use args and kwargs to pass data to thread functions:

import threading

def fetch_data(url, timeout=5):
    print(f"Fetching {url} with timeout {timeout}")
    # ... fetch logic ...

# Positional arguments
t1 = threading.Thread(target=fetch_data, args=("http://example.com",))

# Keyword arguments
t2 = threading.Thread(target=fetch_data, kwargs={"url": "http://example.com", "timeout": 10})

# Mixed
t3 = threading.Thread(target=fetch_data, args=("http://example.com",), kwargs={"timeout": 10})

t1.start()
t2.start()
t3.start()

Note: Tuples with one element need a trailing comma: (value,)

Deep Dive: The GIL (Global Interpreter Lock)

▼

Python's GIL is a mutex that protects access to Python objects. Only one thread can execute Python bytecode at a time, even on multi-core systems.

# GIL Impact: CPU-bound threading does NOT parallelize
import threading
import time

def cpu_work(n):
    """CPU-bound: lots of computation"""
    result = 0
    for i in range(n):
        result += i ** 2
    return result

# Sequential: 4 seconds
start = time.time()
cpu_work(100_000_000)
cpu_work(100_000_000)
print(f"Sequential: {time.time() - start:.2f}s")

# Threading: still ~4 seconds (GIL prevents parallelism)
start = time.time()
t1 = threading.Thread(target=cpu_work, args=(100_000_000,))
t2 = threading.Thread(target=cpu_work, args=(100_000_000,))
t1.start()
t2.start()
t1.join()
t2.join()
print(f"Threading: {time.time() - start:.2f}s")

# But I/O-bound threading DOES parallelize
# While Thread 1 waits for I/O, Thread 2 can run

Key points:

  • Threads don't parallelize CPU-bound work (GIL blocks it)
  • Threads DO parallelize I/O-bound work (GIL is released during I/O)
  • Multiprocessing bypasses the GIL (each process has its own)
  • Python 3.13+ has "free-threaded" mode (no GIL), but it's still experimental

Deep Dive: concurrent.futures

▼

concurrent.futures provides high-level abstractions for thread and process pools:

from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor
import time

def work(task_id):
    print(f"Task {task_id} started")
    time.sleep(1)
    return f"Task {task_id} done"

# ThreadPoolExecutor for I/O-bound
with ThreadPoolExecutor(max_workers=3) as executor:
    futures = [executor.submit(work, i) for i in range(5)]
    for future in futures:
        print(future.result())  # Wait and get result

# ProcessPoolExecutor for CPU-bound
with ProcessPoolExecutor(max_workers=3) as executor:
    futures = [executor.submit(work, i) for i in range(5)]
    for future in futures:
        print(future.result())

# map convenience method
with ThreadPoolExecutor(max_workers=3) as executor:
    results = executor.map(work, range(5))
    for result in results:
        print(result)

Executors manage thread/process pools, handle job queues, and simplify result collection.

Deep Dive: Asyncio (Cooperative Multitasking)

▼

Asyncio is for cooperative concurrency where tasks voluntarily yield control (using await):

import asyncio

async def fetch_data(url):
    print(f"Fetching {url}")
    await asyncio.sleep(2)  # Simulate I/O
    return f"Data from {url}"

async def main():
    # Run multiple coroutines concurrently
    tasks = [
        fetch_data("http://example1.com"),
        fetch_data("http://example2.com"),
        fetch_data("http://example3.com"),
    ]
    results = await asyncio.gather(*tasks)
    for result in results:
        print(result)

# Run the event loop
asyncio.run(main())

# Advantages over threading:
# - No GIL issues (single-threaded)
# - Lighter weight than threads
# - Explicit control with 'await'
# - Better for I/O-heavy applications

Asyncio is more efficient than threading for I/O-bound code with many concurrent operations (thousands of connections).

Which to Use: Decision Tree

Quick Decision Matrix

Problem Type Solution Why
I/O-bound (few tasks) threading Simple, light, works well
I/O-bound (many tasks, 1000+) asyncio Lighter, more scalable
CPU-bound multiprocessing Bypasses GIL, true parallelism
General purpose concurrent.futures Higher-level abstraction
Complex async code asyncio More control, explicit flow

Step-by-Step Decision Process

  1. Is it CPU-bound? (mostly computation)
    • YES → Use multiprocessing
    • NO → Continue
  2. How many concurrent tasks?
    • < 50 tasks → threading (simple)
    • 1000+ tasks → asyncio (scalable)
    • Otherwise → threading
  3. Can you restructure code to async/await?
    • YES → asyncio (better)
    • NO → threading (pragmatic)

Real-World Examples

Web Scraper (I/O-bound, few tasks): threading

import threading
import requests
from concurrent.futures import ThreadPoolExecutor

def download_page(url):
    response = requests.get(url)  # I/O blocking
    return len(response.text)

urls = ["http://example.com"] * 10

with ThreadPoolExecutor(max_workers=5) as executor:
    results = executor.map(download_page, urls)
    print(f"Downloaded {sum(results)} characters")

Data Processing (CPU-bound): multiprocessing

from multiprocessing import Pool

def process_data(chunk):
    # CPU-intensive work
    return sum(x ** 2 for x in chunk)

data = [[1, 2, 3]] * 10

with Pool(processes=4) as pool:
    results = pool.map(process_data, data)
    total = sum(results)

Web Server (I/O-bound, many tasks): asyncio

import asyncio

async def handle_request(request):
    await asyncio.sleep(0.1)  # Simulate database query
    return "Response"

# In reality, you'd use a web framework like FastAPI
# which handles asyncio internally

Socratic Q&A: Test Your Understanding

Can Python threads achieve true parallelism on multi-core systems? Why or why not? ▼

No, due to the GIL (Global Interpreter Lock). Only one thread can execute Python bytecode at a time, even on multi-core systems. The GIL prevents true parallelism for CPU-bound work. However, threads DO parallelize I/O-bound work because the GIL is released during I/O operations (network, disk, etc.), allowing other threads to run while one waits.

What's the difference between `t.start()` and `t.run()` for threads? ▼

`t.start()` launches the thread as a separate concurrent execution. `t.run()` executes the target function directly in the CURRENT thread (blocking). Always use `start()`. Using `run()` defeats the purpose of threading—the code runs sequentially, not concurrently.

What does `t.join()` do? What happens if you don't call it? ▼

`t.join()` blocks the calling thread until thread `t` completes. Without it, your program can exit while threads are still running. If the thread is NOT a daemon, the program waits anyway. If it IS a daemon, it terminates when the main program exits, potentially losing data or leaving tasks incomplete.

What's a daemon thread? When would you use one? ▼

A daemon thread is one that exits when the main program exits, regardless of whether it finished. Use for background tasks that aren't critical (logging, monitoring, cleanup). Normal threads force the program to wait for them; daemon threads don't. Critical work should use normal threads.

When should you use multiprocessing instead of threading? ▼

Use multiprocessing for CPU-bound work. Each process has its own GIL, enabling true parallelism on multi-core systems. Threading only helps for I/O-bound work (where the GIL is released during I/O). Multiprocessing is heavier (more memory, slower startup) but achieves actual parallelism for computation.

Why is asyncio preferred over threading for I/O-bound apps with thousands of concurrent connections? ▼

Each thread consumes memory and OS resources. Creating 10,000 threads is expensive. Asyncio uses a single thread with an event loop managing many "lightweight" coroutines. Coroutines are much cheaper than threads: you can have 100,000+ without resource exhaustion. Asyncio is more scalable for high-concurrency I/O workloads.

What's the difference between `concurrent.futures.ThreadPoolExecutor` and manually creating threads? ▼

ThreadPoolExecutor manages a pool of reusable threads and a job queue. Manual threads require you to create, start, and join each one. Executors handle thread lifecycle, reuse threads to save creation overhead, and provide convenient methods like `map()` and `submit()`. Use ThreadPoolExecutor for cleaner, more efficient code.

How do you pass arguments to a thread's target function? ▼

Use `args` for positional arguments and `kwargs` for keyword arguments. Example: `Thread(target=func, args=(1, 2), kwargs={'x': 3})`. Args are passed as a tuple (note: single-element tuples need a trailing comma: `args=(value,)`). Kwargs are passed as a dict.

Can you safely share mutable objects between threads? What precautions are needed? ▼

Sharing mutable objects is risky—concurrent modifications can corrupt data (race conditions). Use synchronization primitives like `Lock`, `RLock`, `Semaphore`, `Event`, or `Condition` from the `threading` module to protect shared access. Or use thread-safe collections like `queue.Queue`. This is covered in Chapter 28 (Synchronization & Thread Safety).

What happens if a thread raises an unhandled exception? ▼

The thread terminates, but the program continues. The exception doesn't propagate to the main thread. To catch exceptions from threads, wrap the target function in try-except or use executor's `result()` method which re-raises exceptions. This is important: you must explicitly handle exceptions in threads.
