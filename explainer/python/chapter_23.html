Dataclasses, Type Hints, and Metaclasses for Production Code

Beyond Basic OOP: Modern Python Features

While Chapters 20-22 covered OOP fundamentals, modern Python offers powerful features for writing cleaner, more maintainable code. This chapter explores dataclasses for simple data containers, type hints for clarity, and metaclasses for advanced use cases.

When to Use Modern Features

  • Dataclasses: When you need simple data containers with less boilerplate
  • Type hints: Always! They improve code clarity and enable IDE support
  • Metaclasses: Rarely. Only for frameworks and APIs (ORMs, validation, decorators)

@dataclass: Auto-Generating __init__, __repr__, __eq__

The @dataclass decorator automatically generates boilerplate methods (__init__, __repr__, __eq__, etc.) based on class attributes.

Basic Dataclass

from dataclasses import dataclass

# Without dataclass: lots of boilerplate
class PersonManual:
    def __init__(self, name: str, age: int):
        self.name = name
        self.age = age

    def __repr__(self):
        return f"PersonManual(name={self.name!r}, age={self.age!r})"

    def __eq__(self, other):
        if not isinstance(other, PersonManual):
            return NotImplemented
        return self.name == other.name and self.age == other.age

# With dataclass: clean and simple
@dataclass
class Person:
    name: str
    age: int

# All methods auto-generated!
p1 = Person("Alice", 30)
p2 = Person("Alice", 30)

print(repr(p1))      # Person(name='Alice', age=30)
print(p1 == p2)      # True (eq auto-generated)
print(p1.name)       # Alice

Default Values and field()

from dataclasses import dataclass, field

@dataclass
class Student:
    name: str
    age: int
    courses: list = field(default_factory=list)  # Mutable default!

# Using field() for mutable defaults is critical to avoid
# the shared mutable default problem from regular classes

s1 = Student("Alice", 20)
s2 = Student("Bob", 21)

s1.courses.append("Math")
print(s1.courses)  # ['Math']
print(s2.courses)  # []  (Independent!)

# With simple default, would be shared:
# courses: list = []  # WRONG! All instances share this list

Frozen Dataclasses (Immutable)

from dataclasses import dataclass

@dataclass(frozen=True)
class Point:
    x: float
    y: float

p = Point(1, 2)
print(p)  # Point(x=1, y=2)

# p.x = 3  # FrozenInstanceError! Can't modify

# Frozen dataclasses are hashable (can be dict keys)
points = {p: "origin"}
print(points[p])  # origin

# frozen=True makes them immutable

class ImmutablePoint:

Type Hints: Type Safety Without Runtime Enforcement

Type hints are optional annotations that document expected types. Python doesn't enforce them at runtime, but IDEs, linters, and type checkers (like mypy) use them.

Basic Type Hints

# Function signatures
def greet(name: str) -> str:
    return f"Hello, {name}!"

# Variable annotations
age: int = 25
is_student: bool = True

# Class attributes
class Person:
    name: str
    age: int

    def __init__(self, name: str, age: int):
        self.name = name
        self.age = age

# Type hints don't enforce at runtime
result = greet(123)  # Python allows this!
print(result)  # "Hello, 123!"
# But mypy would complain about the type mismatch

Generic Types from typing Module

from typing import List, Dict, Tuple, Optional, Union

# List[T]: list of T
def add_numbers(numbers: List[int]) -> int:
    return sum(numbers)

# Dict[K, V]: dictionary with key type K, value type V
def word_counts(text: str) -> Dict[str, int]:
    counts = {}
    for word in text.split():
        counts[word] = counts.get(word, 0) + 1
    return counts

# Tuple[T1, T2, ...]: specific tuple types
def get_point() -> Tuple[float, float]:
    return (1.0, 2.0)

# Optional[T] = Union[T, None]: value or None
def find_user(user_id: int) -> Optional[str]:
    if user_id == 1:
        return "Alice"
    return None

# Union[T1, T2]: one of multiple types
def process(value: Union[int, str]) -> None:
    if isinstance(value, int):
        print(f"Number: {value}")
    else:
        print(f"String: {value}")

Generics and TypeVar

from typing import TypeVar, Generic, List

T = TypeVar('T')  # Type variable

# Generic class
class Stack(Generic[T]):
    def __init__(self):
        self.items: List[T] = []

    def push(self, item: T) -> None:
        self.items.append(item)

    def pop(self) -> T:
        return self.items.pop()

# Can specialize the generic
int_stack: Stack[int] = Stack()
int_stack.push(1)
int_stack.push(2)

str_stack: Stack[str] = Stack()
str_stack.push("hello")

Special Typing Constructs

from typing import Literal, Final, TypedDict, Annotated

# Literal: specific allowed values
def set_status(status: Literal["active", "inactive", "pending"]) -> None:
    pass

# Final: immutable variable or method
MAX_RETRIES: Final[int] = 3

class Base:
    def method(self) -> Final[str]:
        return "Don't override me"

# TypedDict: dictionary with known keys and types
class UserDict(TypedDict):
    name: str
    age: int
    email: str

user: UserDict = {"name": "Alice", "age": 30, "email": "alice@example.com"}

# Annotated: attach metadata to types
from typing import Annotated

UserId = Annotated[int, "positive integer"]

def get_user(user_id: UserId) -> str:
    return f"User {user_id}"

Type Hints Best Practices

  • Use type hints on public functions and class methods
  • Use Optional[T] for values that might be None
  • Use Union[T1, T2] for multiple possible types
  • Don't over-complicate — simple hints are better than complex ones
  • Run mypy to check types statically

Protocols: Structural Subtyping Without Inheritance

Protocol from the typing module defines a structural type (duck typing with type hints). A class matches a protocol if it has the required methods, even without explicit inheritance. 

from typing import Protocol

class Drawable(Protocol):
    """Any object with a draw method is Drawable"""
    def draw(self) -> None: ...

class Circle:
    def draw(self) -> None:
        print("Drawing circle")

class Square:
    def draw(self) -> None:
        print("Drawing square")

# Both implement Drawable implicitly
def render(obj: Drawable) -> None:
    obj.draw()

render(Circle())   # Works!
render(Square())   # Works!

# No inheritance needed!

Protocols vs ABCs

Protocols: Structural (duck typing with types). No inheritance required. ABCs: Nominal (explicit inheritance required). Enforces contracts. 

from abc import ABC, abstractmethod
from typing import Protocol

# ABC: must explicitly inherit
class Animal(ABC):
    @abstractmethod
    def speak(self) -> str: ...

# This doesn't work without inheriting
class Dog:
    def speak(self) -> str:
        return "Woof"

# obj = Dog()
# isinstance(obj, Animal)  # False!

# Protocol: structure is enough
class Speakable(Protocol):
    def speak(self) -> str: ...

class Cat:
    def speak(self) -> str:
        return "Meow"

cat = Cat()
# isinstance(cat, Speakable)  # False, but type checker accepts it
# mypy knows Cat matches Speakable structurally

Metaclasses: Classes That Create Classes

A metaclass is a class whose instances are classes. It's "a class factory." type is Python's default metaclass. Most of the time, you never need custom metaclasses, but they enable powerful metaprogramming in frameworks.

What is type?

# type is the metaclass of all classes
class MyClass:
    pass

print(type(MyClass))         # 
print(type(MyClass()))       # 

# You can create classes dynamically with type()
MyDynamicClass = type('MyDynamicClass', (), {'attr': 'value'})
obj = MyDynamicClass()
print(obj.attr)  # value

Custom Metaclass

# Define a custom metaclass
class LoggingMeta(type):
    def __new__(mcs, name, bases, namespace):
        print(f"Creating class {name}")
        return super().__new__(mcs, name, bases, namespace)

# Use the metaclass
class MyClass(metaclass=LoggingMeta):
    def method(self):
        pass

# Output: Creating class MyClass

obj = MyClass()
print(type(obj))  # 
print(type(MyClass))  #

Practical Example: Singleton Metaclass

class SingletonMeta(type):
    """Metaclass that enforces singleton pattern"""
    _instances = {}

    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            cls._instances[cls] = super().__call__(*args, **kwargs)
        return cls._instances[cls]

class Database(metaclass=SingletonMeta):
    def __init__(self):
        self.connection = "Connected to DB"

# Only one instance ever created
db1 = Database()
db2 = Database()

print(db1 is db2)  # True! Same object
print(db1.connection)  # Connected to DB

Deep Dive: When to Use Metaclasses (Almost Never)

▼

Metaclasses are powerful but confusing. Use them only for:

  • ORMs (Django, SQLAlchemy): Dynamically create database model classes
  • Validation frameworks: Enforce class-level constraints
  • API frameworks: Auto-generate routes, serializers, etc.

For 99% of use cases, use @dataclass, descriptors, or __init_subclass__ instead. 

# Usually better: __init_subclass__
class Plugin:
    subclasses = []

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        Plugin.subclasses.append(cls)

class PluginA(Plugin):
    pass

class PluginB(Plugin):
    pass

print(Plugin.subclasses)  # [PluginA, PluginB]
# Simpler than metaclasses for most use cases!

__init_subclass__: Lightweight Alternative

class Base:
    def __init_subclass__(cls, greeting="Hello", **kwargs):
        super().__init_subclass__(**kwargs)
        cls.greeting = greeting

class EnglishGreeter(Base, greeting="Hello"):
    pass

class SpanishGreeter(Base, greeting="Hola"):
    pass

print(EnglishGreeter.greeting)  # Hello
print(SpanishGreeter.greeting)  # Hola

# Cleaner than metaclasses for customizing subclasses!

In standard library Yes No No

import attr

Expert Insights: Advanced Metaprogramming

ParamSpec for Decorator Typing

▼

Typing decorators is tricky because they wrap functions with different signatures. ParamSpec captures the original function's parameters. 

from typing import ParamSpec, Callable, TypeVar

P = ParamSpec('P')
T = TypeVar('T')

def timing_decorator(func: Callable[P, T]) -> Callable[P, T]:
    def wrapper(*args: P.args, **kwargs: P.kwargs) -> T:
        print("Starting...")
        result = func(*args, **kwargs)
        print("Finished!")
        return result
    return wrapper

@timing_decorator
def add(a: int, b: int) -> int:
    return a + b

result = add(1, 2)  # Type checker knows result is int

Descriptor-Based Validation

▼

Combine descriptors with validation for powerful attribute control. 

class PositiveInt:
    def __set_name__(self, owner, name):
        self.name = name

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return obj.__dict__.get(self.name, 0)

    def __set__(self, obj, value):
        if not isinstance(value, int) or value <= 0:
            raise ValueError(f"{self.name} must be a positive int")
        obj.__dict__[self.name] = value

class Person:
    age = PositiveInt()
    height = PositiveInt()

p = Person()
p.age = 30       # OK
# p.age = -5     # ValueError!
# p.age = "30"   # ValueError!

type.__new__ Flow and Metaclass Conflict Resolution

▼

When a class is created, type.__new__ and type.__init__ are called. In multiple metaclass inheritance, Python needs to resolve which metaclass to use. 

class MetaA(type):
    pass

class MetaB(type):
    pass

# This works: MetaA is a subclass of type
class Child(metaclass=MetaA):
    pass

# This fails: conflicting metaclasses
# class Conflict(Child, metaclass=MetaB):
#     pass
# TypeError: metaclass conflict

# Solution: create a metaclass that inherits from both
class MetaC(MetaA, MetaB):
    pass

class Works(Child, metaclass=MetaC):
    pass

__prepare__: Controlling the Class Namespace

▼

__prepare__ in a metaclass returns the namespace (dictionary) used during class body execution. This lets you intercept class definitions. 

class OrderedMeta(type):
    @classmethod
    def __prepare__(mcs, name, bases):
        # Use an ordered dict to track definition order
        from collections import OrderedDict
        return OrderedDict()

    def __new__(mcs, name, bases, namespace):
        # namespace is an OrderedDict, preserving order
        return super().__new__(mcs, name, bases, dict(namespace))

class MyClass(metaclass=OrderedMeta):
    field1 = 1
    field2 = 2
    field3 = 3

# Can know definition order of attributes

Socratic Questions: Test Your Understanding

1. What does @dataclass do and why is it useful?

▼
@dataclass auto-generates __init__, __repr__, __eq__, and other methods based on class attributes. It eliminates boilerplate for simple data containers, making code cleaner and less error-prone. Use it instead of manual class definitions for most data containers.

2. Why use field(default_factory=list) instead of courses: list = []?

▼
Default arguments are evaluated once when the class is defined. Using courses: list = [] means all instances share the same list! Using field(default_factory=list) creates a new list for each instance. This prevents the mutable default argument bug.

3. What is the purpose of type hints?

▼
Type hints document expected types and enable static analysis. Python doesn't enforce them at runtime, but IDEs use them for autocomplete, and type checkers (mypy) use them to find bugs. They make code self-documenting and catch errors before runtime.

4. What's the difference between Optional[T] and Union[T, None]?

▼
They're the same! Optional[T] is shorthand for Union[T, None]. Use Optional[T] when a value might be None; use Union[T1, T2, ...] for multiple possible types.

5. What is a Protocol and why use it instead of ABC?

▼
Protocol uses structural subtyping: a class matches if it has the required methods, even without inheritance. Use Protocol for duck typing with type safety. Use ABC when you want to enforce explicit inheritance or prevent instantiation of incomplete classes.

6. What is a metaclass and why are they rarely needed?

▼
A metaclass is a class whose instances are classes (a "class factory"). They're powerful but confusing. Most use cases are better solved with @dataclass, descriptors, or __init_subclass__. Only use metaclasses in frameworks (ORMs, validation APIs).

7. What does frozen=True do in a dataclass?

▼
frozen=True makes instances immutable — you can't modify attributes after creation. This also makes the instance hashable (can be used as dict keys or in sets). Useful for data that shouldn't change or needs to be used as dictionary keys.

8. Why doesn't Python enforce type hints at runtime?

▼
Enforcing type hints at runtime adds overhead and goes against Python's "ask forgiveness, not permission" philosophy. Instead, Python relies on static type checkers (mypy) to catch errors before runtime. This keeps Python fast while still offering type safety benefits.

9. When would you use __init_subclass__ instead of a metaclass?

▼
__init_subclass__ is called when a subclass is created, allowing you to customize subclasses. It's much simpler than metaclasses for most use cases (plugin registration, validation, etc.). Use __init_subclass__ when you just need to react to subclass creation; use metaclasses only when you need to control how classes are created.

10. What are the trade-offs between dataclass, attrs, and Pydantic?

▼
dataclass: Built-in, simple, no validation. Good for basic data containers. attrs: More features (validation, evolution), external dependency. Good for complex dataclasses. Pydantic: Strong validation, type coercion, JSON support. Best for web APIs and external data. Choose dataclass first; upgrade to attrs or Pydantic if you need more features.

11. How do you type a generic class that works with multiple types?

▼
Use TypeVar to create a type variable, then inherit from Generic[T]. Example: class Stack(Generic[T]): def push(self, item: T) -> None: ... This allows specialized types like Stack[int] or Stack[str] while sharing implementation.

12. What does __prepare__ do in a metaclass?

▼
__prepare__ returns the namespace dictionary used during class body execution. Overriding it lets you use a custom dictionary (like OrderedDict) to track the order or behavior of class attributes as they're defined. Very advanced, rarely needed.
