Decorators & Metaprogramming

Learn how to write functions that modify other functions. Master decorators, metaprogramming, and the tools Python provides for introspection and code transformation.

Mental Model: Functions as First-Class Values

The Core Idea: In Python, functions are objects. You can pass them as arguments, return them from other functions, and store them in data structures. A decorator is simply a function that takes a function as input and returns a modified function. The `@decorator` syntax is syntactic sugar for function composition: `@dec def func(): ...` is equivalent to `func = dec(func)`.

Key Insight: Decorators enable cross-cutting concerns: logging, caching, validation, timing, and more—without modifying the original function. They work through closures, capturing the original function and wrapping it with new behavior.

Practical Consequence: Understanding decorators unlocks advanced patterns: parameterized decorators, stacked decorators, class decorators, and the descriptor protocol. Metaprogramming tools like `globals()`, `getattr()`, and `__dict__` allow code to inspect and modify itself at runtime.

What Are Decorators?

Functions as Objects: The Foundation

Python treats functions as first-class objects. You can assign them to variables, pass them as arguments, and return them from other functions.

Functions as objects def greet(name): return f"Hello, {name}!" # Assign function to variable say_hello = greet print(say_hello("Alice")) # Hello, Alice! # Pass function as argument def call_twice(func, value): return [func(value), func(value)] result = call_twice(greet, "Bob") print(result) # ['Hello, Bob!', 'Hello, Bob!']

Basic Decorator Pattern

A decorator is a function that takes a function as input, wraps it with additional behavior, and returns the modified function.

Basic decorator structure def timing_decorator(func): def wrapper(*args, **kwargs): print(f"Calling {func.__name__}") result = func(*args, **kwargs) print(f"Finished {func.__name__}") return result return wrapper # Manual application def add(a, b): return a + b add = timing_decorator(add) add(5, 3) # Prints: Calling add, Finished add

The @ Syntax: Syntactic Sugar

The `@decorator` syntax is equivalent to reassigning the function. It's just cleaner.

Using @ syntax def timing_decorator(func): def wrapper(*args, **kwargs): print(f"Calling {func.__name__}") result = func(*args, **kwargs) return result return wrapper @timing_decorator def add(a, b): return a + b # Equivalent to: add = timing_decorator(add) add(5, 3) # Prints: Calling add
Insight: `@timing_decorator` is just syntactic sugar. Understanding that it's function reassignment makes decorators less magical.

functools.wraps: Preserving Function Metadata

When you decorate a function, the wrapper function replaces it. This loses metadata like `__name__`, `__doc__`, and function signature. `functools.wraps` copies this metadata.

Without functools.wraps def my_decorator(func): def wrapper(*args, **kwargs): return func(*args, **kwargs) return wrapper @my_decorator def add(a, b): """Add two numbers.""" return a + b print(add.__name__) # 'wrapper' - Lost the original name! print(add.__doc__) # None - Lost the docstring!
With functools.wraps (correct) import functools def my_decorator(func): @functools.wraps(func) def wrapper(*args, **kwargs): return func(*args, **kwargs) return wrapper @my_decorator def add(a, b): """Add two numbers.""" return a + b print(add.__name__) # 'add' - Correct! print(add.__doc__) # 'Add two numbers.' - Correct!
Best Practice: Always use `@functools.wraps` in decorators to preserve metadata. This is essential for tools like `help()` and `inspect`.

Deep Dive: Parameterized Decorators

+

Sometimes you want to configure a decorator with parameters. This requires an extra layer of nesting—a decorator factory.

Parameterized decorator import functools def retry(max_attempts=3): """Decorator factory that creates a retry decorator.""" def decorator(func): @functools.wraps(func) def wrapper(*args, **kwargs): for attempt in range(max_attempts): try: return func(*args, **kwargs) except Exception as e: if attempt == max_attempts - 1: raise print(f"Attempt {attempt + 1} failed, retrying...") return wrapper return decorator @retry(max_attempts=5) def unstable_function(): import random if random.random() < 0.7: raise ValueError("Failed!") return "Success!"

The call stack: `@retry(5)` calls `retry(5)`, which returns a decorator function, which then decorates the target function.

Decorating Functions with Arguments

*args and **kwargs: Handling Any Signature

Decorators must work with functions of any signature. Use `*args` and `**kwargs` to capture and pass all arguments.

Decorator with *args and **kwargs import functools def logging_decorator(func): @functools.wraps(func) def wrapper(*args, **kwargs): print(f"Calling {func.__name__}") print(f" args: {args}") print(f" kwargs: {kwargs}") result = func(*args, **kwargs) print(f" result: {result}") return result return wrapper @logging_decorator def greet(name, greeting="Hello"): return f"{greeting}, {name}!" greet("Alice", greeting="Hi") # Output: # Calling greet # args: ('Alice',) # kwargs: {'greeting': 'Hi'} # result: Hi, Alice!

Multiple Decorators: Stacking and Order

You can apply multiple decorators to a single function. They execute bottom-to-top on definition, and the wrapping order determines call order.

Stacking decorators import functools def log_calls(func): @functools.wraps(func) def wrapper(*args, **kwargs): print(f"-> {func.__name__}") result = func(*args, **kwargs) print(f"<- {func.__name__}") return result return wrapper def timing(func): @functools.wraps(func) def wrapper(*args, **kwargs): import time start = time.time() result = func(*args, **kwargs) elapsed = time.time() - start print(f" Took {elapsed:.4f}s") return result return wrapper @log_calls @timing def slow_function(): import time time.sleep(0.1) return 42 slow_function() # Output: # -> slow_function # Took 0.1000s # <- slow_function

The order matters: `@log_calls` on top means logging wraps timing, so timing runs inside the logging wrapper.

Class Decorators: Decorating Entire Classes

Decorators can also work on classes, allowing you to modify class behavior globally.

Class decorator example def add_repr(cls): """Decorator that adds a __repr__ method to a class.""" def __repr__(self): attrs = ', '.join(f"{k}={v!r}" for k, v in self.__dict__.items()) return f"{cls.__name__}({attrs})" cls.__repr__ = __repr__ return cls @add_repr class Point: def __init__(self, x, y): self.x = x self.y = y p = Point(3, 4) print(p) # Point(x=3, y=4)
Use Case: Class decorators are powerful for frameworks. For example, `@dataclass` from the `dataclasses` module automatically generates `__init__`, `__repr__`, and other methods.

Built-in Decorators in Python

@property: Creating Getter/Setter Methods

The `@property` decorator turns a method into an attribute-like accessor without exposing internal state.

@property example class Temperature: def __init__(self, celsius): self._celsius = celsius @property def celsius(self): return self._celsius @property def fahrenheit(self): return self._celsius * 9/5 + 32 @celsius.setter def celsius(self, value): if value < -273.15: raise ValueError("Invalid temperature") self._celsius = value temp = Temperature(25) print(temp.celsius) # 25 (accessed like attribute, not method) print(temp.fahrenheit) # 77.0 temp.celsius = 30 # Uses setter

@staticmethod and @classmethod: Method Types

`@staticmethod` creates methods that don't need `self` or `cls`. `@classmethod` passes the class as the first argument.

@staticmethod and @classmethod class MathOps: @staticmethod def add(a, b): """No access to instance or class.""" return a + b @classmethod def from_string(cls, s): """Factory method using the class.""" x, y = map(int, s.split(',')) return cls(x, y) def __init__(self, x, y): self.x = x self.y = y # staticmethod print(MathOps.add(5, 3)) # 8 # classmethod result = MathOps.from_string("10,20") # Creates MathOps(10, 20)

@functools.lru_cache: Memoization

LRU (Least Recently Used) cache decorator memoizes function results, avoiding recomputation.

@functools.lru_cache import functools @functools.lru_cache(maxsize=128) def fibonacci(n): if n < 2: return n return fibonacci(n-1) + fibonacci(n-2) # First call: recomputes print(fibonacci(30)) # Takes a moment # Second call: cached result print(fibonacci(30)) # Instant
Caution: Only cache pure functions (no side effects). The cached function must be called with hashable arguments.

Deep Dive: Descriptor Protocol and @property

+

Under the hood, `@property` implements the descriptor protocol. When you access an attribute, Python calls special methods (`__get__`, `__set__`, `__delete__`).

Custom descriptor (manual property) class MyProperty: def __init__(self, getter): self.getter = getter def __get__(self, obj, objtype=None): if obj is None: return self return self.getter(obj) class Example: def __init__(self, value): self._value = value @MyProperty def value(self): return self._value e = Example(42) print(e.value) # Calls __get__, returns 42

This is how Python implements attribute access. `@property` is a descriptor class providing `__get__`, `__set__`, and `__delete__` methods.

Docstrings, Arguments, Bitwise Ops & Assertions

Docstrings: Self-Documenting Code

A docstring is a string literal as the first statement in a module, function, class, or method. Tools like `help()` and `pydoc` use it for documentation.

Docstring conventions def divide(a, b): """Divide a by b. Args: a: Dividend (numeric). b: Divisor (numeric, non-zero). Returns: The quotient a/b. Raises: ZeroDivisionError: If b is zero. """ if b == 0: raise ZeroDivisionError("Cannot divide by zero") return a / b help(divide) # Displays the docstring

Command-Line Arguments: sys.argv and argparse

`sys.argv` is a list of command-line arguments. For complex programs, use `argparse` for parsing and validation.

Using argparse import argparse parser = argparse.ArgumentParser(description='Process data files') parser.add_argument('input', help='Input filename') parser.add_argument('--output', '-o', default='output.txt', help='Output file') parser.add_argument('--verbose', '-v', action='store_true', help='Verbose mode') parser.add_argument('--count', type=int, default=1, help='Number of iterations') args = parser.parse_args() print(f"Input: {args.input}") print(f"Output: {args.output}") print(f"Verbose: {args.verbose}") print(f"Count: {args.count}") # Run: python script.py input.txt -o output.txt -v --count 5

Bitwise Operators: Efficient Bit Manipulation

Python supports bitwise operations for low-level manipulation: `&` (AND), `|` (OR), `^` (XOR), `~` (NOT), `<<` (left shift), `>>` (right shift).

Bitwise operations a = 5 # Binary: 0101 b = 3 # Binary: 0011 print(a & b) # 0101 & 0011 = 0001 = 1 (AND) print(a | b) # 0101 | 0011 = 0111 = 7 (OR) print(a ^ b) # 0101 ^ 0011 = 0110 = 6 (XOR) print(~a) # ~0101 = ...1010 = -6 (NOT) print(a << 1) # 0101 << 1 = 1010 = 10 (left shift) print(a >> 1) # 0101 >> 1 = 0010 = 2 (right shift) # Use case: Flag checking READ = 1 << 0 # 0001 WRITE = 1 << 1 # 0010 EXEC = 1 << 2 # 0100 permissions = READ | WRITE print(permissions & READ) # Non-zero if readable

Assertions: Defensive Programming

The `assert` statement checks conditions and raises `AssertionError` if false. Useful for catching bugs during development.

Using assertions def divide(a, b): assert b != 0, "Divisor cannot be zero" assert isinstance(a, (int, float)), "a must be numeric" return a / b divide(10, 2) # OK divide(10, 0) # AssertionError: Divisor cannot be zero divide("10", 2) # AssertionError: a must be numeric # Note: Assertions are removed when Python runs with -O flag # Don't use for user input validation; use exceptions instead.
Caution: Assertions are removed with `python -O`. Never use them for input validation or critical checks. They're for development/debugging only.

Unicode and Bytes: Text vs. Data

Python 3 distinguishes between strings (Unicode text) and bytes (binary data). Encoding converts text to bytes; decoding converts bytes to text.

Unicode and bytes text = "Hello, 世界" print(type(text)) # - Unicode text # Encode to bytes data = text.encode('utf-8') print(type(data)) # print(data) # b'Hello, \xe4\xb8\x96\xe7\x95\x8c' # Decode back to string recovered = data.decode('utf-8') print(recovered) # Hello, 世界 # Bytes literals raw = b'Hello' print(raw[0]) # 72 (byte value)

Creating Executable Files: Entry Points

Modern Python packages use entry points (in `pyproject.toml`) to create command-line tools. When installed, they become system commands.

pyproject.toml [project.scripts] my-tool = "mypackage.cli:main"
mypackage/cli.py def main(): import argparse parser = argparse.ArgumentParser() parser.add_argument('--name', default='World') args = parser.parse_args() print(f"Hello, {args.name}!") if __name__ == '__main__': main()

After installing (`pip install -e .`), `my-tool --name Alice` runs the script.

Advanced Metaprogramming Patterns

Decorator Classes with __call__

A class can act as a decorator if it implements `__call__`. This is useful for stateful decorators.

Decorator class class Counter: def __init__(self, func): self.func = func self.call_count = 0 def __call__(self, *args, **kwargs): self.call_count += 1 print(f"Call #{self.call_count}") return self.func(*args, **kwargs) @Counter def greet(name): return f"Hello, {name}!" greet("Alice") # Call #1 greet("Bob") # Call #2 print(greet.call_count) # 2

Using typing.ParamSpec for Type-Safe Decorators

For advanced type hints, `ParamSpec` preserves the function signature in the decorator.

Type-safe decorator with ParamSpec from typing import ParamSpec, TypeVar import functools P = ParamSpec('P') T = TypeVar('T') def typed_cache(func: Callable[P, T]) -> Callable[P, T]: cache = {} @functools.wraps(func) def wrapper(*args: P.args, **kwargs: P.kwargs) -> T: key = (args, tuple(sorted(kwargs.items()))) if key not in cache: cache[key] = func(*args, **kwargs) return cache[key] return wrapper @typed_cache def expensive(x: int) -> int: return x ** 2

The inspect Module: Introspection

The `inspect` module provides tools to examine functions, classes, and code.

Using inspect for introspection import inspect def example(a, b, c=10, *args, **kwargs): """Example function.""" pass sig = inspect.signature(example) print(sig) # (a, b, c=10, *args, **kwargs) for param_name, param in sig.parameters.items(): print(f" {param_name}: {param.annotation or 'Any'}") # Get source code print(inspect.getsource(example)) # Check if callable print(inspect.isfunction(example)) # True print(inspect.ismethod(example)) # False

Deep Dive: Metaclasses and Type Manipulation

+

A metaclass is a "class of a class." It controls how classes are created. Most code never needs metaclasses, but they're powerful for frameworks.

Simple metaclass example class LoggingMeta(type): """Metaclass that logs class creation.""" def __new__(mcs, name, bases, namespace): print(f"Creating class {name}") return super().__new__(mcs, name, bases, namespace) class MyClass(metaclass=LoggingMeta): pass # Output: Creating class MyClass

Metaclasses are advanced and rarely needed. Class decorators are usually simpler and more readable.

Socratic Q&A: Test Your Understanding

Click each question to explore your understanding of decorators and metaprogramming.

1. What is a decorator, and how does it relate to function composition?
+

A decorator is a function that takes a function as input, wraps it with additional behavior, and returns the modified function. It's function composition: `@dec def f(): ...` is equivalent to `f = dec(f)`. The original function is passed to the decorator, which returns a wrapper function that calls the original while adding extra logic (logging, validation, caching, etc.).

2. Why is `functools.wraps` important in decorators?
+

`functools.wraps` copies metadata from the original function to the wrapper: `__name__`, `__doc__`, `__module__`, `__qualname__`, and `__annotations__`. Without it, `help()`, `inspect`, and other tools see the wrapper's metadata (usually "wrapper"), not the original function's. This breaks introspection and documentation. Always use `@functools.wraps(func)` in your decorator's wrapper function.

3. How do parameterized decorators work, and what's the structure?
+

Parameterized decorators use an extra layer of nesting. `@decorator(param)` calls a decorator factory that returns the actual decorator. Structure: (1) Factory function takes parameters, (2) Returns a decorator function, (3) Decorator takes the target function, (4) Decorator returns a wrapper. Example: `@retry(max_attempts=3)` calls `retry(3)`, returns a decorator, which decorates the function. This allows configuration without modifying the decorator itself.

4. What is the execution order when stacking multiple decorators?
+

Decorators apply bottom-to-top on definition. With `@dec1 @dec2 @dec3 def f(): ...`, the order is: `f = dec1(dec2(dec3(f)))`. At runtime, execution order is top-to-bottom through the call stack: dec1's wrapper runs first, which calls dec2's wrapper, which calls dec3's wrapper, which calls f. The outermost decorator executes first and last, wrapping all inner decorators.

5. How does `@property` enable attribute-like access to methods?
+

`@property` implements the descriptor protocol, converting a method into a getter. When you access the attribute, Python calls the property's `__get__` method, which invokes the original method. This enables `obj.value` to call a method without visible `()`. You can also define setters with `@value.setter`. This provides encapsulation: you can change internal implementation without changing the external interface.

6. What's the difference between `@staticmethod` and `@classmethod`?
+

`@staticmethod` creates a method that doesn't receive `self` or `cls`. It's just a function inside a class. `@classmethod` receives the class as the first argument (`cls`), not an instance. Use `@staticmethod` for utility functions. Use `@classmethod` for factory methods (creating instances) or operations on the class itself. Regular methods receive `self` (the instance).

7. How does `@functools.lru_cache` optimize functions, and what are its limitations?
+

`@functools.lru_cache(maxsize=N)` memoizes function results in an LRU (Least Recently Used) cache. On repeated calls with the same arguments, it returns the cached result instead of recomputing. Limitations: (1) Only caches pure functions (no side effects), (2) Arguments must be hashable (no lists/dicts), (3) Cache grows with unique inputs, (4) Not thread-safe. Use for expensive pure functions like recursive Fibonacci or API calls.

8. How do class decorators modify class behavior?
+

A class decorator takes the class as input and returns a modified class. It can add methods, modify existing methods, or change class attributes. Example: `@add_repr` adds a `__repr__` method. `@dataclass` auto-generates `__init__`, `__repr__`, etc. Class decorators are powerful for frameworks but can be less explicit than explicit methods. Prefer class decorators for automatic/generic functionality, explicit methods for custom behavior.

9. What are assertions, and why shouldn't they be used for input validation?
+

The `assert` statement checks a condition and raises `AssertionError` if false. They're for internal consistency checks during development. They're removed with `python -O`, so production code cannot rely on them. For input validation, use explicit `if` statements and raise `ValueError` or `TypeError` instead. Assertions are for "this should never happen" internal checks; exceptions are for expected errors.

10. How do decorators with `__call__` differ from function decorators?
+

A class with `__call__` can act as a decorator. It's instantiated with the target function, and `__call__` handles invocations. Advantages: (1) Can store state (call count, cache, etc.), (2) More readable for complex logic, (3) Can use instance attributes. Disadvantage: requires class syntax. Example: `class Counter: def __init__(self, func): self.func = func; self.count = 0` enables counting function calls by storing state in the decorator instance.

11. What is the descriptor protocol, and how does it power `@property`?
+

The descriptor protocol is a Python mechanism where objects with `__get__`, `__set__`, or `__delete__` methods intercept attribute access. When you access `obj.attr`, Python calls `descriptor.__get__(obj, type(obj))`. `@property` is a descriptor class implementing `__get__` (and optionally `__set__`, `__delete__`). This enables `@property.setter` and `@property.deleter`. Descriptors are how methods, properties, and class methods work under the hood.

12. How does `argparse` improve over `sys.argv` for command-line arguments?
+

`sys.argv` is a raw list of strings; parsing is manual and error-prone. `argparse` (or `click`) provides: (1) Automatic type conversion, (2) Validation, (3) Help generation (`-h`), (4) Short/long flags, (5) Default values, (6) Subcommands. `argparse.ArgumentParser` defines arguments declaratively, parses the command line, validates types, and raises helpful errors. For any non-trivial CLI, use argparse; `sys.argv` is only for simple scripts.

Key Takeaways

  • A decorator is a function taking a function and returning a modified function. `@decorator` is syntactic sugar for function reassignment.
  • Always use `@functools.wraps` to preserve metadata (`__name__`, `__doc__`) in decorator wrappers.
  • Use `*args` and `**kwargs` in wrapper functions to handle functions of any signature.
  • Parameterized decorators use nesting: decorator factory returns a decorator that decorates the target.
  • Multiple decorators apply bottom-to-top on definition; execution order is top-to-bottom through the call stack.
  • Built-in decorators like `@property`, `@staticmethod`, `@classmethod`, and `@functools.lru_cache` provide common functionality.
  • Class decorators can add or modify class behavior globally; they're powerful for frameworks.
  • Assertions are for development debugging; use exceptions for input validation and error handling.
  • The descriptor protocol (`__get__`, `__set__`, `__delete__`) powers properties, methods, and custom attribute access.
  • Use `argparse` for command-line argument parsing; avoid manual `sys.argv` manipulation.
