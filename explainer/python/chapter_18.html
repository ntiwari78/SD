Namespaces & Scope — The LEGB Rule

Master how Python organizes and resolves names. Understand the LEGB rule that determines where Python looks for variables, and how global and nonlocal keywords bend the rules.

Mental Model: Name Resolution as Layered Search

The Core Idea: Python doesn't store variables in one global bucket. Instead, it organizes them into separate namespaces: local (inside functions), enclosing (in outer functions), global (module level), and built-in (Python keywords). When you reference a name, Python searches these layers in LEGB order—it stops at the first match.

Key Insight: This layered approach enables closures (functions that remember their creation context) and prevents accidental global pollution. The `global` and `nonlocal` keywords override the search order, allowing modification of outer scopes—but use them sparingly.

Practical Consequence: Understanding LEGB explains UnboundLocalError, closures, and why modifying mutable objects from inner functions works differently than modifying primitives.

Namespaces: How Python Tracks Names

What is a Namespace?

A namespace is a dictionary mapping names (strings) to objects in memory. Every scope has its own namespace. Python creates new namespaces for modules, functions, and classes.

x = 5  # Added to global namespace

def func():
    y = 10  # Added to func's local namespace
    print(x)  # Looks up 'x' in global namespace

func()

The LEGB Rule: Name Resolution Order

LEGB Explained

When Python encounters a name, it searches for it in this order:

  1. Local: Inside the current function
  2. Enclosing: Inside outer functions (for nested functions)
  3. Global: At module level
  4. Built-in: Python's builtins (print, len, etc.)

Python stops at the first match. If a name isn't found, Python raises `NameError`.

x = 1

Inner Functions and Closures

Defining Functions Inside Functions

A function defined inside another function is an inner function. It has access to the outer function's variables via LEGB's Enclosing scope.

Inner function accessing enclosing scope def outer(x): def inner(): return x * 2 # Accesses x from enclosing scope return inner func = outer(5) print(func()) # 10

Closures: Functions That Remember

A closure is an inner function that "remembers" variables from its enclosing scope, even after the outer function has finished executing. Python implements this via cell objects.

A closure example def make_multiplier(n): def multiplier(x): return x * n # 'n' is captured from enclosing scope return multiplier times_3 = make_multiplier(3) times_5 = make_multiplier(5) print(times_3(10)) # 30 print(times_5(10)) # 50 # Each closure has its own copy of the captured variable

The inner function captures the variable `n`. Even after `make_multiplier` returns, the returned function remembers `n`. This is powerful for creating specialized functions.

Closures with Mutable State

Closures can modify captured variables using `nonlocal`:

Stateful closure def counter(start=0): count = start def increment(): nonlocal count count += 1 return count def decrement(): nonlocal count count -= 1 return count return {'increment': increment, 'decrement': decrement} c = counter(10) print(c['increment']()) # 11 print(c['increment']()) # 12 print(c['decrement']()) # 11

Deep Dive: Cell Objects and Closure Implementation

+

Python implements closures via cell objects. When a variable is captured by a closure, Python wraps it in a cell—a container that allows sharing the variable across function boundaries.

Inspecting closure cells def make_adder(x): def adder(y): return x + y return adder func = make_adder(5) print(func.__closure__) # Tuple of cell objects print(func.__closure__[0].cell_contents) # 5 # Without closure, __closure__ is None def simple(): return 1 print(simple.__closure__) # None

This mechanism is why closures work efficiently—cells point to the same object, so multiple closures can share state.

Advanced Scope Patterns

Bytecode and the dis Module

Python compiles code to bytecode, which you can inspect with the `dis` module. This reveals scope decisions:

Inspecting bytecode with dis import dis def func(): x = 1 y = 2 return x + y dis.dis(func)

The output shows instructions like `LOAD_FAST` (local variable), `LOAD_GLOBAL` (global), `LOAD_DEREF` (closure cell). This reveals exactly how Python resolves names at runtime.

Scope and Comprehensions

List, dict, and set comprehensions have their own scope (since Python 3). This prevents polluting the local namespace:

Comprehension scope result = [x * 2 for x in range(5)] print(result) # [0, 2, 4, 6, 8] print(x) # NameError: x is in comprehension scope, not local scope # Generator expressions also have their own scope gen = (x * 2 for x in range(5))

Scope Gotchas: Default Arguments

Default arguments are evaluated at definition time, not call time. This can create surprising behavior:

Default argument gotcha def func(x=[]): # Mutable default - BAD! x.append(1) return x print(func()) # [1] print(func()) # [1, 1] - Same list! # Correct approach def func2(x=None): if x is None: x = [] x.append(1) return x print(func2()) # [1] print(func2()) # [1]

Deep Dive: The Late Binding Closure Gotcha

+

When closures capture variables in a loop, they all capture the same variable, leading to unexpected behavior:

Late binding problem funcs = [] for i in range(3): funcs.append(lambda x: x + i) # All closures captured 'i' by reference print([f(10) for f in funcs]) # [12, 12, 12] - all use i=2! # Solution 1: Use default argument (captures by value) funcs = [lambda x, i=i: x + i for i in range(3)] print([f(10) for f in funcs]) # [10, 11, 12] # Solution 2: Use a factory function def make_adder(n): return lambda x: x + n funcs = [make_adder(i) for i in range(3)] print([f(10) for f in funcs]) # [10, 11, 12]

Python uses late binding—closures look up variables when called, not when created. The variable `i` is the same object for all closures; by the time they're called, `i` has its final value.

Socratic Q&A: Test Your Understanding

Click each question to reveal the answer. These questions deepen your understanding of namespaces, scope, and the LEGB rule.

1. What is a namespace, and how does it relate to scope?
+

A namespace is a dictionary mapping names to objects. Scope is the region of code where a name is valid. Each scope has its own namespace: local (inside function), enclosing (outer function), global (module), builtin (Python keywords). A name in one namespace is invisible in another unless accessed through the search order (LEGB).

2. What does `globals()` return, and when would you use it?
+

`globals()` returns a dictionary of the global namespace. Use it for debugging (to see all global variables), metaprogramming (dynamically accessing or creating globals), or introspection (checking if a variable is defined). You can modify `globals()` directly: `globals()['x'] = 5` sets global `x = 5`. Common in dynamic code and frameworks.

3. Explain the LEGB rule and give an example of each level.
+

LEGB is the order Python searches for a name: Local (inside current function), Enclosing (in outer function), Global (module level), Builtin (Python's builtins). Example: In `def outer(): def inner(): print(x)`, Python checks local (x in inner?), enclosing (x in outer?), global (x at module level?), builtin (x is a builtin?). It stops at the first match.

4. What is the difference between `global` and `nonlocal`?
+

`global` allows modifying global (module-level) variables from inside a function. `nonlocal` allows modifying enclosing (outer function) variables from inside a nested function. `global x` in an inner function refers to global x, not to any enclosing x. `nonlocal x` in a nested function refers to x in an enclosing function. You can't use `nonlocal` for globals.

5. What is `UnboundLocalError` and why is it confusing?
+

`UnboundLocalError` occurs when you reference a local variable before assigning it. It's confusing because Python determines scope at compile-time, not runtime. If a function has `x = 5` anywhere, Python marks all references to `x` as local. If you then try to read `x` before the assignment, you get UnboundLocalError. The fix: use `global` to read from global scope, or restructure your code.

6. What is a closure, and how does it enable functional programming patterns?
+

A closure is an inner function that captures variables from its enclosing scope, even after the outer function returns. Closures enable: (1) Data hiding (private state), (2) Partial application (bind some arguments), (3) Decorators (wrap functions), (4) Stateful callbacks. Example: `def make_counter(start=0): count = start; def inc(): nonlocal count; count += 1; return count; return inc`. Each closure has its own captured variables.

7. Explain the "late binding" gotcha in closures.
+

Late binding means closures look up variables when called, not when created. In a loop creating closures, all closures capture the same variable reference. When called, the variable has its final value. Example: `[lambda: i for i in range(3)]` creates 3 closures, but all return 2 when called. Fix: (1) use default arguments to capture by value: `[lambda i=i: i for i in range(3)]`, or (2) use a factory function that creates a new enclosing scope for each iteration.

8. What do cell objects do, and how are they used in closures?
+

Cell objects are containers for variables captured by closures. They allow the inner function to access and modify the outer function's variables. When Python detects a closure, it wraps captured variables in cells. Multiple closures can share the same cell, enabling state sharing. Inspect with `func.__closure__` (tuple of cells) and `cell.cell_contents` (the actual value).

9. How do comprehensions handle scope differently in Python 3?
+

In Python 3, list/dict/set comprehensions and generator expressions have their own scope. The loop variable is local to the comprehension. This prevents polluting the enclosing namespace. Example: `[x for x in range(5)]` doesn't create x in the enclosing scope; x is only in the comprehension scope. This was changed from Python 2 (where the loop variable leaked) to reduce namespace pollution.

10. Why is using mutable default arguments dangerous?
+

Default arguments are evaluated once, at function definition time, not each time the function is called. If the default is mutable (list, dict), all calls share the same object. Example: `def func(x=[]): x.append(1); return x` appends to the same list across calls. Fix: Use None as default and create a new object inside: `def func(x=None): if x is None: x = []; ...`. This is a common Python gotcha.

11. How can you use `dis` module to understand scope?
+

The `dis` module disassembles Python functions to bytecode. Instructions like `LOAD_FAST` (local), `LOAD_GLOBAL` (global), `LOAD_DEREF` (closure cell) reveal how variables are resolved. `dis.dis(func)` shows the bytecode. This reveals the compile-time scope decisions and helps debug scope-related issues. Advanced debugging/learning tool.

12. Why is modifying global state considered bad practice?
+

Global state makes code harder to test, reason about, and maintain. Functions depending on globals have hidden dependencies (tests must set up globals). Globals can be modified anywhere, making bugs hard to track. Multiple functions modifying the same global can create subtle interaction bugs. Better: return values, pass arguments, or use objects to encapsulate state. Use globals only for truly global constants or configuration.

Key Takeaways

  • A namespace is a dict mapping names to objects. Each scope (local, enclosing, global, builtin) has its own namespace.
  • LEGB rule: Python searches Local, Enclosing, Global, Builtin (in that order) when resolving a name.
  • The `global` keyword allows modifying global variables from inside a function.
  • The `nonlocal` keyword allows modifying enclosing scope variables in nested functions—essential for closures.
  • Closures are inner functions that capture variables from their enclosing scope, enabling data hiding and functional patterns.
  • Late binding: closures look up variables when called, not when created. Use default arguments to capture by value.
  • Compile-time scope analysis causes UnboundLocalError—Python marks variables as local if they're assigned anywhere in the function.
  • Minimize global state; prefer returning values or using objects to encapsulate state.
