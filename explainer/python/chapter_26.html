File I/O & Serialization

Read, write, and persist Python objects efficiently using streams, context managers, and serialization formats

🧠 Mental Model: Streams and Serialization

Files are sequences of bytes. Python abstracts them as streams (sequences of data flowing in or out). When you open a file, you get a file object that lets you read/write data.

Serialization converts Python objects (dicts, lists, custom objects) into bytes that can be saved. Deserialization reconstructs objects from bytes.

Think of it like: File = Container. Stream = Pipeline from/to container. Serialization = Converting furniture to boxes. Deserialization = Unpacking boxes to furniture.

File I/O Fundamentals

Opening Files

Use the open() function to open files:

file = open('data.txt', 'r')  # Open file for reading
# ... do something with file ...
file.close()  # Always close when done

The file object represents your connection to the file. Always close it when done to free system resources.

Better: Use the context manager (with statement):

with open('data.txt', 'r') as file:
    content = file.read()
# File is automatically closed, even if an error occurred

The with statement guarantees the file closes, making it much safer.

Reading Operations

Several methods for reading files:

# Read entire file as one string
with open('file.txt', 'r') as f:
    content = f.read()
    print(content)  # Entire file content

# Read one line at a time (lazy, memory-efficient)
with open('file.txt', 'r') as f:
    line = f.readline()  # Returns next line including \n
    print(line)

# Read all lines as a list
with open('file.txt', 'r') as f:
    lines = f.readlines()  # List of lines, each with \n
    for line in lines:
        print(line.strip())  # Remove newline

# Iterate over lines directly (most Pythonic)
with open('file.txt', 'r') as f:
    for line in f:
        print(line.strip())  # Lazy iteration, memory-efficient

# Read first N characters
with open('file.txt', 'r') as f:
    chunk = f.read(100)  # Read first 100 characters

For large files, iterate line-by-line or read in chunks. Don't use read() on gigabyte files—it loads everything into memory.

Writing Operations

Write data to files:

# Write a string (overwrites file)
with open('output.txt', 'w') as f:
    f.write("Hello, World!\n")
    f.write("Second line\n")

# Write multiple lines at once
with open('output.txt', 'w') as f:
    lines = ["Line 1\n", "Line 2\n", "Line 3\n"]
    f.writelines(lines)

# Append to existing file
with open('output.txt', 'a') as f:
    f.write("Appended line\n")

Note: write() doesn't add newlines. You must include \n explicitly.

File Opening Modes

Mode Name Behavior File Must Exist?
'r' Read Read only. Start at beginning. Yes
'w' Write Write only. Truncate (overwrite) existing. No (creates)
'a' Append Write only. Add to end of existing. No (creates)
'r+' Read+Write Read and write, start at beginning. Yes
'w+' Write+Read Read and write, truncate existing. No (creates)
'a+' Append+Read Read and append, add to end. No (creates)
'rb' Read Binary Read binary data (images, pickles). Yes
'wb' Write Binary Write binary data, truncate. No (creates)
'ab' Append Binary Write binary data, append to end. No (creates)

Text mode ('r', 'w') handles encoding. Binary mode ('rb', 'wb') gives raw bytes.

Text vs Binary

Text mode: Reads/writes strings. Python handles encoding/decoding based on system locale (usually UTF-8). Line endings may be normalized (\n vs \r\n on Windows).

Binary mode: Reads/writes raw bytes. No encoding/decoding. No line ending normalization. Use for images, pickled objects, executables, etc.

The with Statement: Context Managers

Why with is Essential

The with statement automatically manages file closing:

# WITHOUT with: manual cleanup needed
file = open('data.txt', 'r')
try:
    content = file.read()
finally:
    file.close()  # Must remember this!

# WITH with: automatic cleanup
with open('data.txt', 'r') as file:
    content = file.read()
# File is guaranteed closed here

Benefits of with:

  • File always closes, even if exception occurs
  • Cleaner, more readable code
  • No resource leaks

Multiple Files

You can open multiple files with one with statement:

with open('input.txt', 'r') as infile, open('output.txt', 'w') as outfile:
    for line in infile:
        processed = line.upper()
        outfile.write(processed)
# Both files automatically closed

Moving Within a File

seek() and tell()

Control your position in the file:

with open('data.txt', 'rb') as f:
    print(f.tell())  # 0, at beginning

    data = f.read(10)  # Read 10 bytes
    print(f.tell())  # 10, position after read

    f.seek(0)  # Go back to beginning
    print(f.tell())  # 0

    data = f.read(5)  # Read first 5 bytes again
    print(data)

    f.seek(-5, 2)  # Seek 5 bytes from end (offset 2)
    end_data = f.read()  # Read last 5 bytes

seek(offset, whence) where whence is:

  • 0 - absolute position from start
  • 1 - relative to current position
  • 2 - relative to end of file

tell() returns the current byte position in the file.

Text vs Binary seek()

In binary mode, you can seek to any byte position. In text mode, seeking is limited—you can only seek to positions returned by tell() (due to encoding complexities). For complex positioning, use binary mode.

Serialization and Deserialization

What is Serialization?

Serialization converts Python objects (lists, dicts, objects) into bytes that can be saved to a file or transmitted. Deserialization reconstructs the objects.

import json

# Serialization: Python object → JSON string
data = {
    'name': 'Alice',
    'age': 30,
    'items': [1, 2, 3]
}
json_str = json.dumps(data)  # Convert to string
print(json_str)  # {"name": "Alice", "age": 30, "items": [1, 2, 3]}

# Deserialization: JSON string → Python object
loaded = json.loads(json_str)
print(loaded['name'])  # Alice

pickle Module: Python Serialization

The pickle module serializes any Python object:

import pickle

# Serialization: dump to file
data = {
    'numbers': [1, 2, 3],
    'text': 'hello',
    'nested': {'key': 'value'}
}

with open('data.pkl', 'wb') as f:
    pickle.dump(data, f)  # Write pickled object to file

# Deserialization: load from file
with open('data.pkl', 'rb') as f:
    loaded_data = pickle.load(f)

print(loaded_data)  # {'numbers': [1, 2, 3], ...}

You can also pickle to/from bytes directly:

import pickle

data = {'a': 1, 'b': 2}
pickled = pickle.dumps(data)  # Convert to bytes
unpickled = pickle.loads(pickled)  # Convert from bytes

JSON vs Pickle

JSON:

  • Human-readable text format
  • Secure (only encodes data, not code)
  • Limited to basic types (dict, list, str, int, float, bool, null)
  • Interoperable (works across languages)

Pickle:

  • Binary format (not human-readable)
  • Can serialize ANY Python object (custom classes, functions, etc.)
  • Python-only format
  • Security risk: only unpickle trusted data (pickle can execute code)
# JSON example
import json
data = {'name': 'Alice', 'age': 30}
json.dump(data, open('data.json', 'w'))

# Pickle example
import pickle
pickle.dump(data, open('data.pkl', 'wb'))

Pickling Custom Objects

Pickle can serialize custom classes:

import pickle

class Person:
    def __init__(self, name, age):
        self.name = name
        self.age = age

    def __repr__(self):
        return f"Person('{self.name}', {self.age})"

# Serialize
person = Person('Alice', 30)
with open('person.pkl', 'wb') as f:
    pickle.dump(person, f)

# Deserialize
with open('person.pkl', 'rb') as f:
    loaded_person = pickle.load(f)

print(loaded_person)  # Person('Alice', 30)
print(loaded_person.name)  # Alice

For custom serialization behavior, implement __getstate__ and __setstate__:

class Person:
    def __init__(self, name, age, password):
        self.name = name
        self.age = age
        self.password = password  # Don't pickle this!

    def __getstate__(self):
        # Return dict of what to pickle (exclude password)
        state = self.__dict__.copy()
        del state['password']
        return state

    def __setstate__(self, state):
        # Restore from pickled state
        self.__dict__.update(state)
        self.password = None  # Set default for missing field

File and Directory Operations

os Module Basics

The os module provides file system operations:

import os

# Check if file exists
if os.path.exists('data.txt'):
    print("File exists")

# Check if path is file or directory
if os.path.isfile('data.txt'):
    print("It's a file")

if os.path.isdir('/home/user'):
    print("It's a directory")

# Get file size in bytes
size = os.path.getsize('data.txt')
print(f"File size: {size} bytes")

# Get absolute path
abs_path = os.path.abspath('data.txt')
print(abs_path)

# List directory contents
files = os.listdir('.')
for filename in files:
    print(filename)

# Delete file
os.remove('temp.txt')

# Delete directory (must be empty)
os.rmdir('empty_dir')

pathlib: Modern File Handling

pathlib.Path is more modern and object-oriented:

from pathlib import Path

# Create Path objects
file_path = Path('data.txt')
dir_path = Path('/home/user/documents')

# Check if exists
if file_path.exists():
    print("File exists")

# Get file properties
print(file_path.name)  # 'data.txt'
print(file_path.stem)  # 'data'
print(file_path.suffix)  # '.txt'
print(file_path.parent)  # Path('.')

# Path operations
new_path = dir_path / 'subfolder' / 'file.txt'  # Join paths

# Read/write convenience methods
content = file_path.read_text()
file_path.write_text('New content')

# Iterate over files in directory
for file in dir_path.glob('*.txt'):
    print(file)

# Delete
file_path.unlink()

pathlib is preferred for new code—it's more intuitive and handles cross-platform paths.

Deep Dive: io Module Architecture

▼

Python's I/O system has three layers (only relevant for advanced use):

# TextIOWrapper (text encoding/decoding)
#    ↓
# BufferedWriter/BufferedReader (buffering)
#    ↓
# FileIO (actual file system)

# Most of the time, open() handles this for you
with open('file.txt') as f:  # Returns TextIOWrapper
    f.read()

# You can access underlying layers
with open('file.txt', 'rb') as f:
    raw_bytes = f.read()  # RawIOBase

For normal file operations, you don't need to think about layers. Understanding them helps with debugging and advanced use cases.

Socratic Q&A: Test Your Understanding

Why is using `with open()` better than manually calling `file.close()`? ▼

The `with` statement guarantees the file closes, even if an exception occurs. Manual `close()` can be skipped if your code raises an error before reaching it, leaking file handles. Additionally, `with` is cleaner and more Pythonic. It's also easy to forget to close when refactoring code.

What's the difference between `readline()` and `readlines()`? When would you use each? ▼

`readline()` reads ONE line (including \n). `readlines()` reads ALL lines into a list. For large files, iterating with a for loop is best: `for line in file:` is lazy and memory-efficient. Use `readlines()` only when you need all lines at once (rare).

Why can't you `seek()` to arbitrary positions in text mode but can in binary mode? ▼

Text mode handles encoding (UTF-8, etc.) where multi-byte characters can span multiple positions. Seeking mid-character would corrupt data. Binary mode works with raw bytes—any position is valid. Use binary mode if you need precise positioning.

What's the security risk with using `pickle.loads()` on untrusted data? ▼

Pickle can deserialize any Python object, including code. Malicious pickled data can execute arbitrary code when unpickled. Never use `pickle.load()` or `pickle.loads()` on untrusted sources (network, user uploads). For untrusted data, use JSON instead—it's restricted to data structures.

When would you use JSON vs pickle? What are the tradeoffs? ▼

JSON: human-readable, secure, works across languages, limited to basic types (dict, list, str, number, bool, null). Use for config files, APIs, untrusted data. Pickle: binary, Python-only, can serialize any object, security risk. Use for caching Python objects internally (not from untrusted sources).

What's the difference between file modes 'w' and 'a'? What happens if you use 'w' on an existing file? ▼

'w' (write) truncates the file—deletes all existing content. 'a' (append) adds to the end, preserving existing content. If you use 'w' on an existing file with data, that data is LOST. Always be careful with 'w'.

What does `tell()` return? How is it different from a line number? ▼

`tell()` returns the byte position in the file (offset from start). It's NOT a line number. If a file has multi-byte characters, line numbers and byte positions diverge. In binary mode, this is crucial for precise positioning.

Why should you use `pathlib.Path` instead of `os.path` for new projects? ▼

`pathlib` is object-oriented, more intuitive, and handles cross-platform paths elegantly (using `/` operator). `os.path` is older and string-based. `pathlib` is also more consistent: `Path('a') / 'b'` works on all platforms, while string concatenation requires manual handling.

How do `write()` and `writelines()` differ? Does `writelines()` add newlines automatically? ▼

`write()` writes a single string. `writelines()` writes an iterable of strings. NEITHER adds newlines automatically—you must include \n in the strings. Despite its name, `writelines()` doesn't add line breaks.

What's the purpose of the `newline=''` parameter in `open()` for CSV writing? ▼

By default, text mode converts \n to platform-specific line endings (\r\n on Windows). For CSV files, this breaks parsing. Setting `newline=''` disables this conversion, letting the CSV module handle line endings correctly. This is a best practice when working with CSV files.
