Making Objects Work Like Built-in Types

Identifier Naming Conventions

PEP 8, Python's style guide, defines naming conventions that make code more readable and consistent across the Python community.

Snake_case vs CamelCase

  • snake_case: Variables, functions, methods. Example: my_variable, calculate_total()
  • PascalCase: Classes. Example: MyClass, DatabaseConnection
  • UPPER_CASE: Constants. Example: MAX_ITERATIONS = 100
  • _leading_underscore: "Protected" — internal implementation detail
  • __double_leading: Name mangling, weak privacy (Chapter 20)
  • __dunder__: "Magic methods" or special methods Python looks for
MAX_ITERATIONS = 100  # Constant

class DataProcessor:  # PascalCase for class
    _internal_state = None  # Protected

    def process_data(self):  # snake_case for method
        pass

    def __init__(self):  # Dunder for special behavior
        pass

Calling Functions and Methods: Bound vs Unbound

In Python 3, there's an important distinction between how methods work when accessed through an instance vs. accessed through a class.

Bound Methods

When you access a method through an instance (obj.method()), you get a bound method. It "remembers" the instance and automatically passes self when called.

Unbound Methods (Functions)

When you access a method through the class (Class.method), you get the raw function. You must manually pass the instance as the first argument. 

class Circle:
    def __init__(self, radius):
        self.radius = radius

    def area(self):
        return 3.14 * self.radius ** 2

circle = Circle(5)

# Bound method (instance access)
method = circle.area
print(method())  # Works: 78.5
print(method.__self__)  # Shows bound instance

# Unbound method (class access)
unbound = Circle.area
# print(unbound())  # TypeError! Missing required argument: self

# But you can call it by passing instance explicitly
print(unbound(circle))  # Works: 78.5

How Method Resolution Works

When Python accesses obj.method, it:

  1. Finds the function in the class
  2. Detects it's being accessed from an instance
  3. Wraps it in a bound method object that remembers the instance
  4. Returns the bound method

This automatic binding is what makes self work seamlessly.

Operator Overloading: Dunders in Action

"Dunder" methods (double underscore) are special methods that allow instances to behave like built-in types. Python calls these methods when you use operators or built-in functions on your objects.

Common Arithmetic Operators

class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __add__(self, other):
        """Overload + operator"""
        return Vector(self.x + other.x, self.y + other.y)

    def __sub__(self, other):
        """Overload - operator"""
        return Vector(self.x - other.x, self.y - other.y)

    def __mul__(self, scalar):
        """Overload * operator (scalar multiplication)"""
        return Vector(self.x * scalar, self.y * scalar)

    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

v1 = Vector(1, 2)
v2 = Vector(3, 4)

print(v1 + v2)      # Vector(4, 6)
print(v1 - v2)      # Vector(-2, -2)
print(v1 * 3)       # Vector(3, 6)

Comparison Operators

class Student:
    def __init__(self, name, gpa):
        self.name = name
        self.gpa = gpa

    def __eq__(self, other):
        """Overload == operator"""
        return self.gpa == other.gpa

    def __lt__(self, other):
        """Overload < operator"""
        return self.gpa < other.gpa

    def __gt__(self, other):
        """Overload > operator"""
        return self.gpa > other.gpa

    def __le__(self, other):
        """Overload <= operator"""
        return self.gpa <= other.gpa

    def __ge__(self, other):
        """Overload >= operator"""
        return self.gpa >= other.gpa

    def __ne__(self, other):
        """Overload != operator"""
        return self.gpa != other.gpa

alice = Student("Alice", 3.8)
bob = Student("Bob", 3.6)

print(alice > bob)      # True (3.8 > 3.6)
print(alice == bob)     # False (different GPAs)
print(alice <= bob)     # False

Reflection and Truthiness

class Container:
    def __init__(self, items=None):
        self.items = items or []

    def __len__(self):
        """len(obj) calls this"""
        return len(self.items)

    def __bool__(self):
        """bool(obj) calls this; also used in if statements"""
        return len(self.items) > 0

    def __repr__(self):
        """str(obj) falls back to this if __str__ not defined"""
        return f"Container({self.items})"

    def __getitem__(self, index):
        """obj[index] calls this"""
        return self.items[index]

    def __setitem__(self, index, value):
        """obj[index] = value calls this"""
        self.items[index] = value

c = Container([1, 2, 3])

print(len(c))           # 3 (__len__)
if c:                   # Calls __bool__
    print("Not empty")

print(c[0])             # 1 (__getitem__)
c[0] = 10
print(c[0])             # 10 (__setitem__)

Deep Dive: Right-Hand Operators and Reflected Methods

▼

Some operators have "reflected" versions used when the left operand doesn't support the operation:

  • a + b tries a.__add__(b) first, then b.__radd__(a)
  • a * b tries a.__mul__(b) first, then b.__rmul__(a)
class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __mul__(self, scalar):
        """Vector * scalar"""
        return Vector(self.x * scalar, self.y * scalar)

    def __rmul__(self, scalar):
        """scalar * Vector (reflected)"""
        return Vector(self.x * scalar, self.y * scalar)

    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

v = Vector(1, 2)

print(v * 3)      # Calls v.__mul__(3)
print(3 * v)      # Calls v.__rmul__(3)

Which Operators to Overload?

Just because you can overload an operator doesn't mean you should. Overload operators when they make intuitive sense for your domain.

Good Candidates

  • Arithmetic: +, -, *, / for math-like classes
  • Comparison: <, >, == for ordered or comparable objects
  • Container: len(), [], in for collection-like classes
  • String representation: str(), repr() for debugging
  • Truthiness: bool() when an object can be logically empty/full

Poor Candidates

  • Bitwise operators (&, |) unless working with bits or sets
  • The tilde operator (~) unless negation is intuitive
  • Unexpected mappings (e.g., using + for something that isn't addition)

Principle: Least Surprise

Overload operators in ways your users would expect. If a + b would confuse someone reading your code, don't overload +. Use explicit method names instead.

Everything is an Object

In Python, everything is an object, including numbers, strings, functions, and even classes themselves. 

# Numbers are objects
x = 42
print(type(x))              # 
print(x.__class__)          # 
print(dir(x))               # All methods int has

# Strings are objects
s = "hello"
print(s.upper())            # Calls s.__str__() etc.

# Functions are objects
def greet():
    pass

print(type(greet))          # 
print(greet.__name__)       # 'greet'

# Classes are objects (instances of type/metaclass)
class MyClass:
    pass

print(type(MyClass))        # 
print(MyClass.__name__)     # 'MyClass'

# This means you can pass functions and classes around
functions = [int, str, float]
for func in functions:
    print(func("42"))       # Each is called

Deep Dive: int, str, float are Classes

▼

int, str, and float aren't keywords — they're classes. Calling int(42) creates an instance of the int class. 

# int is a class
x = int("42")      # Creates an int instance from a string
print(type(x))     # 

# str is a class
s = str(42)        # Creates a str instance from an int
print(type(s))     # 

# float is a class
f = float("3.14")  # Creates a float instance
print(type(f))     # 

# They all have methods
print(x.bit_length())  # int method
print(s.upper())       # str method
print(round(f, 1))     # float has __round__

Type Conversion: __int__, __float__, __str__, __repr__, __bool__

These dunders let your objects convert to built-in types or represent themselves as strings.

String Representation

  • __str__(): Human-readable string (for end users). Called by str() and print()
  • __repr__(): Developer-friendly string (for debugging). Called by repr()

Numeric Conversion

  • __int__(): Convert to int. Called by int()
  • __float__(): Convert to float. Called by float()
  • __bool__(): Convert to bool. Called by bool() and in truthiness checks
class Temperature:
    def __init__(self, celsius):
        self.celsius = celsius

    def __str__(self):
        """User-friendly representation"""
        return f"{self.celsius}°C"

    def __repr__(self):
        """Developer-friendly representation"""
        return f"Temperature(celsius={self.celsius})"

    def __int__(self):
        """Convert to integer (rounded Celsius)"""
        return round(self.celsius)

    def __float__(self):
        """Convert to float (Fahrenheit)"""
        return (self.celsius * 9/5) + 32

    def __bool__(self):
        """True if above absolute zero"""
        return self.celsius > -273.15

temp = Temperature(25)

print(str(temp))            # 25°C (__str__)
print(repr(temp))           # Temperature(celsius=25) (__repr__)
print(int(temp))            # 25 (__int__)
print(float(temp))          # 77.0 (__float__)
print(bool(temp))           # True (__bool__)

# In f-strings, uses __str__
print(f"Temperature: {temp}")  # Temperature: 25°C

# In interactive shell, uses __repr__
# >>> temp
# Temperature(celsius=25)

__str__ vs __repr__ Best Practices

  • __str__: For humans. "1984 by George Orwell"
  • __repr__: For developers. "Book(title='1984', author='George Orwell')" (ideally reproduces creation)
  • If you only define one, define __repr__ — Python falls back to it for __str__

Imitating a Structure: Classes as C-like Structs

Sometimes you just need a simple data container, like a C struct. Python classes work perfectly for this, though for simple cases, you might prefer dataclasses or namedtuple. 

# Simple struct-like class
class Point:
    def __init__(self, x, y, z):
        self.x = x
        self.y = y
        self.z = z

    def __repr__(self):
        return f"Point({self.x}, {self.y}, {self.z})"

p = Point(1, 2, 3)
print(p.x)      # 1
print(p)        # Point(1, 2, 3)

# For truly simple data, consider namedtuple
from collections import namedtuple

Point = namedtuple('Point', ['x', 'y', 'z'])
p = Point(1, 2, 3)
print(p.x)      # 1 (immutable, lightweight)

# Or dataclasses (Chapter 23)
from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int
    z: int

p = Point(1, 2, 3)
print(p.x)      # 1

Expert Insights: Advanced Dunder Methods

The Descriptor Protocol: __get__, __set__, __delete__

▼

Descriptors are advanced attribute access customization. A descriptor is an object that defines __get__, __set__, and/or __delete__. When a descriptor is accessed through an instance, Python calls these methods instead of normal attribute lookup. 

class Descriptor:
    def __get__(self, obj, objtype=None):
        print("Getting...")
        return obj.__dict__.get('value', None)

    def __set__(self, obj, value):
        print(f"Setting to {value}")
        obj.__dict__['value'] = value

class MyClass:
    attr = Descriptor()

obj = MyClass()
obj.attr = 10      # Calls Descriptor.__set__
print(obj.attr)    # Calls Descriptor.__get__

# Practical example: property
class Circle:
    def __init__(self, radius):
        self._radius = radius

    @property
    def radius(self):
        return self._radius

    @radius.setter
    def radius(self, value):
        if value <= 0:
            raise ValueError("Radius must be positive")
        self._radius = value

c = Circle(5)
print(c.radius)     # 5
c.radius = 10       # Validated!
# c.radius = -1     # ValueError!

Data vs Non-Data Descriptors

▼
  • Data descriptor: Has both __get__ and __set__. Takes precedence over instance dict.
  • Non-data descriptor: Has only __get__. Instance dict takes precedence.
# Data descriptor (property is one)
class ValidatedInt:
    def __set_name__(self, owner, name):
        self.name = name

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return obj.__dict__.get(self.name, 0)

    def __set__(self, obj, value):
        if not isinstance(value, int):
            raise TypeError(f"{self.name} must be int")
        obj.__dict__[self.name] = value

class Person:
    age = ValidatedInt()  # Data descriptor

p = Person()
p.age = 30              # Calls __set__
print(p.age)            # Calls __get__
# p.age = "thirty"      # TypeError!

Rich Comparison Methods and Consistency

▼

Python provides "rich comparison" methods: __eq__, __ne__, __lt__, __le__, __gt__, __ge__.

Important: If you define __eq__, Python won't auto-generate __ne__. Define both explicitly or use functools.total_ordering. 

from functools import total_ordering

@total_ordering
class Version:
    def __init__(self, major, minor, patch):
        self.major = major
        self.minor = minor
        self.patch = patch

    def __eq__(self, other):
        return (self.major, self.minor, self.patch) == \
               (other.major, other.minor, other.patch)

    def __lt__(self, other):
        return (self.major, self.minor, self.patch) < \
               (other.major, other.minor, other.patch)

v1 = Version(1, 0, 0)
v2 = Version(1, 0, 1)

print(v1 < v2)      # True
print(v1 <= v2)     # True (auto-generated)
print(v1 > v2)      # False (auto-generated)
print(v1 >= v2)     # False (auto-generated)
print(v1 != v2)     # True (auto-generated)

__slots__ Memory Optimization Revisited

▼

__slots__ replaces the flexible __dict__ with a fixed tuple of slot references. Benefits: less memory (no dict overhead), faster attribute access. Drawback: less flexible. 

# Without __slots__: ~296 bytes per instance (due to __dict__)
class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

# With __slots__: ~56 bytes per instance
class PointSlots:
    __slots__ = ('x', 'y')

    def __init__(self, x, y):
        self.x = x
        self.y = y

import sys
p1 = Point(1, 2)
p2 = PointSlots(1, 2)

print(sys.getsizeof(p1))      # Much larger
print(sys.getsizeof(p2))      # Much smaller

# Trade-off: slots classes can't add attributes dynamically
p1.z = 3    # Works
# p2.z = 3  # AttributeError!

Socratic Questions: Test Your Understanding

1. What is a dunder method and why does Python have them?

▼
Dunder (double underscore) methods are special methods that Python looks for to customize object behavior. They allow instances to work with built-in operations (operators, functions, etc.). For example, __add__ makes + work with your objects, __len__ makes len() work, etc.

2. Explain bound vs unbound methods.

▼
A bound method is accessed through an instance (obj.method) and automatically passes the instance as self. An unbound method is accessed through the class (Class.method) and is just a function — you must manually pass the instance. In Python 3, this distinction is less prominent, but it still matters conceptually.

3. When would you overload the + operator vs using a method like add()?

▼
Overload + when your objects naturally represent something additive (vectors, numbers, strings, etc.). Use explicit methods when the meaning would be ambiguous or surprising. If someone sees a + b and doesn't immediately understand what's happening, use a.combine(b) or similar instead.

4. What's the difference between __str__ and __repr__?

▼
__str__ is for end users (human-readable, called by print() and f-strings). __repr__ is for developers (unambiguous, ideally shows how to recreate the object). If you define only one, define __repr__ — Python falls back to it for __str__.

5. How does operator precedence work with overloaded operators?

▼
Operator precedence doesn't change when you overload operators. a + b * c still evaluates as a + (b * c), whether using built-ins or your custom objects. Python respects the standard precedence rules for all operator overloads.

6. What happens when you use reflected operators like __radd__?

▼
a + b first tries a.__add__(b). If that returns NotImplemented (or doesn't exist), Python tries b.__radd__(a). This allows 3 * vector to work even if int doesn't know how to multiply by your custom type.

7. Everything in Python is an object. What does this really mean?

▼
Every value in Python is an instance of some class. int, str, list, and function are all classes. Even classes themselves are instances of type (the metaclass). This means you can pass functions around, store them in variables, inspect their attributes, etc.

8. What is a descriptor and what problem does it solve?

▼
A descriptor customizes attribute access by defining __get__, __set__, and/or __delete__. It solves the problem of enforcing constraints or computed values. @property is a common descriptor example — it lets you validate assignments or compute values dynamically.

9. Why would you define __bool__ on a class?

▼
__bool__ allows your objects to be used in boolean contexts (if obj:, while obj:). Define it when your object has a natural "empty" or "full" state. For example, a Container with __bool__ returns False if empty, True if it has items.

10. What does the Principle of Least Surprise suggest about operator overloading?

▼
Overload operators in ways your users expect. If a + b isn't intuitively addition-like, don't use __add__. Use explicit method names instead. For example, don't use + to mean "concatenate" in a non-string context; use a method like merge().

11. How would you ensure your comparison methods (__eq__, __lt__, etc.) are consistent?

▼
Use @functools.total_ordering decorator. Define __eq__ and one other comparison method (__lt__, etc.), and the decorator auto-generates the rest. This ensures a < b, a <= b, a > b, a >= b are all consistent.

12. What advantage does __slots__ provide and what's the trade-off?

▼
__slots__ reduces memory usage (no __dict__) and speeds up attribute access by using fixed slots instead. Trade-off: you can't add attributes dynamically to instances. Good for performance-critical code with many instances; bad for flexible class hierarchies.
