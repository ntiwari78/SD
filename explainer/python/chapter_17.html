Modules & Packages

Understand how Python organizes code into reusable, importable units. Learn the mechanics of modules, packages, and the import system that powers Python's ecosystem.

Mental Model: Code Organization & Reusability

The Core Idea: Python's module system transforms scripts into libraries. A single `.py` file can be both a standalone script AND an importable module—Python's `__name__` variable determines which "role" the code plays.

Key Insight: Packages are directories containing modules, creating a hierarchical namespace. This prevents name collisions and allows logical grouping of related code. The import system searches through a sequence of locations (PYTHONPATH, installed packages) to find and load modules.

Practical Consequence: Understanding `import`, module naming, and `__init__.py` unlocks the ability to use the entire Python ecosystem (pip, PyPI) and build modular applications.

The Main Module: __name__ == '__main__'

Script vs. Module: A Dual Nature

Every Python file has a special variable `__name__` that Python sets automatically. When a file runs directly, `__name__` is `'__main__'`. When imported, `__name__` is the module's name. This allows code to detect its execution context.

File: calculator.py def add(a, b): return a + b def multiply(a, b): return a * b # This runs ONLY when executed directly if __name__ == '__main__': print("Running as script") print(add(3, 5)) print(multiply(4, 7))

When you run `python calculator.py` directly, `__name__` is `'__main__'` and the code block executes. When you import it (`import calculator`), `__name__` is `'calculator'` and the code block is skipped.

Best Practice: Always guard your execution code with `if __name__ == '__main__':` to make your code importable without side effects.

How Python Sets __name__

Python's import system sets `__name__` based on how the module is invoked. The entry point file (the one you run directly) always gets `__name__ = '__main__'`, creating the entry point to your program.

Execution Method __name__ Value Use Case
python file.py (direct execution) '__main__' Standalone script, entry point
import file (as module) 'file' Use as library, import functions
from package import file 'package.file' Submodule import, namespacing
python -m module (module mode) '__main__' Run package as script

Deep Dive: sys.argv and Script Entry Points

+

When a script runs, `sys.argv` is a list containing the command-line arguments. `sys.argv[0]` is the script name.

File: process.py import sys if __name__ == '__main__': print(f"Script name: {sys.argv[0]}") print(f"Arguments: {sys.argv[1:]}")

Running `python process.py data.txt output.txt` produces:

Output Script name: process.py Arguments: ['data.txt', 'output.txt']

Importing Modules: The Mechanics

Import Statements: Four Variations

Python supports multiple import syntaxes, each with different implications for your namespace.

Variation 1: Import entire module import math print(math.pi) # Access via module.name
Variation 2: Import specific name from math import pi, sqrt print(pi) # Direct access result = sqrt(16)
Variation 3: Import all names (use sparingly) from math import * print(pi) # Danger: pollutes namespace
Variation 4: Import with aliasing import numpy as np from collections import Counter as Ctr print(np.array([1, 2, 3]))
Warning: Avoid `from module import *` in production code. It obscures what names are in scope and can cause name collisions.

The Search Sequence: Where Python Looks

When you write `import module`, Python searches for `module.py` in this order:

  1. Current directory: The directory containing the script you're running
  2. PYTHONPATH: Environment variable specifying additional directories
  3. Installation-dependent defaults: Standard library, site-packages (where pip installs)
  4. sys.path: The actual list of paths Python uses (modifiable at runtime)
Inspect sys.path import sys for path in sys.path: print(path)
Insight: You can modify `sys.path` at runtime to add custom directories, but this is a code smell in production. Use proper package structure instead.

Deep Dive: PEP 302 and the Import Protocol

+

Python 3.3+ uses a sophisticated import system defined by PEP 451. The `importlib` module provides programmatic access to the import machinery. When you `import foo`, Python:

  1. Checks if `foo` is already in `sys.modules` (the module cache)
  2. Searches for `foo.py` or a package named `foo` using finders
  3. Creates a module object and executes the code
  4. Stores the module in `sys.modules` for future imports
Direct module manipulation with importlib import importlib import sys # Dynamically import a module config = importlib.import_module('my_config') # Reload a module (useful during development) importlib.reload(config) # Check if module is cached print('my_config' in sys.modules) # True after first import

This cache mechanism is why changing a `.py` file doesn't take effect unless you restart the interpreter or call `reload()`.

Multiple Modules: Splitting Code Across Files

Building a Multi-Module Application

Real applications split code into logical modules for maintainability. Each module is responsible for a specific concern.

Project structure app/ ├── main.py ├── data.py ├── utils.py └── config.py
File: data.py - Responsible for data handling def load_data(filename): with open(filename) as f: return [line.strip() for line in f] def save_data(filename, data): with open(filename, 'w') as f: f.write('\n'.join(data))
File: utils.py - General utilities def format_output(text): return text.upper().replace('_', ' ') def validate_input(data): return all(isinstance(item, str) for item in data)
File: main.py - Orchestrates everything import data import utils if __name__ == '__main__': raw = data.load_data('input.txt') if utils.validate_input(raw): processed = [utils.format_output(item) for item in raw] data.save_data('output.txt', processed)

This structure allows each module to be tested independently and reused in other projects.

Packages: Hierarchical Organization

What is a Package?

A package is a directory containing a special `__init__.py` file. Packages organize modules into namespaces and prevent name conflicts. Subdirectories with their own `__init__.py` create sub-packages.

Package structure myproject/ ├── __init__.py ├── core/ │ ├── __init__.py │ ├── models.py │ └── database.py ├── utils/ │ ├── __init__.py │ ├── helpers.py │ └── formatters.py └── api/ ├── __init__.py ├── endpoints.py └── middleware.py

The __init__.py File

The `__init__.py` file marks a directory as a package. It can be empty, or it can define the package's public API. When a package is imported, Python executes `__init__.py` first.

File: myproject/__init__.py # Package initialization __version__ = '1.0.0' # Export public API from .core.models import User, Post from .utils.helpers import format_date __all__ = ['User', 'Post', 'format_date', '__version__']

Now users can import the public API directly:

Usage from myproject import User, Post # Instead of: # from myproject.core.models import User, Post

Absolute vs. Relative Imports

Relative imports use dots to refer to package structure. They're useful within a package but dangerous across packages.

Relative imports (within myproject/core/models.py) # Relative: from a sibling module from ..utils.helpers import format_date # Absolute: always works from myproject.utils.helpers import format_date
Best Practice: Use absolute imports in most code. Use relative imports only within packages, and avoid going up more than 2 levels (`..`).

Deep Dive: Namespace Packages and PEP 420

+

PEP 420 (Python 3.3+) introduced namespace packages—directories without `__init__.py`. They allow multiple locations to contribute to the same package namespace:

Namespace package structure path1/ mypkg/ module_a.py path2/ mypkg/ module_b.py # Both are importable as mypackage.* despite separate directories

This is useful for plugin systems, but introduces implicit coupling. For most use cases, explicit `__init__.py` packages are clearer.

Third-Party Packages: pip and PyPI

Installing Packages with pip

The Python Package Index (PyPI) is a repository of over 500,000 packages. `pip` (Package Installer for Python) downloads and installs them.

Common pip commands pip install requests # Install latest version pip install numpy==1.24.0 # Install specific version pip install "django>=4.0" # Install with constraint pip list # Show installed packages pip show requests # Show package details pip uninstall requests # Remove package

requirements.txt: Dependency Management

The `requirements.txt` file lists all dependencies for reproducibility. It's the industry standard for Python projects.

File: requirements.txt requests==2.31.0 numpy>=1.24.0,<2.0 django==4.2.5 flask>=2.0 pandas
Install from requirements.txt pip install -r requirements.txt
Best Practice: Always commit `requirements.txt` to version control. Use pinned versions (==) for production, and loose constraints (>=) for libraries.

Creating and Distributing Your Own Package

Modern Python uses `pyproject.toml` (PEP 621) to define package metadata. This replaces the older `setup.py` approach.

File: pyproject.toml [build-system] requires = ["setuptools>=61.0", "wheel"] build-backend = "setuptools.build_meta" [project] name = "my-package" version = "0.1.0" description = "A useful package" authors = [{name = "Author", email = "author@example.com"}] dependencies = [ "requests>=2.28", "numpy>=1.20", ] [project.optional-dependencies] dev = ["pytest>=7.0", "black", "mypy"] [project.scripts] my-tool = "my_package.cli:main"

The `[project.scripts]` section creates command-line entry points. When installed, `my-tool` becomes a system command that runs the specified function.

Deep Dive: Entry Points and Plugin Systems

+

Entry points allow packages to define extensible plugin systems. Other packages can register implementations of a plugin interface:

pyproject.toml (plugin definition) [project.entry-points."my_app.exporters"] json = "my_app.exporters:JSONExporter" csv = "my_app.exporters:CSVExporter"
Load plugins at runtime import importlib.metadata eps = importlib.metadata.entry_points(group='my_app.exporters') for ep in eps: exporter_class = ep.load() print(f"{ep.name}: {exporter_class}")

This pattern powers tools like pytest plugins, Flask extensions, and Django apps.

Advanced Import Patterns

Circular Imports: Detection and Resolution

Circular imports occur when module A imports B, and B imports A. Python's module cache partially prevents infinite loops, but creates incomplete modules.

File: a.py from b import helper_b def func_a(): return helper_b()
File: b.py from a import func_a # CIRCULAR! def helper_b(): return "B"

Solutions:

  1. Restructure: Move shared code to a third module
  2. Import inside functions: Defer import until it's needed
  3. Use TYPE_CHECKING guard: Import only for type hints
TYPE_CHECKING pattern (a.py) from typing import TYPE_CHECKING if TYPE_CHECKING: from b import SomeClass # Only imported for type hints def func_a(obj: 'SomeClass'): # String annotation return obj.method()

The sys.modules Cache

Once a module is imported, Python stores it in `sys.modules`. Subsequent imports return the cached module, not a fresh execution.

Module caching example import sys import mymodule # mymodule.py is executed once print(id(mymodule)) # Memory address print(sys.modules['mymodule']) # Same object del mymodule import mymodule # Returns cached version, doesn't re-execute # To force re-execution import importlib importlib.reload(mymodule)

Conditional Imports and Optional Dependencies

Some packages have optional features that depend on third-party libraries. Use try/except to handle missing dependencies gracefully.

Graceful degradation try: import numpy as np HAS_NUMPY = True except ImportError: HAS_NUMPY = False if HAS_NUMPY: def process_array(data): return np.array(data) * 2 else: def process_array(data): return [x * 2 for x in data]
Insight: This pattern allows your package to work with or without optional dependencies, providing basic functionality without heavy requirements.

Socratic Q&A: Test Your Understanding

Engage with these questions to deepen your understanding of modules and packages. Click each question to reveal the answer.

1. Why would you use `if __name__ == '__main__':` in a module?
+

This pattern allows code to detect whether the module is being run directly or imported. When run directly, `__name__` is `'__main__'`, so the guarded code executes. When imported, `__name__` is the module name, so the code is skipped. This prevents side effects (like printing or running tests) when the code is imported as a library. It's the standard way to make modules both executable scripts and importable libraries.

2. What's the difference between `import math` and `from math import pi`?
+

`import math` imports the entire module and adds the name `math` to your namespace. You access items as `math.pi`. `from math import pi` imports only the name `pi` directly into your namespace, so you access it as just `pi`. The first keeps everything under the module name (cleaner for large modules), while the second is more concise for a few items. Avoid `from module import *` because it obscures what names are in scope.

3. What order does Python use to search for a module when you write `import foo`?
+

Python searches in this order: (1) current directory, (2) directories in the PYTHONPATH environment variable, (3) installation-dependent defaults (standard library, site-packages where pip installs), (4) sys.path (which combines all above). You can see the exact order by printing `sys.path`. The first match wins, so a file named `foo.py` in the current directory will shadow a system-wide `foo` package.

4. What is the purpose of `__init__.py` in a package?
+

`__init__.py` marks a directory as a Python package (not just a folder of scripts). When the package is imported, Python executes `__init__.py` first. It can be empty, or it can define the package's public API by importing names from submodules. This allows users to do `from mypackage import User` instead of `from mypackage.models import User`, controlling what's public and simplifying the interface.

5. How do relative imports work, and when should you use them?
+

Relative imports use dots to navigate package hierarchy. A single dot `.` refers to the current package, `..` refers to the parent package. For example, within `mypackage/core/models.py`, you can write `from ..utils import helpers` to import from `mypackage/utils/`. Use relative imports within a package for code that belongs together, but stick to absolute imports in most cases (they're clearer and work everywhere). Avoid going up more than 2 levels as it suggests poor structure.

6. What does `pip install -r requirements.txt` do, and why is this important?
+

`pip install -r requirements.txt` reads a text file listing all project dependencies (with optional version constraints) and installs them. This ensures everyone working on a project uses the same package versions, preventing "it works on my machine" problems. It's the standard way to capture and reproduce a project's environment. Always commit `requirements.txt` to version control; never rely on manual installation instructions.

7. What is a circular import, and how can you fix it?
+

A circular import occurs when module A imports B, and B imports A. Python's module cache prevents infinite loops, but creates incomplete modules (functions/classes might not exist when needed), causing `NameError`. Fix it by: (1) restructuring code to move shared definitions to a third module, (2) importing inside functions (defers import until execution), or (3) using `TYPE_CHECKING` guard for type-only imports. The root cause usually indicates poor separation of concerns.

8. What is `sys.modules` and why should you know about it?
+

`sys.modules` is a dictionary caching all imported modules. The first time you `import foo`, Python executes `foo.py` and stores the module object in `sys.modules['foo']`. Subsequent imports return the cached object without re-executing the code. This is why changes to a `.py` file don't take effect until you restart the interpreter. You can modify `sys.modules` at runtime (rarely needed) or call `importlib.reload()` to force re-execution during development.

9. How does `pyproject.toml` define package metadata, and why is it better than `setup.py`?
+

`pyproject.toml` (PEP 621) uses declarative TOML format to specify package name, version, dependencies, and entry points. It's standardized, human-readable, and works with any build backend. The older `setup.py` was a Python script requiring custom logic to parse; `pyproject.toml` is declarative (less error-prone) and language-agnostic. Modern tools like setuptools, Poetry, and Hatch all support `pyproject.toml`, making it the future of Python packaging.

10. What are entry points, and how do they enable plugin systems?
+

Entry points are named exports defined in `pyproject.toml` under `[project.scripts]` or `[project.entry-points]`. They map a command name to a Python function (for scripts) or group name to implementations (for plugins). For example, `pytest` plugins register themselves as entry points, and pytest discovers them at runtime. This allows third-party packages to extend your application without modifying your code—the package is discovered automatically via the import system and `importlib.metadata`.

11. How do you handle optional dependencies in your package?
+

Use try/except blocks at import time: try to import an optional library, set a flag if it's available, and provide fallback implementations. For example, if `numpy` is optional, try `import numpy as np; HAS_NUMPY = True` and catch `ImportError` to set `HAS_NUMPY = False`. Then define different implementations based on the flag. In `pyproject.toml`, define optional dependencies under `[project.optional-dependencies]` with a name like `dev` or `data`, so users can install with `pip install mypackage[data]` to get optional features.

12. Explain the difference between absolute and relative imports in practice.
+

Absolute imports use the full path from the root package: `from mypackage.utils import helpers`. Relative imports use dots within a package: `from ..utils import helpers` (one level up, then into utils). Absolute imports work everywhere and are clearer; use them as the default. Relative imports are useful within tightly-coupled packages but harder to follow. The rule: use absolute imports, only use relative imports within a package, and avoid going up more than one level (`..`).

Key Takeaways

  • Every Python file can be a script OR a module—`__name__` determines which. Guard code with `if __name__ == '__main__':`
  • Packages are directories with `__init__.py`, organizing modules into namespaces to prevent name collisions
  • Python's import system searches in a defined order: current dir, PYTHONPATH, installation defaults, sys.path
  • Use `requirements.txt` to document dependencies; use `pyproject.toml` to define package metadata
  • Avoid circular imports through restructuring, deferred imports, or TYPE_CHECKING guards
  • Entry points enable plugin systems—packages can register implementations for your application to discover
  • Modern Python packaging emphasizes `pyproject.toml` over `setup.py`, and absolute imports over relative ones
