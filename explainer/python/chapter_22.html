Inheritance & Composition

Code Reuse Through Hierarchy and Containment

Reuse Mechanisms: Inheritance vs Composition

The fundamental question in OOP design: how do you reuse code across multiple classes? Python offers two primary mechanisms: inheritance and composition.

Inheritance: "Is-A" Relationship

A child class inherits from a parent class, gaining all its attributes and methods. The child "is-a" specialized version of the parent. Example: Dog is-a Animal.

Composition: "Has-A" Relationship

A class contains instances of other classes as attributes. The container "has-a" component. Example: Car has-a Engine.

When to Use Which?

  • Inheritance: When you have a natural "is-a" relationship and the child truly specializes the parent
  • Composition: When the relationship is "has-a", or when you need flexibility and to avoid deep hierarchies
  • Rule of thumb: "Prefer composition over inheritance" — it's more flexible and maintainable

Containership: Composition in Action

Composition means building classes by combining other classes as attributes. This is often more flexible than inheritance. 

class Engine:
    def __init__(self, horsepower):
        self.horsepower = horsepower

    def start(self):
        return f"Engine with {self.horsepower} HP starting..."

class Car:
    def __init__(self, make, model, engine):
        self.make = make
        self.model = model
        self.engine = engine  # Composition: Car "has-a" Engine

    def start(self):
        return self.engine.start()

# Create an engine, then use it in a car
v8_engine = Engine(500)
my_car = Car("Ferrari", "F8", v8_engine)

print(my_car.start())  # Delegates to the engine

Advantages of Composition

  • Flexibility: Swap components at runtime
  • Simplicity: Easier to understand — no deep class hierarchies
  • Reusability: Components can be reused in multiple classes
  • Avoids fragile base class problem: Changes to parent classes don't ripple through
# Composition example: flexible component swapping
class Battery:
    def __init__(self, capacity):
        self.capacity = capacity

class ElectricCar:
    def __init__(self, make, battery):
        self.make = make
        self.battery = battery

    def range(self):
        return self.battery.capacity * 5  # 5 km per unit capacity

# You can swap batteries!
standard_battery = Battery(50)
my_car = ElectricCar("Tesla", standard_battery)
print(my_car.range())  # 250 km

# Upgrade the battery
upgraded_battery = Battery(100)
my_car.battery = upgraded_battery
print(my_car.range())  # 500 km

Inheritance: Building Hierarchies

Basic Inheritance Syntax

Use class Child(Parent): to inherit from a parent class. The child class automatically has all the parent's attributes and methods. 

class Animal:
    def __init__(self, name):
        self.name = name

    def speak(self):
        return f"{self.name} makes a sound"

class Dog(Animal):  # Dog inherits from Animal
    def speak(self):  # Override parent method
        return f"{self.name} barks"

class Cat(Animal):
    def speak(self):
        return f"{self.name} meows"

# Creating instances
dog = Dog("Buddy")
cat = Cat("Whiskers")

print(dog.speak())  # Buddy barks (uses Dog's speak)
print(cat.speak())  # Whiskers meows (uses Cat's speak)

# Both inherit __init__ from Animal
print(dog.name)     # Buddy
print(cat.name)     # Whiskers

Calling Parent Methods with super()

Use super() to call the parent's method. This is essential when you override a method but still want the parent's behavior. 

class Vehicle:
    def __init__(self, make, model):
        self.make = make
        self.model = model

    def info(self):
        return f"{self.make} {self.model}"

class Car(Vehicle):
    def __init__(self, make, model, num_doors):
        super().__init__(make, model)  # Call parent's __init__
        self.num_doors = num_doors

    def info(self):
        parent_info = super().info()  # Call parent's info()
        return f"{parent_info} ({self.num_doors} doors)"

car = Car("Toyota", "Camry", 4)
print(car.info())  # Toyota Camry (4 doors)

Inheritance Hierarchy

Inheritance creates a hierarchy of classes. A child inherits from a parent; a grandchild inherits from a child. At the top is object, the root of all Python classes.

What is Accessible Where? Attribute Resolution

When you access obj.attr, Python searches for it in a specific order, following the inheritance chain. This is called the Method Resolution Order (MRO).

The Search Order (MRO)

  1. Instance __dict__ (instance attributes)
  2. Class __dict__ (class attributes, methods)
  3. Parent class __dict__
  4. Continue up the inheritance chain
  5. The object class (root of all classes)
  6. Raise AttributeError if not found
class GrandParent:
    grandparent_attr = "from grandparent"

class Parent(GrandParent):
    parent_attr = "from parent"

class Child(Parent):
    child_attr = "from child"

    def __init__(self):
        self.instance_attr = "from instance"

obj = Child()

# All are accessible
print(obj.instance_attr)      # from instance (instance dict)
print(obj.child_attr)         # from child (Child class dict)
print(obj.parent_attr)        # from parent (Parent class dict)
print(obj.grandparent_attr)   # from grandparent (GrandParent class dict)

The MRO in Multiple Inheritance

Use Class.mro() or Class.__mro__ to see the exact resolution order. 

class A:
    pass

class B(A):
    pass

class C(A):
    pass

class D(B, C):
    pass

# View the Method Resolution Order
print(D.mro())
# [, , , , ]

# Python uses C3 linearization algorithm to determine this order
# (covered in expert section)

isinstance() and issubclass()

Use isinstance(obj, Class) to check if an object is an instance of a class (or a subclass of it). Use issubclass(ChildClass, ParentClass) to check class relationships. 

class Animal:
    pass

class Dog(Animal):
    pass

class Cat(Animal):
    pass

dog = Dog()
cat = Cat()

# isinstance checks instance relationships
print(isinstance(dog, Dog))     # True
print(isinstance(dog, Animal))  # True (Dog is a subclass of Animal)
print(isinstance(cat, Dog))     # False

# issubclass checks class relationships
print(issubclass(Dog, Animal))  # True
print(issubclass(Cat, Animal))  # True
print(issubclass(Dog, Cat))     # False

# Useful in polymorphic code
def feed(animal):
    if isinstance(animal, Dog):
        return "Dog food"
    elif isinstance(animal, Cat):
        return "Cat food"
    else:
        return "Unknown food"

print(feed(dog))  # Dog food

Duck Typing vs Type Checking

Python embraces "duck typing": if it walks like a duck and quacks like a duck, it's a duck. Rather than checking isinstance(obj, Dog), check if obj has the methods you need. This is more Pythonic but sometimes isinstance is clearer.

The object Class: Root of All Classes

In Python, all classes inherit from object, the root class. Every class automatically gets methods from object like __init__, __str__, __repr__, and more. 

# These are equivalent
class A:
    pass

class B(object):
    pass

# Both inherit from object
print(issubclass(A, object))  # True
print(issubclass(B, object))  # True

# So all classes inherit object's methods
class MyClass:
    pass

obj = MyClass()
print(hasattr(obj, '__str__'))     # True (from object)
print(hasattr(obj, '__class__'))   # True (from object)
print(hasattr(obj, '__dict__'))    # True (from object)

This is why you can call str(obj), repr(obj), and use hasattr() on any object — they all have these methods from the object base class.

Features of Inheritance: Overriding and Extending

Method Overriding

When a child class defines a method with the same name as a parent method, the child's version is used. This is called method overriding. 

class Shape:
    def area(self):
        return 0

class Circle(Shape):
    def __init__(self, radius):
        self.radius = radius

    def area(self):  # Override parent's area()
        return 3.14159 * self.radius ** 2

class Square(Shape):
    def __init__(self, side):
        self.side = side

    def area(self):  # Override parent's area()
        return self.side ** 2

circle = Circle(5)
square = Square(4)

print(circle.area())  # Uses Circle's area(): 78.5
print(square.area())  # Uses Square's area(): 16

Method Extension

Use super() to call the parent's method and add additional behavior. 

class Animal:
    def __init__(self, name):
        self.name = name

    def speak(self):
        return f"{self.name} makes a sound"

class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name)      # Call parent __init__
        self.breed = breed          # Add child-specific attribute

    def speak(self):
        base_sound = super().speak()  # Call parent speak()
        return f"{base_sound} and barks"

dog = Dog("Buddy", "Golden Retriever")
print(dog.speak())  # Buddy makes a sound and barks

Types of Inheritance

Single Inheritance

A class inherits from one parent.

class Parent:
    pass

class Child(Parent):
    pass

Multiple Inheritance

A class inherits from multiple parents. Use commas to list all parents.

class Swimmer:
    def swim(self):
        return "Swimming"

class Flyer:
    def fly(self):
        return "Flying"

class Duck(Swimmer, Flyer):  # Duck inherits from both
    pass

duck = Duck()
print(duck.swim())  # Swimming
print(duck.fly())   # Flying

Multilevel Inheritance

A class inherits from a class that inherits from another class.

class Animal:
    pass

class Mammal(Animal):
    pass

class Dog(Mammal):
    pass

# Dog inherits from Mammal, which inherits from Animal

Hierarchical Inheritance

Multiple classes inherit from the same parent.

class Animal:
    pass

class Dog(Animal):
    pass

class Cat(Animal):
    pass

# Both Dog and Cat inherit from Animal

Hybrid Inheritance

A combination of multiple inheritance types.

class Animal:
    pass

class Mammal(Animal):
    pass

class Swimmer:
    pass

class Whale(Mammal, Swimmer):  # Multiple + Multilevel
    pass

The Diamond Problem and C3 Linearization

The "diamond problem" occurs in multiple inheritance when a class inherits from two parents that both inherit from the same grandparent. Which grandparent method should be called? 

#     A
                #    / \
                #   B   C
                #    \ /
                #     D

class A:
    def method(self):
        return "A's method"

class B(A):
    pass

class C(A):
    pass

class D(B, C):  # Diamond!
    pass

d = D()
print(d.method())  # Which A.method? B's or C's?

# Python uses C3 linearization to resolve this
print(D.mro())
# [D, B, C, A, object]
# So it uses B's path first, then C's, avoiding duplicates

Deep Dive: C3 Linearization Algorithm

▼

C3 linearization is the algorithm Python uses to determine the MRO. It guarantees three properties:

  1. Children come before parents
  2. Parent order is preserved
  3. Each class appears exactly once
class A:
    pass

class B(A):
    pass

class C(A):
    pass

class D(B, C):
    pass

# MRO: [D, B, C, A, object]
# D first (child)
# B before C (matches D's parent order)
# A after B and C (parents before grandparents)
# object last (root)
# A appears once (no duplicates)

Abstract Base Classes (ABC)

Abstract base classes define an interface (set of methods) that subclasses must implement. Use the abc module to enforce this. 

from abc import ABC, abstractmethod

class Shape(ABC):  # ABC = Abstract Base Class
    @abstractmethod
    def area(self):
        """Subclasses must implement this"""
        pass

    @abstractmethod
    def perimeter(self):
        """Subclasses must implement this"""
        pass

# Can't instantiate an abstract class
# shape = Shape()  # TypeError: Can't instantiate abstract class

class Circle(Shape):
    def __init__(self, radius):
        self.radius = radius

    def area(self):
        return 3.14159 * self.radius ** 2

    def perimeter(self):
        return 2 * 3.14159 * self.radius

# Can instantiate concrete subclass
circle = Circle(5)
print(circle.area())  # Works!

Enforcing Contracts

If a subclass doesn't implement all abstract methods, Python raises a TypeError. 

class Shape(ABC):
    @abstractmethod
    def area(self):
        pass

class BadShape(Shape):
    pass

# obj = BadShape()
# TypeError: Can't instantiate abstract class BadShape
# with abstract method area

When to Use Abstract Classes

  • Define a common interface for related classes
  • Ensure subclasses implement required methods
  • Use @abstractmethod for methods that must be overridden
  • Consider @abstractproperty for abstract properties

Runtime Polymorphism and Duck Typing

Polymorphism means "many forms." In OOP, it means that the same method call can produce different behavior depending on the object's actual type.

Polymorphism Through Inheritance

class Animal:
    def speak(self):
        pass

class Dog(Animal):
    def speak(self):
        return "Woof!"

class Cat(Animal):
    def speak(self):
        return "Meow!"

# Polymorphic code: works with any Animal subclass
def animal_sound(animal):
    return animal.speak()

dog = Dog()
cat = Cat()

print(animal_sound(dog))   # Woof!
print(animal_sound(cat))   # Meow!

# The function doesn't need to know the specific type
# It works with any object that has a speak() method

Duck Typing

"If it walks like a duck and quacks like a duck, it's a duck." Python doesn't require explicit inheritance — if an object has the right methods, it works. 

class NotAnimal:
    def speak(self):
        return "I'm not an Animal, but I can speak!"

# This works even though NotAnimal doesn't inherit from Animal!
def animal_sound(animal):
    return animal.speak()

obj = NotAnimal()
print(animal_sound(obj))  # I'm not an Animal, but I can speak!

# No inheritance needed — just having the method is enough

Polymorphism vs Duck Typing

  • Polymorphism: Same method, different implementations, based on type hierarchy
  • Duck Typing: Same method, different implementations, based on interface (methods)
  • Python embraces duck typing more than strict polymorphism
  • Both lead to flexible, reusable code

Expert Insights: Advanced Inheritance Patterns

super() Mechanics: MRO in Action

▼

super() doesn't call the parent class — it calls the next class in the MRO. This subtle distinction is crucial for multiple inheritance. 

class A:
    def method(self):
        return "A"

class B(A):
    def method(self):
        return "B -> " + super().method()

class C(A):
    def method(self):
        return "C -> " + super().method()

class D(B, C):
    pass

# MRO: [D, B, C, A, object]
d = D()
print(d.method())
# B -> C -> A
# super() in B goes to C (next in MRO)
# super() in C goes to A (next in MRO)

Liskov Substitution Principle (LSP)

▼

Objects of a subclass should be substitutable for objects of the parent class without breaking the program. 

# Good: LSP adhered
class Bird:
    def move(self):
        return "Moving"

class Sparrow(Bird):
    def move(self):
        return "Flying"

# Bad: violates LSP
class Penguin(Bird):
    def move(self):
        raise NotImplementedError("Penguins don't fly!")

# Code expecting a Bird might call move() and crash
# if it gets a Penguin!

Protocols: Structural Subtyping vs ABCs

▼

Protocol (from typing) is like ABC but doesn't require inheritance. It defines a structural interface — if your class has the methods, it's compatible. 

from typing import Protocol

class Drawable(Protocol):
    def draw(self) -> None: ...

class Circle:
    def draw(self) -> None:
        print("Drawing circle")

class Square:
    def draw(self) -> None:
        print("Drawing square")

# Both Circle and Square implement Drawable (by duck typing)
# No inheritance needed!

def render(obj: Drawable) -> None:
    obj.draw()

render(Circle())   # Works
render(Square())   # Works

Mixin Pattern: Reusing Behavior Across Unrelated Classes

▼

A mixin is a class designed to be mixed into other classes via multiple inheritance, providing reusable functionality without being a parent class. 

class LoggingMixin:
    """Adds logging capability to any class"""
    def log(self, msg):
        print(f"[LOG] {msg}")

class Dog(LoggingMixin):
    def bark(self):
        self.log("Barking!")
        return "Woof!"

class Car(LoggingMixin):
    def start(self):
        self.log("Starting engine!")
        return "Engine started"

dog = Dog()
dog.log("Creating dog...")  # Mixin provides this
dog.bark()

car = Car()
car.log("Creating car...")  # Same mixin works here
car.start()

Socratic Questions: Test Your Understanding

1. What's the difference between "is-a" and "has-a" relationships?

▼
Is-a: Inheritance. "Dog is-a Animal" — Dog inherits from Animal. Has-a: Composition. "Car has-a Engine" — Car contains an Engine instance. Choose inheritance for natural specialization; choose composition for flexibility.

2. When should you use composition over inheritance?

▼
Almost always! Composition is more flexible: you can swap components at runtime, avoid deep hierarchies, and prevent the fragile base class problem. Use inheritance only when you have a clear "is-a" relationship and true specialization.

3. What does super() do and why is it important?

▼
super() calls the next class in the MRO (Method Resolution Order). It's important because it allows child classes to extend parent behavior without hardcoding the parent's name, making code more maintainable and enabling proper multiple inheritance.

4. Explain the Method Resolution Order (MRO).

▼
The MRO is the order in which Python searches for attributes/methods in a class hierarchy. It goes: instance → class → parents → grandparents → object. In multiple inheritance, Python uses C3 linearization to determine the MRO, ensuring children come before parents and respecting parent order.

5. What is method overriding and how does it enable polymorphism?

▼
Method overriding is when a child class defines a method with the same name as the parent. When called, the child's version is used. This enables polymorphism: the same method call produces different behavior depending on the object's actual type.

6. What is the diamond problem and how does Python solve it?

▼
The diamond problem occurs when a class inherits from two classes that both inherit from a common ancestor. Which ancestor's method gets called? Python solves it with C3 linearization, which produces a consistent, predictable MRO that avoids calling methods twice.

7. When would you use an abstract base class (ABC)?

▼
Use ABCs to define a contract: subclasses must implement certain methods. Use @abstractmethod to mark methods that subclasses must override. This prevents instantiation of incomplete classes and documents the required interface.

8. What is duck typing and how does it relate to polymorphism?

▼
Duck typing: "If it walks like a duck and quacks like a duck, it's a duck." Your code doesn't check isinstance(obj, Duck) — it just calls methods and assumes they exist. This is more flexible than strict polymorphism because it doesn't require inheritance or explicit type declarations.

9. What's the difference between isinstance() and issubclass()?

▼
isinstance(obj, Class) checks if obj is an instance of Class (or a subclass). issubclass(Child, Parent) checks if Child is a subclass of Parent. Use isinstance for objects, issubclass for classes.

10. What is the Liskov Substitution Principle?

▼
Objects of a subclass should be substitutable for objects of the parent class without breaking the program. If you have code expecting a Bird, it should work with any Bird subclass. Violating LSP (e.g., a Penguin that can't fly when Bird.move() is expected to fly) breaks polymorphism.

11. What is a mixin and how does it differ from inheritance?

▼
A mixin is a class providing reusable functionality via multiple inheritance, but without being a "parent" in the traditional sense. Mixins add specific capabilities (like logging) to unrelated classes. Difference: inheritance creates an "is-a" relationship; mixins just provide shared functionality.

12. Why use Protocols over ABC for type checking?

▼
Protocol uses structural subtyping (duck typing): if a class has the right methods, it matches the protocol, even without explicit inheritance. ABCs require inheriting from the abstract class. Protocols are more flexible and better for type hints in large codebases.
