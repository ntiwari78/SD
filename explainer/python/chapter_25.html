Exception Handling

Write robust Python code that gracefully handles errors and recovers from unexpected situations

🧠 Mental Model: The Exception Hierarchy

Python has a built-in hierarchy of exceptions. When an error occurs, Python raises an exception (creates an exception object) and looks for a handler. If no handler catches it, the program crashes.

Think of exceptions like emergency protocols: different types of emergencies (syntax errors, runtime errors, custom problems) are handled differently. You write handlers to catch specific emergency types and respond appropriately.

Two categories: Errors (compile-time mistakes like syntax errors) vs Exceptions (runtime problems you can potentially recover from).

What May Go Wrong: Errors vs Exceptions

Syntax Errors

Syntax errors are compile-time errors—Python detects them before your code runs. You can't catch them.

# SyntaxError: invalid syntax
if x = 5:
    print(x)

# SyntaxError: unexpected EOF while parsing
def incomplete_function():
    pass
    (missing closing paren)

# IndentationError (subclass of SyntaxError)
def bad_indent():
print("misaligned")

Syntax errors must be fixed in your source code; you can't handle them with try-except.

Runtime Errors: Exceptions

Runtime errors occur while your program is executing. These are exceptions, and you can potentially handle them:

# ZeroDivisionError
result = 10 / 0

# NameError
print(undefined_variable)

# TypeError
result = "string" + 5

# IndexError
my_list = [1, 2, 3]
print(my_list[10])

# KeyError
my_dict = {'a': 1}
print(my_dict['b'])

# FileNotFoundError
with open('nonexistent.txt') as f:
    pass

# ValueError
number = int("not a number")

# AttributeError
class MyClass:
    pass
obj = MyClass()
print(obj.undefined_attribute)

These exceptions happen at runtime and can be caught and handled gracefully.

Why Exceptions Exist

Exceptions are Python's way of saying "something unexpected happened, but I'm not going to silently fail or return None." They force you to think about error handling explicitly.

Handling Exceptions with try-except

Basic try-except Structure

The try-except block lets you catch and handle exceptions:

try:
    # Code that might raise an exception
    result = 10 / 0
except ZeroDivisionError:
    # Handle the specific exception
    print("Cannot divide by zero!")
    result = 0

print(result)  # 0

If an exception occurs in the try block, Python immediately jumps to the matching except block. Code after the except block continues normally.

Catching Specific vs General Exceptions

Always catch specific exceptions when possible:

# GOOD: Specific exception
try:
    number = int(user_input)
except ValueError:
    print("Please enter a valid number")

# BAD: Too general
try:
    number = int(user_input)
except Exception:
    print("Something went wrong")  # Hides the real problem

# VERY BAD: Catches everything, even SystemExit
try:
    number = int(user_input)
except:
    print("Suppresses KeyboardInterrupt too!")

Specific exceptions help you understand what went wrong and respond appropriately. Generic catching hides bugs.

Multiple except Blocks

Handle different exceptions differently:

try:
    file = open('data.txt')
    content = file.read()
    value = int(content)
except FileNotFoundError:
    print("File doesn't exist")
except ValueError:
    print("File content is not a valid number")
except Exception as e:
    print(f"Unexpected error: {e}")

Python checks except blocks top-to-bottom and executes the first matching one. A catch-all Exception handler at the end is a safety net.

Catching Multiple Exceptions in One Handler

Use a tuple to catch multiple exception types:

try:
    result = dictionary[key]
except (KeyError, TypeError):
    print("Either key doesn't exist or dictionary is wrong type")
    result = None

This is cleaner than repeating the same handler code.

Exception Object

You can access the exception object with as variable_name. This gives you the error message and other information about what went wrong.

try-except Nuances

The else Block

Code in the else block runs only if NO exception occurred:

try:
    file = open('data.txt')
except FileNotFoundError:
    print("File not found")
else:
    # Only runs if file opened successfully
    content = file.read()
    file.close()
    print(f"Read {len(content)} bytes")

The else block clarifies "this code only runs if everything was OK." It's cleaner than putting all success code in the try block.

The finally Block

Code in finally ALWAYS runs, whether an exception occurred or not:

try:
    file = open('data.txt')
    content = file.read()
except FileNotFoundError:
    print("File not found")
finally:
    # Always runs, even if exception occurred
    print("Cleanup code here")

Use finally for cleanup tasks (closing files, releasing locks, etc.):

try:
    connection = open_database()
    result = connection.query("SELECT * FROM users")
except ConnectionError:
    print("Database connection failed")
finally:
    # Ensures cleanup happens even if query fails
    if connection:
        connection.close()

This pattern is so common that Python provides the with statement for automatic cleanup (context managers).

Complete try-except-else-finally

You can combine all blocks:

try:
    file = open('data.txt')
    content = file.read()
except FileNotFoundError:
    print("File not found, using default")
    content = ""
except IOError as e:
    print(f"IO error: {e}")
    content = ""
else:
    print(f"Successfully read {len(content)} bytes")
finally:
    print("Cleanup: closing file")
    if 'file' in locals():
        file.close()

Order: try → except(s) → else → finally. Finally always runs last, even if else ran.

User-Defined Exceptions

Creating Custom Exceptions

Inherit from Exception to create your own exception types:

class InsufficientFundsError(Exception):
    """Raised when account balance is too low"""
    pass

class InvalidAccountError(Exception):
    """Raised when account number is invalid"""
    pass

# Using custom exceptions
def withdraw(account, amount):
    if account.balance < amount:
        raise InsufficientFundsError(f"Need ${amount}, only have ${account.balance}")
    if not account.is_valid():
        raise InvalidAccountError("Account is closed or invalid")
    account.balance -= amount
    return account.balance

# Handling custom exceptions
try:
    balance = withdraw(account, 500)
except InsufficientFundsError as e:
    print(f"Withdrawal failed: {e}")
except InvalidAccountError as e:
    print(f"Account error: {e}")

Exception Hierarchy for Your Code

Create a hierarchy of custom exceptions for complex applications:

# Base exception for your application
class AppError(Exception):
    """Base exception for this application"""
    pass

# Domain-specific exceptions
class ValidationError(AppError):
    """Data validation failed"""
    pass

class DatabaseError(AppError):
    """Database operation failed"""
    pass

class AuthenticationError(AppError):
    """User authentication failed"""
    pass

# Now you can catch all app errors or specific ones
try:
    do_something()
except ValidationError:
    print("Input validation failed")
except DatabaseError:
    print("Database error occurred")
except AppError:
    print("Some other app error")
except Exception:
    print("Unexpected system error")

This hierarchy lets you catch "any app error" with except AppError or specific errors with except ValidationError.

Include useful information in your custom exceptions:

Exception Handling Best Practices

Be Specific, Not Generic

Catch the specific exceptions you expect:

# GOOD: Catches what we expect
try:
    number = int(user_input)
except ValueError:
    number = 0

# BAD: Catches too much
try:
    number = int(user_input)
except:
    number = 0

If your function later introduces a bug (like accessing an undefined variable), the bad version would hide it silently.

Avoid Bare except

Never use bare except: without an exception type:

# TERRIBLE: Catches everything including SystemExit, KeyboardInterrupt
try:
    something()
except:
    pass

# GOOD: Specific exceptions only
try:
    something()
except ValueError:
    pass

# ACCEPTABLE: At least catch Exception, not BaseException
try:
    something()
except Exception:
    pass

EAFP vs LBYL

EAFP (Easier to Ask for Forgiveness than Permission) is the Pythonic way:

# EAFP: Pythonic approach
try:
    value = dictionary['key']
except KeyError:
    value = None

# LBYL (Look Before You Leap): Not Pythonic
if 'key' in dictionary:
    value = dictionary['key']
else:
    value = None

# Why EAFP is better:
# 1. No race condition (key might be deleted between the check and access)
# 2. More efficient (one lookup vs two)
# 3. Pythonic idiom

With EAFP, you assume success and handle the exception if it occurs. With LBYL, you check conditions before acting. Python culture favors EAFP.

When to Use try-except vs Validation

Use try-except for expected operational errors (file not found, invalid user input converted to int). Use validation for checking assumptions in your code logic (ensuring preconditions are met).

Expert Topics

Exception Chaining

When handling an exception, you might raise a different exception. Use raise ... from ... to preserve the original error:

def load_config(filename):
    try:
        with open(filename) as f:
            return json.load(f)
    except FileNotFoundError as e:
        # Chain exceptions: the original FileNotFoundError is preserved
        raise ConfigError(f"Config file {filename} not found") from e

try:
    config = load_config("missing.json")
except ConfigError as e:
    print(f"Error: {e}")
    print(f"Caused by: {e.__cause__}")  # Original FileNotFoundError

Exception chaining preserves the full error traceback, helping with debugging. The __cause__ attribute contains the original exception.

Deep Dive: ExceptionGroup (Python 3.11+)

▼

Python 3.11 introduced ExceptionGroup to handle multiple exceptions at once:

try:
    # Code that might raise multiple exceptions
    with contextlib.suppress(Exception):
        raise ExceptionGroup("multiple errors", [
            ValueError("first error"),
            TypeError("second error"),
            RuntimeError("third error")
        ])
except ExceptionGroup as eg:
    # Handle multiple exceptions
    for exc in eg.exceptions:
        print(f"Caught: {exc}")

# except* syntax for specific exception types
try:
    operations()
except* ValueError as eg:
    print(f"Value errors: {eg.exceptions}")
except* TypeError as eg:
    print(f"Type errors: {eg.exceptions}")
except* Exception as eg:
    print(f"Other errors: {eg.exceptions}")

This is useful for async operations that fail in multiple places simultaneously.

Working with Tracebacks

The traceback module lets you examine exception details:

import traceback
import sys

try:
    problem_function()
except Exception as e:
    # Get the full traceback as a string
    tb_str = traceback.format_exc()
    print(tb_str)

    # Get current exception info
    exc_type, exc_value, exc_traceback = sys.exc_info()
    print(f"Type: {exc_type}")
    print(f"Value: {exc_value}")

    # Print just the traceback
    traceback.print_exc()

Useful for logging detailed error information or implementing custom error reporting.

Context Managers and Exception Safety

The with statement automatically handles exceptions with context managers:

class DatabaseConnection:
    def __enter__(self):
        print("Opening connection")
        self.connection = connect_db()
        return self.connection

    def __exit__(self, exc_type, exc_val, exc_tb):
        print("Closing connection")
        self.connection.close()
        # Return True to suppress the exception, False to propagate
        return False

# Connection is automatically closed even if error occurs
try:
    with DatabaseConnection() as db:
        db.query("SELECT * FROM users")
        raise ValueError("Something went wrong")
except ValueError:
    print("Handled ValueError, but connection was closed")

Context managers guarantee cleanup code runs, even when exceptions occur. Much safer than try-finally in most cases.

Socratic Q&A: Test Your Understanding

What's the difference between a syntax error and an exception? Can you catch either? ▼

Syntax errors occur at compile-time (before code runs) and prevent execution entirely. You cannot catch them with try-except because your code never runs. Exceptions occur at runtime and CAN be caught. If Python detects a syntax error, it won't even try to run the program.

Why does Python prefer "EAFP" over "LBYL"? Give a concrete race condition example. ▼

LBYL (Look Before You Leap) checks first: "if key in dict then access". EAFP (Easier to Ask Forgiveness) assumes it works and catches exceptions. Race condition example: Between checking "if file exists" and opening it, another process deletes the file. EAFP avoids this by doing one atomic operation. EAFP is also faster (one dict lookup vs two) and more Pythonic.

When you use `except Exception`, what exceptions will it NOT catch? ▼

except Exception catches most runtime errors but NOT: SystemExit, KeyboardInterrupt, GeneratorExit, or anything inheriting directly from BaseException. This is intentional—you usually want Ctrl+C to work and SystemExit to propagate. Only bare except: catches everything, which is why it's dangerous.

What order do try-except-else-finally execute? What if the else block raises an exception? ▼

Order: (1) try block, (2) if exception, matching except block, (3) if no exception, else block, (4) finally block always last. If the else block raises an exception, that new exception is not caught by the except handlers—it propagates up (unless caught externally). Finally still runs. Think of finally as "guaranteed cleanup" no matter what happens.

Why would you create a custom exception hierarchy instead of using built-in exceptions? ▼

Custom exceptions make error handling more explicit and domain-specific. A base exception class like `AppError` lets callers catch "any error from my application" with one handler, separate from system exceptions. It also documents what errors your code might raise, improving API clarity. Callers can handle "any database error" vs all exceptions.

What happens if an exception is raised in a finally block? Does it replace the original exception? ▼

Yes, if finally raises an exception, it REPLACES any original exception. The original exception is lost. This is dangerous—cleanup code in finally should be robust and not raise exceptions. If it must handle exceptions, wrap it in its own try-except to avoid masking the original error.

How does exception chaining with `raise ... from ...` improve debugging? ▼

Without chaining, you lose information about what caused an exception. With raise NewException(...) from original_exc, both exceptions appear in the traceback. This shows the full error history: the original problem (FileNotFoundError) and how you handled it (ConfigError). Debugging becomes much easier because you see the root cause.

Why is bare `except:` considered dangerous? What could it hide? ▼

Bare `except:` catches EVERYTHING, including: KeyboardInterrupt (user wants to stop), SystemExit (clean shutdown), and bugs in your own code (NameError, AttributeError). If you have a typo creating an undefined variable, bare except silently hides it instead of crashing. The user can't Ctrl+C to stop. Always catch specific exceptions: ValueError, TypeError, etc.

When should you use `except` with a tuple of exceptions vs multiple except blocks? ▼

Use a tuple when you want to handle multiple exceptions identically: `except (KeyError, IndexError):`. Use multiple except blocks when different exceptions need different handling. Tuples reduce code duplication; multiple blocks are more explicit about different error paths. Choose based on whether the response is the same or different.

What's the relationship between the exception object and the traceback? Can you modify one without the other? ▼

The exception object (what you catch with `as e`) holds the error message and custom data. The traceback is metadata showing where the error occurred (file, line, function calls). They're related but separate—sys.exc_info() returns both. You can access traceback info via the traceback module, but typically you work with the exception object in except blocks.
