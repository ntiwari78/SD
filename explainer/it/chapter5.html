Chapter 5: Data Compression

From Morse code's single dot for 'E' to Huffman's optimal trees: how entropy reveals the irreducible limit of every code.

Feynman Mental Model

Morse code gives the letter 'E' a single dot because it's the most common letter in English — that's data compression from 1838. Huffman coding does the same thing optimally: frequent symbols get short codes, rare ones get long codes. This is the essence of compression.

But there is a brick wall you can never beat: the entropy \(H(X)\). No matter how clever your code, the average code length \(L\) satisfies \(H(X) \leq L\). For sources that are "random enough," you cannot compress below the entropy.

Codes and the Kraft Inequality (5.1–5.2)

Nonsingular and Uniquely Decodable Codes

A code maps symbols from a source alphabet to binary strings. We distinguish three levels of "goodness":

  • Nonsingular: Different symbols map to different codewords.
  • Uniquely decodable: No ambiguity when decoding a sequence of codewords.
  • Prefix (instantaneous): No codeword is a prefix of another. Decoding is immediate, with no lookahead.

Example: For symbols \(\{A, B, C\}\), the code \(\{0, 10, 11\}\) is a prefix code. When you see "0", you immediately know it's 'A' — no need to read further.

The Kraft Inequality

A fundamental theorem:

For any prefix code with codeword lengths \(l_1, l_2, \ldots, l_m\):

$$\sum_{i=1}^{m} 2^{-l_i} \leq 1$$

Conversely: given lengths \(l_1, l_2, \ldots, l_m\) that satisfy Kraft's inequality, there exists a prefix code with exactly those lengths.

Why? Imagine a binary tree. Leaves at depth \(l_i\) "consume" \(2^{-l_i}\) of the tree's capacity. A valid prefix code uses disjoint leaves, so the sum cannot exceed 1.

Optimal Codes (5.3–5.4)

The Optimal Code Length

What is the shortest average code length we can achieve? The answer lies in the optimal lengths:

$$l_i^* = \log_2\left(\frac{1}{p_i}\right)$$

If we could assign these fractional lengths, the average code length would be:

$$L^* = H(X)$$

where \(H(X) = -\sum p_i \log_2 p_i\)

In practice, we cannot use fractional lengths. So we must round up to integers. The fundamental bound is:

$$H(X) \leq L^* < H(X) + 1$$

The left inequality is tight if and only if all probabilities are negative powers of 2.

Huffman Coding (5.6–5.8)

The Algorithm

Huffman coding constructs an optimal prefix code greedily:

  1. Start with one leaf node for each symbol (probability = symbol's probability).
  2. Repeat: Combine the two least probable nodes into a parent (probability = sum of children).
  3. Continue until one root remains.
  4. Label left edges 0, right edges 1 (or vice versa).
  5. Read codewords from root to leaf.

Why is Huffman optimal? It minimizes the expected code length among all prefix codes. No other prefix code assigns shorter average length to the same probability distribution.

Alternative Coding Methods (5.9–5.10)

Shannon-Fano-Elias Coding

An alternative to Huffman that assigns lengths \(\lceil \log_2(1/p_i) \rceil + 1\). It is slightly less efficient than Huffman, but has a simpler structure and plays a key role in the analysis of optimal code length.

Arithmetic Coding

Rather than assigning integer-length codes to individual symbols, arithmetic coding maps the entire source message to a single real number in [0, 1). It achieves an average length arbitrarily close to \(H(X)\) over long sequences, without the integer rounding loss.

Key idea: Partition [0, 1) recursively as you encode each symbol. The final interval's binary representation is the codeword.

Code Lab: Huffman Implementation

Here's a complete Python implementation of Huffman coding with detailed comments: 

import heapq
from collections import defaultdict

class HuffmanNode:
    """A node in the Huffman tree."""
    def __init__(self, char, freq, left=None, right=None):
        self.char = char
        self.freq = freq
        self.left = left
        self.right = right

    def __lt__(self, other):
        return self.freq < other.freq

def build_huffman_tree(frequencies):
    """Build a Huffman tree from symbol frequencies.

    Args:
        frequencies: dict mapping symbols to their counts.

    Returns:
        root: HuffmanNode representing the root of the tree.
    """
    # Create a min-heap of leaf nodes.
    heap = [HuffmanNode(char, freq)
            for char, freq in frequencies.items()]
    heapq.heapify(heap)

    # Combine nodes until one remains.
    while len(heap) > 1:
        left = heapq.heappop(heap)
        right = heapq.heappop(heap)
        parent = HuffmanNode(None, left.freq + right.freq,
                           left, right)
        heapq.heappush(heap, parent)

    return heap[0]

def build_codes(root, prefix="", codes=None):
    """Extract codewords from a Huffman tree.

    Args:
        root: HuffmanNode representing the tree.
        prefix: current binary code (built recursively).
        codes: dict mapping symbols to codewords.

    Returns:
        codes: dict of symbol -> codeword.
    """
    if codes is None:
        codes = {}

    if root.char is not None:
        # Leaf node: assign the code.
        codes[root.char] = prefix if prefix else "0"
    else:
        # Internal node: recurse on children.
        if root.left:
            build_codes(root.left, prefix + "0", codes)
        if root.right:
            build_codes(root.right, prefix + "1", codes)

    return codes

def huffman_encode(message, frequencies):
    """Encode a message using Huffman coding.

    Args:
        message: string to encode.
        frequencies: dict of symbol -> count.

    Returns:
        encoded: binary string.
        codes: dict of symbol -> codeword.
    """
    tree = build_huffman_tree(frequencies)
    codes = build_codes(tree)
    encoded = "".join(codes[char] for char in message)
    return encoded, codes

def huffman_decode(encoded, root):
    """Decode a Huffman-encoded message.

    Args:
        encoded: binary string.
        root: HuffmanNode of the Huffman tree.

    Returns:
        decoded: original message.
    """
    decoded = []
    node = root
    for bit in encoded:
        # Traverse: 0 = left, 1 = right.
        node = node.left if bit == "0" else node.right
        if node.char is not None:
            # Reached a leaf: output the symbol.
            decoded.append(node.char)
            node = root
    return "".join(decoded)

# Example usage
text = "the quick brown fox jumps over the lazy dog"
frequencies = defaultdict(int)
for char in text:
    if char != ' ':
        frequencies[char] += 1

encoded, codes = huffman_encode(text, frequencies)
print("Codewords:", codes)
print("Encoded length:", len(encoded), "bits")
decoded = huffman_decode(encoded, build_huffman_tree(frequencies))
print("Decoded:", decoded)

Q1: Can any uniquely decodable code beat a prefix code in average length?
Answer: No. By the Kraft-McMillan inequality, any uniquely decodable code satisfies \(\sum 2^{-l_i} \leq 1\), the same constraint as prefix codes. Therefore, the optimal prefix code is just as good as any uniquely decodable code. Prefix codes are often simpler to construct, so Huffman is the practical choice.
Q2: For a source with \(H(X) = 2.5\) bits, what's the best and worst average code length?
Answer:
  • Best: 2.5 bits. Achieved only if all symbol probabilities are negative powers of 2 (e.g., 0.5, 0.25, 0.125, ...).
  • Worst (for an optimal code): Less than 3.5 bits. The bound \(L^* < H(X) + 1 = 3.5\) is tight for certain distributions.
In practice, Huffman achieves \(L \leq H(X) + 1\), and often much closer to \(H(X)\) for real-world distributions.
Q3: Why does Huffman coding sometimes waste bits compared to entropy?
Answer: Because Huffman assigns integer-length codes. If the optimal length for a symbol is 2.3 bits, Huffman must choose either 2 or 3 bits. This rounding loss accumulates across symbols.

Arithmetic coding avoids this by encoding the entire message as one number, avoiding integer quantization. It achieves average length arbitrarily close to \(H(X)\) over long blocks.
Q4: If you Huffman-code English character by character, can you reach 1 bit/char?
Answer: No. Character-level Huffman captures only the individual character frequencies (~4 bits/char for English). To approach 1 bit/char, you must exploit multi-character dependencies (e.g., "qu" often appears together, English has predictable letter sequences).

Solutions:
  • Block codes: Huffman on pairs or triples of characters.
  • Universal methods: LZ77/LZ78 (DEFLATE, gzip) learn patterns adaptively without knowing the source model.
  • Entropy-based prediction: PPM (Prediction by Partial Matching) builds a tree of context and adaptively refines codes.
