FEYNMAN BRIDGE

Understanding Words and Tokens

The LEGO Analogy

Imagine you have a pile of LEGO bricks and you want to build something. But here's the problem: what counts as one brick?

Consider the word don't. Is that one brick or two? Should we split it into do and n't? Or keep it whole? What about New York? That's two separate words, but they often go together.

Now imagine you're learning Chinese. There are no spaces between words. So how do you even know where one "brick" ends and another begins? A Chinese speaker reads 我喜欢猫 and instantly recognizes four separate words, but a computer sees it as one continuous string.

This is the fundamental problem: how do we break text into pieces?

The Word Problem

In traditional NLP, we used words as our basic units. We'd run the text through a tokenizer that splits on spaces and punctuation. But words have problems:

  • Vocabulary explosion: English has millions of rare words (technical terms, names, slang).
  • Language-specific: Different languages need different splitting rules.
  • Morphological variation: "Walk", "walks", "walked", "walking" are all variations of the same root.

The Subword Revolution

Modern language models don't use words OR characters. They use subwords: pieces that are somewhere between characters (a, b, c) and full words (walking, restaurant).

Think of it like this: instead of having a LEGO brick for every possible word in English (millions of them!), you have bricks for common parts of words. You can combine these parts to build any word, even ones you've never seen before.

This is called Byte-Pair Encoding (BPE), and it's the secret sauce behind GPT-2, GPT-3, and most modern language models.

The Spelling Bee Analogy: Edit Distance

Imagine a spelling bee where you need to measure how different two words are. If you spell "intention" but the correct answer is "execution", how many letter changes did you get wrong?

intention → execution

The answer isn't just counting mismatches. You have to think about the minimum number of edits (insertions, deletions, substitutions) needed to transform one word into the other. This is the Levenshtein distance, and it's a classic algorithm used everywhere: spell-checkers, speech recognition, DNA sequence alignment, and more.

TECHNICAL ASCENT

The Mathematics of Words

Herdan's Law / Heaps' Law

As you read more text, you keep encountering new words. But how fast does your vocabulary grow?

$$|V| = kN^\beta$$ where:
  • $|V|$ = vocabulary size (number of unique words)
  • $N$ = corpus size (total word count)
  • $k$ and $\beta$ are constants
  • $0 < \beta < 1$ (typically $\beta \approx 0.46$ for English)

Key insight: vocabulary grows sub-linearly. If you double the text, your vocabulary doesn't double. You keep seeing new words, but at a decreasing rate. This is why even with billions of words of text, we still encounter OOV (out-of-vocabulary) words.

Byte-Pair Encoding (BPE): The Algorithm

BPE is beautifully simple. Start with characters. Repeatedly merge the most common adjacent pairs.

Algorithm: BPE(corpus C, merges k) 
V ← all unique characters in C
for i = 1 to k:
    (tL, tR) ← most frequent adjacent pair in C
    tNEW ← tL + tR
    V ← V + tNEW
    Replace all (tL, tR) in C with tNEW
return V

After $k$ merges, you have a vocabulary of size $|\text{chars}| + k$. You've created $k$ new subword units. When you tokenize new text, you apply these merges in order and output the tokens.

BPE is data-driven: it learns what "pieces" are useful for your specific language and domain.

Minimum Edit Distance (Levenshtein Distance)

The problem: transform source string into target string with minimum edits. Three operations: insert, delete, substitute.

Recurrence relation: $$D[i,j] = \min \begin{cases} D[i-1,j] + 1 & \text{(deletion)} \\ D[i,j-1] + 1 & \text{(insertion)} \\ D[i-1,j-1] + \text{cost} & \text{(substitution)} \end{cases}$$ where cost = $\begin{cases} 0 & \text{if } \text{source}[i] = \text{target}[j] \\ 2 & \text{otherwise} \end{cases}$

Base cases: $D[i,0] = i$ (delete all $i$ characters), $D[0,j] = j$ (insert all $j$ characters)

This is dynamic programming (Wagner & Fischer, 1974). Fill the matrix bottom-up, and $D[n,m]$ is your answer. We can also backtrack to find the actual alignment.

Section 1: BPE Tokenizer

# ═══════════════════════════════════════════════════════════════════════════ # Chapter 2: Words and Tokens — Complete Implementation # ═══════════════════════════════════════════════════════════════════════════ # ── Section 1: BPE Tokenizer from Scratch (§2.4) ────────────────────────── import re from collections import Counter def get_pairs(vocab): """Count frequency of adjacent symbol pairs in vocabulary. This is the core counting step of BPE (§2.4.1).""" pairs = Counter() for word, freq in vocab.items(): symbols = word.split() for i in range(len(symbols) - 1): pairs[(symbols[i], symbols[i+1])] += freq return pairs def merge_pair(pair, vocab): """Merge all occurrences of a symbol pair in the vocabulary. After finding the most frequent pair, we join them into one token.""" bigram = re.escape(' '.join(pair)) pattern = re.compile(r'(?) new_vocab = {} for word in vocab: new_word = pattern.sub(''.join(pair), word) new_vocab[new_word] = vocab[word] return new_vocab def train_bpe(corpus, num_merges): """Train BPE tokenizer on a corpus for k merge operations. Returns the learned vocabulary and merge rules (§2.4.1).""" # Step 1: Initialize vocab as character-level with word boundaries vocab = {} for word, count in Counter(corpus.split()).items(): vocab[' '.join(list('_' + word))] = count merges = [] for i in range(num_merges): pairs = get_pairs(vocab) if not pairs: break best = max(pairs, key=pairs.get) vocab = merge_pair(best, vocab) merges.append(best) print(f"Merge {i+1}: {best[0]} + {best[1]} → {''.join(best)} (freq: {pairs[best]})") return vocab, merges

Section 2: Minimum Edit Distance

# ── Section 2: Minimum Edit Distance (§2.9) ────────────────────────────── import numpy as np def min_edit_distance(source, target): """Compute Levenshtein distance using dynamic programming. Uses the recurrence from Eq. 2.20 in §2.9: D[i,j] = min(D[i-1,j]+1, D[i,j-1]+1, D[i-1,j-1]+sub_cost) where sub_cost = 0 if source[i]==target[j], else 2.""" n, m = len(source), len(target) D = np.zeros((n + 1, m + 1), dtype=int) # Base cases: distance from empty string for i in range(1, n + 1): D[i, 0] = i # i deletions from source prefix for j in range(1, m + 1): D[0, j] = j # j insertions into empty source # Fill the matrix using the recurrence relation for i in range(1, n + 1): for j in range(1, m + 1): sub_cost = 0 if source[i-1] == target[j-1] else 2 D[i, j] = min( D[i-1, j] + 1, # Deletion D[i, j-1] + 1, # Insertion D[i-1, j-1] + sub_cost # Substitution (or match) ) return D[n, m], D

THE 2026 BRIDGE

Modern Challenges and Solutions

The Tokenization Bias Problem

English-centric BPE vocabularies create a serious bias against non-English languages. Consider the Spanish word hondo (deep). In GPT-2's BPE vocabulary (trained primarily on English), this gets split into h + ondo. Why? Because "h" at the start and "ondo" at the end are rare in English, so the algorithm never learned to merge them.

The consequence: A Spanish sentence uses nearly double the tokens of its English translation. For users paying per token, this is a real economic cost. For models, it wastes context window capacity on repetitive tokens instead of actual semantic content.

Recent Solutions (2025)

Recent work addresses this:

  • SuperBPE (Liu et al., 2025): Allow BPE merges to cross word boundaries. This lets Spanish "h" and "ondo" merge even if they span multiple words in the training data.
  • BoundlessBPE (Schmidt et al., 2025): Remove the assumption that merges happen within a single word. Learn optimal subwords globally across the corpus.
  • Multilingual vocabularies: Train separate BPE vocabularies for different language clusters, or use language-aware weighting when building the shared vocabulary.

Edit Distance is Still Everywhere

You might think edit distance is "old" algorithm. But it's everywhere:

  • Speech Recognition (Ch 15): Word Error Rate (WER) is computed using edit distance alignment between hypothesized and reference transcripts.
  • Machine Translation: BLEU scores and other metrics use n-gram matching, which can be expressed as edit distance.
  • Spell Checking: Find the closest word in the dictionary by edit distance.
  • Bioinformatics: Align DNA sequences; edit distance is fundamental.

Regular Expressions in Modern NLP

You might also think regex is "old". But every major tokenizer uses regex:

  • Pre-tokenization: Before BPE, GPT-2 uses regex to split on whitespace and punctuation.
  • Cleaning: Remove URLs, emails, special characters using regex patterns.
  • Language-specific rules: Different languages need different regex for contractions, hyphens, etc.

BPE is the "what". Regex is the "how to prepare it".

SOCRATIC KNOWLEDGE CHECK

Guided Discovery: The Tokenization Bias Scenario

The Scenario: A Spanish user types a recipe into a GPT-based system:

"En un recipiente hondo, mezclar el jugo de naranja con la harina tamizada."

The English translation: "In a deep bowl, mix the orange juice with the sifted flour."

Token counts (GPT-2 tokenizer):
English: 18 tokens
Spanish: 33 tokens
Ratio: 1.83× more tokens for Spanish

Question 1: Why the Disparity?

Why does the same semantic content require nearly twice as many tokens in Spanish? What does this tell us about how BPE vocabularies are built?

Key insight: BPE vocabularies are typically trained on English-dominant corpora. The most frequent bigrams and trigrams in English are learned first. When BPE encounters Spanish, it sees many word combinations it never saw during training.

For example, "hondo" (common in Spanish) might not appear frequently enough in the training corpus for BPE to keep it as a single token. Instead, it gets split into subword pieces: "h" + "ondo". This is not a flaw in BPE; it's a consequence of English-centric training data.

The algorithm is data-driven: it learns what's frequent. Frequency reflects the training distribution. English-dominant training = English-favoring vocabulary.

Question 2: Economic Implications

If a language model charges per token (as OpenAI's API does), what are the economic implications of this tokenization bias for non-English users?

Direct impact: A Spanish user pays ~1.8× more for the same content compared to an English user. This is not reflected in the actual computational cost—both take roughly the same time to process.

Equity concern: This creates a hidden "language tax" on non-English users. Speakers of morphologically rich languages (Turkish, Finnish, German) suffer even worse penalties.

Scale: For large organizations, this can be millions of dollars. For individuals, it disproportionately affects non-English-speaking users in lower-income countries who already have lower digital access.

Question 3: Morphological Tokenization

The word "hondo" is split into "h" + "ondo". Would a morphologically-aware tokenizer do better? What are the tradeoffs between BPE and morphological tokenization?

Morphological tokenization: Split words into meaningful parts (stems and affixes). For example, "walked" → "walk" + "-ed" (past tense). This is linguistically meaningful.

Advantage: Morphological tokenization would keep "hondo" as one unit (it has no suffix), and handle morphological variants (hondos, hondísimo) more systematically.

Disadvantage: You need linguistic resources (morphological analyzers) for each language. BPE is completely language-agnostic—it works on any raw text. Also, morphological boundaries don't always align with what's semantically important for neural models.

Modern synthesis: Systems like Unigram LM and WordPiece (used in BERT) try to blend the data-driven nature of BPE with linguistic structure.

Question 4: Model Performance Impact

How might this tokenization disparity affect model PERFORMANCE (not just cost) on Spanish text? Think about the Transformer's context window.

Context window is finite. GPT-4 has a context window of ~8000 tokens (or more with extensions). If Spanish uses 1.8× more tokens, a Spanish user gets 1.8× less context.

Performance degradation:

  • Less history to condition on → harder to maintain context and coherence
  • Less room for long-form examples (few-shot prompts)
  • Harder to fit documents into context without summarization

Research evidence: Studies show that languages with higher token-to-character ratios perform worse on downstream tasks, even controlling for model size and training data.

Question 5: A Concrete Solution

Propose a concrete solution to reduce this bias. What would SuperBPE or BoundlessBPE do differently? What are implementation challenges?

SuperBPE approach: Allow BPE merges to cross word boundaries. Standard BPE treats each word independently (after adding a word-start marker "_"). SuperBPE removes this restriction.

How it helps: The sequence "h ondo" might appear frequently across multiple Spanish words (hondo, ahonda, rehecho). By allowing cross-boundary merges, the algorithm learns that "h" + "ondo" should be one token, even if they span word boundaries.

Challenges:

  • Larger search space → more computation to find best merges
  • Tokens might span linguistic boundaries → harder to interpret
  • Need to recompute frequency counts across all words

Better approach: Use separate, language-balanced vocabularies. Train BPE on equally-weighted multilingual data. Or use dynamic vocabulary selection based on the user's language.

Historical Context

ELIZA (1966)

Weizenbaum, MIT

The first chatbot. Used pattern matching and string substitution—no real language understanding. Yet people attributed human-like understanding to it. This highlights why tokenization matters: even simple string operations can feel like intelligence.

Unicode (1988–1991)

Xerox/Apple Engineers

Before Unicode, text was fragmented into incompatible encodings. Unicode solved it: one standard for all human writing systems. But even Unicode requires careful tokenization (combining characters, right-to-left scripts, etc.).

UTF-8 Design (1992)

Ken Thompson, Plan 9

Thompson designed UTF-8 on a placemat. It encodes Unicode characters as variable-length byte sequences. Elegant: ASCII remains a prefix, backward compatible. This is why computers can process multilingual text.

Minimum Edit Distance (1974)

Wagner & Fischer

Formalized the dynamic programming algorithm for computing Levenshtein distance. Used in spell-checking, speech recognition, DNA sequencing. One of the most useful algorithms in computational biology and NLP.

Dynamic Programming Name (1950s)

Richard Bellman

Bellman coined the term "dynamic programming" during Cold War. The "programming" means "planning", not "coding". He chose the name because it sounded authoritative and was hard to attack politically!

Regular Expressions (1951–1956)

Stephen Cole Kleene, McCulloch-Pitts Neurons

Kleene formalized regular expressions as a notation for describing finite automata. He was studying the computational power of McCulloch-Pitts artificial neurons. Regex is fundamentally a model of computation.

BPE for Compression (1994)

Philip Gage

Gage published "A New Algorithm for Data Compression" in C Users Journal. BPE was born as a data compression technique. No one knew it would revolutionize tokenization 22 years later.

BPE for NMT (2016)

Sennrich, Haddow, Birch

"Neural Machine Translation of Rare Words with Subword Units" (2016). Applied BPE to neural machine translation, solving the OOV (out-of-vocabulary) problem. This paper paved the way for GPT.

GPT-2 BPE (2019)

OpenAI

OpenAI used BPE to tokenize the 1.5B parameter GPT-2 model. With 50K tokens, it balanced efficiency and expressiveness. This became the standard tokenization for the GPT family.

SuperBPE & BoundlessBPE (2025)

Liu et al., Schmidt et al.

Recent papers propose removing the word-boundary constraints of BPE. This allows learning better subwords across languages, addressing the tokenization bias problem. The arms race continues!
