Information Extraction: Relations, Events, and Time

Chapter 20 · Turning Unstructured Text into Structured Knowledge

From Jurafsky & Martin: Speech and Language Processing (3rd Ed)

The Journalist's Notebook

Imagine you're a journalist reading thousands of news articles and your job is to extract structured facts: WHO did WHAT to WHOM, and WHEN and WHERE did it happen? You start with raw, unstructured text and need to output clean, queryable information—a knowledge graph linking entities and their relationships, a timeline of events, a database of transactions.

Information extraction (IE) bridges the gap between human-readable text and machine-readable data. We ask: What semantic relationships hold between entities? What events occur, and in what temporal order? Can we automatically fill in structured templates from documents? These are the questions of Chapter 20.

The Core Challenge

Text is rich, ambiguous, and context-dependent. Extracting structured facts requires solving multiple sub-problems simultaneously: entity recognition, relation classification, event identification, and temporal reasoning.

Real-World Impact

IE powers knowledge graphs (Google, DBpedia), medical record processing, financial document analysis, and social media monitoring. Without IE, vast amounts of textual knowledge remain inaccessible to machines.

20.1: Relation Extraction

What Is a Relation?

A relation is a semantic connection between two entities. Relations are the edges in a knowledge graph.

Example:

Sentence: "Steve Jobs founded Apple in 1976."

Extracted relations:

  • (Steve Jobs, founder-of, Apple)
  • (Apple, founded-by, Steve Jobs)
  • (Steve Jobs, date-of-founding, 1976)

Relation Types: ACE Relations

The ACE (Automatic Content Extraction) program defined 17 major relation types:

Category Example Relations Example
Person-Social family, friendship, contact-info John is married to Mary
Physical location-of, near, part-of Paris is in France
Org-Affiliation employment, membership, founder Gates founded Microsoft
Agent-Artifact user-of, inventor-of Tim invented the web
Event-Related located, in-region, participant Protest in downtown Boston

Knowledge Graphs & RDF Triples

Relations can be represented as RDF triples (Resource Description Framework):

Subject — Predicate — Object

Example:

  • <dbr:Steve_Jobs> dbp:founded <dbr:Apple_Inc>
  • <dbr:Paris> dbo:country <dbr:France>

TACRED Dataset

TACRED is a large-scale supervised dataset for relation extraction:

  • 106,264 sentences from newswire
  • 41 relation types + no_relation class
  • Entity pairs: persons and organizations
  • Benchmark: used to evaluate supervised neural methods

Example: (Barack Obama, org:founded, Democratic Party)

20.2: Relation Extraction Algorithms

20.2.1: Hearst Patterns (1992)

One of the earliest and simplest approaches: use lexico-syntactic patterns to extract semantic relations.

The Hypernym Pattern

For hypernymy (e.g., "dog" is a type of "animal"):

NP₀ such as NP₁, NP₂, ..., (and|or) NPᵢ, i ≥ 1

∀ NPᵢ, i ≥ 1, hyponym(NPᵢ, NP₀)

Example:

Text: "We have seen a similar pattern in other European countries such as France, Belgium, and Spain."

Extracted:

  • hyponym(France, European countries)
  • hyponym(Belgium, European countries)
  • hyponym(Spain, European countries)

Five Classic Patterns:

  1. NP₀ such as {NP₁, NP₂, ..., NP_n}
  2. such NP₀ as {NP₁, NP₂, ..., NP_n}
  3. NP₀ {, NP₁, ..., NP_n} or other NP
  4. NP₀ {, NP₁, ..., NP_n} and other NP
  5. NP₀ {,} including {NP₁, NP₂, ..., NP_n}

Strengths: High precision, simple to implement, interpretable. Weaknesses: Low recall, requires hand-crafted patterns, doesn't scale to new relation types.

20.2.2: Supervised Learning

Train a classifier on labeled data to predict relation types.

Feature-Based Approach

Extract features from the context and train a linear classifier:

  • Word features: lemmas, POS tags, dependency paths between entities
  • Named entity features: entity types (PERSON, ORG, LOCATION)
  • Syntactic features: shortest path in dependency tree, subtree patterns
  • Lexical features: surrounding word windows, entity mention characteristics

Neural Approach: BERT + Linear Classifier

Modern supervised methods use pre-trained language models:

# Simplified BERT-based relation classifier import torch from transformers import AutoTokenizer, AutoModel tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased") bert_model = AutoModel.from_pretrained("bert-base-uncased") # Input: sentence with marked entity positions sentence = "[CLS] Steve Jobs [unused1] founded [unused2] Apple [SEP]" tokens = tokenizer.encode(sentence, return_tensors="pt") # Get [CLS] representation (whole sentence encoding) with torch.no_grad(): outputs = bert_model(tokens) cls_repr = outputs.last_hidden_state[:, 0, :] # [batch, 768] # Linear layer for 41 relation types (TACRED) relation_classifier = torch.nn.Linear(768, 41) logits = relation_classifier(cls_repr) predicted_relation = torch.argmax(logits, dim=1)

Advantages: High accuracy on benchmark datasets, leverages large pre-trained models. Limitations: Requires labeled training data, domain transfer can be difficult.

20.2.3: Semi-Supervised Bootstrapping

Start with a small set of seed tuples/patterns and iteratively grow the extraction set.

The Bootstrapping Loop

  1. Initialize: Start with a few hand-picked seed tuples or patterns (e.g., "Steve Jobs", "Apple")
  2. Find sentences: Search a corpus for sentences containing the seed tuple members
  3. Extract patterns: Identify textual patterns in those sentences (e.g., "[X] founded [Y]")
  4. Find new tuples: Apply patterns to find new entity pairs in the corpus
  5. Score & filter: Rank new tuples by confidence and add high-confidence ones back to seeds
  6. Repeat: Continue until convergence or manual stop

Confidence Scoring

Two key confidence measures:

Pattern confidence (RlogF): $Conf_{RlogF}(p) = \frac{|hits(p)|}{|finds(p)|} \log(|finds(p)|)$

where |hits(p)| = correct extractions from pattern p, |finds(p)| = total extractions from p

Tuple confidence (noisy-or): $Conf(t) = 1 - \prod_{p \in P'} (1 - Conf(p))$

where P' = set of patterns that extracted tuple t

The Semantic Drift Problem

As bootstrapping progresses, noisy extractions can contaminate the seed set. For example, starting with "founded", the algorithm might learn the polysemous "set" pattern, leading to incorrect relations like ("foundation", "set", "goal") instead of founder relations.

20.2.4: Distant Supervision

Use a large knowledge base (KB) like Freebase or DBpedia to automatically generate training data.

The Key Idea (Mintz et al. 2009)

If a (subject, relation, object) triple exists in the KB, then any sentence mentioning both the subject and object might contain evidence for that relation.

Example

KB triple: (Barack Obama, spouse, Michelle Obama)

Sentence 1: "Barack Obama married Michelle in 1992."

Label: spouse relation (correct)

Sentence 2: "Barack and Michelle Obama visited Kenya last year."

Label: spouse relation (noisy—they're together, but sentence doesn't express spouse relation)

Challenge: Noisy labels. The assumption that co-occurrence implies the relation is too weak. Multi-instance learning approaches address this by only requiring one sentence per tuple to express the relation.

20.2.5: Unsupervised / Open IE

Extract relations without predefined relation types. The system learns to identify and extract arbitrary relations.

ReVerb System

The ReVerb (Relation Verb) system extracts relations based on the structure of English verbs:

Pattern: (Arg1) — (Relation Phrase) — (Arg2)

  • Arg1 (Argument 1): Noun phrase (subject)
  • Relation Phrase: Verb-based phrase with constraints (stops before nested clauses)
  • Arg2 (Argument 2): Noun phrase (object)

Examples:

  • "Einstein won the Nobel Prize" → (Einstein, won, Nobel Prize)
  • "Obama is president of USA" → (Obama, is president of, USA)
  • "The Nile flows through Egypt" → (Nile, flows through, Egypt)

Advantage: No predefined relation types needed. Works across domains. Tradeoff: Less structured relations, harder to use in downstream applications that expect specific relation types.

20.2.6: Evaluation Metrics

How do we evaluate IE systems?

Supervised (Labeled Test Set)

Standard metrics: Precision, Recall, F₁ Score

  • Precision: Of extracted relations, how many are correct?
  • Recall: Of all relations in the gold standard, how many did we find?

Semi-Supervised & Unsupervised (No Complete Gold Standard)

Sampled Precision: Randomly sample 100 extractions, manually judge them, estimate precision.

$\hat{P} = \frac{\text{correctly extracted tuples in sample}}{\text{total extracted tuples in sample}}$

20.3–20.7: Extracting Events and Representing Time

20.3: Event Extraction

An event is an occurrence that happens at a specific time and place, typically involving entities as participants.

Event Mention

A textual reference to an event, often centered on a verb or action noun.

Example: "Apple announced the iPhone in 2007."

  • Event mention: "announced"
  • Event type: ANNOUNCEMENT
  • Participants: Apple (agent), iPhone (theme), 2007 (temporal)

Light Verbs & Complex Predicates

Not all events are expressed as main verbs. English uses light verbs to create complex predicates:

  • "make a decision" → event DECISION, light verb "make"
  • "take a trip" → event TRAVEL, light verb "take"
  • "have a conversation" → event COMMUNICATION, light verb "have"

Extraction Approach: IOB Tagging

Use sequence labeling to identify event trigger words:

Sentence: "The president announced a major policy yesterday." IOB tags: The O (Outside) president B-ARG (Begin argument) announced B-TRIG (Begin trigger) a I-TRIG (Inside trigger) major I-TRIG policy I-ARG (Inside argument) yesterday B-TIME (Time expression)

20.4: Representing Time

Allen's 13 Temporal Interval Relations (1984)

Fundamental relations between two time intervals A and B:

before
A < B
after
A > B
meets
A.end = B.start
met-by
B.end = A.start
overlaps
A.start < B.start < A.end
overlapped-by
B.start < A.start < B.end
starts
A.start = B.start
started-by
B.start = A.start
finishes
A.end = B.end
finished-by
B.end = A.end
during
A inside B
contains
B inside A
equals
A.start = B.start AND A.end = B.end

Reichenbach's Reference Time (1947)

Analyze tense using three time points:

  • E (Event time): When the event occurred
  • R (Reference time): The perspective point from which we evaluate the event
  • S (Speech time): The moment of utterance (now)

Example analyses:

  • Past simple: "I went home" → E < R = S (event before now)
  • Present perfect: "I have gone home" → E < S = R (event before now, result in present)
  • Future: "I will go home" → S = R < E (now before event)
  • Past perfect: "I had gone home" → E < R < S (event before reference, reference before now)

20.5: Representing Aspect

Aspect describes the internal temporal structure of an event (whether it's ongoing, habitual, completed, etc.).

Vendler's 4 Classes (1957)

Class Definition Example
Stative No change; no internal phases "know", "love", "be tall"
Activity Ongoing; can be interrupted "run", "eat", "swim" (for a time)
Accomplishment Has result; reaches an endpoint "build a house", "run a mile"
Achievement Instantaneous; no duration "find", "reach", "win", "arrive"

Aspect classification is important for NLU systems because it affects inference. "Mary is writing a letter" suggests an incomplete activity, while "Mary wrote a letter" suggests a completed accomplishment.

20.6: TimeBank and TimeML

TimeML is an XML markup language for annotating temporal information in text.

Key Annotations:

  • TIMEX3: Temporal expressions (dates, times, durations)
  • EVENT: Event mentions (verbs, action nouns, states)
  • TLINK: Temporal link between events or times
  • ALINK: Aspect link (how aspect applies)
  • SLINK: Subordination link (how events embed)

Example TimeML Annotation:

<text> John <EVENT eid="e1" class="OCCURRENCE">started</EVENT> his job on <TIMEX3 tid="t1" type="DATE" value="2020-01-15"> January 15, 2020</TIMEX3>. </text> <TLINK eventInstanceID="e1" relType="IS_INCLUDED" timeID="t1"/>

TimeBank: A corpus of 183 documents from newswire, fully annotated with TimeML. Used as benchmark for event extraction and temporal relation classification tasks.

20.7.1: Extracting Temporal Expressions

Recognize and classify temporal expressions in text.

Types of Temporal Expressions:

  • Absolute: "January 15, 2020", "Tuesday", "2:30 PM"
  • Relative: "yesterday", "next week", "last month", "in 3 days"
  • Duration: "3 hours", "2 weeks", "5 years"

Extraction Method: IOB Sequence Labeling

Sentence: "The meeting is scheduled for next Tuesday at 2:00 PM." IOB: The O meeting O is O scheduled O for O next B-TIMEX3 Tuesday I-TIMEX3 at O 2:00 B-TIMEX3 PM I-TIMEX3

Lexical triggers: Words that often signal temporal expressions (after, before, during, while, yesterday, tomorrow, etc.)

20.7.2: Temporal Normalization

Convert temporal expressions to ISO 8601 standard format.

ISO 8601 Format:

  • Date: YYYY-MM-DD (e.g., 2024-03-15)
  • Time: HH:MM:SS (e.g., 14:30:00)
  • DateTime: YYYY-MM-DDTHH:MM:SS (e.g., 2024-03-15T14:30:00)
  • Duration: P[n]Y[n]M[n]DT[n]H[n]M[n]S (e.g., P1Y2M3DT4H5M6S = 1 year, 2 months, 3 days, 4 hours, 5 minutes, 6 seconds)

Examples of Normalization:

Text Expression ISO 8601 Value Reference Date
"January 15, 2024" 2024-01-15 —
"yesterday" 2024-03-14 2024-03-15
"next week" 2024-03-24 2024-03-15
"2 hours" PT2H —
"Q3" 2024-Q3 2024

Challenge: Relative expressions require a reference date (from context) and calendar knowledge (e.g., what date is "next Tuesday"?).

20.8: Template Filling & Slot Filling

Slot Filling and Template Filling

Extract information to fill in a structured template or form from unstructured text.

Example: Airline Incident Template

Template schema:

INCIDENT:
  Lead Airline: ___________
  Incident Type: ___________
  Amount Damaged: ___________
  Date: ___________
  Follower Airline: ___________

Text: "United Airlines collided with a small jet yesterday, causing $2 million in damages. American Airlines was also involved."

Filled template:

INCIDENT:
  Lead Airline: United Airlines
  Incident Type: collision
  Amount Damaged: $2 million
  Date: yesterday
  Follower Airline: American Airlines

Machine Reading Approaches

Rule-Based / Pattern-Based

Define extraction rules for each slot. Simple but requires manual engineering for each template.

Example rule for Lead Airline:

IF sentence contains verb "collided" or "crashed" AND sentence contains airline name THEN Lead Airline = first airline mentioned IF sentence contains "United" or "American" or "Delta" ... AND within 5 tokens of collision verb THEN Lead Airline = that airline

Neural / Reading Comprehension Approaches

Train a model (e.g., BERT-based span extraction) to predict which text span fills each slot.

# Simplified span-based template filling from transformers import AutoTokenizer, AutoModelForQuestionAnswering tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased") model = AutoModelForQuestionAnswering.from_pretrained("bert-base-uncased") text = "United Airlines collided with a jet, causing $2M in damages." question = "Which airline is the lead airline?" inputs = tokenizer.encode_plus(question, text, return_tensors="pt") outputs = model(**inputs) # Extract answer span answer_start = torch.argmax(outputs.start_logits) answer_end = torch.argmax(outputs.end_logits) + 1 answer_tokens = inputs.input_ids[0][answer_start:answer_end] answer = tokenizer.decode(answer_tokens) print(answer) # "United Airlines"

Zero-Shot with Large Language Models

Use instruction-tuned LLMs (ChatGPT, Claude, etc.) with prompts to extract structured information.

prompt = """ Extract the following information from the text below. Return the result as JSON. Text: "United Airlines collided with a jet yesterday, causing $2 million in damages. American Airlines was also involved." Extract: - Lead Airline (the primary airline in the incident) - Incident Type - Amount Damaged - Date - Follower Airline Output JSON: """ # LLM responds: { "Lead Airline": "United Airlines", "Incident Type": "collision", "Amount Damaged": "$2 million", "Date": "yesterday", "Follower Airline": "American Airlines" }

Modern advantage: LLMs require no task-specific training and generalize well to unseen template types. Tradeoff: Cost, latency, and hallucination risks.

      Code Lab: Practical Implementations

      Code 1: Hearst Pattern Extractor

      Extract hypernymy relations from text using Hearst patterns.

      import re def extract_hearst_patterns(text): """ Extract hypernym relations using Hearst patterns. Patterns: 1. NP0 such as NP1, NP2, ..., (and|or) NPn 2. such NP0 as NP1, NP2, ..., NPn 3. NP0, NP1, ..., or other NP 4. NP0, NP1, ..., and other NP """ # Simplified NP pattern (word or phrase) NP = r"([A-Z][a-z\s]+)" patterns = [ # Pattern 1: NP such as NP, NP, ... (and|or) NP re.compile( NP + r"\s+such as\s+" + r"((?:" + NP + r"(?:,|and|or)\s*)+)" ), # Pattern 2: such NP as NP, NP, ... re.compile( r"such\s+" + NP + r"\s+as\s+" + r"((?:" + NP + r"(?:,|and|or)\s*)+)" ), ] relations = [] for pattern in patterns: for match in pattern.finditer(text): hypernym = match.group(1).strip() hyponym_str = match.group(2) hyponyms = re.findall(NP, hyponym_str) for hyponym in hyponyms: relations.append((hyponym.strip(), "hypernym", hypernym)) return relations # Example usage text = """ Pets such as dogs, cats, and rabbits require regular care. We have various fruits including apples, oranges, and bananas. """ results = extract_hearst_patterns(text) for hyponym, rel, hypernym in results: print(f"({hyponym}, {rel}, {hypernym})")

      Note: This is a simplified implementation. Production systems would use proper NLP parsers (spaCy, Stanford CoreNLP) to identify noun phrases more accurately.

      Code 2: Relation Classifier with BERT

      Classify relations between entity pairs using BERT embeddings.

      import torch from transformers import AutoTokenizer, AutoModel class BERTRelationClassifier(torch.nn.Module): def __init__(self, num_relations=41): super().__init__() self.bert = AutoModel.from_pretrained("bert-base-uncased") self.hidden_size = self.bert.config.hidden_size # Concatenate [CLS], entity1, entity2 representations self.classifier = torch.nn.Linear(self.hidden_size * 3, num_relations) def forward(self, input_ids, entity1_mask, entity2_mask): # Get BERT representations outputs = self.bert(input_ids, output_hidden_states=True) last_hidden = outputs.last_hidden_state # [batch, seq, 768] # Extract [CLS] representation (index 0) cls_repr = last_hidden[:, 0, :] # [batch, 768] # Extract entity representations (average pooling over entity span) entity1_repr = (last_hidden * entity1_mask.unsqueeze(-1)).sum(dim=1) / \ entity1_mask.sum(dim=1, keepdim=True) # [batch, 768] entity2_repr = (last_hidden * entity2_mask.unsqueeze(-1)).sum(dim=1) / \ entity2_mask.sum(dim=1, keepdim=True) # [batch, 768] # Concatenate combined = torch.cat([cls_repr, entity1_repr, entity2_repr], dim=1) logits = self.classifier(combined) # [batch, 41] return logits # Example usage tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased") model = BERTRelationClassifier(num_relations=41) sentence = "Steve Jobs founded Apple Inc." tokens = tokenizer.encode(sentence, return_tensors="pt") # Mark entity positions: Steve Jobs (tokens 1-2), Apple Inc (tokens 3-4) entity1_mask = torch.tensor([[0, 1, 1, 0, 0, 0, 0]]) # Steve Jobs entity2_mask = torch.tensor([[0, 0, 0, 1, 1, 0, 0]]) # Apple Inc logits = model(tokens, entity1_mask, entity2_mask) predicted_relation = torch.argmax(logits, dim=1) print(f"Predicted relation ID: {predicted_relation.item()}")

      This code shows the core architecture. A full implementation would include training loops, loss functions, and evaluation metrics on the TACRED dataset.

      Code 3: Temporal Expression Normalizer

      Normalize temporal expressions to ISO 8601 format.

      from datetime import datetime, timedelta import re def normalize_temporal_expression(expr, reference_date=None): """ Normalize temporal expressions to ISO 8601 format. Args: expr: Temporal expression string (e.g., "yesterday", "Q3 2024") reference_date: Reference date for relative expressions (datetime object) Returns: ISO 8601 formatted string """ if reference_date is None: reference_date = datetime.now() expr = expr.lower().strip() # Absolute dates (YYYY-MM-DD format) if re.match(r"\d{4}-\d{2}-\d{2}", expr): return expr # Relative dates if expr == "yesterday": return (reference_date - timedelta(days=1)).strftime("%Y-%m-%d") elif expr == "today": return reference_date.strftime("%Y-%m-%d") elif expr == "tomorrow": return (reference_date + timedelta(days=1)).strftime("%Y-%m-%d") # Relative weeks/months/years if expr == "next week": return (reference_date + timedelta(weeks=1)).strftime("%Y-%m-%d") elif expr == "last week": return (reference_date - timedelta(weeks=1)).strftime("%Y-%m-%d") elif expr == "next month": # Approximate: add 30 days return (reference_date + timedelta(days=30)).strftime("%Y-%m-%d") elif expr == "last month": return (reference_date - timedelta(days=30)).strftime("%Y-%m-%d") # Duration expressions duration_pattern = r"(\d+)\s+(hour|day|week|month|year)s?" match = re.match(duration_pattern, expr) if match: amount = int(match.group(1)) unit = match.group(2) if unit == "hour": return f"PT{amount}H" elif unit == "day": return f"P{amount}D" elif unit == "week": return f"P{amount*7}D" elif unit == "month": return f"P{amount}M" elif unit == "year": return f"P{amount}Y" # Month and year (e.g., "January 2024") month_year_pattern = r"([A-Z][a-z]+)\s+(\d{4})" match = re.match(month_year_pattern, expr) if match: month_str = match.group(1) year = match.group(2) months = { "january": "01", "february": "02", "march": "03", "april": "04", "may": "05", "june": "06", "july": "07", "august": "08", "september": "09", "october": "10", "november": "11", "december": "12" } month = months.get(month_str.lower()) if month: return f"{year}-{month}" # Quarters (e.g., "Q3 2024") quarter_pattern = r"Q(\d)\s+(\d{4})" match = re.match(quarter_pattern, expr) if match: quarter = match.group(1) year = match.group(2) return f"{year}-Q{quarter}" return None # Could not normalize # Example usage ref_date = datetime(2024, 3, 15) test_exprs = [ "yesterday", "next week", "3 days", "January 2024", "Q3 2024", "2024-03-15" ] for expr in test_exprs: normalized = normalize_temporal_expression(expr, ref_date) print(f"{expr:20} -> {normalized}")

      Production systems would use libraries like dateparser or natty to handle more complex temporal expressions and ambiguities.

      The Modern Era: Neural and Large Language Models

      LLMs for Zero-Shot Relation and Event Extraction

      Large language models like GPT-4, Claude, and others excel at information extraction with minimal supervision.

      Zero-Shot Extraction with Instructions

      No task-specific training required—just write a prompt:

      prompt = """ You are an expert information extraction system. Extract all semantic relations from the following text. Return results as a JSON list of {subject, relation, object} triples.       Knowledge Graph Completion

      Neural methods can predict missing edges in knowledge graphs using graph embeddings.

      Graph Embedding Approaches

      • TransE: Model relations as translations in embedding space. If (h, r, t) is a fact, then h + r ≈ t.
      • Graph Neural Networks (GNNs): Learn entity and relation embeddings by message-passing over the graph structure.
      • Transformer-based: Encode multi-hop paths through the KB using attention mechanisms.

      Example: If we know (Apple, founded-by, Steve Jobs) and (Steve Jobs, year-of-birth, 1955), we can infer (Apple, founded-by-someone-born-in, 1955).

      Event Extraction in Low-Resource and Multilingual Settings

      Modern approaches leverage multilingual pre-trained models for zero-shot and few-shot event extraction.

      Cross-Lingual Transfer

      Train on English data, test on other languages. Multilingual BERT and mT5 enable this:

      • Fine-tune mBERT on English event extraction
      • Apply directly to Chinese, Arabic, Hindi, etc. without retraining
      • Performance depends on language similarity and linguistic overlap

      This democratizes IE for under-resourced languages and domains where labeled data is scarce.

      Multimodal Information Extraction

      Extract relations and events from text + images + video.

      Emerging Frontiers

      • Visual relations: Detect object-object relations in images ("woman riding a horse")
      • Video event detection: Temporal localization of events in video streams
      • Vision-language models: CLIP, LLaVA enable joint reasoning over text and images

      Historical Notes & Evolution

      The Pattern Era: Hearst (1992) to Early 2000s ▼

      The field of information extraction began with hand-crafted patterns. Marti Hearst's 1992 paper showed that simple lexico-syntactic patterns could reliably extract hypernym relations from raw text. This sparked decades of pattern engineering:

      • MUC (Message Understanding Conferences, 1987-1997): First systematic evaluation of IE systems. Participants hand-built pattern-based extractors for structured templates (e.g., terrorist attacks, corporate management changes).
      • JAPE (Jape is Another Pattern Engine): Grammatical-engine-based patterns for Java-based IE.
      • Limitations: Labor-intensive, low recall, difficult to port to new domains or languages.
      The Learning Era: 2000s–2010s (Supervised and Semi-Supervised) ▼

      As labeled data became available, machine learning approaches dominated:

      • Feature-based classifiers: SVM and logistic regression on hand-engineered features (word, syntactic, dependency path).
      • Bootstrapping (Hearst, 1992; Brin, 1998; Agichtein & Gravano, 2000): Start with seed patterns, iteratively expand. Confidence scoring methods developed (RlogF, noisy-or).
      • Distant supervision (Mintz et al., 2009): Revolutionary idea: use knowledge bases (Freebase) to automatically generate training data at scale. Enabled large-scale RE.
      • ACE / TAC campaigns (2004–2012): Standard benchmarks for relation, event, and slot-filling tasks. TACRED (2017) extended ACE to 41 relations and 106k sentences.
      The Neural Era: 2015–Present (Deep Learning & Transformers) ▼

      Deep learning transformed IE:

      • Recurrent Neural Networks (RNNs) & LSTMs (2015–2016): Sequence labeling for event extraction and NER. Bidirectional LSTMs with CRF decoding became standard.
      • Convolutional Neural Networks for RE (Zeng et al., 2014): CNN over dependency trees for relation classification. High performance on TACRED (80%+ F₁).
      • Attention mechanisms (Bahdanau et al., 2014): Focus on relevant context. Enabled pointer networks for span extraction.
      • BERT & pre-trained transformers (Devlin et al., 2018): Transfer learning breakthrough. Fine-tuning BERT on TACRED achieved 88%+ F₁. Massively reduced need for task-specific architectures.
      • Knowledge graph embeddings (Bordes et al., 2013 TransE; etc.): Learned representations for link prediction and completion.
      Temporal Reasoning & TimeML ▼

      Temporal reasoning has a rich history:

      • Allen's Interval Algebra (1984): Foundational work on reasoning about time. The 13 relations remain the standard in NLP today.
      • Reichenbach's Tense Theory (1947): Pre-computational but deeply influential. E/R/S framework essential for understanding tense and aspect.
      • TimeML (2003–present): Developed collaboratively. Initial focus on event and temporal expression annotation. Evolved into TempEval shared tasks (2007–2013, assessing temporal relation extraction).
      • TimeBank (Pustejovsky et al., 2003): First large-scale corpus with rich temporal annotations. 183 documents, ~2,000 events, ~5,000 temporal relations.
      • Modern challenges: Implicit temporal relations (inferred but not explicitly stated), cross-document temporal ordering, temporal reasoning over narratives.
      Modern Era: 2018–Present (Large Language Models & Zero-Shot) ▼

      Recent years have seen rapid progress toward more flexible, data-efficient IE:

      • Pre-trained language models: GPT-2, RoBERTa, ELECTRA, T5 further improved performance. In-context learning (prompt-based) enabled few-shot extraction.
      • Large language models (GPT-3, GPT-4, Claude, etc.): Enabled true zero-shot IE without task-specific training. Prompt engineering became a critical skill.
      • Unified models: T5, BART, and others frame IE as text-to-text generation. A single model can perform relation extraction, event extraction, and slot filling.
      • Knowledge graph construction at scale: Companies like Google, Microsoft, Amazon use neural IE pipelines to build and maintain KGs with millions of entities and billions of relations.
      • Challenges remain: Out-of-distribution generalization, hallucinations in LLM-based extraction, resource efficiency, interpretability, and handling of adversarial/noisy text.

      Summary & Key Takeaways

      Relations & Extraction

      • Relations are edges in knowledge graphs linking entities.
      • From Hearst patterns (high precision) to neural methods (high recall) to LLMs (flexible).
      • Distant supervision leverages KBs; bootstrapping iteratively grows with confidence scoring.

      Events & Time

      • Events are state changes at specific times/places; temporal expressions require normalization.
      • Allen's 13 interval relations and Reichenbach's E/R/S framework structure temporal reasoning.
      • TimeML enables rich annotation; TempEval assesses temporal reasoning systems.

      Templates & Slot Filling

      • Structured templates capture facts from text (airline incident, financial news, etc.).
      • Modern LLMs excel at zero-shot template filling via prompt engineering.
      • Spans of text are extracted and mapped to template slots.

      Evaluation & Impact

      • Precision/recall/F₁ for supervised; sampled precision for unsupervised.
      • IE powers knowledge graphs, QA systems, document analysis, and decision support.
      • Modern systems blend neural embeddings, large models, and structured reasoning.
