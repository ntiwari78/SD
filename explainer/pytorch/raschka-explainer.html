Module 1

Installing PyTorch

Setting up your PyTorch environment with CPU and GPU support, and verifying your installation.

Why Installation Matters

Understanding the Setup

PyTorch is a Python library, but it's special: it wraps high-performance C++ and CUDA kernels underneath. Your installation choice determines whether PyTorch uses your CPU, NVIDIA GPU, or both. Getting this right at the start means your code will run as fast as your hardware allows, without extra configuration later.

Installation Methods

PyTorch offers multiple installation paths. The official approach uses either pip or conda and depends on your OS (Linux, macOS, Windows) and hardware (CPU-only, CUDA GPU, Metal).

# Method 1: pip install for CPU-only (fastest download)
pip install torch

# Method 2: pip install with CUDA 12.1 support (for NVIDIA GPUs)
pip install torch --index-url https://download.pytorch.org/whl/cu121

# Method 3: conda install with GPU support
conda install pytorch pytorch-cuda=12.1 -c pytorch -c nvidia
bash

Choose the method based on:

  • CPU-only if you're developing and testing models on a laptop
  • CUDA if you have an NVIDIA GPU (check compatibility: NVIDIA GPUs support compute capability 3.5+)
  • Metal on M1/M2 Macs via conda (accelerated on Apple Silicon)

Verifying Your Installation

import torch
print("PyTorch version:", torch.__version__)
print("CUDA available:", torch.cuda.is_available())
print("CUDA version:", torch.version.cuda)
print("Number of GPUs:", torch.cuda.device_count())
if torch.cuda.is_available():
    print("GPU device:", torch.cuda.get_device_name(0))
python
Pause & Think

If you run the verification code above and torch.cuda.is_available() returns False, what could the causes be?

Possible causes:

  • You installed the CPU-only version by mistake
  • Your NVIDIA driver is outdated (check with nvidia-smi)
  • Your GPU doesn't support CUDA (check NVIDIA documentation)
  • You're in an environment (Docker, WSL) where GPU access isn't exposed
  • The CUDA version in your PyTorch install doesn't match your driver
Check Your Understanding

What is the primary advantage of installing PyTorch with CUDA support over CPU-only?

Correct! GPUs excel at parallel computation. Modern NVIDIA GPUs have thousands of cores, making them ideal for the matrix operations at the heart of neural networks. A single GPU often matches or beats a CPU by orders of magnitude on deep learning workloads.
Pro Tip

After installing PyTorch, run a simple benchmark: create a large tensor on CPU and GPU, perform the same operation, and measure the time difference. This builds intuition for when to use GPU acceleration.

Module 2

Tensors

The fundamental data structure in PyTorch. Understanding tensors is understanding PyTorch.

What Is a Tensor?

Understanding Tensors

A tensor is a multi-dimensional array of numbers. Think of it as a generalization: a scalar (single number) is a 0-D tensor, a list is a 1-D tensor (vector), a table is a 2-D tensor (matrix), and a stack of tables is a 3-D tensor. PyTorch tensors are NumPy arrays' sophisticated cousins: they live on GPUs, track how they were created (for backprop), and are optimized for deep learning math.

Creating Tensors

import torch
import numpy as np

# From Python list
t1 = torch.tensor([1, 2, 3])

# From NumPy array
np_array = np.array([[1, 2], [3, 4]])
t2 = torch.from_numpy(np_array)

# Zeros, ones, random
t3 = torch.zeros(2, 3)  # 2x3 matrix of zeros
t4 = torch.ones(3, 4)   # 3x4 matrix of ones
t5 = torch.randn(2, 3)  # 2x3 matrix of random values (normal dist)

# Arange and linspace
t6 = torch.arange(0, 10, 2)  # [0, 2, 4, 6, 8]
t7 = torch.linspace(0, 1, 5)  # [0, 0.25, 0.5, 0.75, 1.0]
python

Check Your Understanding

You have a tensor t of shape (128, 64) on CPU. You want to move it to GPU and perform matrix multiplication with a tensor of shape (64, 32) also on the GPU. What is the resulting shape?

Correct! When multiplying a (128, 64) tensor with a (64, 32) tensor, the inner dimensions must match (they do: 64), and the result is (128, 32). The computation will succeed because both tensors are on the same device (GPU).
Note

PyTorch tensors are the first of PyTorch's three major components. The other two are Computation Graphs (how operations are tracked) and Autograd (automatic differentiation). All three work together to train neural networks.

Module 3

Computation Graphs

How PyTorch records operations and prepares for automatic differentiation.

What Is a Computation Graph?

Understanding Computation Graphs

A computation graph is a directed acyclic graph (DAG) where nodes represent tensors and edges represent operations. When you compute y = (a + b) * c, PyTorch builds a graph that records: "a and b were added, then the result was multiplied by c." This record is essential because it lets PyTorch later ask: "How much did changing a affect the final output?" That's backpropagation.

Building a Simple Computation Graph

import torch

# Define inputs
a = torch.tensor(2.0, requires_grad=True)
b = torch.tensor(3.0, requires_grad=True)
c = torch.tensor(4.0, requires_grad=True)

# Forward pass: build the graph
sum_ab = a + b              # operation 1
result = sum_ab * c         # operation 2

print(result)              # tensor(20., grad_fn=)
print(result.grad_fn)      # 
python

Visualizing the Computation Graph

For the expression y = (a + b) * c, the graph looks like:

a (2.0) b (3.0) \ / \ / Add (+) | sum_ab (5.0) | \ | \ | c (4.0) | / \ / Mul (*) | y (20.0)

Each box is a node (a tensor), and each arrow is an operation. During backward pass, gradients flow backward through these arrows.

Pause & Think

In the graph above, if we compute .backward() starting from y, which operation will be traversed first during backpropagation?

The Mul operation. Backpropagation traverses the graph backward, so it starts at the output (y) and goes to its immediate parent operation (Mul). Then it goes to Add, and finally to the leaf nodes (a, b, c).

Dynamic Computation Graphs

PyTorch builds a fresh computation graph on every forward pass. This is different from TensorFlow (pre-2.x), which used static graphs. The advantage: your code can use Python control flow (if/for/while) naturally.

import torch

def forward_pass(x):
    result = x * 2
    if result.sum() > 0:         # dynamic control flow
        result = result + 1
    else:
        result = result - 1
    return result

x = torch.tensor([1.0, 2.0], requires_grad=True)
y = forward_pass(x)
y.sum().backward()  # builds graph on-the-fly based on actual execution
python
Key Insight

Dynamic graphs are rebuilt each forward pass. This means if your code takes different branches (due to if statements or while loops), the graph adapts. This flexibility is crucial for models like RNNs that process variable-length sequences.

Module 4

Automatic Differentiation (Autograd)

How PyTorch computes gradients automatically using the chain rule.

Why Automatic Differentiation?

Understanding Autograd

Training neural networks requires computing gradients: how much does each parameter affect the loss? With 100 million parameters, computing these by hand (or numerical approximation) is infeasible. Automatic differentiation (autograd) uses the chain rule to compute exact gradients in one backward pass. PyTorch's autograd engine is the engine powering deep learning.

The Chain Rule in Action

If y = (a + b) * c, then the gradient of the loss L with respect to a is:

$$\frac{\partial L}{\partial a} = \frac{\partial L}{\partial y} \cdot \frac{\partial y}{\partial (a+b)} \cdot \frac{\partial (a+b)}{\partial a}$$

Breaking this down:

  • $\frac{\partial y}{\partial (a+b)} = c$ (the derivative of multiplication)
  • $\frac{\partial (a+b)}{\partial a} = 1$ (the derivative of addition)
  • $\frac{\partial L}{\partial y}$ is computed by the operation above (backprop flow)

Computing Gradients with requires_grad

import torch

# Create parameters that need gradients
x = torch.tensor(3.0, requires_grad=True)
y = torch.tensor(2.0, requires_grad=True)

# Forward pass
z = x**2 + 2*x*y + y**2   # z = (x + y)^2

# Backward pass
z.backward()

# Access gradients
print("dz/dx =", x.grad)   # should be 2*(3+2) = 10
print("dz/dy =", y.grad)   # should be 2*(3+2) = 10
python

Scalar vs. Vector Outputs

The .backward() method expects a scalar. If your output is a vector (e.g., loss per batch), sum or mean it first:

# If loss is a vector
loss_vec = model(x)  # shape: (batch_size,)
loss = loss_vec.mean()  # scalar
loss.backward()

# OR specify gradient manually for advanced use
y = model(x)  # shape: (batch_size,)
y.backward(torch.ones_like(y))  # "distribute" gradient equally
python

Zeroing Gradients (Critical!)

Gradients accumulate by default. In a training loop, you must zero them before each backward pass:

# Wrong (gradients accumulate)
for epoch in range(10):
    y = model(x)
    loss = loss_fn(y, target)
    loss.backward()  # dw keeps accumulating!
    optimizer.step()

# Correct (zero before backward)
for epoch in range(10):
    optimizer.zero_grad()  # CRUCIAL: reset gradients to zero
    y = model(x)
    loss = loss_fn(y, target)
    loss.backward()
    optimizer.step()
python

Disabling Gradients (torch.no_grad)

During inference or when you don't need gradients, disable automatic differentiation to save memory and speed:

with torch.no_grad():
    predictions = model(test_data)
    # No gradients computed here; faster and lighter
python
Check Your Understanding

In a training loop, if you forget to call optimizer.zero_grad(), what happens?

Correct! By default, PyTorch accumulates gradients. If you don't zero them, the gradient from step 2 gets added to step 1's gradient, corrupting the learning signal. This is a classic mistake that causes models to fail to learn.
Pro Tip

Always structure your training loop with optimizer.zero_grad() first, followed by forward pass, loss computation, and backward. This muscle-memory pattern prevents gradient accumulation bugs.

Module 5

Implementing Deep Neural Networks

Building multi-layer perceptrons and other architectures using nn.Module.

nn.Module: The Building Block

Understanding nn.Module

nn.Module is PyTorch's way of wrapping reusable neural network components. It's a base class that handles parameter management, device movement, training/eval mode, and more. Every model you build inherits from nn.Module. The pattern: define layers in __init__, implement the forward pass in forward(), and PyTorch automatically handles backward by tracing your forward pass.

Building a Multilayer Perceptron

import torch
import torch.nn as nn

class MLP(nn.Module):
    def __init__(self, num_features, num_hidden, num_classes):
        super().__init__()

        # Define layers
        self.layers = nn.Sequential(
            nn.Linear(num_features, num_hidden),
            nn.ReLU(),
            nn.Linear(num_hidden, num_classes),
        )

    def forward(self, x):
        return self.layers(x)

# Instantiate and test
model = MLP(num_features=28*28, num_hidden=128, num_classes=10)
x = torch.randn(4, 784)  # batch of 4 MNIST images
output = model(x)  # shape: (4, 10)
print(output.shape)  # torch.Size([4, 10])
python

In the MLP above with 784 inputs, 128 hidden units, and 10 outputs, how many parameters does the first linear layer have?

Common Mistake

Forgetting to call model.eval() before validation causes inflated performance metrics. Dropout will randomly zero activations, making your validation metrics misleadingly low.

Module 6

Training Neural Networks

The complete training loop: forward pass, loss computation, backward pass, and optimization.

The Training Loop Template

Understanding the Training Loop

Training a neural network is a loop: (1) forward pass—push data through the model; (2) compute loss—measure how wrong the prediction is; (3) backward pass—trace back through the graph to find gradients; (4) optimizer step—update parameters in the direction of lower loss. Repeat thousands of times. That's it. Modern deep learning is automated differentiation applied repeatedly.

A Complete Training Loop

import torch
import torch.nn as nn
from torch.optim import SGD, Adam

# Setup
model = MLP(784, 128, 10)
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model = model.to(device)

loss_fn = nn.CrossEntropyLoss()
optimizer = Adam(model.parameters(), lr=1e-3)

# Training loop
num_epochs = 10
for epoch in range(num_epochs):
    model.train()
    total_loss = 0

    for batch_idx, (features, labels) in enumerate(train_loader):
        # Move data to device
        features = features.to(device)
        labels = labels.to(device)

        # Forward pass
        logits = model(features)
        loss = loss_fn(logits, labels)

        # Backward pass
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

        total_loss += loss.item()

        if batch_idx % 100 == 0:
            print(f"Epoch {epoch}, Batch {batch_idx}, Loss: {loss.item():.4f}")

    # Validation
    model.eval()
    val_loss = 0
    correct = 0
    total = 0

    with torch.no_grad():
        for features, labels in val_loader:
            features = features.to(device)
            labels = labels.to(device)

            logits = model(features)
            loss = loss_fn(logits, labels)
            val_loss += loss.item()

            predictions = logits.argmax(dim=1)
            correct += (predictions == labels).sum().item()
            total += labels.size(0)

    avg_val_loss = val_loss / len(val_loader)
    accuracy = correct / total
    print(f"Epoch {epoch} - Val Loss: {avg_val_loss:.4f}, Accuracy: {accuracy:.4f}")
python

Check Your Understanding

In the training loop above, the line optimizer.zero_grad() appears before the backward pass. What is the reason?

Correct! By default, PyTorch accumulates gradients. If you don't zero them before the backward pass, the new gradients add to the old ones, causing incorrect updates. The position matters: zero → forward → loss → backward → step.

Module 7

Multi-GPU Training

Scaling training across multiple GPUs with DataParallel and DistributedDataParallel.

Why Multi-GPU?

Understanding Distributed Training

A single NVIDIA A100 GPU can train maybe 1,000 examples per second. Large models and datasets require hours or days on one GPU. Multi-GPU training splits the data (or model) across multiple GPUs, computing gradients in parallel and synchronizing updates. It's the standard way to train LLMs and large vision models today.

DataParallel (Simple Approach)

nn.DataParallel is the easiest way to use multiple GPUs on a single machine:

import torch
import torch.nn as nn

# Create model on primary GPU
model = MLP(784, 128, 10)
model = model.to("cuda:0")

# Wrap with DataParallel
model = nn.DataParallel(model)

# Now training proceeds normally, but data is split across GPUs
for features, labels in train_loader:
    features = features.to("cuda:0")
    labels = labels.to("cuda:0")

    logits = model(features)  # automatically splits & gathers
    loss = loss_fn(logits, labels)
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()
python

How DataParallel works:

  1. You have a batch of 64 samples on GPU 0
  2. DataParallel splits it: 32 to GPU 0, 32 to GPU 1
  3. Each GPU computes forward pass on its chunk in parallel
  4. Outputs are gathered back to GPU 0, loss is computed
  5. Gradients are computed and synchronized across GPUs

DistributedDataParallel (Production Approach)

For serious multi-GPU or multi-node training, use DistributedDataParallel:

import torch
import torch.nn as nn
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DistributedSampler

# Initialize distributed training
torch.distributed.init_process_group("nccl")
rank = torch.distributed.get_rank()
world_size = torch.distributed.get_world_size()

# Set device
device = torch.device(f"cuda:{rank}")

# Create model
model = MLP(784, 128, 10)
model = model.to(device)
model = DDP(model, device_ids=[rank])

# Use DistributedSampler for data
sampler = DistributedSampler(
    train_dataset,
    num_replicas=world_size,
    rank=rank,
    shuffle=True
)
train_loader = torch.utils.data.DataLoader(
    train_dataset,
    batch_size=64,
    sampler=sampler
)

# Training proceeds normally
for epoch in range(num_epochs):
    sampler.set_epoch(epoch)  # shuffle differently each epoch
    for features, labels in train_loader:
        features = features.to(device)
        labels = labels.to(device)

        logits = model(features)
        loss = loss_fn(logits, labels)
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()
python

Pause & Think

If you have 4 GPUs and a batch size of 64, and you use DistributedDataParallel, how many samples does each GPU process in a single forward pass?

16 samples per GPU (64 ÷ 4). Each GPU gets a shard of the batch. All 4 GPUs compute gradients in parallel, then synchronize before the optimizer step.

DDP vs DataParallel
  • DataParallel: Single machine, multiple GPUs. Simpler. Slower (bottleneck on GPU 0).
  • DistributedDataParallel: Multi-machine, multi-GPU. More setup. Much faster. Production-ready.
Module 8

From PyTorch to LLMs

How the PyTorch fundamentals you've learned power large language models and beyond.

The LLM Training Pipeline

Understanding LLMs Through PyTorch

Large Language Models—like GPT—are just a sequence of tensor operations built with nn.Module, trained using the exact training loop you learned in Module 6. The scale is larger (billions of parameters, trillions of tokens), and the architecture is more sophisticated (Transformers with attention), but the fundamentals are identical. Install PyTorch, build an nn.Module, define a loss, loop forward → backward → step. That's it.

The Training Recipe for LLMs

  1. Data Pipeline: Load raw text, tokenize it into token IDs, batch into sequences of fixed length (e.g., 4096 tokens).
  2. Model Definition: Build a Transformer using nn.Module with multi-head self-attention, feed-forward layers, and layer normalization.
  3. Loss Function: Compute next-token prediction loss: $\mathcal{L} = -\sum_i \log P(\text{token}_{i+1} | \text{token}_1, \ldots, \text{token}_i)$
  4. Training Loop: Use the exact template from Module 6, but with mixed precision (fp16) and gradient accumulation to fit larger models.
  5. Multi-GPU: Use DistributedDataParallel to parallelize across many GPUs/TPUs.
  6. Inference: Set model.eval(), use torch.no_grad(), and generate tokens autoregressively.

A Minimal Transformer Block

import torch
import torch.nn as nn
import torch.nn.functional as F

class TransformerBlock(nn.Module):
    def __init__(self, hidden_dim, num_heads):
        super().__init__()
        self.attn = nn.MultiheadAttention(hidden_dim, num_heads)
        self.mlp = nn.Sequential(
            nn.Linear(hidden_dim, 4 * hidden_dim),
            nn.GELU(),
            nn.Linear(4 * hidden_dim, hidden_dim)
        )
        self.ln1 = nn.LayerNorm(hidden_dim)
        self.ln2 = nn.LayerNorm(hidden_dim)

    def forward(self, x):
        # Self-attention with residual
        attn_out, _ = self.attn(x, x, x)
        x = x + attn_out
        x = self.ln1(x)

        # MLP with residual
        mlp_out = self.mlp(x)
        x = x + mlp_out
        x = self.ln2(x)

        return x

# Usage
block = TransformerBlock(hidden_dim=768, num_heads=12)
x = torch.randn(2, 512, 768)  # (batch, seq_len, hidden_dim)
output = block(x)  # same shape
python

Mixed Precision & Gradient Accumulation

At large scale, you use techniques to make training faster and more memory-efficient:

# Mixed precision: use fp16 for forward/backward, fp32 for param updates
from torch.cuda.amp import autocast, GradScaler

scaler = GradScaler()

for epoch in range(num_epochs):
    for features, labels in train_loader:
        optimizer.zero_grad()

        # Compute in fp16 (faster, less memory)
        with autocast():
            logits = model(features)
            loss = loss_fn(logits, labels)

        # Scale loss before backward
        scaler.scale(loss).backward()
        scaler.step(optimizer)
        scaler.update()
python
