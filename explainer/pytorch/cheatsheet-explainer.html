Module 1

Tensors — The Foundation

Creating, manipulating, and understanding PyTorch's core data structure.

🧠 Feynman Explanation

A PyTorch tensor is a multi-dimensional array. A scalar (single number) is 0D. A list of numbers is 1D (a vector). A grid of numbers is 2D (a matrix). A stack of grids is 3D. And so on. Unlike regular Python lists, tensors are homogeneous (all elements must be the same type) and can live on a GPU for lightning-fast computation. They also automatically track every operation done to them, enabling automatic gradients.

Creating Tensors

python
import torch

# From a Python list
tensor = torch.tensor([1, 2, 3])
print(tensor)  # tensor([1, 2, 3])

# Random tensor (standard normal)
rand_tensor = torch.rand(3, 3)  # shape (3, 3)

# Zeros and ones
zeros = torch.zeros(2, 4)
ones = torch.ones(2, 4)

# Tensor properties
print(tensor.shape)   # torch.Size([3])
print(tensor.dtype)   # torch.int64
print(tensor.device)  # cpu

Element-Wise Operations

python
a = torch.tensor([1.0, 2.0, 3.0])
b = torch.ones(3)

# Arithmetic: all element-wise
result_add = a + b       # [2., 3., 4.]
result_mul = a * 2       # [2., 4., 6.]
result_pow = a ** 2      # [1., 4., 9.]

# In-place operations (modify the tensor directly)
a.add_(b)  # modifies a in-place; slow down by .grad computation
⏸ Pause & Think

What's the difference between a + b and a.add_(b)? Which should you use in a training loop?

a + b creates a new tensor without modifying a. a.add_(b) modifies a in-place. In-place ops (_ suffix) can save memory but can break the computation graph for gradients. In a training loop, use a + b so gradients flow correctly. In-place ops are useful only for operations that won't be differentiated (like updating running statistics during inference).

Key Point

Avoid in-place operations on tensors that require gradients. The computation graph relies on the original values, and in-place modifications can corrupt gradient computation.

Module 2

Automatic Differentiation (Autograd)

The magic behind PyTorch: automatic computation of gradients.

🧠 Feynman Explanation

Imagine a recipe that takes inputs and produces an output by applying a series of transformations. Autograd is like having a video camera that records every transformation. When you want to know "how much would the output change if I tweaked input #3 by 0.001?", the camera replays the video backward and computes the sensitivity. This is called reverse-mode differentiation.

Enabling and Computing Gradients

python
# Enable gradient tracking
x = torch.tensor(2.0, requires_grad=True)

# Compute a function: y = x^2
y = x ** 2

# Compute the gradient dy/dx at x=2
y.backward()

# Access the gradient: dy/dx = 2x = 4
print(x.grad)  # tensor(4.)

Gradient Accumulation and Zeroing

python
# Gradients ACCUMULATE by default
x = torch.tensor(3.0, requires_grad=True)
y = x ** 2
y.backward()
print(x.grad)  # tensor(6.) — dy/dx = 2*3

y = x ** 2
y.backward()
print(x.grad)  # tensor(12.) — accumulated! 6 + 6

# Zero gradients before each training step
x.grad.zero_()  # or x.grad = None
Common Pitfall

Forgetting to zero gradients is the #1 source of training bugs. Every call to .backward() adds to the existing .grad. In a training loop, always call optimizer.zero_grad() before loss.backward().

🏛 Socratic Check

You compute loss = (model(x) - y).sum() and call loss.backward(). What's required for gradients to propagate back to the model's weights?

Correct! Parameters in nn.Module have requires_grad=True by default. As long as the forward pass builds a computation graph connecting the output to the parameters, .backward() will compute gradients. You don't need to manually create anything—PyTorch tracks it automatically.
Module 3

Neural Networks (nn.Module)

Building trainable models with PyTorch's module system.

🧠 Feynman Explanation

An nn.Module is a container for neural network components. It's like a factory blueprint: it specifies what parts you have (layers, activations) and how to assemble them into a final output. Each nn.Module has parameters (weights to learn) and a forward method (how data flows through). When you call model(input), PyTorch automatically routes it to the forward method and records the computation for autograd.

Defining and Using a Simple Model

python
import torch.nn as nn

class SimpleNet(nn.Module):
    def __init__(self, input_dim, hidden_dim, output_dim):
        super().__init__()
        self.fc1 = nn.Linear(input_dim, hidden_dim)
        self.fc2 = nn.Linear(hidden_dim, output_dim)
        self.relu = nn.ReLU()

    def forward(self, x):
        x = self.fc1(x)
        x = self.relu(x)
        x = self.fc2(x)
        return x

# Create an instance
model = SimpleNet(input_dim=10, hidden_dim=64, output_dim=2)

# Forward pass
input_data = torch.randn(32, 10)  # batch of 32 samples, 10 features
output = model(input_data)         # shape: (32, 2)

# Access and inspect weights
print(model.fc1.weight.shape)  # torch.Size([64, 10])
print(model.fc1.weight.data)   # the actual weight tensor
⏸ Pause & Think

Why do we call model(input) instead of model.forward(input)?

When you call model(input), Python invokes __call__, which is a wrapper that handles important bookkeeping before and after forward(). It records the computation for autograd, applies hooks, and manages training/eval modes. Calling forward() directly bypasses this machinery, breaking gradient tracking and other features. Always use model(input).

Pro Tip

Use model.parameters() to iterate over all learnable weights, and sum(p.numel() for p in model.parameters()) to count the total number of parameters—useful for understanding model size.

🏛 Socratic Check

You create a custom parameter with self.W = torch.randn(64, 64) instead of self.W = nn.Parameter(torch.randn(64, 64)). Will an optimizer update W?

Exactly! nn.Parameter is a subclass of Tensor that signals to PyTorch: "I'm a learnable weight." The optimizer only updates parameters returned by model.parameters(). A plain Tensor` stored as an attribute is invisible to the optimizer.
Module 4

Optimization — Training Loop Essentials

Optimizers adjust model weights using computed gradients.

🧠 Feynman Explanation

An optimizer is like a hiker trying to reach the bottom of a valley in the fog. They can feel the slope beneath their feet (the gradient) but can't see the full landscape. Each step, they take a step downhill proportional to the slope's steepness. PyTorch provides different "stepping strategies": SGD takes simple steps, Adam adapts step size per parameter, etc. The key insight: gradients tell you which direction to move, and the optimizer decides how far.

Setting Up an Optimizer

python
import torch.optim as optim

model = SimpleNet(input_dim=10, hidden_dim=64, output_dim=2)

# SGD with learning rate 0.01
optimizer = optim.SGD(model.parameters(), lr=0.01)

# Adam optimizer (adaptive learning rate per parameter)
optimizer = optim.Adam(model.parameters(), lr=0.001)

# AdamW (Adam with weight decay, often better)
optimizer = optim.AdamW(model.parameters(), lr=0.001, weight_decay=1e-4)

The Three-Step Training Loop

python
# Typical training iteration:

# 1. Zero gradients from previous step
optimizer.zero_grad()

# 2. Forward pass and loss computation
outputs = model(inputs)
loss = loss_fn(outputs, labels)

# 3. Backward pass: compute gradients
loss.backward()

# 4. Update weights
optimizer.step()
Critical Order

zero_grad() → forward() → backward() → step() is the only correct order. If you call backward() before zero_grad(), gradients accumulate incorrectly. If you call step() before backward(), you're updating weights with stale gradients from the previous iteration.

🏛 Socratic Check

You forgot to call optimizer.zero_grad() in your training loop. What happens on iteration 2?

Correct! Gradients accumulate by default. This is useful for mini-batch gradients, but between training steps, you must zero them. Your model would train incorrectly, with effective learning rates bouncing erratically.
Module 5

Data Handling — DataLoaders

Efficiently feeding data to your model in batches.

🧠 Feynman Explanation

A Dataset is a collection: given an index, it returns one sample. A DataLoader wraps the dataset and handles the logistics: how many samples per batch, whether to shuffle, how many CPU cores to use for parallel loading, and whether to cache data in GPU memory. It yields batches (multiple samples stacked) automatically, so you can iterate over your entire dataset efficiently.

Creating a DataLoader

python
from torch.utils.data import DataLoader, TensorDataset

# Simple case: tensors you already have
X = torch.randn(1000, 10)  # 1000 samples, 10 features
y = torch.randint(0, 2, (1000,))  # 1000 binary labels

dataset = TensorDataset(X, y)
loader = DataLoader(
    dataset,
    batch_size=32,
    shuffle=True,           # randomize order for training
    num_workers=2,         # parallel data loading
    pin_memory=True,      # faster GPU transfer
)

# Iterate over batches
for batch_x, batch_y in loader:
    # batch_x shape: (batch_size, 10)
    # batch_y shape: (batch_size,)
    print(batch_x.shape, batch_y.shape)
⏸ Pause & Think

Why should you set shuffle=True during training but shuffle=False during validation/testing?

Shuffling during training breaks correlation between consecutive samples, helping the model generalize better. During validation/testing, you want reproducibility. If the data is shuffled each time you evaluate, metrics become noisy. More importantly, a fixed order lets you debug which samples are being misclassified.

Key Point

num_workers > 0 uses separate processes to load data in parallel, keeping the GPU fed while your model computes. On a slow disk or complex preprocessing, this is a huge speedup. Start with num_workers=2 or 4.

Module 6

Model Training — Putting It All Together

A complete training loop: data, model, loss, optimizer.

🧠 Feynman Explanation

Training is a repeating cycle: (1) load a batch of data, (2) run the model on it, (3) compute the loss (how wrong it was), (4) compute gradients, (5) update weights toward reducing loss. Repeat thousands of times until the loss stops decreasing meaningfully. The loss function is your compass: it tells you if you're heading in the right direction.

Complete Training Loop

python
import torch.nn as nn
import torch.optim as optim

model = SimpleNet(input_dim=10, hidden_dim=64, output_dim=2)
loss_fn = nn.CrossEntropyLoss()
optimizer = optim.Adam(model.parameters(), lr=0.001)

# Training loop
num_epochs = 10
for epoch in range(num_epochs):
    total_loss = 0.0
    for batch_x, batch_y in loader:
        # Zero gradients
        optimizer.zero_grad()

        # Forward pass
        outputs = model(batch_x)
        loss = loss_fn(outputs, batch_y)

        # Backward pass
        loss.backward()

        # Update weights
        optimizer.step()

        total_loss += loss.item()

    print(ff"Epoch {epoch+1}, Loss: {total_loss/len(loader):.4f}")

Common Loss Functions

python
# Classification: softmax + cross-entropy (combines sigmoid and BCE)
loss_fn = nn.CrossEntropyLoss()  # multi-class, outputs must be logits (not softmax!)

# Binary classification
loss_fn = nn.BCEWithLogitsLoss()  # expects logits (not probabilities)

# Regression: mean squared error
loss_fn = nn.MSELoss()

# Regression: mean absolute error
loss_fn = nn.L1Loss()
Critical Tip

CrossEntropyLoss and BCEWithLogitsLoss expect logits (raw scores), not probabilities! Do not apply softmax or sigmoid to your output before passing to these losses. They handle the normalization internally for numerical stability.

🏛 Socratic Check

You're training a binary classifier. Your model outputs predictions in [0, 1] (probabilities), but you pass them to nn.BCEWithLogitsLoss instead of nn.BCELoss. What's the problem?

Correct! BCEWithLogitsLoss combines sigmoid + BCE for numerical stability. If you feed it probabilities, it applies another sigmoid, squashing values into an even smaller range. Your gradients become tiny (vanishing gradient problem), and training fails. Always match loss functions to whether your output is logits or probabilities.
Module 7

Saving and Loading Models

Persist your trained models for later use.

🧠 Feynman Explanation

After training, you want to save your model so you can use it later without retraining. PyTorch offers two approaches: (1) save only the weights (the model's "learned knowledge"), or (2) save the entire model architecture + weights. Saving just weights is smaller and safer; loading requires you to rebuild the architecture first. Saving the whole model is convenient but risks version incompatibilities.

Save and Load Model Weights

python
# Save only the weights (recommended)
torch.save(model.state_dict(), 'model_weights.pth')

# Load weights into a new model instance
model = SimpleNet(input_dim=10, hidden_dim=64, output_dim=2)
model.load_state_dict(torch.load('model_weights.pth'))
model.eval()  # set to eval mode (disables dropout, etc.)

Save and Load Entire Model

python
# Save the entire model (architecture + weights)
torch.save(model, 'model.pth')

# Load it directly (no need to rebuild architecture)
model = torch.load('model.pth')
model.eval()

Save Training Checkpoint

python
# Save model + optimizer state for resuming training
checkpoint = {
    'epoch': epoch,
    'model_state_dict': model.state_dict(),
    'optimizer_state_dict': optimizer.state_dict(),
    'loss': loss,
}
torch.save(checkpoint, 'checkpoint.pth')

# Restore training
checkpoint = torch.load('checkpoint.pth')
model.load_state_dict(checkpoint['model_state_dict'])
optimizer.load_state_dict(checkpoint['optimizer_state_dict'])
epoch = checkpoint['epoch']
Important Warning

Avoid saving the entire model object with torch.save(model, 'model.pth') for production. If your code structure changes, the pickled object may fail to load. Always use state_dict() and rebuild the architecture manually.

Pro Tip

When loading, always set the model to evaluation mode with model.eval(). This disables layers like Dropout and BatchNorm that behave differently during training vs. inference.

Module 8

CUDA — GPU Acceleration

Harness the GPU for massive speedups.

🧠 Feynman Explanation

GPUs excel at parallel arithmetic. A CPU is good at sequential logic; a GPU has thousands of tiny cores working in lockstep on the same operation. Tensors live on either the CPU (slow but universal) or a GPU (fast but limited). Moving data to the GPU involves a small overhead, but once there, matrix operations (which dominate neural network math) run 10x–100x faster. PyTorch lets you move tensors and models to the GPU with a single method call: .to('cuda').

Checking GPU Availability and Moving Tensors

python
# Check if GPU is available
print(torch.cuda.is_available())    # True or False
print(torch.cuda.get_device_name(0))  # e.g., 'NVIDIA A100'

# Automatically choose device
device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')

# Move tensor to GPU
tensor = torch.randn(1000, 1000)
tensor = tensor.to(device)
# or: tensor = tensor.cuda()

# Move model to GPU
model = SimpleNet(...)
model = model.to(device)
# or: model = model.cuda()

# Check where a tensor lives
print(tensor.device)  # device(type='cuda', index=0)

GPU Training Template

python
device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')

model = SimpleNet(...)
model = model.to(device)  # move model to GPU

for epoch in range(num_epochs):
    for batch_x, batch_y in loader:
        # Move batch to same device as model
        batch_x = batch_x.to(device)
        batch_y = batch_y.to(device)

        # Forward and backward as usual
        optimizer.zero_grad()
        outputs = model(batch_x)
        loss = loss_fn(outputs, batch_y)
        loss.backward()
        optimizer.step()
Pro Tip

Move both the model and data to the GPU. A common mistake is moving the model but forgetting batches, causing a "tensor is on different device" error. Use a variable device and consistently apply .to(device).

⏸ Pause & Think

Your model is on GPU, but you forgot to move a batch to the GPU before passing it. What error will you see?

You'll get an error like RuntimeError: Expected all tensors to be on the same device. PyTorch can't perform operations between a tensor on GPU and a tensor on CPU. The fix: always move batches with batch.to(device) before passing to the model.

Memory Warning

GPU memory is limited. If you allocate huge batches or very large models, you'll get an OutOfMemoryError. Start with small batch sizes and increase until your GPU is full. Use torch.cuda.empty_cache() to free unused memory (rarely needed).
