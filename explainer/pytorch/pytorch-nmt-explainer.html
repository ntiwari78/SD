Module 1

Tensors & Autograd — The Engine

Understanding the fundamental data structure and automatic differentiation that powers every PyTorch model.

Tensors: NumPy's GPU-Accelerated Cousin

🧠 Feynman Explanation

Imagine a spreadsheet. A single cell is a scalar (0-D tensor). A row is a vector (1-D tensor). The whole sheet is a matrix (2-D tensor). Now imagine a stack of spreadsheets—that's a 3-D tensor. PyTorch tensors are just this: multi-dimensional arrays that know how to live on a GPU and track every math operation done to them, so they can later compute gradients automatically.

If you've used NumPy, PyTorch tensors will feel familiar. The critical differences are: (1) tensors can move to a GPU for parallel computation, and (2) tensors can record a computation graph for automatic differentiation.

python
import torch
import numpy as np

# Creating tensors — almost identical to NumPy
a_np = np.array([[1.0, 2.0], [3.0, 4.0]])
a_pt = torch.tensor([[1.0, 2.0], [3.0, 4.0]])

# Zero-copy conversion between the two
a_from_np = torch.from_numpy(a_np)   # shares memory!
a_to_np   = a_pt.numpy()             # shares memory on CPU

# Device management — move to GPU if available
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
a_gpu = a_pt.to(device)  # now lives on GPU (if available)

print(a_gpu.shape)   # torch.Size([2, 2])
print(a_gpu.device)  # cuda:0 or cpu
Key Point

torch.from_numpy() shares memory with the original array. If you modify one, the other changes too. This is efficient but can surprise you. When moving to GPU, .to(device) always creates a copy.

Autograd: The Dynamic Computation Graph

🧠 Feynman Explanation

Imagine you're baking a cake and you write down every single step: "I mixed flour and eggs, then I added sugar, then I baked it." If the cake tastes bad and you want to know "how much would it improve if I used slightly more sugar?", you could trace your steps backward through your notes to figure it out. That's exactly what autograd does. Every time you do math on a tensor marked with requires_grad=True, PyTorch secretly writes down that operation in a directed acyclic graph (DAG). When you call .backward(), it walks that graph in reverse to compute gradients.

This is called "reverse-mode automatic differentiation." Unlike symbolic differentiation (which manipulates math expressions) or numerical differentiation (which uses finite differences), autograd computes exact gradients efficiently by replaying the computation backward.

python
# Tell PyTorch: "track all operations on this tensor"
x = torch.tensor([2.0, 3.0], requires_grad=True)

# Forward pass: compute y = x^2 + 3x
y = x**2 + 3 * x   # PyTorch records: pow, mul, add
loss = y.sum()        # reduce to scalar (backward needs a scalar)

# Backward pass: compute dy/dx = 2x + 3
loss.backward()

print(x.grad)  # tensor([7., 9.])  — i.e., 2*2+3=7, 2*3+3=9 ✓
Common Pitfall

Gradients accumulate by default. If you call .backward() twice without zeroing, x.grad doubles. In a training loop, always call optimizer.zero_grad() before the backward pass.

⏸ Pause & Think

Consider: z = (3 * x + 1).sum() where x = tensor([1.0, 2.0], requires_grad=True). What will x.grad be after z.backward()?

The derivative of \(z = 3x + 1\) with respect to \(x\) is simply \(3\). Since x has two elements, x.grad will be tensor([3., 3.]). The constant \(+1\) vanishes in the derivative, and the .sum() distributes the gradient equally (each element contributes independently).

Dynamic vs. Static Graphs

PyTorch uses a dynamic computation graph (define-by-run): the graph is rebuilt every forward pass. This means you can use standard Python control flow (if, for, while) inside your models, and the graph adapts. TensorFlow 1.x used static graphs (define-then-run), where you built the graph once and executed it repeatedly. PyTorch's approach feels more Pythonic and makes debugging trivial—you can use print() and pdb mid-forward-pass.

🏛 Socratic Check

Why must you call .backward() on a scalar tensor, not a vector?

Correct! The Jacobian of a vector-valued function w.r.t. a vector input is a matrix, not a simple gradient vector. PyTorch's .backward() computes the vector-Jacobian product, which requires a "starting vector" (defaulting to 1.0 for scalar outputs). You can pass a gradient argument to .backward() for non-scalar outputs, but in practice we always reduce to a scalar loss first.
Module 2

The Building Blocks — nn.Module

Every neural network in PyTorch is built by composing nn.Module objects. Learn to create your own.

🧠 Feynman Explanation

Think of nn.Module like a LEGO brick. Each brick does one thing (a linear layer, a ReLU, a dropout). You snap bricks together to build bigger structures (a Transformer block), and those bigger structures snap together to build an entire architecture. Every brick knows two things: (1) what parameters it owns, and (2) how to compute its output given an input (the forward method).

python
import torch.nn as nn

class TranslationBlock(nn.Module):
    """A simple feed-forward block we might use inside a bigger model."""

    def __init__(self, d_model, d_ff, dropout=0.1):
        super().__init__()
        # nn.Linear stores a weight matrix W and bias b
        # It computes: output = input @ W.T + b
        self.linear1 = nn.Linear(d_model, d_ff)
        self.linear2 = nn.Linear(d_ff, d_model)
        self.dropout = nn.Dropout(dropout)
        self.norm    = nn.LayerNorm(d_model)
        self.relu    = nn.ReLU()

    def forward(self, x):
        # x shape: (batch, seq_len, d_model)
        residual = x
        x = self.linear1(x)   # → (batch, seq_len, d_ff)
        x = self.relu(x)
        x = self.dropout(x)
        x = self.linear2(x)   # → (batch, seq_len, d_model)
        x = self.norm(x + residual)  # residual connection + layer norm
        return x

# PyTorch automatically discovers all parameters
block = TranslationBlock(d_model=512, d_ff=2048)
print(f"Parameters: {sum(p.numel() for p in block.parameters()):,}")
# → Parameters: 2,362,880

Three rules to remember: (1) always call super().__init__(), (2) register sub-modules and parameters in __init__, and (3) define the forward computation in forward(). Never call forward() directly—use model(input) instead, because PyTorch hooks and other machinery wrap the forward call.

Why nn.ModuleList instead of a plain Python list? ▶

If you store sub-modules in a regular Python list, PyTorch won't "see" them. model.parameters() won't find their weights, .to(device) won't move them, and state_dict() won't save them. Use nn.ModuleList (or nn.ModuleDict) so PyTorch properly registers them.

# ❌ WRONG — parameters are invisible
self.layers = [nn.Linear(512, 512) for _ in range(6)]

# ✅ CORRECT — parameters are registered
self.layers = nn.ModuleList([nn.Linear(512, 512) for _ in range(6)])
🏛 Socratic Check

You define a custom module with self.W = torch.randn(512, 512) in __init__. Will the optimizer update W during training?

Exactly. A plain torch.Tensor stored as an attribute is invisible to model.parameters(). You must wrap it in nn.Parameter(), which tells PyTorch: "this tensor is a learnable weight — include it in the parameter list and update it during optimization."
Module 3

Data Pipelines for Translation

How to feed parallel sentence pairs into a model using Dataset, DataLoader, and custom collate_fn.

🧠 Feynman Explanation

Imagine you're a librarian. You have a huge catalog of book pairs (ancient text → English translation). A Dataset is your catalog: given an index, it returns one pair. A DataLoader is your assistant who grabs batches of pairs, shuffles them, and hands them to you efficiently. The tricky part? Ancient sentences have different lengths. You can't stack jagged arrays. A collate_fn is the assistant's padding tool—it pads shorter sentences with a special token so every sentence in the batch is the same length.

python
from torch.utils.data import Dataset, DataLoader

class TranslationDataset(Dataset):
    """Custom dataset for ancient → English sentence pairs."""

    def __init__(self, src_sentences, tgt_sentences, src_vocab, tgt_vocab):
        self.src = src_sentences  # list of tokenized source sentences
        self.tgt = tgt_sentences  # list of tokenized target sentences
        self.src_vocab = src_vocab
        self.tgt_vocab = tgt_vocab

    def __len__(self):
        return len(self.src)

    def __getitem__(self, idx):
        # Convert tokens to integer IDs
        src_ids = [self.src_vocab[t] for t in self.src[idx]]
        tgt_ids = [self.tgt_vocab[t] for t in self.tgt[idx]]
        return torch.tensor(src_ids), torch.tensor(tgt_ids)


def collate_fn(batch):
    """Pad variable-length sequences so they can be stacked into a batch tensor."""
    src_batch, tgt_batch = zip(*batch)

    # pad_sequence pads shorter tensors with 0 (our PAD token ID)
    # batch_first=True → output shape: (batch_size, max_seq_len)
    src_padded = nn.utils.rnn.pad_sequence(src_batch, batch_first=True, padding_value=0)
    tgt_padded = nn.utils.rnn.pad_sequence(tgt_batch, batch_first=True, padding_value=0)

    return src_padded, tgt_padded


# DataLoader handles batching, shuffling, and multi-process loading
loader = DataLoader(
    dataset,
    batch_size=32,
    shuffle=True,
    collate_fn=collate_fn,
    num_workers=2,      # parallel data loading
    pin_memory=True,     # faster CPU→GPU transfer
)
⏸ Pause & Think

Why do we need pin_memory=True? What happens during CPU → GPU data transfer without it?

Normally, CPU memory is pageable—the OS can move it to disk. Before transferring to GPU, CUDA must first copy data to a pinned (non-pageable) staging area, then transfer. With pin_memory=True, the DataLoader pre-allocates data in pinned memory, skipping the extra copy. The GPU transfer then uses DMA (Direct Memory Access) and becomes asynchronous, overlapping with computation. This can significantly speed up training when GPU utilization is high.

Translation-Specific Tip

For ancient languages, your vocabulary will likely be small (5k–20k tokens). Consider using BPE (Byte Pair Encoding) via sentencepiece to handle rare words and morphological richness. Train a shared BPE model on both your ancient corpus and the English side for a shared subword vocabulary.

Module 4

Sequential Modeling — LSTMs, GRUs & Teacher Forcing

Before Transformers, these recurrent architectures were the workhorses of NMT. Understanding them builds crucial intuition.

🧠 Feynman Explanation

An RNN reads a sentence one word at a time, updating a "hidden state" that acts as a running summary. Problem: by the time it reaches word 50, it has largely forgotten word 1. An LSTM fixes this with a "cell state"—think of it as a conveyor belt running alongside the hidden state. Information can be placed on the belt (input gate), removed from it (forget gate), or read from it (output gate). A GRU simplifies this to two gates (reset, update) and is faster to train, often with similar performance.

python
class Seq2SeqRNN(nn.Module):
    def __init__(self, src_vocab_size, tgt_vocab_size, emb_dim, hid_dim):
        super().__init__()
        self.encoder_emb = nn.Embedding(src_vocab_size, emb_dim)
        self.decoder_emb = nn.Embedding(tgt_vocab_size, emb_dim)

        # LSTM encoder: reads source sentence
        self.encoder = nn.LSTM(emb_dim, hid_dim, batch_first=True)
        # LSTM decoder: generates target sentence
        self.decoder = nn.LSTM(emb_dim, hid_dim, batch_first=True)
        # Project hidden state to vocabulary logits
        self.fc_out  = nn.Linear(hid_dim, tgt_vocab_size)

    def forward(self, src, tgt, teacher_forcing_ratio=0.5):
        # Encode: compress source into a context vector
        src_emb = self.encoder_emb(src)            # (B, S, E)
        _, (hidden, cell) = self.encoder(src_emb)  # hidden: (1, B, H)

        # Decode: generate one token at a time
        batch_size, tgt_len = tgt.shape
        outputs = torch.zeros(batch_size, tgt_len, self.fc_out.out_features)

        # First input to decoder is <BOS> token
        input_tok = tgt[:, 0].unsqueeze(1)  # (B, 1)

        for t in range(1, tgt_len):
            dec_emb = self.decoder_emb(input_tok)
            out, (hidden, cell) = self.decoder(dec_emb, (hidden, cell))
            logits = self.fc_out(out.squeeze(1))
            outputs[:, t] = logits

            # Teacher forcing: use true token or model's prediction?
            use_teacher = torch.rand(1).item() < teacher_forcing_ratio
            input_tok = tgt[:, t].unsqueeze(1) if use_teacher \
                        else logits.argmax(-1).unsqueeze(1)

        return outputs

Why Teacher Forcing?

🧠 Feynman Explanation

Imagine teaching a child to recite a poem. If they get word 3 wrong, every word after that will likely be wrong too (because each word depends on the previous one). Teacher forcing means: at training time, even when the model guesses wrong, you feed it the correct previous word so it can still learn the rest of the sequence. Without it, one early mistake cascades and the model learns almost nothing. The catch? At inference time, you don't have the answers, so the model must rely on its own predictions. That's why we often anneal the teacher forcing ratio during training.

🏛 Socratic Check

What is the "exposure bias" problem with teacher forcing?

Exactly. During training with teacher forcing, the decoder always sees ground-truth tokens as input. But at inference, it feeds its own predictions, which may contain errors. The model has never "practiced" recovering from its own mistakes. Scheduled sampling (linearly reducing teacher forcing ratio) and techniques like sequence-level training help mitigate this.
Module 5

The Transformer from Scratch

The architecture that revolutionized NLP. We build Multi-Head Attention, the Encoder, and the Decoder using vanilla PyTorch.

Self-Attention: The Core Mechanism

🧠 Feynman Explanation

Imagine you're reading the sentence: "The cat sat on the mat because it was tired." What does "it" refer to? To answer, you mentally attend to other words in the sentence and decide "it" most strongly connects to "cat." Self-attention does this mechanically. For each word, it asks: "How relevant is every other word to me?" It computes a relevance score (attention weight) for each pair, then produces a new representation of each word as a weighted sum of all other words' representations.

Mathematically, given queries \(Q\), keys \(K\), and values \(V\):

$$\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right) V$$

The \(\sqrt{d_k}\) scaling prevents dot products from growing too large (which would push softmax into regions with vanishing gradients).

Tensor Shapes Through Self-Attention

Let's trace every shape. Assume batch size \(B=2\), sequence length \(S=10\), model dimension \(d_{model}=512\), and \(h=8\) heads, so \(d_k = 512/8 = 64\):

Input \(X\): (2, 10, 512) → \(Q, K, V\) after linear projections: (2, 10, 512) → Reshape to multi-head: (2, 8, 10, 64) → \(QK^\top\): (2, 8, 10, 10) → Attention weights (after softmax): (2, 8, 10, 10) → Weighted values: (2, 8, 10, 64) → Concatenate heads: (2, 10, 512) → Output projection: (2, 10, 512) 

python
import math

class MultiHeadAttention(nn.Module):
    def __init__(self, d_model, n_heads):
        super().__init__()
        assert d_model % n_heads == 0
        self.d_k = d_model // n_heads
        self.n_heads = n_heads

        # Three separate linear projections for Q, K, V
        self.W_q = nn.Linear(d_model, d_model)
        self.W_k = nn.Linear(d_model, d_model)
        self.W_v = nn.Linear(d_model, d_model)
        self.W_o = nn.Linear(d_model, d_model)  # output projection

    def forward(self, query, key, value, mask=None):
        B = query.size(0)

        # Step 1: Project inputs
        Q = self.W_q(query)  # (B, S, d_model)
        K = self.W_k(key)
        V = self.W_v(value)

        # Step 2: Reshape into (B, n_heads, S, d_k)
        # .view splits d_model into (n_heads, d_k)
        # .transpose swaps seq_len and n_heads dimensions
        Q = Q.view(B, -1, self.n_heads, self.d_k).transpose(1, 2)
        K = K.view(B, -1, self.n_heads, self.d_k).transpose(1, 2)
        V = V.view(B, -1, self.n_heads, self.d_k).transpose(1, 2)

        # Step 3: Scaled dot-product attention
        # K.transpose(-2,-1) flips the last two dims for matrix multiply
        scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.d_k)
        # scores shape: (B, n_heads, S_q, S_k)

        if mask is not None:
            # .masked_fill() sets positions where mask==0 to -inf
            # This makes softmax assign ~0 weight to those positions
            # Used for: (a) padding masks, (b) causal masks in decoder
            scores = scores.masked_fill(mask == 0, float('-inf'))

        attn_weights = torch.softmax(scores, dim=-1)

        # Step 4: Weighted sum of values
        context = torch.matmul(attn_weights, V)  # (B, n_heads, S, d_k)

        # Step 5: Concatenate heads back together
        # .transpose then .contiguous().view merges the head dimension
        context = context.transpose(1, 2).contiguous().view(B, -1, self.n_heads * self.d_k)

        return self.W_o(context)  # (B, S, d_model)
⏸ Pause & Think

Why do we call .contiguous() before .view() after transposing? What would happen without it?

.transpose() doesn't physically rearrange memory—it just changes how PyTorch indexes into the same data (modifying strides). But .view() requires that the tensor is stored contiguously in memory. Without .contiguous(), you'd get a runtime error: "view size is not compatible with input tensor's size and stride." Calling .contiguous() copies the data into a fresh, contiguous block. Alternatively, you can use .reshape() which handles this automatically (but may silently copy).

The Full Encoder & Decoder Blocks

python
class TransformerEncoderBlock(nn.Module):
    def __init__(self, d_model, n_heads, d_ff, dropout):
        super().__init__()
        self.self_attn = MultiHeadAttention(d_model, n_heads)
        self.ffn = nn.Sequential(
            nn.Linear(d_model, d_ff),
            nn.ReLU(),
            nn.Dropout(dropout),
            nn.Linear(d_ff, d_model),
        )
        self.norm1 = nn.LayerNorm(d_model)
        self.norm2 = nn.LayerNorm(d_model)
        self.dropout = nn.Dropout(dropout)

    def forward(self, x, src_mask):
        # Self-attention with residual connection
        attn_out = self.self_attn(x, x, x, src_mask)
        x = self.norm1(x + self.dropout(attn_out))
        # Feed-forward with residual connection
        ffn_out = self.ffn(x)
        x = self.norm2(x + self.dropout(ffn_out))
        return x


class TransformerDecoderBlock(nn.Module):
    def __init__(self, d_model, n_heads, d_ff, dropout):
        super().__init__()
        self.self_attn  = MultiHeadAttention(d_model, n_heads)
        self.cross_attn = MultiHeadAttention(d_model, n_heads)
        self.ffn = nn.Sequential(
            nn.Linear(d_model, d_ff),
            nn.ReLU(),
            nn.Dropout(dropout),
            nn.Linear(d_ff, d_model),
        )
        self.norm1 = nn.LayerNorm(d_model)
        self.norm2 = nn.LayerNorm(d_model)
        self.norm3 = nn.LayerNorm(d_model)
        self.dropout = nn.Dropout(dropout)

    def forward(self, x, enc_out, src_mask, tgt_mask):
        # Masked self-attention (causal: can't look at future tokens)
        attn1 = self.self_attn(x, x, x, tgt_mask)
        x = self.norm1(x + self.dropout(attn1))
        # Cross-attention: decoder attends to encoder output
        # Query=decoder, Key/Value=encoder
        attn2 = self.cross_attn(x, enc_out, enc_out, src_mask)
        x = self.norm2(x + self.dropout(attn2))
        # Feed-forward
        ffn_out = self.ffn(x)
        x = self.norm3(x + self.dropout(ffn_out))
        return x
🏛 Socratic Check

In the decoder's cross-attention, why does the query come from the decoder but the key and value come from the encoder?

Think of it as a database lookup. The decoder's current state is the "search query." The encoder's output is the "database" where keys help find relevant entries and values contain the actual information. The decoder asks: "Given what I've generated so far, which source words should I focus on next?" This is the fundamental mechanism that lets the model align source and target words.
Module 6

Advanced Research Patterns — PyTorch Lightning

Reduce boilerplate, add logging, checkpointing, and multi-GPU support with minimal code changes.

🧠 Feynman Explanation

PyTorch gives you total control—but that means you also have to write the training loop, handle device placement, implement checkpointing, set up logging, and manage distributed training yourself. PyTorch Lightning is like hiring a very competent assistant. You tell it what your model does (the training_step, validation_step, etc.), and it handles how to run training. You keep full control over the science; Lightning handles the engineering.

python
import lightning as L

class TranslationModel(L.LightningModule):
    def __init__(self, model, lr=1e-4):
        super().__init__()
        self.model = model       # your Transformer from Module 5
        self.lr = lr
        self.criterion = nn.CrossEntropyLoss(ignore_index=0)  # ignore PAD

    def training_step(self, batch, batch_idx):
        src, tgt = batch
        # tgt[:, :-1] = decoder input (all tokens except last)
        # tgt[:, 1:]  = labels (all tokens except first <BOS>)
        output = self.model(src, tgt[:, :-1])
        loss = self.criterion(
            output.reshape(-1, output.size(-1)),
            tgt[:, 1:].reshape(-1)
        )
        self.log("train_loss", loss, prog_bar=True)
        return loss

    def validation_step(self, batch, batch_idx):
        src, tgt = batch
        output = self.model(src, tgt[:, :-1])
        loss = self.criterion(
            output.reshape(-1, output.size(-1)),
            tgt[:, 1:].reshape(-1)
        )
        self.log("val_loss", loss, prog_bar=True)

    def configure_optimizers(self):
        return torch.optim.Adam(self.parameters(), lr=self.lr)


# Lightning's Trainer handles EVERYTHING else
trainer = L.Trainer(
    max_epochs=50,
    accelerator="auto",          # auto-detect GPU/CPU
    callbacks=[
        L.pytorch.callbacks.ModelCheckpoint(
            monitor="val_loss",     # save best model
            mode="min",
        ),
        L.pytorch.callbacks.EarlyStopping(
            monitor="val_loss",     # stop if no improvement
            patience=5,
        ),
    ],
)
trainer.fit(lit_model, train_loader, val_loader)

Transfer Learning for Low-Resource Languages

How to "warm-start" a model using pre-trained weights from a related modern language to bootstrap translation for an ancient language.

🧠 Feynman Explanation

Imagine you speak modern Italian fluently and now want to learn Latin. You don't start from zero—Italian shares roots, grammar patterns, and vocabulary with Latin. Transfer learning works the same way. We take a model already trained on millions of sentences in modern languages (like Meta's NLLB-200, which covers 200 languages), and we fine-tune it on our small ancient language dataset. The model already understands sentence structure, morphology, and alignment patterns. It just needs to adapt those skills to the specific ancient language.

Strategy: Warm-Start with NLLB-200

NLLB-200 (No Language Left Behind) is a massively multilingual model from Meta. For ancient languages, you'd:

Step 1: Identify the closest living relative (e.g., Modern Greek → Ancient Greek, Hindi → Sanskrit, Icelandic → Old Norse).

Step 2: Fine-tune NLLB on your ancient ↔ English parallel data, using the related language's token embeddings as initialization.

Step 3: Apply techniques to prevent catastrophic forgetting: freeze lower encoder layers, use a low learning rate, and employ gradual unfreezing. 

python
from transformers import AutoModelForSeq2SeqLM, AutoTokenizer

# Load pre-trained NLLB-200 (distilled 600M parameter version)
model_name = "facebook/nllb-200-distilled-600M"
tokenizer  = AutoTokenizer.from_pretrained(model_name)
model      = AutoModelForSeq2SeqLM.from_pretrained(model_name)

# Key insight: NLLB uses language codes like "hin_Deva" (Hindi)
# For Sanskrit, we can START with Hindi embeddings since they
# share the Devanagari script and many cognates

# Strategy 1: Freeze encoder (preserve multilingual knowledge)
for param in model.model.encoder.parameters():
    param.requires_grad = False

# Strategy 2: Gradual unfreezing — unfreeze top 2 encoder layers
for layer in model.model.encoder.layers[-2:]:
    for param in layer.parameters():
        param.requires_grad = True

# Strategy 3: Different learning rates for different parts
optimizer = torch.optim.Adam([
    {"params": model.model.encoder.layers[-2:].parameters(), "lr": 1e-5},
    {"params": model.model.decoder.parameters(), "lr": 5e-5},
    {"params": model.lm_head.parameters(), "lr": 1e-4},
])
Low-Resource Reality Check

With fewer than 10,000 parallel sentences, you're in seriously low-resource territory. Consider data augmentation techniques: back-translation (translate English → ancient using your draft model, then use those as training data), lexicon injection (embed a bilingual dictionary as training pairs), and noise injection (randomly dropping/swapping tokens to make the model robust).

🏛 Socratic Check

Why do we freeze the lower encoder layers but unfreeze the upper ones?

Research consistently shows that lower Transformer layers capture universal linguistic features — phonological patterns, basic morphology, word order regularities — that transfer well across languages. Upper layers increasingly specialize in language-specific semantics. By freezing the lower layers, we preserve the general linguistic knowledge from pre-training while allowing the upper layers to adapt to the new language's specific properties.

"""
Capstone: Seq2Seq Transformer for Ancient Language → English
Complete training pipeline with BLEU evaluation (~145 lines)
"""
import math, torch, torch.nn as nn
from torch.utils.data import Dataset, DataLoader
from collections import Counter

# ── Positional Encoding ──────────────────────────────────────
class PositionalEncoding(nn.Module):
    def __init__(self, d_model, max_len=512, dropout=0.1):
        super().__init__()
        self.dropout = nn.Dropout(dropout)
        pe = torch.zeros(max_len, d_model)
        pos = torch.arange(max_len).unsqueeze(1).float()
        div = torch.exp(torch.arange(0, d_model, 2).float() * -(math.log(10000.0) / d_model))
        pe[:, 0::2] = torch.sin(pos * div)
        pe[:, 1::2] = torch.cos(pos * div)
        self.register_buffer("pe", pe.unsqueeze(0))  # (1, max_len, d_model)

    def forward(self, x):
        return self.dropout(x + self.pe[:, :x.size(1)])

# ── Seq2Seq Transformer ──────────────────────────────────────
class Seq2SeqTransformer(nn.Module):
    def __init__(self, src_vocab, tgt_vocab, d=256, heads=4, N=3, ff=512, drop=0.1):
        super().__init__()
        self.src_emb = nn.Embedding(src_vocab, d); self.tgt_emb = nn.Embedding(tgt_vocab, d)
        self.pos_enc = PositionalEncoding(d, dropout=drop)
        enc_layer = nn.TransformerEncoderLayer(d, heads, ff, drop, batch_first=True)
        dec_layer = nn.TransformerDecoderLayer(d, heads, ff, drop, batch_first=True)
        self.encoder = nn.TransformerEncoder(enc_layer, N)
        self.decoder = nn.TransformerDecoder(dec_layer, N)
        self.fc_out = nn.Linear(d, tgt_vocab)
        self.d_model = d

    def forward(self, src, tgt, src_mask=None, tgt_mask=None,
                src_pad_mask=None, tgt_pad_mask=None):
        # Scale embeddings by sqrt(d_model) as in "Attention Is All You Need"
        src_e = self.pos_enc(self.src_emb(src) * math.sqrt(self.d_model))
        tgt_e = self.pos_enc(self.tgt_emb(tgt) * math.sqrt(self.d_model))
        memory = self.encoder(src_e, src_key_padding_mask=src_pad_mask)
        out = self.decoder(tgt_e, memory, tgt_mask=tgt_mask,
                           tgt_key_padding_mask=tgt_pad_mask,
                           memory_key_padding_mask=src_pad_mask)
        return self.fc_out(out)

# ── Dataset ───────────────────────────────────────────────────
class AncientLangDataset(Dataset):
    def __init__(self, pairs, src_v, tgt_v):
        self.pairs, self.sv, self.tv = pairs, src_v, tgt_v

    def __len__(self): return len(self.pairs)

    def __getitem__(self, i):
        s, t = self.pairs[i]
        src = [1] + [self.sv.get(w, 3) for w in s.split()] + [2]  # 1=BOS 2=EOS 3=UNK
        tgt = [1] + [self.tv.get(w, 3) for w in t.split()] + [2]
        return torch.tensor(src), torch.tensor(tgt)

def collate(batch):
    s, t = zip(*batch)
    return (nn.utils.rnn.pad_sequence(s, batch_first=True),
            nn.utils.rnn.pad_sequence(t, batch_first=True))

# ── Vocab builder ─────────────────────────────────────────────
def build_vocab(sentences, min_freq=1):
    c = Counter(w for s in sentences for w in s.split())
    v = {"<PAD>": 0, "<BOS>": 1, "<EOS>": 2, "<UNK>": 3}
    for w, cnt in c.items():
        if cnt >= min_freq: v[w] = len(v)
    return v

# ── BLEU Score (simplified 4-gram) ────────────────────────────
def bleu_score(pred_tokens, ref_tokens, max_n=4):
    from collections import Counter
    import math
    bp = math.exp(min(0, 1 - len(ref_tokens) / max(len(pred_tokens), 1)))
    scores = []
    for n in range(1, max_n + 1):
        pred_ng = Counter(tuple(pred_tokens[i:i+n]) for i in range(len(pred_tokens)-n+1))
        ref_ng  = Counter(tuple(ref_tokens[i:i+n]) for i in range(len(ref_tokens)-n+1))
        clip = sum(min(pred_ng[g], ref_ng[g]) for g in pred_ng)
        total = max(sum(pred_ng.values()), 1)
        scores.append(math.log(max(clip / total, 1e-10)))
    return bp * math.exp(sum(scores) / max_n)

# ── Training Loop ─────────────────────────────────────────────
def train():
    # Example data — replace with your ancient language pairs
    pairs = [
        ("arma virumque cano", "I sing of arms and the man"),
        ("veni vidi vici", "I came I saw I conquered"),
        ("alea iacta est", "the die is cast"),
        ("cogito ergo sum", "I think therefore I am"),
        ("carpe diem", "seize the day"),
    ] * 100  # repeat for demo; use real data in practice

    src_v = build_vocab([p[0] for p in pairs])
    tgt_v = build_vocab([p[1] for p in pairs])
    tgt_iv = {v: k for k, v in tgt_v.items()}  # inverse vocab for decoding

    ds = AncientLangDataset(pairs, src_v, tgt_v)
    dl = DataLoader(ds, batch_size=16, shuffle=True, collate_fn=collate)

    device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
    model = Seq2SeqTransformer(len(src_v), len(tgt_v)).to(device)
    opt = torch.optim.Adam(model.parameters(), lr=3e-4)
    loss_fn = nn.CrossEntropyLoss(ignore_index=0)  # ignore PAD=0

    for epoch in range(30):
        model.train(); total_loss = 0
        for src, tgt in dl:
            src, tgt = src.to(device), tgt.to(device)
            tgt_in = tgt[:, :-1]  # decoder input: all but last token
            tgt_out = tgt[:, 1:]  # labels: all but first token (<BOS>)

            # Causal mask: prevents attending to future tokens
            sz = tgt_in.size(1)
            tgt_mask = torch.triu(torch.ones(sz, sz, device=device), diagonal=1).bool()
            # Padding masks: True where token is PAD
            src_pad = (src == 0); tgt_pad = (tgt_in == 0)

            logits = model(src, tgt_in, tgt_mask=tgt_mask,
                          src_pad_mask=src_pad, tgt_pad_mask=tgt_pad)

            opt.zero_grad()
            loss = loss_fn(logits.reshape(-1, logits.size(-1)), tgt_out.reshape(-1))
            loss.backward(); opt.step()
            total_loss += loss.item()

        # Evaluate BLEU every 5 epochs
        if (epoch + 1) % 5 == 0:
            model.eval()
            avg = total_loss / len(dl)
            # Greedy decode first example
            with torch.no_grad():
                s = torch.tensor([[1] + [src_v.get(w,3) for w in pairs[0][0].split()] + [2]]).to(device)
                dec = torch.tensor([[1]]).to(device)
                for _ in range(20):
                    m = torch.triu(torch.ones(dec.size(1),dec.size(1),device=device),1).bool()
                    o = model(s, dec, tgt_mask=m)
                    nxt = o[:, -1].argmax(-1, keepdim=True)
                    dec = torch.cat([dec, nxt], 1)
                    if nxt.item() == 2: break  # EOS
                pred = [tgt_iv.get(t.item(), "?") for t in dec[0, 1:] if t.item() not in (0,1,2)]
                ref = pairs[0][1].split()
                b = bleu_score(pred, ref)
            print(f"Epoch {epoch+1:3d} | Loss: {avg:.4f} | BLEU: {b:.4f} | Pred: {' '.join(pred)}")

if __name__ == "__main__":
    train()
⏸ Pause & Think

In the capstone code, the causal mask is created with torch.triu(..., diagonal=1).bool(). Why diagonal=1 and not diagonal=0?

torch.triu(matrix, diagonal=1) returns the upper triangle excluding the main diagonal (which represents a token attending to itself). We want True in positions that should be blocked (future tokens), and False where attention is allowed. A token should be able to attend to itself and all previous tokens. If we used diagonal=0, the main diagonal would be True, meaning each token couldn't even look at itself—which breaks self-attention entirely.
