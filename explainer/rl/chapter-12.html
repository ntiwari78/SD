🎯 The Feynman Bridge

Credit at a Dinner Party

Imagine a dinner party where an amazing dessert is served. Everyone loved it. The question: who deserves credit?

The baker? She made the dessert. The ingredient buyer? They selected premium chocolate. The host? They created the atmosphere and invited the right people. The sommelier? They chose the perfect wine pairing.

The λ Parameter

λ = 0 (TD): Give all credit to the most recent contributor — the dessert baker alone.

λ = 0.9: Distribute credit backward in time: 90% to the baker, 81% to the ingredient buyer, 73% to the host, etc. Older decisions matter but less.

λ = 1 (Monte Carlo): Distribute credit equally to everyone who was involved, no matter how far back you go.

In RL, eligibility traces are the "credit distribution mechanism." When the agent receives a reward (the delicious dessert), we need to decide which past states and actions were responsible. The parameter λ controls how far back in the trajectory we assign credit.

The magic of eligibility traces is that they provide two equivalent ways to think about this:

  • Forward view: After the episode ends, compute the weighted average of all n-step returns (weights controlled by λ).
  • Backward view: At each time step, maintain a "trace" z(s) that decays over time, and apply all prediction errors to the trace (eligibility).

Both views give the same answer—but the backward view works online, learning as you go, without waiting for the episode to end.

📐 The Technical Ascent

The λ-Return and Forward View

In Chapter 7, we learned that an n-step return looks ahead n steps:

$$G_{t:t+n} = R_{t+1} + \gamma R_{t+2} + \cdots + \gamma^{n-1} R_{t+n} + \gamma^n V(S_{t+n})$$

The λ-return is a weighted average of all possible n-step returns, where the weights decay geometrically with λ:

$$G_t^\lambda = (1 - \lambda) \sum_{n=1}^{T-t-1} \lambda^{n-1} G_{t:t+n} + \lambda^{T-t-1} G_t$$

The factor $(1 - \lambda)$ normalizes the weights so they sum to 1 (except for the last term). Notice the pattern:

  • 1-step return gets weight $(1 - \lambda)$
  • 2-step return gets weight $(1 - \lambda)\lambda$
  • 3-step return gets weight $(1 - \lambda)\lambda^2$
  • n-step return gets weight $(1 - \lambda)\lambda^{n-1}$
🧮 Special Cases: λ=0 and λ=1 ▼

When λ=0: All weight goes to the 1-step return (TD error):

$$G_t^0 = G_{t:t+1} = R_{t+1} + \gamma V(S_{t+1})$$

When λ=1: All weight goes to the final return (Monte Carlo), weighted equally across all steps:

$$G_t^1 = G_t = R_{t+1} + \gamma R_{t+2} + \cdots + \gamma^{T-t-1} R_T$$

For 0 < λ < 1, we interpolate between these extremes.

The forward view TD(λ) update uses the λ-return:

$$V(S_t) \leftarrow V(S_t) + \alpha \big[ G_t^\lambda - V(S_t) \big]$$

This is elegant but has one critical problem: it requires the full episode to be complete before we can compute \(G_t^\lambda\). We must wait until the end (time T) to know all future rewards. This is offline learning.

📐 The Technical Ascent

The Backward View: Eligibility Traces

Here's the breakthrough: we can compute the same updates as the forward view, but online, using eligibility traces. The backward view maintains a trace vector z(s) for each state:

Eligibility Trace (Accumulating)

$$z_t(s) = \begin{cases} \gamma \lambda z_{t-1}(s) + 1 &\text{if } s = S_t \\ \gamma \lambda z_{t-1}(s) &\text{otherwise} \end{cases}$$

The trace z(s) tracks "how recent and how often" state s was visited. After each step, all traces decay by factor γλ. When we visit state s again, we increment its trace by 1 (add 1). The decay constant γλ means: if s was visited long ago, its trace is nearly zero; if visited recently, its trace is large.

At each time step, we observe a TD error:

$$\delta_t = R_{t+1} + \gamma V(S_{t+1}) - V(S_t)$$

Then we apply this error to all states in proportion to their eligibility:

$$V(S) \leftarrow V(S) + \alpha \delta_t z_t(S) \quad \text{for all } S$$

This is the heart of the backward view: when we get a TD error, every state that was recently visited shares in the credit/blame proportional to how recently it was visited.

🧮 Accumulating vs Replacing Traces ▼

Accumulating traces (above) add 1 each time a state is visited:

$$z_t(s) = \gamma \lambda z_{t-1}(s) + 1 \text{ if } s = S_t$$

If we visit the same state twice in quick succession, its trace grows. This can help when the same state appears multiple times in an episode.

Replacing traces reset to 1 each time a state is visited:

$$z_t(s) = \begin{cases} 1 &\text{if } s = S_t \\ \gamma \lambda z_{t-1}(s) &\text{otherwise} \end{cases}$$

Replacing traces can be more stable with function approximation and off-policy learning. They reset the eligibility each visit rather than accumulating it.

The Equivalence Theorem

For finite MDPs, the backward view (eligibility traces) and the forward view (λ-return) produce identical expected weight updates. This is one of the most beautiful results in RL: two conceptually different algorithms are mathematically equivalent!

💻 Algorithm Implementation

TD(λ) with Accumulating Traces

Below is a complete implementation of TD(λ) for a random walk task, with accumulating eligibility traces. The algorithm updates the value function online at each time step:

Python 
import numpy as np
from collections import defaultdict

class TDLambda:
    """
    TD(λ) agent with accumulating eligibility traces.

    Core idea:
      1. Maintain eligibility trace z(s) for each state
      2. At each step, observe TD error δ_t
      3. Update all states: V(s) += α δ_t z(s)
      4. Decay and update traces: z(s) *= γλ, then z(S_t) += 1
    """

    def __init__(self, num_states, alpha=0.1, gamma=0.99, lambda_=0.9):
        """
        Args:
          num_states: Number of states in the MDP
          alpha: Step-size parameter (learning rate)
          gamma: Discount factor (importance of future rewards)
          lambda_: Trace decay parameter (λ)
            λ=0: pure TD(0)
            λ=1: pure Monte Carlo
            0<λ<1: interpolate
        """
        self.num_states = num_states
        self.alpha = alpha
        self.gamma = gamma
        self.lambda_ = lambda_

        # Value function: V(s) for each state
        self.V = np.zeros(num_states)

        # Eligibility trace: z(s) for each state
        self.z = np.zeros(num_states)

    def update(self, state, reward, next_state, done=False):
        """
        Perform one TD(λ) update step.

        Args:
          state: current state index
          reward: immediate reward signal
          next_state: next state index
          done: whether the episode terminated
        """
        # 1. Compute TD error: δ = r + γV(s') - V(s)
        if done:
            td_error = reward - self.V[state]
        else:
            td_error = reward + self.gamma * self.V[next_state] - self.V[state]

        # 2. Update all states proportional to eligibility
        self.V += self.alpha * td_error * self.z

        # 3. Decay all traces and add current state
        self.z *= self.gamma * self.lambda_
        self.z[state] += 1.0

        return td_error

    def reset_trace(self):
        """Reset eligibility trace at episode start."""
        self.z = np.zeros(self.num_states)


# Example: Learning a 5-state random walk
def run_random_walk_episode(agent):
    """Run one episode of a 5-state random walk."""
    state = 2  # Start in middle
    agent.reset_trace()

    for t in range(20):
        # Random walk: move left or right with equal probability
        next_state = state + np.random.choice([-1, 1])

        # Boundary: episode ends if we leave [0, 4]
        if next_state < 0 or next_state >= 5:
            reward = 1.0 if next_state >= 5 else 0.0
            agent.update(state, reward, next_state, done=True)
            break
        else:
            reward = 0.0
            agent.update(state, reward, next_state, done=False)
            state = next_state


# Run multiple episodes to learn
if __name__ == "__main__":
    agent = TDLambda(num_states=5, alpha=0.01, gamma=0.99, lambda_=0.9)

    num_episodes = 100
    for episode in range(num_episodes):
        run_random_walk_episode(agent)

    print("Learned values:", agent.V)
    # Expected: higher values toward the right (reward at rightmost boundary)
Key Insight: Why Traces Work Online

Without eligibility traces, we could only update the current state based on the TD error. With traces, we update all recently visited states at once, which is equivalent to using the λ-return but without waiting for the episode to end. This is the power of the equivalence theorem.

Practical Insight

TD(λ) with 0.9 ≤ λ ≤ 0.95 is often a "sweet spot" — it balances the low variance of TD(0) with the unbiased estimates of Monte Carlo. In practice, λ should be tuned as a hyperparameter alongside α.

📦 Socratic Sandbox

Deep Questions About Eligibility Traces

1
Why are the forward view and backward view mathematically equivalent?
▼
The forward view computes a weighted average of n-step returns: \(G_t^\lambda = (1-\lambda)\sum_{n=1}^{T-t-1} \lambda^{n-1} G_{t:t+n}\). The backward view, through the eligibility trace mechanism, distributes the TD error \(\delta_t\) to all past states according to their recency. When you work through the algebra, the cumulative effect of all TD errors applied to traces produces the exact same expected updates as the forward view. This equivalence is profound: two different computational paths lead to the same destination.
2
When would λ=0.9 be better than λ=0, and why?
▼
TD(0) (λ=0) only uses 1-step returns, so it converges fast and has low variance, but it may have high bias if the value function is poorly initialized. TD(0.9) looks further ahead, reducing initial bias while keeping variance reasonable. In a problem where several steps are "necessary" to understand the value of a state (e.g., the state has no immediate reward but leads to high-reward states several steps away), TD(0.9) will learn faster and more accurately than TD(0). The tradeoff: slight increase in variance for significant bias reduction.
3
What's the practical difference between accumulating and replacing traces?
▼
Accumulating traces: If you visit a state twice quickly, the trace grows (z += 1 each visit). This means revisiting a state amplifies its eligibility for the next error, which can be useful if the same state appears multiple times and should receive double credit. Replacing traces: The trace resets to 1 each visit (z = 1), then decays. This is more stable with function approximation and off-policy learning, where accumulating traces can become numerically unstable or misleading. In tabular methods on-policy, accumulating traces are more faithful to the theory; replacing traces are a practical hack that often works better in practice.
4
Why is the forward view offline but the backward view is online?
▼
The forward view computes \(G_t^\lambda\), which depends on all future rewards: \(R_{t+1}, R_{t+2}, \ldots, R_T\). You cannot know the full return \(G_t^\lambda\) until the episode ends at time T. So you must wait until the episode is over to update. The backward view, however, processes TD errors as they arrive. Each step computes \(\delta_t\) (which only needs \(R_{t+1}\) and \(V(S_{t+1})\)), applies it to all traced states, and updates the value function immediately. No waiting required. This is why the backward view is the practical choice for online, streaming learning.
5
How do eligibility traces solve the credit assignment problem?
▼
Credit assignment asks: which actions (or states) in the past were responsible for the outcome? Eligibility traces answer this by maintaining a "memory" of recent states (the trace) and distributing the TD error proportionally to that memory. If a state was visited recently, it has a high trace and receives more of the error. If it was visited long ago, the trace has decayed and it receives less. The parameter λ controls the decay rate: high λ spreads credit far back in time; low λ concentrates it near the most recent states. This is elegant: a single scalar parameter controls the temporal scope of credit assignment.
