🎯 The Feynman Bridge

RL Is Like Learning to Cook

Imagine you've just moved into your first apartment and you need to learn to cook. There's no cookbook — just a kitchen, some ingredients, and your taste buds.

Every time you cook a meal, you take actions (add salt, turn up the heat, stir more), then you taste the result (the reward). Over time, you build up intuition about which actions lead to delicious meals and which lead to burnt disasters. You learn from the consequences of your own actions.

The Core Analogy

You = the agent. The kitchen = the environment. Taste = the reward signal. Your recipe intuition = the value function. Your cooking strategy = the policy.

This is the heart of reinforcement learning: an agent learns to make good decisions by interacting with an environment and receiving reward signals — not by being shown the "correct" action at each step (that would be supervised learning), and not by just finding hidden patterns (that would be unsupervised learning).

Two special challenges make RL unique from other types of machine learning:

1. Trial-and-error search. Nobody tells you the right answer. You have to try things and see what happens. Sometimes you add too much cayenne pepper and learn the hard way.

2. Delayed reward. The meal's taste (reward) comes at the end, but the critical decisions were made earlier — choosing to marinate the chicken overnight was the key move, not the final garnish. Credit must be assigned backward in time.

And then there's the fundamental dilemma that every learner faces:

Exploration vs. Exploitation

Should you cook your best recipe again tonight (exploit what you know works), or try a brand-new ingredient you've never used before (explore the unknown)?

Too much exploitation and you eat the same meal forever, never discovering something even better. Too much exploration and you suffer a lot of bad meals. Every RL agent faces this tension. Getting the balance right is one of the deepest open questions in the field.

📐 The Technical Ascent

Formalizing the RL Problem

At its core, RL is about an agent interacting with an environment over a series of time steps. At each step, the agent observes the current state, takes an action, and receives a reward.

Agent
Action At →
Environment
St+1, Rt+1 ←

At time step \(t\), the agent is in state \(S_t\), takes action \(A_t\), then transitions to state \(S_{t+1}\) and receives reward \(R_{t+1}\). This loop continues until the task ends (episodic) or forever (continuing).

🧮 Mathematical Curiosity: The Return ▼

The agent's goal is to maximize the expected return — the total accumulated reward from time \(t\) onward. For episodic tasks with a terminal state at step \(T\):

$$G_t = R_{t+1} + R_{t+2} + R_{t+3} + \cdots + R_T$$

For continuing tasks (or to weight nearer rewards more), we introduce a discount factor \(\gamma \in [0, 1]\):

$$G_t = R_{t+1} + \gamma R_{t+2} + \gamma^2 R_{t+3} + \cdots = \sum_{k=0}^{\infty} \gamma^k R_{t+k+1}$$

When \(\gamma = 0\), the agent is completely myopic — it only cares about the very next reward. When \(\gamma = 1\), it weights all future rewards equally. Most practical settings use \(\gamma\) between 0.9 and 0.99.

The critical insight from the Tic-Tac-Toe example in the book is the temporal-difference (TD) update rule, which updates the value of a state based on the value of the next state:

$$V(S_t) \leftarrow V(S_t) + \alpha \Big[ V(S_{t+1}) - V(S_t) \Big]$$

where \(\alpha\) is the step-size parameter (learning rate). This simple rule is the seed from which all of TD learning grows — it will blossom into SARSA, Q-learning, and eventually deep RL algorithms like DQN.

🧮 Mathematical Curiosity: Why RL ≠ Supervised ≠ Unsupervised ▼

Supervised learning optimizes: \(\min_\theta \sum_i L(f_\theta(x_i), y_i)\) — you have labels \(y_i\).

Unsupervised learning finds structure: \(\max_\theta \sum_i \log p_\theta(x_i)\) — no labels, find patterns.

Reinforcement learning maximizes cumulative reward: \(\max_\pi \mathbb{E}_\pi \left[\sum_{t=0}^{T} \gamma^t R_{t+1}\right]\) — no labels, just a scalar reward signal. The agent must discover good behavior through trial and error.

The key difference: in RL, the data distribution depends on the agent's own actions. The agent generates its own training data by interacting with the environment. This creates a feedback loop that doesn't exist in supervised or unsupervised learning.

🧱 Core Concepts

The Four Elements of RL

Every RL system is built from four fundamental pieces:

1. Policy (π)

The agent's strategy — a mapping from states to actions. It can be deterministic (\(\pi(s) = a\)) or stochastic (\(\pi(a|s) = \text{probability}\)). The policy is the core of the agent: given only the policy, the agent knows how to behave.

2. Reward Signal (R)

A single number the environment gives the agent at each step. This defines the goal. The agent's sole objective is to maximize total reward over time. Rewards are immediate and given by the environment — think pleasure/pain.

3. Value Function (V or Q)

An estimate of how much total future reward the agent can expect from a given state (or state-action pair). While rewards are immediate, values capture the long-term desirability. A state might yield low immediate reward but have high value because it leads to reward-rich states. Estimating values is the central challenge of RL.

4. Model (optional)

A representation of how the environment works — predicting the next state and reward. Agents with models can plan ahead (model-based RL), while agents without models learn purely from trial-and-error (model-free RL). The Tic-Tac-Toe player in the book uses a model (it knows the rules), but many powerful RL algorithms don't.

💻 Algorithm Implementation

Tic-Tac-Toe Value Learning in Python

Below is the complete value-learning algorithm from Section 1.5, with every line annotated to map back to the concepts above:

Python / NumPy 
import numpy as np
from collections import defaultdict

class TicTacToeRL:
    """
    A reinforcement learning agent that learns to play Tic-Tac-Toe
    using the temporal-difference (TD) value-learning method from
    Sutton & Barto, Section 1.5.

    Core idea: V(S_t) <- V(S_t) + alpha * [V(S_{t+1}) - V(S_t)]
    """

    def __init__(self, symbol='X', alpha=0.1, epsilon=0.1):
        # alpha (α): step-size parameter — controls how fast we learn
        #   Large α → fast learning but noisy estimates
        #   Small α → slow learning but stable estimates
        self.alpha = alpha

        # epsilon (ε): exploration rate — probability of taking a
        #   random action instead of the greedy (best-known) action
        #   This is how we balance EXPLORATION vs EXPLOITATION
        self.epsilon = epsilon

        self.symbol = symbol          # 'X' or 'O'
        self.value_table = {}         # V(s): maps state → estimated value
        self.history = []             # sequence of states visited in one game

    def state_key(self, board):
        """Convert 3x3 board to a hashable tuple (the 'state' s)."""
        return tuple(board.flatten())

    def get_value(self, board):
        """
        V(s): Look up the estimated value of state s.
        If never seen before, initialize to 0.5 (uncertain).
        States where we've won → 1.0, lost → 0.0.
        """
        key = self.state_key(board)
        if key not in self.value_table:
            # Optimistic initialization: start at 0.5 (50% win estimate)
            self.value_table[key] = 0.5
        return self.value_table[key]

    def choose_action(self, board, available_moves):
        """
        The POLICY π(s): choose which action to take in state s.

        With probability ε → EXPLORE (random move)
        With probability 1-ε → EXPLOIT (greedy: pick highest V(s'))
        """
        if np.random.random() < self.epsilon:
            # ── EXPLORATION: random action to discover new states ──
            return available_moves[np.random.randint(len(available_moves))]

        # ── EXPLOITATION: pick the action leading to highest V(s') ──
        best_value = -np.inf
        best_move = available_moves[0]

        for move in available_moves:
            # Simulate taking this action → get next state s'
            next_board = board.copy()
            next_board[move] = self.symbol

            # V(s'): value of the state we'd transition to
            value = self.get_value(next_board)

            if value > best_value:
                best_value = value
                best_move = move

        return best_move

    def update_values(self, reward):
        """
        After a game ends, apply the TD update rule to all states
        visited during the episode, working backwards.

        The key equation:
          V(S_t) ← V(S_t) + α [ V(S_{t+1}) - V(S_t) ]
                                  ╰──── TD Error (δ) ────╯

        - V(S_t):   value of the state before the move
        - V(S_{t+1}): value of the state after the move (or reward)
        - α:        step-size parameter (learning rate)
        - δ:        temporal-difference error — the "surprise"
        """
        # Set the value of the terminal state to the actual reward
        # (1.0 for win, 0.0 for loss, 0.3 for draw)
        if self.history:
            self.value_table[self.history[-1]] = reward

        # Walk backwards through the game history
        for t in range(len(self.history) - 2, -1, -1):
            state_t = self.history[t]         # S_t
            state_next = self.history[t + 1]  # S_{t+1}

            v_t = self.value_table[state_t]           # V(S_t)
            v_next = self.value_table[state_next]     # V(S_{t+1})

            # ── THE TD UPDATE ──
            td_error = v_next - v_t                   # δ = V(S_{t+1}) - V(S_t)
            self.value_table[state_t] = v_t + self.alpha * td_error
            #                           ╰─── V(S_t) + α·δ ───╯

        self.history = []  # Reset for next episode

    def record_state(self, board):
        """Remember each state we visit during a game (for the update)."""
        self.history.append(self.state_key(board))


def train_agent(episodes=50000):
    """
    Train two RL agents by playing them against each other.
    This is the "learning from experience" loop.
    """
    agent_x = TicTacToeRL(symbol='X', alpha=0.1, epsilon=0.1)
    agent_o = TicTacToeRL(symbol='O', alpha=0.1, epsilon=0.1)

    wins = {'X': 0, 'O': 0, 'draw': 0}

    for ep in range(episodes):
        board = np.full((3, 3), ' ')  # Empty 3x3 board = initial state S_0
        current = agent_x             # X goes first
        other = agent_o

        game_over = False
        while not game_over:
            # Current agent observes state S_t
            current.record_state(board)

            # Find available actions A(s)
            available = [(r, c) for r in range(3) for c in range(3)
                        if board[r, c] == ' ']

            if not available:
                # Draw
                agent_x.update_values(0.3)  # Draw reward
                agent_o.update_values(0.3)
                wins['draw'] += 1
                game_over = True
                break

            # Agent selects action A_t using ε-greedy policy
            move = current.choose_action(board, available)
            board[move] = current.symbol  # Environment transitions to S_{t+1}

            # Check if this action won the game (terminal reward R)
            if check_winner(board, current.symbol):
                current.record_state(board)
                current.update_values(1.0)   # Winner gets reward = 1
                other.update_values(0.0)     # Loser gets reward = 0
                wins[current.symbol] += 1
                game_over = True
            else:
                current, other = other, current  # Switch turns

        # Decay exploration over time (anneal ε)
        if ep % 10000 == 0 and ep > 0:
            agent_x.epsilon *= 0.5
            agent_o.epsilon *= 0.5

    return agent_x, agent_o, wins


def check_winner(board, symbol):
    """Check rows, columns, and diagonals for three in a row."""
    for i in range(3):
        if all(board[i, :] == symbol) or all(board[:, i] == symbol):
            return True
    if board[0,0] == board[1,1] == board[2,2] == symbol:
        return True
    if board[0,2] == board[1,1] == board[2,0] == symbol:
        return True
    return False


if __name__ == '__main__':
    agent_x, agent_o, wins = train_agent(50000)
    total = sum(wins.values())
    print(f"After 50,000 games of self-play:")
    print(f"  X wins: {wins['X']/total:.1%}")
    print(f"  O wins: {wins['O']/total:.1%}")
    print(f"  Draws:  {wins['draw']/total:.1%}")
    print(f"  States learned: {len(agent_x.value_table)}")

🧪 Socratic Sandbox

Debug the Agent

You observe that your Tic-Tac-Toe RL agent, after 50,000 training episodes, still loses frequently to a simple rule-based opponent. It seems to repeat the same mistakes. Let's diagnose the problem together.

Scenario: The Stuck Agent

The agent wins about 35% of games against a rule-based player, draws 25%, and loses 40%. It keeps making the same opening move (top-left corner) even though the opponent has learned to counter it. What's going wrong?

1 The agent always plays the same opening move. Which of the four elements (policy, reward, value, model) is most likely misconfigured? ▼

The policy — specifically the exploration component. If ε (epsilon) is too low or has decayed to zero, the agent is purely exploiting: it always picks the greedy action. Since it converged early on a sub-optimal value table, it keeps repeating the same opening. The fix: ensure ε doesn't decay to zero too quickly, or use a more sophisticated exploration strategy like optimistic initialization (covered in Chapter 2).

2 The agent's value estimates for many board positions are still at the initial value of 0.5. Why hasn't it learned these states, and what does this suggest about the training process? ▼

Many states remain at 0.5 because the agent has never visited them. Tic-Tac-Toe has 5,478 valid board positions, but a low-exploration policy will only visit a narrow subset. This is a symptom of insufficient exploration. It also reveals why function approximation (Part II) becomes critical for larger problems: instead of learning a value for every individual state, the agent can generalize from similar states it has seen.

3 If we increase α (the learning rate) from 0.1 to 0.9, will the agent learn faster? What are the risks? ▼

A higher α means the agent overwrites old estimates more aggressively with new information. In a stationary environment (one that doesn't change), this leads to noisy, oscillating value estimates — the agent "forgets" good estimates when it occasionally has a bad game. However, in a non-stationary environment (like playing against an opponent who adapts), a higher α is actually beneficial because it lets the agent track the shifting dynamics. This tradeoff between tracking and stability is a recurring theme throughout the book (see Chapter 2.5).

4 The TD update only occurs after greedy moves in the book's version. Should we also update after exploratory moves? What would change? ▼

This is actually Exercise 1.4 from the book! If we update after all moves (including exploratory ones), the agent learns a different set of values — they converge to the values under the current ε-greedy policy (which includes random moves), rather than the values under the optimal (fully greedy) policy. In technical terms: updating only on greedy moves learns off-policy (the target values are for the greedy policy), while updating on all moves learns on-policy (the values include the effects of exploration). This distinction becomes central in Chapters 5 and 6.

5 Instead of using a value function, could we use an evolutionary approach — maintaining a population of policies and selecting the fittest? What would we lose? ▼

Yes, evolutionary methods work — they would maintain a population of strategies, play games to evaluate fitness, and breed/mutate the best ones. But we'd lose a key advantage: per-move credit assignment. The evolutionary approach only gets feedback at the end of a game (win or lose), and credits the entire strategy equally. The value function approach, by contrast, evaluates individual states, learning that some positions are strong and others are weak. This allows much faster learning because information from each game update flows through every move. As the book notes, this is why value function methods often vastly outperform evolutionary approaches in practice.

📜 Historical Context

A Brief History of RL

RL emerged from three independent intellectual threads that converged in the late 1980s:

1911
Thorndike's "Law of Effect" — actions followed by satisfying consequences become more likely. The psychological root of the reward signal.
1950s
Bellman & Dynamic Programming — Richard Bellman formalizes optimal control with the Bellman equation, providing the mathematical foundation for value functions.
1959
Samuel's Checkers Player — Arthur Samuel builds a program that learns to play checkers by adjusting evaluation weights through self-play. An early form of TD learning.
1972
Klopf's "Heterostatic" Theory — A. Harry Klopf argues that individual neurons are "hedonistic" — seeking to maximize their own reward. This inspires the modern RL research program.
1988
Sutton's TD(λ) — Sutton publishes the formal TD learning framework, unifying ideas from animal learning, dynamic programming, and adaptive systems.
1989
Watkins' Q-Learning — Chris Watkins develops Q-learning, a simple off-policy TD method that converges to the optimal policy. This becomes the workhorse algorithm of RL.
1992
TD-Gammon — Gerry Tesauro combines TD learning with a neural network to create a world-class backgammon player, demonstrating that RL + function approximation can scale to complex domains.
