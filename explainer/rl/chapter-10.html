🎯 The Feynman Bridge

Control with Approximation: Generalization Under Action

Imagine you're a self-driving car learning to navigate without a pre-recorded map. You can't memorize every road configuration — there are infinitely many. Instead, you must generalize from similar situations. When you learn that increasing the steering angle helps on sharp curves, you apply that insight to all sharp curves, not just the exact one you just experienced.

In RL, this is the core challenge of on-policy control with approximation. You don't just estimate value functions; you learn policies directly, adjusting both what you know about value (the approximator) and what you do (the policy) simultaneously.

The Core Analogy

Control with approximation is like a self-driving car that can't memorize every possible road situation — it must generalize from similar experiences, adjusting both what it knows (value) and what it does (policy) simultaneously.

The challenge? You're learning your policy from incomplete, biased feedback. The car learns from its own driving, which means it might never explore regions of the road where it would crash. This is the central tension: you need exploration to learn good policies, but your approximator must stay stable as you update it.

Semi-gradient methods solve this by treating the target (bootstrapped value) as fixed, not differentiating through it. This sacrifice — not using true gradients — buys stability and convergence.

⚡ Technical Ascent

Semi-gradient Control Methods

We now move from prediction (estimating a given policy's value) to control (finding a better policy). The key methods form a progression:

  1. Episodic Semi-gradient Sarsa: One-step bootstrap for episodic tasks
  2. Semi-gradient n-step Sarsa: Multi-step returns before bootstrapping
  3. Average-reward formulation: For continuing tasks without artificial termination
  4. Differential semi-gradient n-step Sarsa: Scaling average reward with temporal differences
📐 Semi-gradient Sarsa Update Rule ▼

Episodic semi-gradient Sarsa: Update weights based on one-step bootstrapped reward.

$$w_{t+1} = w_t + \alpha[R_{t+1} + \gamma \hat{q}(S_{t+1}, A_{t+1}, w_t) - \hat{q}(S_t, A_t, w_t)]\nabla_w \hat{q}(S_t, A_t, w_t)$$

Key insight: The gradient is computed at the current state-action pair. The bootstrap target is held fixed (not differentiated), making this "semi" gradient.

Update frequency: After every action. Typically used with ε-greedy policy derived from approximate action values.

📐 Semi-gradient n-step Sarsa ▼

Multi-step bootstrapping: Wait n steps before bootstrapping.

$$G_t^{(n)} = R_{t+1} + \gamma R_{t+2} + \cdots + \gamma^{n-1} R_{t+n} + \gamma^n \hat{q}(S_{t+n}, A_{t+n}, w_t)$$
$$w_{t+n} = w_{t+n-1} + \alpha[G_t^{(n)} - \hat{q}(S_t, A_t, w_{t+n-1})]\nabla_w \hat{q}(S_t, A_t, w_{t+n-1})$$

Trade-off: Larger n reduces bias (less bootstrapping) but increases variance. Optimal n depends on the task horizon and function approximation quality.

📐 Average Reward Definition ▼

For continuing tasks: Don't discount infinitely; measure long-run average reward.

$$r(\pi) = \lim_{h \to \infty} \frac{1}{h} \sum_{t=1}^h \mathbb{E}[R_t | S_0, A_{0:t-1} \sim \pi]$$

Intuition: Over a long trajectory, what is the average reward per step? This avoids the discounting artifice and is natural for continuing processes (manufacturing, robotics).

📐 Differential Value Function ▼

Value relative to average reward: How much better/worse than the baseline?

$$v_\pi(s) = \mathbb{E}_\pi\left[\sum_{k=1}^\infty (R_{t+k} - r(\pi)) \,\Big|\, S_t = s\right]$$

Meaning: The value function measures deviation from the long-run average. A state is valuable if it leads to rewards better than the steady-state average.

📐 Differential Temporal Difference Error ▼

Bootstrap target adjusted for average reward:

$$\delta_t = R_{t+1} - \bar{R}_t + \hat{v}(S_{t+1}, w_t) - \hat{v}(S_t, w_t)$$

where $\bar{R}_t$ is an estimate of the average reward.

Use case: In continuing tasks, update the value function and the average reward estimate together. The differential error measures surprise relative to the long-run baseline.

Why Semi-Gradient?

True gradient descent would differentiate through the bootstrap target, but we typically don't know the true target. By treating the bootstrap as fixed, we sacrifice gradient accuracy but gain convergence guarantees and stability. This is a pragmatic trade-off in RL.

⚙️ Algorithm

Semi-gradient Sarsa for Mountain Car

Below is a complete, production-ready implementation of semi-gradient Sarsa on the Mountain Car problem (~110 lines).

python 
import numpy as np
from collections import defaultdict

class SemiGradientSarsa:
    """Semi-gradient Sarsa for continuous state spaces using tile coding."""

    def __init__(self, alpha=0.1, epsilon=0.1, gamma=0.99, tiles=8):
        self.alpha = alpha
        self.epsilon = epsilon
        self.gamma = gamma
        self.tiles = tiles
        self.w = defaultdict(float)

    def get_tiles(self, s):
        """Tile coding: discretize continuous state into features."""
        position, velocity = s
        # Normalize to [0, 1]
        pos_norm = (position + 1.2) / 1.8
        vel_norm = (velocity + 0.07) / 0.14

        features = []
        for i in range(self.tiles):
            # Offset tiling
            offset = i / self.tiles
            pos_idx = int((pos_norm + offset) * self.tiles) % self.tiles
            vel_idx = int((vel_norm + offset) * self.tiles) % self.tiles
            features.append((i, pos_idx, vel_idx))
        return features

    def q_hat(self, s, a, w=None):
        """Approximate action value using tile coding."""
        if w is None:
            w = self.w
        features = self.get_tiles(s)
        return sum(w[(f, a)] for f in features)

    def grad_q_hat(self, s, a):
        """Gradient of q_hat w.r.t. w: returns feature tiles."""
        return self.get_tiles(s)

    def select_action(self, s):
        """ε-greedy action selection."""
        if np.random.rand() < self.epsilon:
            return np.random.randint(3)  # 0: left, 1: coast, 2: right

        # Greedy: best of 3 actions
        q_vals = [self.q_hat(s, a) for a in range(3)]
        return np.argmax(q_vals)

    def update(self, s, a, r, s_next, a_next):
        """Semi-gradient Sarsa update."""
        q_current = self.q_hat(s, a)
        q_next = self.q_hat(s_next, a_next)
        td_error = r + self.gamma * q_next - q_current

        # Update weights: w += alpha * delta * grad_q
        features = self.grad_q_hat(s, a)
        for f in features:
            self.w[(f, a)] += self.alpha * td_error

    def train(self, env, episodes=500):
        """Train the agent on Mountain Car."""
        episode_rewards = []

        for ep in range(episodes):
            s, _ = env.reset()
            a = self.select_action(s)
            episode_reward = 0

            for t in range(200):
                s_next, r, done, _, _ = env.step(a)
                episode_reward += r

                a_next = self.select_action(s_next)
                self.update(s, a, r, s_next, a_next)

                s, a = s_next, a_next
                if done:
                    break

            episode_rewards.append(episode_reward)

            if (ep + 1) % 100 == 0:
                avg_reward = np.mean(episode_rewards[-100:])
                print(f"Episode {ep+1}: avg reward = {avg_reward:.1f}")

        return episode_rewards

Key implementation details:

  • Tile coding: Discretizes the continuous state space into overlapping tiles. This creates linear features that scale well.
  • ε-greedy policy: Balances exploration (random) and exploitation (greedy w.r.t. approximate values).
  • Semi-gradient update: Only the current state-action gradient is used; the bootstrap target is treated as fixed.
  • Convergence: Under suitable conditions (decreasing α, bounded rewards), the weights converge to a fixed point.
🌉 Deep RL Bridge

From Semi-gradient Sarsa to Modern Methods

Semi-gradient Sarsa is the conceptual grandfather of modern deep reinforcement learning. Here's the progression:

The Progression

Semi-gradient Sarsa (linear approximation) → DQN (non-linear, off-policy) → PPO/A3C (policy gradient)

Semi-gradient Sarsa → DQN: The jump from tile coding to neural networks. DQN adds experience replay (decorrelate samples) and target networks (stabilize the bootstrap), solving the instability that pure gradient descent suffered.

DQN → PPO: DQN learns a value function to guide action selection. PPO learns the policy directly, using the value function only as a baseline to reduce variance. This is more flexible and works in continuous action spaces.

What persists: All modern methods still rely on bootstrapping (using estimated values as targets) and semi-gradient updates (not differentiating through the target). The core trade-off — stability vs. true gradient — remains central.

1
Why is the Mountain Car problem "hard" for a naive agent?
▼

Think first: What makes one-shot actions (always accelerate forward) insufficient? What has to happen in the environment?

Answer: The Mountain Car has negative (fail) rewards until the goal is reached. Gravity pulls the car back, so the agent must learn to oscillate left and right to build momentum. A policy that always accelerates forward will never crest the peak. The reward signal is sparse (only at goal), so the agent must learn a long-horizon strategy. This requires credit assignment over many steps.

2
What is the difference between "semi-gradient" and "true gradient"?
▼

Think first: In the Sarsa update, we have a target $R_{t+1} + \gamma \hat{q}(S_{t+1}, A_{t+1}, w_t)$. What if we computed gradients through this entire expression?

Answer: True gradient would differentiate through the bootstrap value, treating the target as if it depends on w. This leads to exponential blow-up in the computational graph and instability—we'd be chasing a moving target. Semi-gradient fixes the target as a constant, so we only differentiate $\hat{q}(S_t, A_t, w)$. This is "semi" because it's not a true gradient of the loss, but it's stable and converges under mild conditions.

3
When would you use average-reward methods instead of discounted?
▼

Think first: In what kinds of problems does an artificial discount factor feel wrong? When is the long-run average more natural than future discounting?

Answer: Use average-reward for continuing tasks where there's no natural termination: a robot that must operate indefinitely, a manufacturing process, a thermostat. Discounting future rewards makes sense for finite-horizon problems (games, navigation to a goal), but for continuous operation, the long-run average reward per step is the right objective. Average-reward also avoids tuning γ, which can be hard in continuing tasks.

4
How does tile coding enable linear approximation in continuous spaces?
▼

Think first: A continuous state like position ∈ [-1.2, 0.6] can't be a table index. How do you turn it into discrete features?

Answer: Tile coding discretizes the continuous state space into overlapping grids (tiles). Each grid has a different offset, so each continuous state falls into a small set of tiles. Each tile is then treated as a feature (a binary indicator: "is state in this tile?"). The value is a linear combination of these tile features. This is efficient, generalizes across nearby states, and allows gradient-based updates. Multiple tilings increase resolution.

5
What happens if you set α (learning rate) too high? Too low?
▼

Think first: In the update $w := w + \alpha \cdot \text{(TD error)} \cdot \text{(features)}$, what role does α play?

Answer: α controls the step size. Too high (e.g., α=1): each update is large, causing oscillations and divergence. The agent "overshoots" and never settles. Too low (e.g., α=0.001): updates are tiny, learning is slow, and the agent may get stuck in local optima or not converge in reasonable time. The optimal α balances fast learning with stability. In practice, use a schedule: start high, decay over time.
