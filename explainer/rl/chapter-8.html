🎯 The Feynman Bridge

Dyna: Chess Player with a Memory

Imagine a chess player who masters the game in two complementary ways:

Learning from real games: Every time they play a real opponent, they experience the consequences of their moves directly. They win or lose, and these lessons stick.

Mental replay: Between games, they don't just rest — they replay the games in their mind. "What if I had played the knight to d4 instead?" They construct imaginary positions, imagine responses, and learn from these simulated games without risk.

The Core Insight

Dyna is exactly this: an agent that learns from both real experience (direct RL) and simulated experience (model-based planning) simultaneously. The model acts like the player's memory and imagination — it predicts what will happen next, enabling mental rehearsal.

This combination is powerful: direct RL learns from the environment's true distribution, while planning exploits the model to generate additional training data at nearly zero cost. The agent learns faster and more efficiently than either approach alone.

Three key intuitions about Dyna:

1. Models are cheaper than reality. Once you have a model, you can simulate millions of imaginary trajectories in seconds. Real interactions are slow and costly.

2. More updates = faster learning. With Dyna, every real experience generates multiple value updates (one direct, plus n planning updates). This acceleration is why agents trained with Dyna converge faster.

3. Model errors matter. If your model is wrong, planning can reinforce bad beliefs. Dyna-Q+ addresses this by adding an exploration bonus when the model hasn't been updated recently.

📐 The Technical Ascent

Models, Planning, and the Dyna Loop

Every Dyna agent has two parallel learning processes:

Direct RL (Actor)

When the agent acts in the real environment, it updates its value function based on actual experience: \(V(S_t) \leftarrow V(S_t) + \alpha [R + \gamma V(S') - V(S_t)]\). This is standard model-free learning (e.g., Q-learning).

Model Learning

Simultaneously, the agent learns a model \(M\) that predicts the next state and reward: \(M(s, a) \rightarrow (s', r)\). This is typically supervised learning — given state-action pairs, predict the resulting state and immediate reward.

Planning (Search)

Between or after real actions, the agent uses its model to generate simulated experience and updates the value function on imaginary trajectories. The same learning rule applies, but with synthetic transitions from the model instead of the real environment.

The core loop:

$$\text{Real action} \rightarrow \text{Observe } (S', R) \rightarrow \text{Update } V \text{ and } M \rightarrow \text{Plan: loop } n \text{ times} \rightarrow \text{Repeat}$$

Key insight: The more planning steps per real action, the faster the agent learns — up to a point. Diminishing returns kick in when the model becomes the bottleneck (if the model is wrong, more planning doesn't help).

🧮 Mathematical Curiosity: Dyna-Q Equations ▼

Direct RL step (standard Q-learning):

$$Q(S_t, A_t) \leftarrow Q(S_t, A_t) + \alpha [R_{t+1} + \gamma \max_a Q(S_{t+1}, a) - Q(S_t, A_t)]$$

Model learning:

$$M(S_t, A_t) \leftarrow (S_{t+1}, R_{t+1})$$

Planning step (simulated Q-learning): For \(n\) iterations, pick a random visited state-action pair \((S, A)\) from the history:

$$S', R \sim M(S, A)$$ $$Q(S, A) \leftarrow Q(S, A) + \alpha [R + \gamma \max_a Q(S', a) - Q(S, A)]$$

Dyna-Q+ (exploration bonus): When the model hasn't predicted a state-action pair recently (time since last update = \(\tau\)), add an exploration bonus:

$$r^+ = R + \beta \sqrt{\tau}$$

where \(\beta > 0\). This encourages the agent to revisit underexplored state-action pairs in the model.

Prioritized sweeping: Instead of randomly sampling state-action pairs for planning, prioritize those whose value has changed significantly. Maintain a priority queue and plan on high-value states first. This speeds up value propagation and reduces the number of planning steps needed.

🧠 Core Algorithm

Dyna-Q: The Complete Algorithm

Dyna-Q integrates three components at each time step:

  1. Direct RL: Take a real action, observe the result, update the value function.
  2. Model Learning: Add the observed transition to the model (state-action → next state, reward).
  3. Planning: Perform \(n\) simulated rollouts using the model to update values on imaginary experience.
Why Dyna-Q Learns Faster

With \(n\) planning steps per real action, the agent generates \(n + 1\) value updates from a single real experience. This amplification effect dramatically accelerates learning. In gridworlds, Dyna-Q with even modest \(n\) (e.g., 10-50) can converge in a fraction of the time required by pure Q-learning.

When the Model Is Wrong: If the learned model doesn't accurately represent the environment, planning can lead the agent astray. The simulated experiences will be biased, reinforcing incorrect value estimates. This is why Dyna-Q+ includes exploration bonuses — they encourage re-exploration of state-action pairs when the model hasn't been updated recently, serving as a safeguard against model drift.

Prioritized Sweeping: A refinement of Dyna-Q that maintains a priority queue of state-action pairs. After each real action, only the states with the largest recent value changes are selected for planning updates. This focuses computation on high-impact states and dramatically reduces the number of planning steps needed.

🎳 Planning at Decision Time

Monte Carlo Tree Search: Planning When It Matters

Dyna performs planning in the background, between and after actions. But there's another strategy: planning at decision time — when you're about to make a move, use your model to search ahead and find the best action right now.

Monte Carlo Tree Search (MCTS) is the gold standard for decision-time planning. It builds a search tree by repeatedly simulating rollouts from the current state and backing up the results. Unlike exhaustive search (which explores every possible path), MCTS focuses computation on the most promising branches.

MCTS is Like a Fortune Teller's Strategy

Instead of reading all possible futures exhaustively, a fortune teller looks at the most likely paths first. MCTS does the same: it samples many possible futures (by simulating random playouts), identifies which branches lead to the best outcomes, and focuses more computation there. AlphaGo used MCTS to beat world champions at Go.

The MCTS Loop (4 phases):

  1. Selection: Navigate down the tree using a selection policy (e.g., UCB, upper confidence bound) that balances exploitation (best-known moves) and exploration (under-visited moves).
  2. Expansion: When you reach a node with unvisited children, add one new child node.
  3. Simulation: From the new node, play out the game randomly (or using a fast heuristic policy) until termination.
  4. Backpropagation: Back up the result of the simulation through the tree, updating visit counts and value estimates.
🧮 Mathematical Curiosity: UCB for MCTS ▼

Selection policy (UCB1): At each node, select the child with highest:

$$a^* = \arg\max_a \left[ \frac{Q(a)}{N(a)} + C \sqrt{\frac{\ln N(\text{parent})}{N(a)}} \right]$$

where \(Q(a)\) = total return from action \(a\), \(N(a)\) = visit count, \(C\) = exploration constant. The first term is exploitation (average reward), the second is exploration (bonus for less-visited actions).

Simulation: Random playout from a leaf node until terminal state is reached. Simple but effective: the randomness explores diverse futures.

Backpropagation: Update the tree from leaf to root:

$$N(a) \leftarrow N(a) + 1, \quad Q(a) \leftarrow Q(a) + R$$

where \(R\) is the simulated return. After enough iterations, the most visited child at the root corresponds to the best move.

Expected vs Sample Updates: Dyna and MCTS use different update strategies. Dyna performs expected updates on states where it has learned the transition model. MCTS performs sample updates based on random rollouts. When the model is accurate, expected updates are more sample-efficient. When planning depth is limited, sample updates (rollouts) are more robust.

Trajectory Sampling: MCTS only explores trajectories that can actually occur — it naturally focuses on the reachable state space. This is more efficient than dynamic programming approaches that must consider all states, even unreachable ones.

💻 Algorithm Implementation

Dyna-Q in Python: Complete Implementation

Below is a full Dyna-Q implementation on a gridworld. This algorithm combines direct RL, model learning, and planning in ~120 annotated lines:

Python 
import numpy as np
from collections import defaultdict

class DynaQ:
    """
    Dyna-Q agent: integrates direct reinforcement learning,
    model learning, and model-based planning.

    Each step: (1) Real action → update Q and model
              (2) Plan: n simulated steps using the model
    """

    def __init__(self, state_size, action_size, alpha=0.1, gamma=0.9,
                 epsilon=0.1, n_planning=10):
        # Learning parameters
        self.alpha = alpha              # Step size for Q-learning
        self.gamma = gamma              # Discount factor
        self.epsilon = epsilon          # Exploration rate
        self.n_planning = n_planning    # Number of planning steps per real action

        # Value function (state → action → Q-value)
        self.Q = defaultdict(lambda: np.zeros(action_size))

        # Model: (state, action) → (next_state, reward)
        # We store this as a dict of dicts for easy lookup
        self.model = defaultdict(dict)

        # Track which state-action pairs we've seen (for prioritized sampling)
        self.state_actions = set()

    def select_action(self, state, training=True):
        """Epsilon-greedy action selection"""
        if training and np.random.rand() < self.epsilon:
            return np.random.randint(len(self.Q[state]))
        return np.argmax(self.Q[state])

    def step(self, state, action, reward, next_state, done=False):
        """
        Execute one full Dyna-Q step:
        (1) Direct RL update
        (2) Model update
        (3) Planning: n simulated steps
        """

        # ━━━ STEP 1: DIRECT RL ━━━
        # Standard Q-learning update
        max_next_q = np.max(self.Q[next_state]) if not done else 0
        td_error = reward + self.gamma * max_next_q - self.Q[state][action]
        self.Q[state][action] += self.alpha * td_error

        # ━━━ STEP 2: MODEL LEARNING ━━━
        # Store the observed transition
        self.model[state][action] = (next_state, reward)
        self.state_actions.add((state, action))

        # ━━━ STEP 3: PLANNING ━━━
        # Perform n simulated Q-learning updates using the model
        for _ in range(self.n_planning):
            # Randomly sample a previously observed (state, action)
            if self.state_actions:
                s, a = list(self.state_actions)[
                    np.random.randint(len(self.state_actions))
                ]

                # Get the simulated next state and reward from the model
                if a in self.model[s]:
                    s_prime, r = self.model[s][a]

                    # Q-learning update on simulated experience
                    max_next_q = np.max(self.Q[s_prime])
                    td_error = r + self.gamma * max_next_q - self.Q[s][a]
                    self.Q[s][a] += self.alpha * td_error

    def train_episode(self, env, max_steps=100):
        """Run one training episode"""
        state, _ = env.reset()
        total_reward = 0

        for step in range(max_steps):
            action = self.select_action(state)
            next_state, reward, done, _, _ = env.step(action)

            self.step(state, action, reward, next_state, done)
            total_reward += reward

            if done:
                break
            state = next_state

        return total_reward

How to read this code: The core is the step() method. After each real action, it performs one Q-learning update (direct RL), stores the transition in the model (model learning), then loops n_planning times, each time randomly selecting a previously visited (state, action) pair and performing another Q-learning update using the model's prediction. This amplifies the learning signal from a single real experience.

📊 Comparative Analysis

Model-Free vs Model-Based Methods

Here's how the major approaches compare on key dimensions:

Dimension Model-Free (Q-learning) Dyna-Q (Hybrid) MCTS (Planning) DP (Exhaustive)
Model Required No Learned Simulator only Perfect model
Sample Efficiency Low High (amplified updates) Medium (focused search) N/A (no samples)
Computational Cost Low per step Medium (plan n steps) Variable (depth-dependent) Exponential
Convergence Speed Slow (1 update/step) Fast (n+1 updates/step) Fast (search-guided) Instant (exhaustive)
Model Errors N/A Can compound; use Dyna-Q+ Corrected by true rollouts Breaks if model wrong
Exploration Explicit (epsilon-greedy) Explicit + bonus (Dyna-Q+) Built-in (UCB) N/A
Best For Offline learning, complex envs Small-to-medium spaces, known dynamics Games with strong simulators Tabular, finite MDPs
Key Takeaway

Dyna-Q sits at the sweet spot: it doesn't require a perfect model (unlike DP), it's more sample-efficient than pure model-free RL, and it handles model errors better than forward search methods. In practice, Dyna-Q is often the best choice for tabular problems where you have or can learn an approximate model.

💡 Socratic Sandbox

Test Your Understanding: 5 Questions

Click on each question to reveal the answer. Push yourself to think through the reasoning before peeking.

1
Why does Dyna-Q learn faster than pure Q-learning, even when the model is approximate?
▼
Answer: Dyna-Q generates \(n + 1\) value updates from a single real experience (one direct update plus n planning updates). Even if the model is imperfect, these additional updates on the learned model usually provide useful learning signal — they point in roughly the right direction, accelerating convergence. The key is that planning updates are free (or nearly free) compared to gathering more real data. Only when the model is very wrong does planning hurt; in most practical cases, even approximate models speed learning considerably.
2
What happens to a Dyna-Q agent when the environment dynamics change (e.g., a door gets blocked)?
▼
Answer: The learned model becomes outdated. The agent will continue to plan using the old, incorrect model, potentially reinforcing bad value estimates. Dyna-Q+ solves this by adding an exploration bonus to state-action pairs that haven't been updated in a while: \(r^+ = R + \beta\sqrt{\tau}\), where \(\tau\) is the time since last update. This bonus encourages the agent to re-explore and re-learn the dynamics of changed regions, preventing the model from becoming completely stale.
3
How does prioritized sweeping improve upon standard Dyna-Q?
▼
Answer: Instead of randomly sampling state-action pairs for planning, prioritized sweeping maintains a priority queue and focuses planning on states whose values have changed the most. This has two benefits: (1) Value propagation is faster — high-value changes propagate backward more quickly, (2) Planning is more efficient — fewer planning steps are needed because computation is concentrated on high-impact states. In complex environments, prioritized sweeping can reduce the planning overhead from hundreds of steps per real action down to tens, with better results.
4
When is MCTS better than Dyna-Q, and when is it worse?
▼
Answer: MCTS is better when: You have a simulator but no learned model (e.g., board games with known rules). It's sample-efficient because it focuses search on reachable states. MCTS is worse when: You need to make many decisions per second (planning is expensive and depth-limited). Dyna-Q is better if you have time to learn a model and do background planning between decisions. In the limit of infinite planning time per decision, MCTS approaches exhaustive search, while Dyna-Q excels at background learning for continuous decision-making.
5
What's the difference between planning "at decision time" vs "in the background"?
▼
Answer: Background planning (Dyna-Q): Between decisions, the agent uses spare computation to improve value estimates through the model. This is efficient if you have CPU available offline and decisions must be fast. Planning at decision time (MCTS, AlphaGo): When a decision is needed, the agent spends available time searching the game tree from the current state. This is better if you must make a single high-stakes decision and have time to think. In practice, many systems combine both: AlphaGo uses background neural network training (like Dyna-Q) plus decision-time tree search (MCTS).
