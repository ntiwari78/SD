🌉 Feynman Bridge

Monte Carlo is Like Learning from Your Trade History

The Investor's Analogy

Imagine you're a trader learning to invest. You don't need to understand the market mechanics (the underlying model). Instead, you simply look back at your complete trade histories: "When I bought tech stocks with $10K, I ended up with $15K. When I bought energy stocks with $10K, I ended up with $12K." After many trades, you average the results and learn: tech stocks are better.

That's Monte Carlo: Learn by averaging the actual outcomes (returns) of complete episodes, without needing a model of the market.

Why This Matters

  • No model needed: Unlike DP which requires P(s'|s,a) and R, MC just needs experience.
  • Simple & intuitive: Average what happens. Nothing fancy.
  • Works in any environment: You don't need to know how your actions affect the world—just observe the results.
  • High variance: Since you wait for complete episodes, estimates fluctuate more than DP (which bootstraps).
  • Low bias: You're averaging real outcomes, not bootstrapping estimates.

In this chapter, we learn how to systematically extract value estimates from experience without bootstrapping, handle the challenge of exploring all state-action pairs, and learn even when we can only observe a different (behavior) policy.

⛰️ Technical Ascent

Core Concepts & Equations

First-Visit vs Every-Visit MC Prediction

📐 First-Visit MC Prediction ▼

Definition: Average returns from the first time we visit state s in each episode.

$$V(s) \approx \frac{1}{|{first\ visits\ to\ s}|} \sum_{visits} G_t$$

Algorithm: For each episode, track the first occurrence of each state. Accumulate its return. Update V as average of all first-visit returns.

📐 Every-Visit MC Prediction ▼

Definition: Average returns from every visit to state s across all episodes.

$$V(s) \approx \frac{1}{|{all\ visits\ to\ s}|} \sum_{all\ visits} G_t$$

Why different? If state s is visited twice in the same episode, every-visit counts both. Theoretically converges to V(s) under both definitions, but convergence rates differ slightly.

MC Estimation of Action Values

To learn a policy, we must estimate Q(s,a), not just V(s). The approach is identical: average returns following each state-action pair.

📐 Q-Value Estimation ▼
$$Q(s,a) \approx \frac{1}{|{visits\ to\ (s,a)}|} \sum_{t: S_t=s, A_t=a} G_t$$

Key requirement: We must visit enough (s,a) pairs to estimate their values. Problem: some pairs might never be visited under a deterministic or nearly-deterministic policy!

MC Control with Exploring Starts

📐 Policy Improvement in MC Control ▼

Greedy Policy Improvement:

$$\pi(s) = \arg\max_a Q(s,a)$$

After estimating Q, improve the policy by selecting the action with highest Q-value at each state. Guarantee: Under "exploring starts" (every state-action pair has non-zero probability of being the start), policy improvement converges.

ε-Soft Policies (Without Exploring Starts)

Problem: Exploring starts is unrealistic. Solution: Use a stochastic policy that maintains exploration.

📐 ε-Soft Policy Definition ▼

ε-Soft Policy: A policy that assigns non-zero probability to all actions:

$$\pi(a|s) \geq \frac{\varepsilon}{|A(s)|} \quad \text{for all } s, a$$

The optimal action gets the most probability, non-optimal actions share the remaining:

$$\pi(a|s) = \begin{cases} 1 - \varepsilon + \frac{\varepsilon}{|A(s)|} & \text{if } a = \arg\max_a Q(s,a) \\ \frac{\varepsilon}{|A(s)|} & \text{otherwise} \end{cases}$$

Off-Policy Prediction via Importance Sampling

Challenge: We observe behavior from a behavior policy b, but want to evaluate a target policy π. Importance sampling corrects for this mismatch.

📐 Importance Sampling Ratio ▼

Definition: Weight each trajectory by how likely it is under π vs b:

$$\rho_{t:T-1} = \prod_{k=t}^{T-1} \frac{\pi(A_k|S_k)}{b(A_k|S_k)}$$

This ratio becomes very small if b takes actions that π never would. If b(a|s) = 0 and π(a|s) > 0, the ratio is infinite!

📐 Ordinary Importance Sampling ▼

Estimate:

$$V(s) = \frac{\sum_{t \in \tau(s)} \rho_{t:T(t)-1} G_t}{|\{t: S_t = s\}|}$$

Problem: High variance. A single large ρ can dominate the average.

📐 Weighted Importance Sampling ▼

Estimate:

$$V(s) = \frac{\sum_{t \in \tau(s)} \rho_{t:T(t)-1} G_t}{\sum_{t \in \tau(s)} \rho_{t:T(t)-1}}$$

Much lower variance! The denominator normalizes the weights. This is the preferred method despite introducing bias.

⚙️ Implementation

Python: First-Visit MC & ε-Soft MC Control

Complete, annotated implementation of Monte Carlo methods for policy evaluation and control.

Python (120 lines) 
import numpy as np
from collections import defaultdict

class MonteCarloAgent:
    """First-Visit MC Prediction and ε-Soft MC Control."""

    def __init__(self, env, epsilon=0.1, gamma=0.99):
        """Initialize MC agent.

        Args:
            env: OpenAI Gym-like environment
            epsilon: Exploration rate for ε-soft policy
            gamma: Discount factor
        """
        self.env = env
        self.epsilon = epsilon
        self.gamma = gamma

        # State-action value function
        self.Q = defaultdict(lambda: np.zeros(env.action_space.n))
        # Visit counts for each state-action pair
        self.N = defaultdict(lambda: np.zeros(env.action_space.n))

    def epsilon_greedy_policy(self, state):
        """ε-soft policy: mostly greedy, sometimes random.

        π(a|s) = 1 - ε + ε/|A|  if a = argmax Q(s,a)
        π(a|s) = ε/|A|          otherwise
        """
        num_actions = self.env.action_space.n

        # Probabilities
        probs = np.ones(num_actions) * (self.epsilon / num_actions)
        best_action = np.argmax(self.Q[state])
        probs[best_action] += 1 - self.epsilon

        return probs

    def generate_episode(self):
        """Generate one complete episode following current policy.

        Returns:
            episode: List of (state, action, reward) tuples
        """
        episode = []
        state, _ = self.env.reset()
        done = False

        while not done:
            probs = self.epsilon_greedy_policy(state)
            action = np.random.choice(len(probs), p=probs)
            next_state, reward, done, _, _ = self.env.step(action)

            episode.append((state, action, reward))
            state = next_state

        return episode

    def mc_prediction(self, num_episodes=1000):
        """First-Visit MC Prediction: estimate V(s) and Q(s,a).

        Algorithm:
        1. Generate episode
        2. For each state-action in episode (first visit only):
           - Compute return G_t = discounted sum of future rewards
           - Update Q(s,a) ← average of all returns for (s,a)
        """
        for episode_num in range(num_episodes):
            episode = self.generate_episode()

            # Compute returns (backward pass)
            G = 0
            visited = set()

            # Process episode backward
            for t in range(len(episode) - 1, -1, -1):
                state, action, reward = episode[t]
                G = reward + self.gamma * G

                # First-visit: only update on first occurrence
                if (state, action) not in visited:
                    visited.add((state, action))
                    self.N[state][action] += 1
                    # Incremental update: Q ← Q + (G - Q) / N
                    self.Q[state][action] += (G - self.Q[state][action]) / self.N[state][action]

        return self.Q

    def mc_control(self, num_episodes=10000):
        """MC Control: Learn optimal ε-soft policy.

        Algorithm:
        1. Generate episode following current policy π
        2. For each (s,a) in episode (first visit):
           - Update Q(s,a) with observed return
        3. Update π to be ε-greedy w.r.t. Q
        Repeat until convergence
        """
        for episode_num in range(num_episodes):
            episode = self.generate_episode()

            # Compute returns backward
            G = 0
            visited = set()

            for t in range(len(episode) - 1, -1, -1):
                state, action, reward = episode[t]
                G = reward + self.gamma * G

                if (state, action) not in visited:
                    visited.add((state, action))
                    self.N[state][action] += 1
                    self.Q[state][action] += (G - self.Q[state][action]) / self.N[state][action]

            # Policy automatically improves (implicit in ε-greedy)
            if episode_num % 1000 == 0:
                print(f"Episode {episode_num}: Avg reward = {np.mean(self.compute_policy_returns(100)):.2f}")

        return self.Q

    def extract_policy(self):
        """Extract greedy policy from Q values."""
        policy = {}
        for state in self.Q.keys():
            policy[state] = np.argmax(self.Q[state])
        return policy

    def compute_policy_returns(self, num_episodes=100):
        """Evaluate current policy: return cumulative rewards."""
        returns = []
        for _ in range(num_episodes):
            episode = self.generate_episode()
            episode_return = sum(r * (self.gamma ** t) for t, (_, _, r) in enumerate(episode))
            returns.append(episode_return)
        return returns
Key Implementation Details
  • First-visit tracking: The `visited` set ensures we only update the first occurrence of each (s,a) per episode.
  • Incremental averaging: Q(s,a) ← Q(s,a) + (G - Q(s,a)) / N(s,a) avoids storing all returns.
  • ε-soft policy: Always maintains exploration: smallest action probability is ε/|A|.
  • Implicit improvement: As Q improves, the ε-greedy policy automatically becomes better without explicit policy updates.
📊 Comparison

Dynamic Programming vs Monte Carlo Methods

How do MC methods compare to DP, the boostrapping method from Chapter 3?

Dimension Dynamic Programming (DP) Monte Carlo (MC)
Model Knowledge Requires full model: P(s'|s,a), R(s,a) Model-free: only needs experience
Data Type Bootstraps from estimated values Uses actual returns from complete episodes
Convergence May not converge if bootstrap estimates are poor Always converges (by law of large numbers)
Episode Requirement Can improve V within an episode Must complete full episodes before updating
Bias Biased (uses estimates of other states) Unbiased (averages real outcomes)
Variance Lower (bootstrapping reduces variance) Higher (full episode returns vary more)
Computational Cost One sweep per iteration (in-place possible) One update per episode (must run to completion)
Function Approximation Works well with linear & non-linear FA Works with FA; high variance can be problematic
Offline Learning Can use any model Can learn from historical data (with importance sampling)
Real-World Fit Model often unavailable Natural: just collect experience, average
Bias-Variance Tradeoff

MC is unbiased but high-variance: Each return G_t is a true sample from the environment. Averaging true samples will converge to the right answer.

DP is low-variance but biased: Using V(s') to estimate returns introduces bias (what if V(s') is wrong?), but reduces variance by "sharing" information across states.

Later (Chapter 6) we'll see Temporal-Difference learning, which combines the best of both: unbiased bootstrapping with lower variance than MC.

Test Your Understanding: 5 Key Questions

Click each question to reveal the answer. Challenge yourself first!

1
Why must Monte Carlo methods wait for complete episodes, whereas DP can update mid-episode?
▼

Key Insight: MC estimates V(s) from actual returns G_t, which is the discounted sum of rewards from that point to episode end. You cannot compute G_t until you reach the terminal state. DP, by contrast, bootstraps: it uses V(s') as an estimate for future returns, so it can update any state as soon as it sees a transition.

In code: MC computes returns backward (from end to start), DP computes forward (using current value estimates).

2
Why is Monte Carlo's high variance not just a drawback—how can it actually be an advantage?
▼

The Unbiased Promise: High variance means value estimates fluctuate a lot. But those fluctuations are around the true value, not a biased estimate. With enough data, high variance is just noise that averages out.

More importantly: MC doesn't compound errors. DP's estimates depend on other estimates (V(s') affects V(s)). If V(s') is wrong, all states depending on it are wrong too (a bootstrapping bias). MC estimates only depend on observed returns—true data, not estimates of estimates.

3
Why is "exploring starts" impractical, and how do ε-soft policies solve this?
▼

The Problem: Exploring starts means: start each episode from a uniformly random state-action pair. Real agents cannot control their own starting state! If an agent always starts in state S₀, it may never explore states reachable only from other starting points.

The ε-Soft Solution: Instead of controlling where you start, ensure your policy always maintains exploration. With probability at least ε/|A(s)|, you take a random action. Over infinite episodes, this guarantees visiting all (s,a) pairs (with high probability). No magic restart button needed.

4
What is importance sampling, and why does the ratio ρ become explosive in off-policy learning?
▼

The Setup: You want to learn policy π but only have data from policy b. Importance sampling reweights trajectories: weight = ρ = ∏ π(a|s) / b(a|s).

The Explosion: If b is very different from π (e.g., b is random, π is deterministic), the ratio explodes. Consider a 10-step trajectory: if each action has π(a|s)=1.0 and b(a|s)=0.1, the product is (1/0.1)^10 = 10^10! A few "lucky" trajectories dominate the average.

Why Weighted IS Helps: Weighted importance sampling normalizes by the sum of weights, bringing high-weight trajectories back to earth. It introduces bias but drastically reduces variance.

5
In what sense is every-visit MC still "correct" despite updating from repeated visits within one episode?
▼

Apparent Problem: If state s appears twice in one episode, every-visit counts both occurrences. But they return different values G_t! Aren't we mixing incompatible data?

The Resolution: Both are unbiased samples of V(s). The first visit to s in an episode returns G_{t1}, the second returns G_{t2}—both are valid, independent samples from the return distribution. Averaging them is fine! The law of large numbers still applies. First-visit and every-visit both converge, though potentially at different rates. Every-visit may actually converge faster in some settings.
