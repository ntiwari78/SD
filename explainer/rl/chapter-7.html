🌉Feynman Bridge

The Day Review Principle

Core Intuition

n-step methods are like reviewing your day: 1-step TD looks at the very next moment (like reacting to immediate feedback), Monte Carlo reviews the whole day (like reflecting at bedtime with full hindsight). n-step lets you choose: review the last hour, the last 3 hours — finding the sweet spot between immediacy and hindsight.

In temporal-difference (TD) learning, you bootstrapped using the next state's estimated value—you looked just one step ahead. In Monte Carlo, you waited for the full trajectory to completion—the entire episode.

n-step methods split the difference. Instead of looking 1 step or ∞ steps, you look exactly n steps into the future, then bootstrap from that distant state's value. This creates a spectrum of algorithms:

  • n = 1: Pure TD—bootstraps immediately on next state
  • n = 3: Look 3 steps ahead, then bootstrap
  • n → ∞: Pure Monte Carlo—no bootstrap, use actual returns
Why This Matters

TD(0) has low variance (stable, quick) but high bias (bootstrapping from estimates). MC has zero bias (uses true returns) but high variance (needs many episodes). n-step methods let you tune this bias-variance tradeoff by choosing n—giving you algorithmic flexibility.

⚙️Technical Ascent

Core Equations & Derivations

The n-step Return

The fundamental quantity is the n-step return, which collects actual rewards for n steps, then bootstraps:

✓ Show Math ▼

$$G_{t:t+n} = R_{t+1} + \gamma R_{t+2} + \gamma^2 R_{t+3} + \cdots + \gamma^{n-1} R_{t+n} + \gamma^n V_{t+n-1}(S_{t+n})$$

Breakdown:

  • First n terms: actual rewards from the trajectory
  • Last term: bootstrapped value of state n steps away
  • When n → ∞: last term vanishes, becomes MC return
  • When n = 1: only $R_{t+1}$ is actual, rest is $\gamma V_t(S_{t+1})$ (pure TD)

n-step TD Prediction

Update the value function using the n-step return:

✓ Show Math ▼

$$V_{t+n}(S_t) = V_{t+n-1}(S_t) + \alpha \left[ G_{t:t+n} - V_{t+n-1}(S_t) \right]$$

This is the same form as TD(0)! The only difference: the target $G_{t:t+n}$ incorporates n steps of actual experience before bootstrapping.

n-step Sarsa Control

For action-value methods, apply the same principle to Q-values:

✓ Show Math ▼

$$G_{t:t+n} = R_{t+1} + \gamma R_{t+2} + \cdots + \gamma^{n-1} R_{t+n} + \gamma^n Q_{t+n-1}(S_{t+n}, A_{t+n})$$

$$Q_{t+n}(S_t, A_t) = Q_{t+n-1}(S_t, A_t) + \alpha \left[ G_{t:t+n} - Q_{t+n-1}(S_t, A_t) \right]$$

Bootstrap from the next action's Q-value. Still on-policy (follows behavior policy to generate trajectory).

n-step Off-Policy with Importance Sampling

If you're learning from a different behavior policy, weight the return by importance sampling ratios. This accounts for the probability difference between the target and behavior policy:

✓ Show Math ▼

$$\rho_{t:t+n-1} = \prod_{s=t}^{t+n-1} \frac{\pi(A_s | S_s)}{b(A_s | S_s)}$$

$$V_{t+n}(S_t) = V_{t+n-1}(S_t) + \alpha \rho_{t:t+n-1} \left[ G_{t:t+n} - V_{t+n-1}(S_t) \right]$$

Issue: If any $\rho_s = 0$ (an action had zero probability under target), the whole product is zero—high variance!

Tree Backup (No Importance Sampling)

Alternative: avoid importance sampling ratios. Instead, weight the backup by the probability of taking actions under the target policy:

✓ Show Math ▼

$$G_t^{(n)} = R_{t+1} + \gamma \sum_{a'} \pi(a' | S_{t+1}) Q_{t+n-1}(S_{t+1}, a') + \pi(A_{t+1} | S_{t+1}) \gamma \left( R_{t+2} + \gamma \sum_{a''} \pi(a'' | S_{t+2}) Q_{t+n-1}(S_{t+2}, a'') + \cdots \right)$$

Conceptually: build a tree of possible actions, but weight them by policy probabilities instead of importance ratios. No division by behavior policy = more stable, lower variance.

Q(σ): Unifying Off-Policy Algorithm

A continuum parameter $\sigma \in [0, 1]$ smoothly interpolates between importance sampling and tree backup:

✓ Show Math ▼

$$G_t^{(n)} = R_{t+1} + \gamma \left[ \sigma \rho_{t+1} + (1 - \sigma) \pi(A_{t+1}|S_{t+1}) \right] \left[ G_{t+1}^{(n-1)} \right]$$

Where each update step mixes importance sampling ($\rho_{t+1}$) with tree backup ($\pi(A_{t+1}|S_{t+1})$).

  • $\sigma = 1$: pure importance sampling (high variance, unbiased)
  • $\sigma = 0$: pure tree backup (lower variance, biased)
  • $\sigma = 0.5$ (or adaptive): tradeoff
The Spectrum

As you vary $n$ from 1 to ∞, you move from bootstrapping immediately to using full Monte Carlo returns. As you vary the off-policy method, you tradeoff variance (importance sampling) vs. bias (tree backup). These are two separate tuning dimensions for the engineer to explore.

🔧Algorithm

n-step Sarsa Implementation

Below is a complete, annotated implementation of n-step Sarsa for control in Python. This is the on-policy variant (uses the same policy for both exploration and learning):

python 
import numpy as np
from collections import defaultdict

class NStepSarsa:
    """
    n-step Sarsa for on-policy control.

    Learns an epsilon-greedy policy by looking n steps ahead,
    then bootstrapping from the Q-value at step n.
    """

    def __init__(self, n_steps=3, alpha=0.1, epsilon=0.1, gamma=0.99):
        """
        Args:
            n_steps: Number of steps to look ahead before bootstrapping
            alpha: Learning rate
            epsilon: Exploration rate for ε-greedy
            gamma: Discount factor
        """
        self.n = n_steps
        self.alpha = alpha
        self.epsilon = epsilon
        self.gamma = gamma

        # Q-value table: Q[state][action]
        self.Q = defaultdict(lambda: defaultdict(float))

        # Store trajectory for n-step backup
        self.trajectory_states = []
        self.trajectory_actions = []
        self.trajectory_rewards = []
        self.trajectory_dones = []

    def choose_action(self, state, available_actions):
        """ε-greedy action selection."""
        if np.random.random() < self.epsilon:
            return np.random.choice(available_actions)
        else:
            # Greedy: pick action with highest Q-value
            q_values = [self.Q[state][a] for a in available_actions]
            max_q = max(q_values)
            best_actions = [a for a, q in zip(available_actions, q_values)
                          if q == max_q]
            return np.random.choice(best_actions)

    def store_transition(self, state, action, reward, next_state, done):
        """Buffer a single transition."""
        self.trajectory_states.append(state)
        self.trajectory_actions.append(action)
        self.trajectory_rewards.append(reward)
        self.trajectory_dones.append(done)

    def n_step_return(self, tau):
        """
        Compute n-step return starting from time tau.

        G_{t:t+n} = R_{t+1} + γR_{t+2} + ... + γ^(n-1)R_{t+n} + γ^n Q(S_{t+n}, A_{t+n})

        Args:
            tau: Time index to compute return for

        Returns:
            n-step return value
        """
        # Determine horizon: min(n, steps until episode end)
        horizon = min(self.n, len(self.trajectory_rewards) - tau)

        # Accumulate rewards
        G = 0.0
        for i in range(horizon):
            G += self.gamma ** i * self.trajectory_rewards[tau + i]

        # Bootstrap from Q-value at step tau + n
        if tau + self.n < len(self.trajectory_states):
            # Still within trajectory: use Q-value
            next_state = self.trajectory_states[tau + self.n]
            next_action = self.trajectory_actions[tau + self.n]
            G += self.gamma ** self.n * self.Q[next_state][next_action]
        # else: episode ended, no bootstrap needed

        return G

    def update(self):
        """
        Perform n-step updates after episode completion.

        Update all transitions that can be backed up now.
        """
        # Number of states in trajectory
        T = len(self.trajectory_states)

        # Update each transition
        for tau in range(T - self.n + 1):
            G = self.n_step_return(tau)
            state = self.trajectory_states[tau]
            action = self.trajectory_actions[tau]

            # TD update
            self.Q[state][action] += self.alpha * (G - self.Q[state][action])

    def episode(self, env, max_steps=500):
        """
        Run one episode and learn.

        Args:
            env: Environment with (state, available_actions) and step(action)
            max_steps: Maximum steps per episode
        """
        self.trajectory_states = []
        self.trajectory_actions = []
        self.trajectory_rewards = []
        self.trajectory_dones = []

        # Initialize state
        state = env.reset()

        for step in range(max_steps):
            # Choose and take action
            available_actions = env.available_actions(state)
            action = self.choose_action(state, available_actions)

            # Step environment
            next_state, reward, done = env.step(action)

            # Store transition
            self.store_transition(state, action, reward, next_state, done)

            state = next_state

            if done:
                break

        # Update Q-values using all buffered transitions
        self.update()

        return len(self.trajectory_states)

# Example Usage
# ─────────────────────────────────

class SimpleGridWorld:
    """Minimal environment for testing."""

    def __init__(self, size=5):
        self.size = size
        self.goal = (size - 1, size - 1)
        self.start = (0, 0)
        self.state = self.start

    def reset(self):
        self.state = self.start
        return self.state

    def available_actions(self, state):
        return ['up', 'down', 'left', 'right']

    def step(self, action):
        x, y = self.state

        if action == 'up' and x > 0:
            x -= 1
        elif action == 'down' and x < self.size - 1:
            x += 1
        elif action == 'left' and y > 0:
            y -= 1
        elif action == 'right' and y < self.size - 1:
            y += 1

        self.state = (x, y)

        # Reward: +1 at goal, -0.01 per step (encourages efficiency)
        reward = 1.0 if self.state == self.goal else -0.01
        done = self.state == self.goal

        return self.state, reward, done


# Run training
if __name__ == '__main__':
    env = SimpleGridWorld(size=5)

    # Try different n values
    for n in [1, 3, 5]:
        agent = NStepSarsa(n_steps=n, alpha=0.1, epsilon=0.1)

        total_return = 0
        for episode in range(100):
            steps = agent.episode(env, max_steps=100)
            # Simple return: -1 per step, +1 if goal reached
            episode_return = 1.0 - 0.01 * steps
            total_return += episode_return

        avg_return = total_return / 100
        print(f"n={n}: Avg return over 100 episodes = {avg_return:.4f}")
Key Implementation Details
  • Trajectory Buffering: Store states, actions, rewards in lists until episode end
  • n_step_return(): Accumulate rewards, then bootstrap from Q(s_{t+n}, a_{t+n})
  • update(): Apply updates for all tau that can complete an n-step backup
  • ε-greedy: Simple exploration strategy; modify for your use case
⚖️Comparison

TD(0) vs n-step TD vs Monte Carlo

Here's how the three approaches compare across key dimensions:

Dimension TD(0) [n=1] n-step TD [n=3-10] Monte Carlo [n=∞]
Lookhead Steps 1 n (tunable) Until episode end
Bootstrap Source Immediate next state n-steps ahead state None (actual return)
Bias High (estimates used) Medium (n steps of actual) Zero (true return)
Variance Low (stable) Medium (n-step average) High (full return variability)
Learning Speed Fast (online) Medium (n-step delay) Slow (episode-end batch)
α Range Wide (0.01 to 0.5) Moderate (0.001 to 0.1) Narrow (0.001 to 0.01)
Convergence Guaranteed (certain conditions) Guaranteed (certain conditions) Guaranteed
Typical Use Online, stable learning Balanced performance Batch, policy eval
Memory Constant (one state) O(n) (buffer transitions) O(episode length)
Sensitivity to α Robust Moderate Very sensitive
Choosing n in Practice
  • Start with n = 3-5: Often a good default for control problems
  • If learning is unstable: Reduce n (more bias, less variance)
  • If learning is slow: Increase n (collect more actual rewards)
  • If episodes are short: Keep n ≤ average episode length
  • For wall-clock time: Consider the delay: n-step methods add n-step latency
❓Socratic Sandbox

Conceptual Questions to Deepen Your Understanding

Click each question to reveal a detailed answer. Try to reason through them yourself first:

▼ Why does n-step have lower variance than MC but higher than TD(0)?
Key insight: Variance comes from random rewards and estimates.

TD(0): Bootstrap almost immediately, so most of the target is your own estimate. You're not exposed to much randomness—low variance.

MC: Wait for the entire trajectory. Each reward in the full return introduces variance. The longer the episode, the more random noise accumulates—high variance.

n-step: You sample n actual rewards (introducing some noise) but then bootstrap, cutting off the tail. You get a middle ground: sample some real data (reducing bias) but not all (keeping variance manageable).

Formula perspective: The return's variance is roughly the sum of squared reward variances over n steps. With n rewards vs. T (episode length) rewards in MC, you have less exposure to noise.
▼ In off-policy learning, why is importance sampling necessary for n-step methods?
The problem: In off-policy learning, you collect data using a behavior policy (e.g., random exploration) but want to learn about a target policy (e.g., greedy).

Why it matters for n-step: Over n steps, the trajectory you observe might have actions that are unlikely under the target policy. If action $A_t$ has probability $b(A_t | S_t)$ under behavior but $\pi(A_t | S_t)$ under target, then your return is biased unless you correct for this difference.

Importance sampling ratio: $$\rho_{t:t+n-1} = \frac{\pi(A_t|S_t) \pi(A_{t+1}|S_{t+1}) \cdots \pi(A_{t+n-1}|S_{t+n-1})}{b(A_t|S_t) b(A_{t+1}|S_{t+1}) \cdots b(A_{t+n-1}|S_{t+n-1})}$$ This ratio reweights the return to make it an unbiased estimate under the target policy.

Cost: If any action was impossible under behavior ($b(A_i|S_i) = 0$), the ratio blows up or becomes undefined. This is the variance problem with importance sampling!
▼ How does tree backup avoid the importance sampling variance problem?
Key idea: Instead of dividing by behavior policy probabilities, tree backup multiplies by target policy probabilities.

No division: Since you're multiplying (not dividing), you never get division-by-zero errors. Even if an action was unlikely under behavior, you don't blow up.

Conceptually: Tree backup builds a hypothetical tree of what could happen next, weighted by the target policy's preferences. You don't "correct" for the behavior policy—instead, you construct a target-policy-aligned backup that only uses information available (the observed actions and their values).

Tradeoff: Tree backup is more stable (lower variance) but biased (not guaranteed to converge to the target policy's true value function, though in practice it's quite good).

Intuition: It's like saying "I observed this trajectory under a different policy, but let me imagine what my target policy would value from this state"—without trying to exactly correct for the history.
▼ Why can't you always use n = ∞ (MC) if it has zero bias?
Three practical problems:

1. Variance: If episodes are long, returns accumulate a lot of random noise. This makes learning unstable. You need smaller learning rates (slower learning) to stabilize it.

2. Latency: You can't update until the episode ends. For very long episodes or real-time control, this is unacceptable. n-step methods let you update after n steps, even within an episode.

3. Credit assignment: The farther back you assign credit, the noisier the gradient. Early-episode states get credit from 100+ steps of future rewards. This diffuses the learning signal and can slow convergence compared to n-step (which focuses updates on more recent, relevant experience).

In practice: When you tune an algorithm, you often find intermediate n (like n=3 or n=8) outperforms both n=1 (high bias) and n=∞ (high variance).
▼ What does Q(σ) with σ = 0.5 represent? Is it truly a "50-50 mix"?
Not exactly 50-50 per step. Q(σ) blends importance sampling and tree backup at each step of the n-step return.

The parameter σ: $$\text{weight} = \sigma \cdot \rho_t + (1 - \sigma) \cdot \pi(A_t | S_t)$$
- When $\sigma = 1$: use importance sampling ratio (high variance, unbiased)
- When $\sigma = 0$: use target policy probability (low variance, biased)
- When $\sigma = 0.5$: weight them equally at each step

Why "per step"? Because you apply this mix at each intermediate backup step, not globally. Over n steps, the compounding effect is subtle—it's not a simple linear blend of pure importance sampling vs. pure tree backup.

Advantage: It gives you a continuum of off-policy methods. You can tune σ to find your own bias-variance sweet spot without choosing between two discrete algorithms.
